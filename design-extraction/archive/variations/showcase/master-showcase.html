When to Use Each Variation

Code Blocks

  • Terminal: DevOps docs, CLI tutorials, hacker/gaming aesthetic
  • Zine: Creative tech content, indie projects, tutorials with personality
  • Brutalist: Enterprise docs, no-nonsense technical reference
  • Editorial: Published articles, books, premium documentation

Cards

  • Typography: Landing pages, marketing sites, hero features
  • Badge: Gamified learning, achievement systems, collectible content
  • Editorial: Blog posts, article previews, content-heavy layouts
  • Textured: Premium products, physical goods, handcrafted feel

Callouts

  • Stamp: Official documentation, certification content
  • Brutal: Developer tools, technical warnings
  • Typography: Editorial content, articles, essays
  • Frame: Premium documentation, formal content

Design System Integration

CSS Variables

All variations use consistent CSS custom properties for colors, typography, and spacing. Override these at the root level to theme the entire system.

Component Architecture

  • Each variation is self-contained with no external dependencies
  • BEM-style class naming prevents conflicts
  • Semantic color tokens (primary, success, warning, error) enable easy theming

Accessibility

  • All color pairings meet WCAG AA minimum (4.5:1)
  • Focus states are clearly visible in all variations
  • Text sizes meet minimum readability requirements

Performance

  • SVG textures are inline data URIs (no HTTP requests)
  • Google Fonts can be self-hosted for faster loading
  • CSS animations use GPU-accelerated properties

Migration Path

Start with one variation per component type. Gradually introduce alternatives for different contexts (e.g., Terminal for CLI docs, Editorial for guides).
