Lesson 1 Objective: Observe the Master's Stance

Who Is Boris Cherny?

Boris Cherny is a Staff Engineer at Anthropic and the creator of Claude Code. His role extends beyond engineering leadership into product vision and design philosophy. He is the canonical source for understanding how Claude Code should be used because its design directly reflects his mental model.

His background in TypeScript and developer tooling explains Claude Code's emphasis on type safety, developer experience over AI complexity, and simple composable primitives over opinionated frameworks.

"I'm Boris and I created Claude Code. [...] My setup might be surprisingly vanilla!" -- Boris Cherny, Twitter

Lesson 2 Objective: Learn the Master's Tools

The Canonical Workflow

Every element of Boris's workflow reflects intentional design decisions. This is not a collection of tips -- it is a system where each element amplifies the others.

Parallel Sessions

Five terminal instances, five to ten web sessions, mobile sessions started in morning. Boris doesn't use one extremely capable agent -- he uses many independent workers, each a separate context window.

"He's distributing cognition like compute: allocate it, queue it, keep it hot, switch contexts only when value is ready."

Always Opus + Extended Thinking

The bottleneck isn't generation speed -- it's correction time. Opus requires less steering, makes fewer errors, produces better first drafts.

Total Time = (Generation Time) + (Correction Time) + (Re-prompting Time) Opus: Higher generation, MUCH lower correction/re-prompting Sonnet: Lower generation, significantly higher correction loops
"I use Opus 4.5 with thinking for everything. Even though it's bigger and slower than Sonnet, since you have to steer it less and it's better at tool use, it is almost always faster than using a smaller model in the end." -- Boris Cherny

Shared CLAUDE.md in Git

Single file at repository root. Team contributes weekly. Every mistake becomes a rule. GitHub Action enforces updates.

"Anytime we see Claude do something incorrectly we add it to the CLAUDE.md, so Claude knows not to do it next time." -- Boris Cherny
Exercise 2A

Lesson 3 Objective: Understand the Master's Technique

Design Philosophy

Technique 1: "Do the Simple Thing First"

The team considered embeddings, vector databases, semantic search for memory. They shipped a markdown file. Simplicity won.

Feature Simple Choice Rejected
MemoryMarkdown fileVector databases
SearchripgrepSemantic search
ExecutionLocal terminalContainers
StorageLocal filesCloud sync
"CLAUDE.md is another example of this idea of doing the simple thing first. We had all these crazy ideas about memory architectures -- but in the end, the thing we did is ship the simplest thing: a file that has some stuff that's auto-read into context." -- Boris Cherny
Technique 2: "Build for the Model Six Months From Now"

Don't add workarounds for current limitations. Build the interface for future capability. When capability arrives, the interface is ready.

"Honestly, for a long time, Claude Code was not a great product. I used it for maybe 10% of my code because the model wasn't capable enough." -- Boris Cherny
Technique 3: Raw Model Access

Boris rejected building "a Cursor or Windsurf." Minimal abstraction between user intent and model capability. No opinionated enforcement.

Technique 4: Unix Philosophy

Do one thing well. Compose freely. Text as interface. Minimal surprise. A programmable CLI, not just a chat.

Exercise 3A

Lesson 4 Objective: Understand the System

The Compounding System

Claude Code is not a one-shot tool. It is an evolving teammate whose effectiveness compounds over time.

Linear usage: Output = Capability x Time Compounding usage: Output = Capability x Time x (1 + Learning Rate)^Sessions

Each session that updates CLAUDE.md increases effective capability for all future sessions. Shared knowledge persists. Every failure improves the future. Verification confirms improvements.

"Every correction becomes permanent context. The cost of a mistake pays dividends forever." -- Boris Cherny
Exercise 4A

Calculate your current compounding rate. How often does your CLAUDE.md get updated? How many team members contribute? If the answer is "never" or "just me," your Learning Rate is near zero.

Lesson 5 Objective: Learn From the Master's Corrections

Anti-Patterns: The Master's Corrections

Skipping Plan Mode. "The ones struggling often skip planning to save time, then spend more time fixing mistakes. Plan Mode isn't training wheels -- it's measuring before you cut."

Sequential Instead of Parallel. Run five Claudes, context-switch between them. Don't wait for one to finish before starting the next.

Optimizing for Token Speed. Using smaller models to "save time" is false economy. Correction time dominates generation time.

Over-Customization. "My setup might be surprisingly vanilla! Claude Code works great out of the box."

Complex Magic Command Systems. "Having a long list of complex, custom slash commands is an anti-pattern. The entire point of an agent like Claude is that you can type almost whatever you want."
