Who Is Boris Cherny?

Boris Cherny is a Staff Engineer at Anthropic and the creator of Claude Code. His role extends beyond engineering leadership into product vision and design philosophy. His pre-Anthropic work included contributions to the TypeScript ecosystem and developer experience tools.

Classification: Public Identity

Anti-complexity is core to his worldview. Boris explicitly describes his setup as "surprisingly vanilla" despite being the creator.

The Canonical Workflow

Boris's workflow is the reference implementation. Every element reflects intentional design. It is a system where each element amplifies the others.

WF-001 -- Parallel Sessions
5-10 Concurrent Instances
Terminal (5 parallel), web (5-10 sessions), mobile (async). Distributing cognition like compute: allocate, queue, keep hot, switch when value is ready.
WF-002 -- Model Selection
Always Opus 4.5 + Extended Thinking
Bottleneck is correction time, not generation speed. Opus requires less steering, fewer tool errors, better first drafts.
Total Time = (Generation Time) + (Correction Time) + (Re-prompting Time) Opus: Higher generation, MUCH lower correction/re-prompting Sonnet: Lower generation, significantly higher correction loops
WF-003 -- Shared CLAUDE.md
Single File at Git Root
Team contributes weekly. Every mistake becomes a rule. GitHub Action enforces updates. This is the core of institutional knowledge architecture.
"Anytime we see Claude do something incorrectly we add it to the CLAUDE.md, so Claude knows not to do it next time." -- Document BC-007, Boris Cherny
WF-004 -- Verification
The Most Important Thing
Chrome extensions, bash scripts, simulators, background agents. Not optional. The difference between 1x and 2-3x quality.
"Probably the most important thing to get great results out of Claude Code: give Claude a way to verify its work." -- Document BC-011, Boris Cherny

Design Philosophy

Principle BC-P-001

"Do the Simple Thing First" -- foundational principle inherited from Anthropic's engineering culture.

"CLAUDE.md is another example of this idea of doing the simple thing first. We had all these crazy ideas about memory architectures -- but in the end, the thing we did is ship the simplest thing." -- Document BC-015, Boris Cherny
Principle BC-P-002

"Build for the Model Six Months From Now" -- don't add workarounds. Build for future capability.

Principle BC-P-003

"Raw Model Access as Design Goal" -- rejected building "a Cursor or Windsurf." Minimal abstraction.

Principle BC-P-004

"Unix Philosophy as Foundation" -- do one thing well. Compose freely. Text as interface.

The Compounding System

Linear usage: Output = Capability x Time Compounding usage: Output = Capability x Time x (1 + Learning Rate)^Sessions

Each session that updates CLAUDE.md increases effective capability for all future sessions. Shared knowledge persists across team and time. Every failure improves the future. Verification confirms improvements.

Enables
Shared knowledge, mistake-to-rule, team contribution, verification
Breaks
Individual files, ignoring failures, bloat, skipping verification
"Every correction becomes permanent context. The cost of a mistake pays dividends forever." -- Document BC-022, Boris Cherny
