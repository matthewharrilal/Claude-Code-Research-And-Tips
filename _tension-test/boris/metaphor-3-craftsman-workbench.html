The Craftsman

Who Is Boris Cherny?

Boris Cherny is a Staff Engineer at Anthropic and the creator of Claude Code. His role extends beyond engineering leadership into product vision and design philosophy. He is the canonical source for understanding how Claude Code should be used because its design directly reflects his mental model.

His pre-Anthropic work included contributions to the TypeScript ecosystem and developer experience tools. This background explains Claude Code's emphasis on type safety as verification, developer experience over AI complexity, and simple, composable primitives over opinionated frameworks.

"I'm Boris and I created Claude Code. [...] My setup might be surprisingly vanilla!" -- Boris Cherny, Twitter
The "Vanilla" Philosophy

Boris explicitly describes his own setup as "surprisingly vanilla" despite being the creator. If the creator doesn't need heavy customization, that signals the tool works well out of the box. Anti-complexity is core to his worldview.

Tool Rack -- The Canonical Workflow

The Complete 13-Point System

Every element reflects intentional design decisions. It is not a list of tips -- it is a system where each element amplifies the others.

Parallel Sessions (5-10)
Distributing cognition like compute. Five terminal instances, five to ten web sessions. Each is a separate context window -- a separate worker with its own focus.
Always Opus + Thinking
The bottleneck isn't generation speed, it's correction time. Opus requires less steering, fewer tool use errors, higher-quality first drafts.
Shared CLAUDE.md
Single file at git root. Team contributes weekly. Every mistake becomes a rule. GitHub Action enforces updates on relevant PRs.
Plan Mode Workflow
Start in Plan Mode. Iterate until solid. Switch to auto-accept. Claude executes without interruption. Prevents 40 unwanted changes.
"He's distributing cognition like compute: allocate it, queue it, keep it hot, switch contexts only when value is ready."

The Total Time Equation

Total Time = (Generation Time) + (Correction Time) + (Re-prompting Time) Opus: Higher generation, MUCH lower correction/re-prompting Sonnet: Lower generation, significantly higher correction loops
"I use Opus 4.5 with thinking for everything. Even though it's bigger and slower than Sonnet, since you have to steer it less and it's better at tool use, it is almost always faster than using a smaller model in the end." -- Boris Cherny
Verification -- The Most Important Tool

Chrome extensions for UI tests, bash scripts for automated checks, mobile simulators for iOS/Android, background verification agents. Verification is not optional. It is the difference between 1x quality and 2-3x quality.

"Probably the most important thing to get great results out of Claude Code: give Claude a way to verify its work." -- Boris Cherny
The Craft -- Design Philosophy

What the Craftsman Believes

Principle 1: "Do the Simple Thing First"

Boris's foundational principle. The team considered embeddings, vector databases, semantic search. They shipped a markdown file.

Feature Simple Choice Rejected
MemoryMarkdown file (CLAUDE.md)Vector databases, embeddings
Searchripgrep (regex)Semantic search, code graphs
ExecutionLocal terminalSandbox, containers
StorageLocal filesCloud sync, databases
"CLAUDE.md is another example of this idea of doing the simple thing first. We had all these crazy ideas about memory architectures -- there's so much literature about this, so many external products -- but in the end, the thing we did is ship the simplest thing." -- Boris Cherny
Principle 2: "Build for the Model Six Months From Now"

Don't add workarounds for current model limitations. When Sonnet and Opus 4 released, Claude Code "just worked" because the interface was already built for future capabilities.

"Honestly, for a long time, Claude Code was not a great product. I used it for maybe 10% of my code because the model wasn't capable enough." -- Boris Cherny
Principle 3: Raw Model Access

Boris explicitly rejected building "a Cursor or Windsurf." The goal is minimal abstraction between user intent and model capability.

Principle 4: Unix Philosophy

Do one thing well. Compose freely. Text as interface. Minimal surprise. Claude Code is a programmable CLI, not just a chat interface.

Inside the Drawer -- The Compounding System

How the Tools Compound

Linear usage: Output = Capability x Time Compounding usage: Output = Capability x Time x (1 + Learning Rate)^Sessions

Each session that updates CLAUDE.md increases effective capability for all future sessions. Shared, git-checked knowledge persists across team and time. Every failure improves the future. Multiple perspectives contribute. Verification loops confirm improvements work.

"Every correction becomes permanent context. The cost of a mistake pays dividends forever." -- Boris Cherny
Enables Compounding
Shared knowledge, mistake-to-rule conversion, team contribution, verification loops
Breaks Compounding
Individual files, ignoring failures, CLAUDE.md bloat, skipping verification
Bottom Drawer -- Things Only The Creator Would Know

Hidden Knowledge

Insight 1

CLAUDE.md Was "The Simple Thing" -- the team considered every memory architecture. They shipped a markdown file. Simplicity won.

Insight 2

Claude Code deletes code on model upgrades. Most teams add complexity. Anthropic removes it -- half the system prompt per upgrade.

"Claude Code was not a great product." The bet was on future model capability, not current workarounds.

Insight 4

50-100 PRs per week. 100% AI-contributed code. The entire system working together.

"Think of Claude Code as a compounding system -- not a one-shot tool, but an evolving teammate." -- Boris Cherny
Warning Labels -- Anti-Patterns

Marks on the Bench

Worn Mark 1

Skipping Plan Mode. "The ones struggling often skip planning to save time, then spend more time fixing mistakes. Plan Mode isn't training wheels -- it's measuring before you cut."

Worn Mark 2

Sequential Instead of Parallel. Run five Claudes, context-switch between them. Don't wait for one to finish before starting the next.

Worn Mark 3

Optimizing for Token Speed. Using smaller models to "save time" is false economy -- correction time dominates generation time.

Worn Mark 4

Over-Customization. "My setup might be surprisingly vanilla! Claude Code works great out of the box."
