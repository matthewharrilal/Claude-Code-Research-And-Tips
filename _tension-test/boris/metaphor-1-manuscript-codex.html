Part I

Who Is Boris Cherny?

Boris Cherny is a Staff Engineer at Anthropic and the creator of Claude Code. His role extends beyond engineering leadership into product vision and design philosophy. He is the canonical source for understanding how Claude Code should be used because its design directly reflects his mental model.

Boris's approach reflects deep experience with developer tooling and type systems. His pre-Anthropic work included contributions to the TypeScript ecosystem and developer experience tools -- explaining Claude Code's emphasis on type safety as verification, developer experience over AI complexity, and simple, composable primitives over opinionated frameworks.

The "Vanilla" Philosophy

Boris explicitly describes his own setup as "surprisingly vanilla" despite being the creator. If the creator doesn't need heavy customization, that signals the tool works well out of the box. Anti-complexity is core to his worldview.

Part II

The Canonical Workflow

Boris's workflow is the reference implementation for Claude Code usage. Every element reflects intentional design decisions. It is not a list of tips -- it is a system where each element amplifies the others.

Parallel Sessions

Five parallel Claude instances in terminal with system notifications. Five to ten sessions on claude.ai/code. Sessions started in morning, checked later on mobile. This number represents the maximum a human can context-switch between effectively, the minimum to achieve parallelism benefits.

"He's distributing cognition like compute: allocate it, queue it, keep it hot, switch contexts only when value is ready."

Model Selection: Always Opus

The bottleneck isn't generation speed -- it's correction time. Opus requires less steering, makes fewer tool use errors, produces higher-quality first drafts, and reduces iteration loops.

Total Time = (Generation Time) + (Correction Time) + (Re-prompting Time) Opus: Higher generation, but MUCH lower correction/re-prompting Sonnet: Lower generation, but significantly higher correction loops
"I use Opus 4.5 with thinking for everything. Even though it's bigger and slower than Sonnet, since you have to steer it less and it's better at tool use, it is almost always faster than using a smaller model in the end." -- Boris Cherny

Shared CLAUDE.md in Git

Single shared file at repository root. Entire team contributes multiple times weekly. Every mistake becomes a rule. During code review, tag @.claude to trigger updates. A GitHub Action enforces that relevant PRs update the file.

"Anytime we see Claude do something incorrectly we add it to the CLAUDE.md, so Claude knows not to do it next time." -- Boris Cherny

Plan Mode Workflow

Start in Plan Mode. Iterate back-and-forth until the plan is solid. Switch to auto-accept mode. Claude executes the entire implementation without interruption. This prevents the classic failure: when the agent makes forty changes you didn't want.

"Don't let a system act before you've agreed on intent and constraints." -- Boris Cherny

Verification: The Most Important Thing

Verification is not optional. It is the difference between 1x quality and 2-3x quality. Boris uses Chrome extensions for UI tests, bash scripts for automated checks, mobile simulators, and background verification agents.

"Probably the most important thing to get great results out of Claude Code: give Claude a way to verify its work. If Claude has that feedback loop, it will 2-3x the quality of the final result." -- Boris Cherny

Part III

Design Philosophy

Principle 1

"Do the Simple Thing First"

This is Boris's foundational principle, directly inherited from Anthropic's engineering culture. The team considered embeddings, vector databases, semantic search for memory. They shipped a markdown file.

Feature Simple Choice Complex Alternative Rejected
Memory Markdown file (CLAUDE.md) Vector databases, embeddings
Search ripgrep (regex) Semantic search, code graphs
Execution Local terminal Sandbox, containers
Storage Local files Cloud sync, databases
"CLAUDE.md is another example of this idea of doing the simple thing first. We had all these crazy ideas about memory architectures -- there's so much literature about this, so many external products -- but in the end, the thing we did is ship the simplest thing: a file that has some stuff that's auto-read into context." -- Boris Cherny
Principle 2

"Build for the Model Six Months From Now"

Perhaps Boris's most counterintuitive principle. He didn't add workarounds for model limitations. He built the interface assuming future capabilities. When Sonnet and Opus 4 released, Claude Code "just worked."

"Honestly, for a long time, Claude Code was not a great product. I used it for maybe 10% of my code because the model wasn't capable enough." -- Boris Cherny
Principle 3

"Raw Model Access as Design Goal"

Boris explicitly rejected building "a Cursor or Windsurf." The goal is minimal abstraction between user intent and model capability. No complex workflow enforcement, no opinionated structure requirements, direct access to all tools.

Principle 4

"Unix Philosophy as Foundation"

Do one thing well. Compose freely. Text as interface. Minimal surprise. Claude Code isn't just a chat interface -- it's a programmable CLI that can be composed with other tools, integrated into CI/CD, and scripted.

Part V

The Compounding System

Boris describes Claude Code not as a one-shot tool but as an evolving teammate. The mathematics are explicit:

Linear usage: Output = Capability x Time Compounding usage: Output = Capability x Time x (1 + Learning Rate)^Sessions

Each session that updates CLAUDE.md increases effective capability for all future sessions. Shared, git-checked knowledge persists across team and time. Every failure improves the future. Multiple perspectives contribute. Verification loops confirm improvements work.

"Every correction becomes permanent context. The cost of a mistake pays dividends forever." -- Boris Cherny

Part IX -- Things Only The Creator Would Know

Unique Insights

These are revelations from the deepest stratum -- knowledge available only because the source is the creator himself.

Revelation 1

CLAUDE.md Was "The Simple Thing" -- the team considered embeddings, vector databases, semantic search. They shipped a markdown file. Simplicity won.

Revelation 2

Claude Code Deletes Code on Model Upgrades -- most teams add complexity over time. Anthropic removes it as models improve, deleting half the system prompt when upgrading.

Revelation 3

"Claude Code Was Not a Great Product" -- Boris admitted the tool didn't work well initially. The bet was on future model capability, not current workarounds.

Revelation 4

50-100 PRs per week. 100% AI-contributed code. Not 80%, not 90%. Enabled by the entire system working together.

"Think of Claude Code as a compounding system -- not a one-shot tool, but an evolving teammate." -- Boris Cherny

Part VII

Anti-Patterns Boris Warns Against

Anti-Pattern 1

Skipping Plan Mode. "The ones struggling often skip planning to save time, then spend more time fixing mistakes. Plan Mode isn't training wheels -- it's measuring before you cut."

Anti-Pattern 2

Sequential Instead of Parallel. Run one Claude, wait for it, start another. Wrong. Run five, context-switch between them as they work.

Anti-Pattern 3

Optimizing for Token Speed Over Quality. Using smaller models to "save time" is false economy -- correction time dominates generation time.

Anti-Pattern 4

Over-Customization. "My setup might be surprisingly vanilla! Claude Code works great out of the box."

Part XI

Boris's 8 Core Principles

  1. Simplicity over sophistication The simplest solution that works is the right solution
  2. Trust the model Build for future capability, not current limitations
  3. Raw access over opinion Let users feel the model directly
  4. Verification as force multiplier Feedback loops 2-3x quality
  5. Parallelization over optimization Run many sessions, not one perfect one
  6. Institutional memory compounds CLAUDE.md turns mistakes into permanent improvements
  7. Planning prevents waste Agree on intent before execution
  8. Agents are team members Integrate into workflows, not alongside them
"Start with verification loops + a simple CLAUDE.md -- that's where most of the magic happens." -- Boris Cherny, The One-Sentence Summary
