## 2. The Parts of R5 -- Comprehensive Summary

### Part 1: HARMONY (H1-H4)

Core insight: Harmony is not sameness. Components harmonize through **shared structural DNA** (callout family), **functional contrast** (philosophy explaining code), **shared visual anchors** (left-border alignment), and **editorial complementarity** (one sets up, the other pays off).

Key specifics:
- Callout family shares: zone structure, spacing rhythm (24px top, 28px bottom asymmetric), typography pattern
- High-harmony contrasting pairs: Code Snippet + Essence, File Tree + Decision Matrix, Task + Reasoning
- Left-anchored components (Callouts, File Tree, Decision Matrix) create reading momentum
- Framed components (Core Abstraction, Reasoning) signal "pause and consider"
- "Editorial Spread" pairs: File Tree + Core Abstraction, Decision Matrix + Reasoning, Code + Tip, Essence + Challenge

### Part 2: TENSION (T1-T4)

Core insight: Tension is useful when it creates **cognitive pause** without **cognitive dissonance**. Three dimensions of tension: velocity mismatch, semantic temperature, and callout saturation.

Key specifics:
- Productive tension examples: Gotcha + Code (warning before example), Essence + Reasoning (axiom + deliberation)
- Jarring collisions: Same-type consecutive (Essence + Essence, Gotcha + Gotcha, Code + Code, Reasoning + Reasoning)
- Component velocities: FAST (Callouts, Tasks), MEDIUM (Decision Matrix, File Tree), SLOW (Code, Reasoning, Core Abstraction)
- **Velocity rule:** Never place two SLOW components consecutively without a FAST buffer
- Component temperatures: Warm (Tip, Challenge, Essence), Neutral (Info, File Tree, Decision Matrix, Code), Cold (Gotcha, Task)
- **Temperature rule:** Don't oscillate rapidly between warm and cold; use neutral buffers
- **2-Callout Limit:** Never more than 2 consecutive callouts (sidebar fatigue)

### Part 3: SEQUENCE (S1-S5)

Core insight: Documentation sections are **narrative arcs** with setup, payoff, and resolution. The first component sets tone; the last component must provide resolution.

Key specifics:
- **Setup-Payoff Arc (S1):** Context (Info) -> Map (File Tree/Decision Matrix) -> Principle (Essence/Core Abstraction) -> Demo (Code) -> Action (Task/Challenge) -> Hazards (Gotcha). Not all 6 needed, but this ORDER when present.
- **Density Wave (S2):** Page density should wave, not plateau. Start LOW, build to HIGH, release to LOW-MED, build again. Anti-pattern: starting with Code Snippet (no context) or ending with Code Snippet (no resolution).
- **First Component Rule (S3):** First component sets section tone deliberately. Info = helpful guide, Code = technical reference, Essence = philosophical treatise, Task = practical tutorial.
- **Resolution Component (S4):** Sections end with resolution (Task, Challenge, Tip, prose summary), NOT tension (Gotcha, Reasoning without recommendation, Code without interpretation, Decision Matrix without guidance).
- **Teaching Moment (S5):** Proven sequence: Essence -> Core Abstraction -> Code Snippet -> Tip. Moves abstract -> concrete -> practical.

### Part 4: GROUPING (G1-G4)

Core insight: Spatial proximity communicates semantic relationship. Different component types signal different things about containment.

Key specifics:
- **Spacing taxonomy:** Tight (8-16px) for related components, Standard (24-32px) between topics, Section break (48-64px) between major sections.
- **Container Signal (G2):** Framed = "bounded thought" (Core Abstraction, Reasoning). Anchored = "annotation" (Callouts, Decision Matrix). Don't place two framed components adjacent without non-framed connective tissue.
- **Visual Weight (G3):** Heavy (Code, Reasoning, Core Abstraction), Medium (Decision Matrix, Task, File Tree), Light (all Callouts). Golden Ratio: 1-2 light components for every heavy component.
- **Advisory Cluster (G4):** Group related advisories together (Info + Tip + Gotcha about same concept) rather than scattering them.

### Part 5: NESTING (N1-N4)

Core insight: KortAI components are designed to **stack vertically, not nest**. The flat editorial design philosophy rejects deep nesting. Maximum effective depth is 1 level.

Key specifics:
- Only two pre-composed composites: Core Abstraction (Essence quote + Code) and Reasoning Component (Question + Columns + Summary).
- When nesting feels necessary, re-sequence instead (place Tip before/after Code, not inside it).
- Maximum depth: Page -> Components (1 level). Component -> Embedded zone (1 level). Never deeper.

### Part 6: RELATIONSHIPS (R1-R3)

Core insight: Components have **personality** -- kinship, mood, and voice -- that governs combination chemistry.

Key specifics:
- **Kinship (R1):** Siblings (same family DNA), Parent-Child (composite containment), Cousins (shared traits, different families). Example cousins: Essence Callout & Core Abstraction (both serif italic, reverent).
- **Moods (R2):** Full 11-component mood taxonomy with primary/secondary moods. Mood combinations: Contemplative + Teaching (Essence -> Core Abstraction), Urgent + Productive (Gotcha -> Task), Inviting + Technical (Challenge -> Code).
- **Voices (R3):** Full 11-component voice register and speaking style. Voice Harmony Rule: Don't follow enthusiastic with authoritative without transition prose.

### Part 7: ANTI-PATTERNS (A1-A7)

Seven specific anti-patterns with diagnosis and fix:
1. **Callout Cacophony (A1):** 3+ consecutive callouts = sidebar blindness. Fix: limit to 2, interleave.
2. **Code Wall (A2):** Multiple consecutive code snippets without interpretation. Fix: interleave with callouts/prose, max 40-50 lines per snippet.
3. **Decision Paralysis (A3):** Multiple Reasoning after Decision Matrix. Fix: 1 Matrix -> 1 Reasoning with recommendation.
4. **Orphaned Heavy (A4):** Heavy component followed by section break without decompression. Fix: add Tip/prose after heavy components.
5. **Temperature Whiplash (A5):** Rapid warm/cold oscillation. Fix: neutral buffers.
6. **Missing Context Start (A6):** Section starts with Code without orientation. Fix: lead with Info Callout.
7. **Anxiety Ending (A7):** Section ends on Gotcha without resolution. Fix: follow Gotcha with Tip/resolution.

### Part 8: RECIPES (Recipes 1-6)

Six pre-composed component sequences for common documentation use cases:
1. **Quick Reference:** Info -> Code -> Tip (3 components)
2. **Deep Dive:** Info -> Essence -> Core Abstraction -> Code -> Gotcha -> Task (6 components)
3. **Decision Guide:** Info -> Decision Matrix -> Reasoning -> Task (4 components)
4. **Troubleshooting Guide:** Gotcha -> Decision Matrix -> Code -> Task -> Tip (5 components)
5. **Tutorial Section:** Info -> Task -> [Code -> Tip]* -> Gotcha -> Challenge (7+ components)
6. **Architecture Overview:** File Tree -> Core Abstraction -> Decision Matrix -> Tip (4 components)

### Part 9: QUANTITATIVE GUIDELINES (Q1-Q3)

Hard numbers for component density and spacing:
- **Q1 Density Limits:** Short page (<500 words): 1 heavy, 3 callouts, 2 code. Medium (500-1500): 2-3 heavy, 5-6 callouts, 4-5 code. Long (>1500): 4-5 heavy, 8-10 callouts, 6-8 code.
- **Q2 Spacing Standards:** Related components: 16px. Standard separation: 24px. Different topics: 32px. Section break: 48px.
- **Q3 Reading Rhythm Ratios:** Heavy 20-30%, Medium 30-40%, Light 30-40%.

### Appendix: Compatibility Matrix

11x11 matrix rating every component pair as Yes (excellent), OK (acceptable with buffer), or No (avoid consecutive). Key No-go pairs: Gotcha+Gotcha, Essence+Essence, Gotcha+Challenge, Challenge+Challenge, FileTree+FileTree, Matrix+Matrix, Reasoning+Reasoning, Core+Core, Code+Code (all self-adjacent = No or OK).

## 6. Gaps in R5 That CD Explorations Might Fill

### Identified Gaps

1. **No multi-axis combination theory.** R5 assumes vertical stacking. It has no rules for combining components in 2D layouts (Bento grids, sidebars, split views). CD explorations, especially CD-003 (File Tree + Callouts) and CD-004 (Essence as background), will need to discover these rules.

2. **No responsive combination rules.** R5's spacing standards (Q2) and density limits (Q1) are static. How do combination rules adapt at different viewport widths? Does the 2-callout limit become a 1-callout limit on mobile?

3. **No animation/transition combination theory.** R5 addresses static placement. It has no guidance for how components should combine with scroll-based reveals, entrance animations, or progressive disclosure. AD explored scroll choreography; CD needs to test combination rules during animated sequences.

4. **No cross-page combination theory.** R5 treats combinations within a single page/section. It doesn't address how a combination on Page 1 affects expectations for Page 2 (reader learns patterns). CD could discover consistency rules across page boundaries.

5. **No component density ceiling.** Q1 gives maximums per page length, but doesn't define the absolute density ceiling -- at what point does ANY combination of components become overwhelming regardless of composition quality? CD stress tests could establish this.

6. **No partial-component combination theory.** R5 treats components as atomic units. But components have internal zones (Core Abstraction has quote zone + code zone). How do the zones of adjacent components interact visually? E.g., does the code zone of Core Abstraction create visual confusion with an adjacent Code Snippet?

7. **No reader expertise modifier.** R5's rules are presented as universal. Expert readers may tolerate higher velocity sequences, more density, and less decompression. CD could test whether rules need an expertise coefficient.

8. **No voice sequencing beyond pairs.** R3 (Component Voices) establishes voice register but only discusses pairs. What happens in a sequence of 5+ components? Does voice need to follow an arc (like narrative voice in fiction)? CD-005 (all callouts as conversation) could test multi-voice sequencing.

9. **Incomplete Gotcha semantics.** Gotcha is classified as "Cold" temperature and "Urgent" mood, but it appears in BOTH "bad endings" (S4) and productive tension pairs (T1). The rules for WHEN Gotcha is productive vs. anxiety-inducing are underspecified. CD should establish clearer Gotcha placement guidelines.

10. **No combination rules for data-heavy content.** R5's examples use conceptual/tutorial content. How do combination rules apply when components contain data tables, metrics dashboards, or quantitative analysis? This is a gap CD could fill if any exploration uses data-rich content.
