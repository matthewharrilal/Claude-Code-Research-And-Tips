## The Nine Principles

### 1. Start with what exists, not what you want.

Most design systems start with "what should it look like?" This one started with "what does it ALREADY look like, and what is it trying to tell us?" That audit-first approach is the foundation. If you skip it and start making decisions, you have nothing to discover. You're just writing rules. Rules don't compound. Discoveries do.

### 2. Audit perceptually, not just technically.

A technical audit says "this component has border-radius: 4px, that one has border-radius: 0, there's inconsistency, let's pick one." A perceptual audit says "when I LOOK at these components as a family, the ones with sharp edges feel more like KortAI. Why? What does sharpness mean for this product? Precision. Authority. The codebase is telling me something." The perceptual layer is where soul pieces come from. Without it, you get a style guide, not a design system with identity.

### 3. Document findings with IDs from the first discovery.

This is the one thing you CAN do from day one that saves you from having to retroactively activate a backbone later. The moment you discover something â€” "sharp edges are identity, not style" â€” give it an ID. COMP-F-001. Write it down with: what was discovered, how it was discovered, what it means, what should consume it next. If you start doing this from the first audit, the provenance chain builds itself. You never have a "scattered markdown files" problem because every file was born connected.

### 4. Build explorations that formally CONSUME prior findings.

This is the gate file concept. Before you build anything, write down: "I am building this exploration. It consumes these specific findings [IDs]. It will test these specific hypotheses. It will be validated by these specific criteria." That gate file IS the muscle tissue. It's the document that says "this exploration exists BECAUSE those findings exist." Without it, you build explorations that are informed by prior work but don't formally reference it. The connection is in someone's head, not in the system.

### 5. Score and validate through perceptual auditing, not just review.

The /perceptual-deepening skill is what turns subjective design review into scored, iterable assessment. DD-006 didn't score 36/40 because someone thought it looked nice. It scored 36/40 because a systematic perceptual evaluation found that fractal self-similarity produced the strongest cognitive response across multiple assessment dimensions. That score IS evidence. It's what makes the finding validated rather than opined.

### 6. Generate outbound findings after every stage.

This is the step most people would skip. You finish DD, you have your density patterns, you move on to OD. What you DON'T naturally do is sit down and ask: "What did DD teach me that ISN'T about density? What did I learn about component behavior under pressure? About iteration methodology? About what makes the perceptual skill work better?"

Those cross-domain insights are the outbound findings. They're what makes the chain COMPOUND rather than just accumulate. DD-F-006 (fractal self-similarity) isn't a density finding â€” it's a UNIVERSAL finding that happened to be discovered during density work. It applies to OD, AD, CD, everything. If you don't extract it, it stays buried in a DD checkpoint file and OD never benefits.

### 7. Write the handoff note before you move on.

Not after. Before. While you still remember what the next stage needs. The handoff is the bridge. It says "here's what I consumed, here's what I produced, here's what the next stage MUST consume from me, and here's what the next stage is responsible for producing." It's the connective tissue between stages. Without it, each stage starts fresh and has to rediscover what the previous stage already knew.

### 8. Each new stage's gate formally references prior stage findings.

This is what makes the compounding real. Stage 3's research gate doesn't just say "I've read the DD patterns." It says "I am consuming DD-F-001 (finding text) applied to OD-003 (specific exploration) because (specific reasoning)." Finding-by-finding. The gate is the proof that prior research is actually flowing forward, not just acknowledged.

### 9. The chain builds itself because every step formally references what came before.

If you do all eight steps above from the start, you never need a retroactive activation protocol. The backbone exists from day one. The activation protocol only exists because we didn't do steps 3, 6, and 7 from the beginning â€” we discovered their importance THROUGH the process. That discovery is itself a finding worth capturing.
