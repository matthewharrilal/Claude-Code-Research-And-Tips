Act I
The Promise
Tension

You have a database full of users. You need an API. Not a theoretical one from a tutorial — a real one, with authentication, validation, error handling, and the kind of edge cases that wake you up at 3 AM.

This is the story of building that API. It starts simple and gets complicated, because that is how real software works. The easy part is the beginning.

The Starting Point

Every API begins with a question: what does the outside world need from your system? Not what your database can do — what your users actually need. The answer shapes everything that follows.

For a user management API, the answer is deceptively simple: create users, read users, update users, delete users. Four verbs. Four endpoints. The simplicity is the trap.

Essence
The distance between "four endpoints" and "production-ready API" is where every junior developer becomes a senior one.
Essence

Every API tells a story. The first chapter is always easy to write. The hard part is what happens when real users arrive.

Act II
The Complications
Tension
First Contact With Reality

The basic route structure seems straightforward. Express gives you a router, you define your endpoints, you connect to the database. Twenty lines of code and you have a working API. This is where tutorials end and reality begins.

typescript — routes/users.ts
1 // The naive version — what tutorials show you
2 router.get('/users/:id', async (req, res) => {
3 const user = await db.users.findById(req.params.id)
4 res.json(user)
5 })

Five lines. Clean. Elegant. And completely wrong for production. What happens when the ID does not exist? When the database is down? When someone sends a string where you expect a UUID? The complications start here.

The Validation Problem

Input validation is the first wall. Every field needs constraints: email format, password strength, name length. You can write these checks inline, scatter them through your handlers, and create a maintenance nightmare. Or you can centralize them with a schema.

typescript — schemas/user.ts
1 // Centralized validation — the schema approach
2 const CreateUserSchema = z.object({
3 email: z.string().email(),
4 name: z.string().min(2).max(100),
5 password: z.string().min(8).regex(/[A-Z]/).regex(/[0-9]/),
6 })
Info
Zod schemas serve double duty: runtime validation and TypeScript type inference. Define the schema once, get both the validator and the type with z.infer<typeof CreateUserSchema>.
The Error Handling Problem

Now every handler needs error handling. Not just "something went wrong" — structured errors that tell the client exactly what failed and how to fix it. 404 for missing resources. 422 for validation failures. 401 for unauthorized access. Each error is a conversation with the caller.

The tension builds: validation, error handling, authentication — each layer adds complexity. The simple four-endpoint API now has middleware, schemas, error formatters, and authentication guards. And we have not even reached the hard part yet.

Act III
The Battle
Tension
Authentication and Authorization

Authentication answers "who are you?" Authorization answers "what can you do?" Both must be solved, and they must be solved differently. Here is where the architecture decisions compound.

typescript — middleware/auth.ts
1 // Authentication middleware — verify identity
2 export const authenticate = async (req, res, next) => {
3 const token = req.headers.authorization?.replace('Bearer ', '')
4 if (!token) throw new AuthError('No token provided')
5
6 const payload = await verifyJWT(token)
7 req.user = await db.users.findById(payload.sub)
8 if (!req.user) throw new AuthError('User not found')
9 next()
10 }
11
12 // Authorization — verify permissions
13 export const authorize = (...roles: Role[]) => (req, res, next) => {
14 if (!roles.includes(req.user.role))
15 throw new ForbiddenError('Insufficient permissions')
16 next()
17 }
Gotcha
Never store JWTs in localStorage — they are accessible to any script on the page (XSS vulnerable). Use httpOnly cookies with the Secure flag. The token should be invisible to JavaScript entirely.
The Race Condition

Two requests arrive simultaneously: both try to update the same user's email. Without concurrency control, the last write wins — silently overwriting the first. This is the moment where naive implementations break in production.

typescript — services/user.ts
1 // Optimistic locking — prevent silent overwrites
2 async function updateUser(id: string, data: UpdateInput, version: number) {
3 const result = await db.execute(`
4 UPDATE users SET email = $1, version = version + 1
5 WHERE id = $2 AND version = $3`,
6 [data.email, id, version]
7 )
8 if (result.rowCount === 0)
9 throw new ConflictError('Resource was modified by another request')
10 }

The version column acts as a guard. If someone else modified the row since you read it, the WHERE clause finds zero rows, and you get a clean 409 Conflict instead of corrupted data.

The Full Route — Everything Combined

Here is what the naive five-line handler looks like after surviving reality: validation, authentication, authorization, error handling, and concurrency control all woven together.

The Architecture
Tension
The Final Shape

After the battle, the dust settles. The four simple endpoints are still there, but now they sit inside a layered architecture that handles the real world. Here is what the final project looks like.

Project Architecture src/
├── routes/ ← Thin handlers, delegates to services
│   ├── users.ts ← CRUD endpoints
│   └── auth.ts ← Login, register, refresh
├── services/ ← Business logic, transactions
│   └── user.service.ts ← Create, update, delete (with locking)
├── middleware/ ← Cross-cutting concerns
│   ├── auth.ts ← authenticate + authorize
│   ├── validate.ts ← Schema validation
│   └── errors.ts ← Centralized error handler
├── schemas/ ← Zod schemas (validation + types)
│   └── user.ts ← Create, Update, Query schemas
└── errors/ ← Typed error classes
    ├── auth.error.ts
    ├── validation.error.ts
    └── conflict.error.ts
The Lesson

The naive version was not wrong. It was incomplete. Every layer you added — validation, authentication, error handling, concurrency — exists because a real user would have encountered a real failure without it.

Production readiness is not about adding complexity for its own sake. It is about anticipating the ways reality will test your assumptions, and having an answer ready for each one.

Challenge
Take an existing API endpoint in your codebase. Walk through the narrative arc: what is its "tutorial version"? What complications has it absorbed? What layers are still missing?
The Story So Far
  • APIs start as four verbs but become layered architectures under production pressure
  • Validation, authentication, and error handling are not optional — they are the real API
  • Concurrency control prevents the silent data corruption that tutorials never show
  • The narrative arc mirrors real development: simple beginning, complex middle, architectural resolution
  • Each layer of complexity exists because a real failure demanded it
