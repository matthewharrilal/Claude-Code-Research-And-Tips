The Testing Landscape

Every software project faces the same fundamental question: how do you know your code works? Testing is not merely a quality gate -- it is a design discipline. The way you test reveals the architecture of your system. The tests you write become the specification your code must honor.

Essence
Tests are not proof that code works. They are the language in which you express what "working" means.
Term
Unit Test

A unit test verifies a single function, method, or module in isolation from its dependencies. The scope is narrow: one input, one expected output, one assertion. Unit tests run in milliseconds and form the foundation of the testing pyramid. They catch regressions at the lowest cost.

Context
The testing pyramid places unit tests at the base (many, fast, cheap), integration tests in the middle (fewer, moderate speed), and end-to-end tests at the top (few, slow, expensive). Each layer serves a different verification purpose.
Term
Integration Test

An integration test verifies that multiple units work together correctly. It tests the boundaries between modules -- the API contracts, database queries, message passing, and shared state. Integration tests catch the errors that unit tests cannot: mismatched interfaces, incorrect serialization, and broken assumptions about external services.

Term
End-to-End Test

An end-to-end test simulates a real user journey through the entire application stack -- from browser click to database write and back. These tests are the most expensive to write and maintain, but they verify the one thing no other test can: that the system works as a whole, from the user's perspective.

A minimal unit test establishes the pattern you will build on throughout this guide. Every test follows the same three-phase structure: arrange, act, assert. 

// Arrange-Act-Assert: the universal test structure
import { describe, it, expect } from 'vitest';
import { calculateTotal } from './cart';

describe('calculateTotal', () => {
  it('sums item prices with quantity', () => {
    // Arrange
    const items = [
      { price: 10, quantity: 2 },
      { price: 5, quantity: 3 },
    ];

    // Act
    const result = calculateTotal(items);

    // Assert
    expect(result).toBe(35);
  });
});

Deep Dive: Unit Testing Patterns

Structuring Test Suites

A well-structured test suite mirrors the architecture of the code it validates. Each module gets a corresponding test file. Each public function gets a describe block. Each behavior gets an it assertion. This correspondence is not accidental -- it is the F-pattern of testing: scan the module interface, then descend into specific behaviors.

The naming convention matters. Test names should read as specifications: "it calculates total with discount applied" is a specification. "it works" is not. When a test fails, the name should tell you what broke without reading the test body.

Tip
Use the pattern it('[verb]s [noun] when [condition]') for test names. This reads as a specification: "it calculates total when discount is applied," "it throws error when input is null." 
// Mirror module structure in test files
import { describe, it, expect, beforeEach } from 'vitest';
import { UserService } from './user-service';

describe('UserService', () => {
  let service: UserService;

  beforeEach(() => {
    service = new UserService();
  });

  describe('createUser', () => {
    it('creates user with valid data', async () => {
      const user = await service.createUser({
        name: 'Ada Lovelace',
        email: 'ada@example.com'
      });
      expect(user.id).toBeDefined();
      expect(user.name).toBe('Ada Lovelace');
    });

    it('throws error when email is missing', async () => {
      await expect(
        service.createUser({ name: 'Ada' })
      ).rejects.toThrow('Email is required');
    });
  });
});

Mocking Strategies

Mocking replaces real dependencies with controlled substitutes. A mock database returns predictable data. A mock HTTP client responds without network latency. Mocking isolates the unit under test from the chaos of the external world.

But mocking carries risk. Over-mocking creates tests that pass even when the real integration is broken. The mock says "yes" while the real service says "404." This is the central tension of test design: isolation versus integration fidelity.

Core Principle

Mock at the boundary, not at the implementation. Replace the database driver, not the business logic. Replace the HTTP client, not the service method. The boundary is where your code meets someone else's code. 

// Mock at the boundary: replace the repository, not the logic
import { vi } from 'vitest';
import { OrderService } from './order-service';
import { OrderRepository } from './order-repo';

// Mock the BOUNDARY (repository), not the SERVICE
const mockRepo = {
  findById: vi.fn(),
  save: vi.fn(),
} as OrderRepository;

const service = new OrderService(mockRepo);
Gotcha
Never mock what you do not own. If you mock a third-party library's internals, your tests become coupled to its implementation. When the library updates, your mocks break silently -- tests pass, but production fails.
Reasoning

When to Mock vs. When to Use Real Dependencies

Mock When
  • External API calls (HTTP, third-party services)
  • Database operations in unit tests
  • File system operations
  • Time-dependent behavior (Date.now())
  • Non-deterministic outputs (random, UUID)
Use Real When
  • Testing business logic (pure functions)
  • Integration tests with test databases
  • Utility function validation
  • Type conversion and serialization
  • Algorithm correctness verification
Summary

The decision comes down to determinism. If calling the real dependency produces the same result every time, use the real thing. If the result varies based on external state, mock it. The boundary between deterministic and non-deterministic IS the boundary between real and mock.

Organizing Test Files

Two dominant conventions exist for test file placement. Co-location places test files next to the source files they test: user-service.ts alongside user-service.test.ts. Separation places all tests in a dedicated __tests__ directory. Co-location makes the mapping obvious; separation keeps the source tree clean. 

// Co-location pattern (recommended)
// Each source file has a sibling test file
src/
  services/
    user-service.ts
    user-service.test.ts     // co-located
    order-service.ts
    order-service.test.ts   // co-located
  utils/
    format.ts
    format.test.ts          // co-located
  __tests__/
    integration/             // integration tests separated
      user-flow.test.ts
      order-flow.test.ts
Tip
Use co-location for unit tests and separation for integration tests. This gives you the best of both patterns: unit tests are discoverable next to their source, integration tests live where they can share setup fixtures and test databases.

Pipeline

CI Integration

  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm test
      - run: npm run test:e2e
Gotcha
Never run E2E tests in CI without explicit browser installation. npx playwright install --with-deps must run before any Playwright tests, or the CI job fails with a cryptic "browser not found" error.

Build Your Test Suite

You have traversed the testing landscape from foundational concepts through deep implementation to infrastructure reference. The path forward is practice. The task below gives you a concrete starting point; the challenge invites you to go further.

Setup Checklist
  • Install Vitest: npm install -D vitest
  • Create vitest.config.ts with jsdom environment
  • Write your first test for a pure utility function
  • Add a test script to package.json
  • Run npm test and verify green output
  • Add coverage reporting with @vitest/coverage-v8
Challenge
Extend your test suite beyond unit tests. Write an integration test that verifies your API handler connects to a test database, creates a record, and returns it correctly. Then write an E2E test with Playwright that fills a form, submits it, and asserts the result appears on the page. Measure your coverage -- aim for 80% line coverage on your first pass.
