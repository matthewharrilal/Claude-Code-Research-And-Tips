2 The Core Abstraction

"Think of extensions as LEGO blocks. Each provides one capability. Power comes from combination."
Single Extension = Capability | Multiple Extensions = Workflow | Coordinated Extensions = System
Extensions transform Claude Code from a chat interface into a composable agentic platform.

Extensions are not monolithic features - they are composable primitives that stack. Each extension provides exactly ONE capability. Claude HUD provides observability. Claude-Mem provides persistence. Ralph provides iteration. Code-Simplifier provides polish.

Use in Compositions: Monitor Ralph loops for context exhaustion, debug multi-agent workflows, track token budget during long sessions.

5 Composition Patterns

Extensions combine in four fundamental patterns. Understanding these patterns lets you architect custom workflows.

PATTERN 1: CHAIN (A -> B -> C)
Sequential processing. Ralph iterates on feature, then triggers code-simplifier, then commits. 
/ralph-loop "Build user auth. When done, use code-simplifier agent. Output <promise>COMPLETE</promise> after polish."
PATTERN 2: PARALLEL
Multiple extensions running simultaneously in different terminals. 
# Terminal 1
tmux new-session -d -s backend "cd ~/project/backend && ./ralph.sh 20"

# Terminal 2
tmux new-session -d -s frontend "cd ~/project/frontend && ./ralph.sh 20"

# Terminal 3
rpai  # Monitor all
PATTERN 3: CONDITIONAL
Extensions triggered based on conditions (via HUD observation or hooks). 
IF context > 80% (via HUD observation)
THEN trigger new Ralph iteration (fresh context)
ELSE continue current session
PATTERN 4: STACKED PERSISTENCE
Multiple persistence layers with different granularities. 
Claude-Mem: Long-term semantic memory
     |
Ralph prd.json: Task state tracking
     |
progress.txt: Append-only session learnings
     |
Git: Code change history

Proven Combinations

AGI Stack (Opus 4.5 + Ralph + Playwright)
Source: @weswinder

Why It Works: Ralph feeds same prompt repeatedly until success. Playwright enables real-world UI verification. Opus 4.5 reduces iterations needed.

Critical Optimization: "Put costly tools like browser control in subagents to protect your main context window tokens" - @TendiesOfWisdom

Claude-Mem + Claude HUD + Code-Simplifier + Auto-formatting hooks

Claude-Mem loads relevant context from past sessions. HUD shows real-time context usage. At 85%, HUD token breakdown appears. Every N edits, trigger code-simplifier. PostToolUse hooks auto-format files.

6 Gotchas

Real problems you'll hit when installing and using extensions:

Plugin installation fails
Symptom: /plugin install <name> returns error or plugin doesn't appear in /plugin list
Cause: Plugin not in local cache, or marketplace add step was skipped
Fix: For marketplace plugins, always add first: /plugin marketplace add author/plugin-name THEN /plugin install plugin-name
Claude-Mem not capturing sessions
Symptom: search() returns no results from past sessions
Cause: Worker service not running, or lifecycle hooks not installed correctly
Fix: Check lsof -i :37777 for worker process. Restart Claude Code after installation. Verify hooks in settings.json.
CC Mirror orchestration skill not loading
Symptom: After npx cc-mirror quick, orchestration skill doesn't activate
Cause: Alias not set up correctly, or skill load command not sent
Fix: Verify alias with which mclaude. Explicitly load: "Load the orchestration skill and initiate"
MCP server connection errors
Symptom: MCP tools show in /mcp but fail with "connection refused" or "timeout"
Cause: Server process crashed, or configuration has wrong path/command
Fix: Test server manually: npx -y @modelcontextprotocol/server-github. Check environment variables are set.
Conflicting hooks from multiple extensions
Symptom: Hooks fire multiple times, or one extension's hooks block another's
Cause: Multiple extensions registered hooks for same events
Fix: List hooks: cat ~/.claude/settings.json | jq '.hooks'. Order intentionally - hooks execute in array order.

7 What's Hard

These are fundamental ecosystem gaps and tensions, not bugs to fix:

Extension Dependency Resolution
The tension: Extensions have no dependency management. If plugin A requires plugin B, you must know to install B first.
Symptoms: Cryptic errors when extension expects another extension. Order-dependent installation failures.
Mitigation: Read extension docs carefully. Install in recommended order. Restart Claude Code between installations.
Hook Conflict Resolution
The tension: Multiple extensions can register for the same hook events. No built-in priority or conflict resolution.
Symptoms: One extension's hook blocks another. Unexpected behavior when combining extensions.
Mitigation: Manual hook ordering in settings.json. Consolidate into single wrapper scripts. Test combinations in isolation.
Missing Ecosystem Infrastructure
The gap: No visual diff viewer, no cost tracker, no session recording, no team collaboration, no rollback manager.
Current workarounds: Manual estimation for cost. Git reset for rollback. External DB for team sharing.
Status: Active community development. Windows support improving. Plugin marketplace UI requested.

8 When to Use / When Not

ADD EXTENSIONS WHEN
You need capabilities Claude Code lacks natively (observability, memory)
Your workflow has repeatable patterns (automation candidates)
You're building for overnight/autonomous execution
External integrations are required (GitHub, databases, browsers)
Token efficiency matters (Claude-Mem provides 95% reduction)
STAY VANILLA WHEN
Simple one-off tasks (complexity overhead not worth it)
Exploratory work with unclear end state
Security-sensitive environments (verify extension sources)
Learning Claude Code (understand base before extending)
Debugging extension conflicts (simplify first)
