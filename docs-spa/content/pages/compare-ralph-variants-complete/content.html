Essence (15 words)
Five Ralph variants share fresh-context DNA but optimize for different use cases: simplicity to validation.
"Fresh context beats accumulated context. Memory lives in files, not in the model." - Geoffrey Huntley
All five variants inherit this fundamental insight. Their differences are optimizations.

Ralph Wiggum is not one pattern but a family of related patterns for autonomous AI coding. Original Ralph is the foundation. Production Ralph adds structure. Marathon Ralph adds SDLC automation. Lisa handles planning. Zeroshot provides validation guarantees.

Understanding the variant spectrum helps you choose the right tool for your use case - neither over-engineering simple tasks nor under-engineering complex ones.

2 The Five Variants

ORIGINAL Original Ralph (Geoffrey Huntley)
Level: 3 Cost: $ Setup: 15 min Source: ghuntley.com/ralph 
while :; do cat PROMPT.md | claude ; done

The foundational loop. Everything else is scaffolding around this core. Each iteration spawns a NEW Claude instance with fresh context. State persists in files, not in model memory.

Best for: Learning Ralph concepts, simple tasks, quick prototypes, budget-constrained projects.

PRODUCTION Production Ralph (Ryan Carson / snarktank)
Level: 3-4 Cost: $ Setup: 30 min Source: github.com/snarktank/ralph (4.9k stars)

Transforms the basic loop into a repeatable, reliable system. Uses prd.json for structured task format, AGENTS.md for long-term patterns, and an archive system for cross-run learning.

The Iron Rule: "Each story must be completable in ONE iteration (~one context window)." If you can't describe it in 2-3 sentences, split it.

Best for: Feature development, teams with existing tests, learning to scale.

MARATHON Marathon Ralph (Stephen Rayner / gruckion)
Level: 5 Cost: $$ Setup: 1+ hour Requires: Linear

Automates the complete software development lifecycle: VERIFY → PLAN → CODE → TEST → QA → EXIT. Six phases per issue, with mandatory test generation (unit, integration, E2E).

Uses Linear for project creation, issue decomposition, and state persistence. Circuit breakers prevent infinite retry loops (3-5 attempts per phase).

Best for: Building complete applications, mandatory test coverage, Linear users.

LISA Lisa (BLEN Engineering / Mike Endale)
Level: 3.5 Cost: $ Setup: 10 min Output: prd.json

"Lisa plans. Ralph does." Most Ralph failures trace back to poor specifications. Lisa conducts systematic, in-depth interviews (15-30 questions) to extract scope, acceptance criteria, edge cases, and implementation phases. 

/lisa:plan "Feature Name" --first-principles

Best for: Creating specifications before Ralph, complex requirements, uncertain scope.

ZEROSHOT Zeroshot (covibes)
Level: 5-6 Cost: $$$ Setup: 30 min Features: Blind validation, crash recovery, provider agnostic

"Validators can't lie about code they didn't write." Multi-agent topology with Conductor (classifier), Planner (strategist), Worker (implementer), and multiple Validators (blind assessment).

Validators check work WITHOUT access to the worker's reasoning - they see only current files. This removes self-validation bias and catches issues fresh perspective reveals. 

zeroshot run 123 --pr

Best for: Validation guarantees, independent review, multi-provider experimentation, compliance.

3 Decision Tree: Which Ralph?

DO YOU HAVE A CLEAR SPECIFICATION? ├── NO → Use LISA to create the specification │ └── YES → DO YOU NEED INDEPENDENT VALIDATION? ├── YES → Use ZEROSHOT (multi-agent blind validation) │ └── NO → DO YOU NEED AUTOMATIC TEST GENERATION? ├── YES → Use MARATHON RALPH (requires Linear) │ └── NO → IS THIS A MULTI-STORY FEATURE? ├── YES → Use PRODUCTION RALPH (PRD-driven) │ └── NO → Use ORIGINAL RALPH (simple loop)

For most individual developers, Lisa + Production Ralph is the sweet spot. Planning investment reduces execution rework. Moderate cost, moderate complexity, good reliability.

6 Migration Paths

Original → Production

1
Convert PROMPT.md tasks to prd.json user stories with id, title, passes: false
2
Add AGENTS.md for long-term patterns: echo "# AGENTS.md\n\n## Codebase Patterns" > AGENTS.md
3
Create archive directory: mkdir -p scripts/ralph/archive
4
Update prompt.md to reference prd.json instead of inline tasks

Any → Lisa (Planning Enhancement)

1
Install Lisa plugin: /plugin marketplace add blencorp/lisa and /plugin install lisa
2
Run Lisa interview BEFORE your current workflow: /lisa:plan "Feature" --first-principles
3
Answer 15-30 interview questions (scope, acceptance criteria, edge cases, phases)
4
Copy Lisa output to your Ralph setup: cp docs/specs/feature.json scripts/ralph/prd.json
5
Run your existing Ralph variant as before

7 Integration Patterns

Lisa + Ralph Pipeline

The most common integration - planning followed by execution. 

# Step 1: Run Lisa interview
/lisa:plan "My Feature" --first-principles
# [Answer 15-30 questions]
# [Say "done" when complete]

# Step 2: Copy spec to Ralph location
cp docs/specs/my-feature.json scripts/ralph/prd.json

# Step 3: Run Ralph loop
./scripts/ralph/ralph.sh 25

Ralph + Git Worktrees (Parallel Ralph)

Run multiple Ralph instances on separate branches for true parallelization. 

# Create worktrees
git worktree add ../project-auth feature/auth
git worktree add ../project-api feature/api

# Terminal 1
cd ../project-auth && ./scripts/ralph/ralph.sh 25

# Terminal 2
cd ../project-api && ./scripts/ralph/ralph.sh 25

# Cleanup when done
git worktree remove ../project-auth
git worktree remove ../project-api

Anti-Patterns to Avoid
Marathon + Zeroshot: Redundant validation, excessive cost
Original + Gas Town: Factory overhead for simple loop
Multiple Ralphs same branch: Git conflicts, state confusion

8 When to Use / When Not

USE ORIGINAL RALPH WHEN
Learning Ralph concepts - minimal complexity
Simple, well-defined tasks - low overhead
Quick prototypes - fast setup
Budget-constrained projects
DON'T USE ORIGINAL WHEN
Complex multi-story features - needs story sizing
Team environments - no built-in coordination
Need test generation - not included
USE ZEROSHOT WHEN
Complex implementations - validation guarantees
Need independent review - blind validators
Multi-provider experimentation - agnostic design
Compliance requirements - SQLite audit trail
DON'T USE ZEROSHOT WHEN
Simple tasks - overhead not justified
Exploratory work - needs clear done criteria
Cost-constrained - multiple agents = more cost
The Optimal Progression
Start: Original Ralph (learn concepts) → Add planning: Lisa + Original → Add structure: Lisa + Production Ralph → Fork based on need: Worktrees (parallel) | Marathon (SDLC) | Zeroshot (validation) → Scale: Gas Town Factory
