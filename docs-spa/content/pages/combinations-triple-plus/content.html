Essence (15 words)
Triple+ stacks combine patterns multiplicatively: 1+1+1 equals 10x, not 3x productivity.

2 The Core Abstraction

"Individual patterns add. Combined patterns multiply."
Ralph (2x) + Hooks (1.5x) + Subagents (2x) = 6-10x (not 5.5x)
The multiplication happens because each pattern removes friction that was limiting the others.

Why multiplicative? Hooks remove friction that slows Ralph loops. Subagents handle complexity that would break Ralph's simple iteration. Ralph's file-based state gives subagents persistent memory. Each pattern enables the others to work at full capacity.

This is advanced territory. Only attempt triple+ stacks after mastering individual patterns and pairwise combinations. Prerequisites: familiarity with Ralph, Hooks, Subagents, and CC Mirror.

# ralph.sh with Archiving
#!/bin/bash
set -e

MAX_ITERATIONS=${1:-50}
RUN_ID=$(date +%Y%m%d_%H%M%S)

for (( i=1; i<=$MAX_ITERATIONS; i++ )); do
  echo "=== Iteration $i / $MAX_ITERATIONS ===" | tee -a plans/progress.txt

  claude "Review plans/prd.json and plans/progress.txt.
    Check plans/archive/learnings.md for previous insights.
    Mark completed PRD items as passes: true.
    Append your progress to progress.txt.
    Make a git commit for completed features.
    Work on ONE task at a time.
    If PRD is complete, output PROMISE COMPLETE HERE." | tee output.txt

  if grep -q "PROMISE COMPLETE HERE" output.txt; then
    echo "=== COMPLETE at iteration $i ===" >> plans/progress.txt
    cp plans/progress.txt "plans/archive/run-$RUN_ID-progress.txt"
    break
  fi
done

6 Anti-Patterns: What NOT to Combine

These combinations seem logical but break in production:

Opus Everything + No Budget
Components: Full Opus + Auto-accept + No limits
Why it fails: $500+/day costs. Unsustainable for anything but funded teams.
Fix: Tiered model selection - Haiku for research, Sonnet for implementation, Opus for architecture only.
Parallel + Shared State
Components: Multiple agents + Same branch
Why it fails: Merge conflicts, file corruption, agents overwriting each other's work.
Fix: Git worktrees. Each agent gets its own branch: git worktree add ../session-1 -b feat/a
Ralph + No Verification
Components: Infinite loops + No tests
Why it fails: Errors compound. One bad commit at iteration 5 poisons iterations 6-20.
Fix: Never skip verification. If tests don't exist, add them before starting Ralph.
Mobile + Complex Tasks
Components: Voice-only + Architecture work
Why it fails: Architecture needs visual context - diagrams, code structure, dependencies.
Fix: Use mobile for direction ("fix the auth bug"), keyboard for precision ("use bcrypt, not argon2").
Haiku + Complex Reasoning
Components: Cheapest model + Hard problems
Why it fails: Haiku fails on complex tasks, wastes iterations, costs MORE in aggregate.
Fix: Match model to task. Haiku for file reading, Sonnet for coding, Opus for architecture decisions.
Adversarial Without Main
Components: Review agent + No implementation
Why it fails: All criticism, no output. Review agent blocks everything, nothing ships.
Fix: Calibrate rejection thresholds. Reject for security/logic errors only, not style preferences.

7 What's Hard

Fundamental tensions that don't have perfect solutions:

Coordination Overhead
The tension: More agents = more capability, but also more coordination cost. At some point, coordination overhead exceeds productivity gain.
Symptoms: Agents waiting for each other. Orchestrator spending more tokens managing than doing. Conflicts that require human intervention.
Mitigation: Start with 2-3 agents max. Add more only when you've extracted full value from current setup. Independent tasks don't need orchestration - just parallel Ralph loops.
Cost vs Quality vs Speed
The tension: Pick two. Opus is quality but expensive. Haiku is cheap but limited. Parallel is fast but multiplies cost. No free lunch.
Symptoms: Budget overruns. Quality complaints. Missed deadlines from cost-cutting.
Mitigation: Define your constraint first. Budget-constrained? Use tiered models. Quality-critical? Accept higher cost. Deadline? Accept both higher cost AND some quality risk.
Stack Complexity vs Maintainability
The tension: Every added component increases capability but also increases failure modes and debugging difficulty.
Symptoms: Mysterious failures. "It worked yesterday." Hours spent debugging infrastructure instead of shipping features.
Mitigation: Add components one at a time. Verify each works before adding the next. Document your stack configuration. Have a "strip to basics" fallback for emergencies.

8 When to Use / Real-World Examples

USE TRIPLE+ STACKS WHEN
You've mastered individual patterns (don't start here)
Project is large enough to justify setup overhead
You have budget for experimentation and iteration
Tasks can be parallelized (multiple independent features)
You're willing to invest 5 weeks building up the stack
DON'T USE TRIPLE+ STACKS WHEN
You're new to Claude Code (master singles first)
Project is small (<1 week of work)
Budget is tight (stack setup + optimization costs)
Work is inherently sequential (no parallelization benefit)
You need results THIS WEEK (stack building takes time)
