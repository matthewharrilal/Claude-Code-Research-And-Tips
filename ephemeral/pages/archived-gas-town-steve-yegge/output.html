Zone 1 / Situation Brief

The Dispatch

Gas Town is not a better single agent. It is a factory for agents. While the industry fixates on making Claude Code run longer, handle bigger files, and survive more tokens, Steve Yegge looked at the problem from a completely different angle: what if the unit of work isn't the ant, but the colony?

Yegge is a 40-year veteran -- ex-Google, ex-Amazon, ex-Sourcegraph. He built Gas Town in three weeks of pure vibe coding, never once looking at the source. It is 225,000 lines of Go orchestrating multiple Claude Code instances through a tmux-based command structure. The code is fragile. The idea is not.

"Claude Code is the world's biggest fuckin' ant. Everyone is focused on making their ant run longer... when nature prefers colonies."
Steve Yegge -- @steve_yegge

This is the thesis that animates Gas Town: stop optimizing the individual agent and start building the infrastructure that lets agents cooperate, specialize, and scale. The orchestrator manages the colony. You manage the orchestrator. Your role shifts from coder to product manager.

Zone 2 / Operational Readiness

The 8 Stages of Dev Evolution

Gas Town is for Stage 7+. If you are not there, Yegge warns, you will get your face ripped off by superintelligent chimpanzees. This is not a joke -- it is a calibration device. Each stage represents a qualitative shift in how you relate to AI tooling.

Stage Description
1 Zero or near-zero AI usage
2 Agent in IDE, permissions on
3 Agent in IDE, YOLO mode
4 Agent fills screen, code is just diffs
5 CLI single agent, YOLO
6 CLI multi-agent (3-5 parallel)
7 10+ agents, hand-managed
8 Building your own orchestrator

Architecture

The system is organized around two concepts: the Town (your orchestrator headquarters at ~/.gt) and Rigs (each project repo under management). The gt binary orchestrates everything.

Town Structure
~/.gt/ config.yaml # Town configuration rigs/ project-a/ # Each managed repo project-b/ logs/ # Agent activity

The 7 Worker Roles + You

Gas Town defines a chain of command. Each role is a specialized agent type with distinct responsibilities. The hierarchy encodes both authority and scope.

Rank 1 -- Coordination
Mayor
Town-level coordination across all rigs. The central nervous system. Manages inter-project dependencies and resource allocation.
Rank 2 -- Monitoring
Deacon
Monitoring and handshakes. Watches agent health, confirms task completion.
Rank 3 -- Quality
Dogs
Quality gates and watchdogs. Enforce standards, catch regressions.
Rank 4 -- Decomposition
Refinery
Task decomposition. Breaks large work into assignable units.
Rank 5 -- Persistent
Polecat
Named persistent workers. Long-running specialists with project memory.
Rank 6 -- Observation
Witness
Per-rig observer. Passive monitoring of each project's state.
Rank 7 -- Ephemeral
Crew
Ephemeral workers. Spun up for tasks, discarded after completion.
Human -- Boss
Overseer (You)
Human boss with inbox. You review, approve, and direct. Your role is product management, not coding.
Zone 3 / Field Intelligence

Key Concepts & Intel

Vibe Coding Philosophy

Doctrine
"Work becomes fluid, an uncountable that you sling around freely, like slopping shiny fish into wooden barrels at the docks."

Some bugs get fixed two or three times. Some fixes get lost entirely. The focus is throughput, not perfection. You are a product manager. Gas Town is an idea compiler.

Beads Integration

Gas Town runs on Beads (github.com/steveyegge/beads) -- a git-backed data plane for universal context management. Approximately 225,000 lines of Go, entirely vibe-coded. Beads provides the shared memory layer that agents use to coordinate without explicit message-passing.

MCP Agent Mail

Agents collaborate via email-like interfaces through MCP Agent Mail. No special training is required -- agents naturally adopt "village" behavior when given mailbox primitives. This is social infrastructure for AI agents, treating communication as a first-class concern.

Why Go?

Yegge's findings from approximately 1 million lines vibe-coded across 2025:

Language Issue
TypeScript 1/3 to 1/2 of diffs are type wrestling. The type system fights the agent.
Python "Fine" but feels like scripts. Lacks the structural confidence for large systems.
Go "Boring" -- and boring is an evolutionary advantage. Diffs are always understandable.
Advantage

Go's boringness is the point. When agents generate code, you need to be able to read the diffs. TypeScript diffs are half type gymnastics. Go diffs are the actual logic. For vibe coding at scale, readability of generated output matters more than language expressiveness.

Warnings from the Author

Yegge attaches five explicit warnings to Gas Town. These are not caveats -- they are threat assessments.

Threat Assessment

1. Code is 3 weeks old -- "smuggled 400 miles upriver in my ass." The codebase is embryonic and volatile.

2. 100% vibe-coded -- Steve has never looked at the code. This is not an exaggeration.

3. Expensive as hell -- Multiple Claude accounts required. Token costs are substantial at scale.

4. Requires tmux -- The primary UI. No GUI unless you build one or use community extensions.

5. Can wreck your shit -- Superintelligent chimpanzees with access to your filesystem.

Caution
"If you have any doubt whatsoever, then you can't use it."

This is not modesty. This is an honest assessment of the readiness requirement. Stage 7+ is not a recommendation -- it is a minimum viable competence threshold.
