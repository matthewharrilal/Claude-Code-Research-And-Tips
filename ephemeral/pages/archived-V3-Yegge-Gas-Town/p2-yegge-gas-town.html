Section 03 -- Architecture

The 8 Roles Architecture

8

Eight specialized roles organized into three tiers: Town Level (global coordination), Rig Level (per-project operations), and Human Level (you). Each role has a specific function, a model recommendation, and a set of commands.

Role Deep Dives

Mayor -- Town-Level Coordination Opus 4.5

Your concierge and chief-of-staff. Receives high-level directives from Overseer. Routes work to appropriate rigs. Manages cross-rig coordination. Handles town-level decisions. Primary AI coordinator for the entire town.

gt mayor up        # Start mayor
gt mayor status    # Check mayor state
gt mayor down      # Stop mayor
Deacon -- Monitoring & Handshakes Haiku

Health monitoring and agent coordination. Monitors agent health across the town. Handles agent-to-agent handshakes. Reports anomalies to Mayor. Tracks work progress. Manages handoff protocols and worker recycling.

gt deacon up       # Start deacon
gt deacon status   # View monitoring state
Dogs -- Quality Gates Sonnet

Quality assurance and safety gates -- the town-level grunts. Reviews work before acceptance. Enforces quality standards. Guards against regressions. Can reject and rollback work. Handles stale branch cleanup.

gt dogs up         # Start dogs
gt dogs bark       # Trigger quality check
Refinery -- Task Decomposition Opus 4.5

Breaks down large tasks into agent-sized work. Receives feature requests from Mayor. Decomposes into smaller, atomic tasks. Assigns to Polecats or Crew. Tracks decomposition progress. Manages the merge queue with sequential rebasing.

gt refinery up <rig>           # Start refinery for rig
gt refinery decompose <task>   # Manually trigger decomposition
Polecat -- Named Persistent Workers Sonnet

Named, persistent agent workers with specializations. Execute assigned tasks. Maintain context across sessions. Specialize in rig-specific knowledge. Naming convention: human names (alice, bob, carol) help with mental model.

gt polecat up <rig> alice      # Start polecat named "alice"
gt polecat up <rig> bob        # Start polecat named "bob"
gt polecat list <rig>          # List polecats for rig
gt polecat assign alice <task> # Assign task to alice
Witness -- Per-Rig Observer Haiku

Observes and records rig activity. Watches all work in assigned rig. Records history and context. Provides state to new workers. Reports anomalies to Deacon. Unsticks drifting agents.

gt witness up <rig>    # Start witness for rig
gt witness report      # Get current state report
Crew -- Ephemeral Burst Workers Haiku

Short-lived, task-specific burst workers. Execute single tasks. No persistent state. Cheap and disposable. Used for parallel bursts. Spawn, complete, disappear.

gt crew spawn <rig> --task "fix this bug"
gt crew count <rig>    # Count active crew
gt crew purge <rig>    # Kill all crew
Overseer -- Human Boss (You) Human

Product Manager and decision maker. Define features and priorities. Review inbox decisions. Approve major changes. Provide human judgment. You have an identity in the system with your own inbox. You can send and receive town mail.

# Your tools:
gt inbox              # Check your inbox
gt inbox send <rig> "Build auth"   # File a feature
gt inbox respond <id> --approve   # Approve work
Section 04 -- Memory Infrastructure

Beads: The Memory System

Agents suffer from "50 First Dates" syndrome -- they wake up with no memory of yesterday. Every new Claude session starts fresh. This is the fundamental problem Beads solves.

Beads is a ~225K line Go project providing git-backed storage, universal context across all agents, hash-based IDs (e.g., bd-a1b2) preventing merge conflicts, dependency tracking, and memory decay through compaction.

The Three-Layer Memory Model

Layer 1: Hot Context (RAM-cached)

Current task state, last N interactions, active file references. Immediate access.

Layer 2: Warm Context (SQLite + Git index)

Recent commit history, cross-agent message log, rig state snapshots. Quick queries.

Layer 3: Cold Context (Git history)

Full project history, previous run outcomes, archived decisions. Only query if needed.

bd ready           # List tasks with zero blockers (what can be done NOW)
bd create "Title"  # Create new task
bd dep add X Y     # X is blocked by Y
bd show <id>       # Full task state and history
Why Git as Database?

Git as database provides durable, diffable, auditable storage with zero infrastructure. Survives crashes. Full audit trail. Roll back to any point. Multiple agents share the same store. No heavyweight message bus required.

Section 06 -- Principles

Core Principles

GUPP: The Universal Propulsion Principle

"If there is work on your hook, YOU MUST RUN IT." This solves the fundamental problem that Claude Code sessions end. Work is slung to agent hooks -- persistent git worktrees. When an agent starts, it checks its hook. If work exists, execution is immediate. No confirmation, no waiting. Work survives crashes and restarts.

Zero Framework Cognition

Claude makes all decisions. The orchestrator executes. No heuristics, regex, or parsing in the framework. The infrastructure provides plumbing, not intelligence.

Agents as Cattle, Not Pets

Kubernetes thinking applied to agents. Individual sessions are ephemeral. The work graph is persistent. The agent is not the session. The agent is a Bead -- an identity with a global address.

Nondeterministic Idempotence

Workflows can be interrupted and resumed at any time. The path is fully nondeterministic -- different runs take different routes. The workflow eventually finishes as long as you keep throwing agents at it.

Implementation Guide

Prerequisites Checklist

  • Go 1.21+ installed (go version)
  • tmux installed and proficient (tmux -V)
  • Multiple Claude accounts for parallel agents
  • Stage 7+ experience (100+ hours multi-agent)
  • High chaos tolerance (some bugs, some lost work)
  • Budget: $50-200/day

Installation

Step 1: Install Gas Town 
# Step 1: Install Gas Town
go install github.com/steveyegge/gastown/cmd/gt@latest

# Step 2: Verify installation
gt version

# Step 3: Initialize your town (creates ~/.gt)
gt init
Checkpoint

You should see Town initialized at ~/.gt

Adding Your First Rig

# Add a project rig
gt rig add /path/to/your/project

# Verify
gt rig list
Checkpoint

Your project should appear in the rig list.

Starting the Core Roles

Recommended startup sequence -- town-level first, then rig infrastructure, then named workers:

# 1. Start town-level roles first
gt mayor up
gt deacon up
gt dogs up

# 2. Start rig-level infrastructure
gt witness up my-project
gt refinery up my-project

# 3. Start named workers
gt polecat up my-project alice
gt polecat up my-project bob
gt polecat up my-project carol
Checkpoint

gt status should show all roles running.

Typical Daily Workflow

# Morning: Start town
./gastown-start.sh

# Review overnight status
gt status
gt inbox

# File new feature

# Refinery decomposes, assigns to polecats

Section 08 -- Troubleshooting

Troubleshooting

Context Collapse -- agents stop making sense, circular reasoning

Cause: Beads state corruption or no auto-compact.

gt town reset --soft   # Keep config, clear state
gt beads sync --force  # Force context resync
Race Conditions -- conflicting commits, overwritten work

Cause: Multiple Polecats editing same files. Use git worktrees for file isolation. Implement file ownership rules. Use the Refinery's merge queue.

gt refinery resolve my-project
Runaway Crew -- cost spike, dozens of Crew spawned unexpectedly

Cause: Refinery misconfiguration or task explosion.

gt crew purge --all
gt config set crew.max_per_rig 5
gt config set crew.idle_timeout 15
Mayor Deadlock -- no work getting assigned, town seems stuck

Cause: Mayor waiting for human decision. Check your inbox and respond to pending items.

gt inbox
gt inbox respond <id> --approve
Beads Desync -- agents have stale context

Cause: Git conflicts in Beads repo.

gt beads sync --force
# If that fails:
gt beads reset
Agent Won't Start -- gt polecat up returns error

Cause: Claude account issue, rate limiting, or config problem.

gt config list
gt config set polecat.alice.account secondary
claude --version

Recovery Command Reference

gt town reset --soft       # Keep config, clear state
gt town reset --hard       # Full reinstall
gt town down               # Kill all agents
gt polecat down <rig> <name>  # Restart specific role
gt beads sync --force      # Force Beads sync
gt inbox clear --all       # Clear inbox backlog
gt pause --all             # Emergency cost stop
gt resume --all            # Resume when ready
Section 09 -- Decision Guide

When to Use Gas Town

Use Gas Town When

You are at Stage 7+ (10+ agents, comfortable with chaos). Budget is not a concern ($50-200/day). You want team-scale output as a solo developer. You have clear product vision but need implementation velocity. You have tmux proficiency.

Do Not Use Gas Town When

You are below Stage 6. Budget is constrained. You need polish and stability. You need predictable, controlled outcomes. The project is simple enough for Ralph loops. You have not mastered Levels 0-6 on the complexity ladder.
