Section 01 -- The Architect

Who is Steve Yegge?

Steve Yegge is a 40-year software development veteran whose career arc -- from Amazon early employee to Google senior engineer to full-time AI development advocate -- gives him unique credibility for building Gas Town. He has vibe-coded roughly one million lines of code in 2025 alone, built Beads (225,000 lines of Go), and built Gas Town using Gas Town itself -- recursive validation at industrial scale.

The Vibe Coding Philosophy

Yegge didn't just adopt AI-assisted development -- he pioneered "vibe coding" as a philosophy. The principles are radical: describe what you want (the "vibe"), let AI generate the code, never read the code directly, judge by outcomes not implementation, and accept some bugs in exchange for throughput. This is not carelessness. This is a deliberate optimization for a world where code is cheap and ideas are expensive.

Section 02 -- The Factory Floor

The Gas Town Mental Model

Core Insight

"Claude Code is the world's biggest fuckin' ant. Everyone is focused on making their ant run longer... when nature prefers colonies."

Gas Town represents a fundamental shift in thinking about AI-assisted development. Instead of making one agent work longer (bigger context window, longer sessions), you build a factory of specialized agents that share the workload. The developer stops being the programmer and becomes the product manager.

Traditional Dev Gas Town
You write code You file features
You debug Agents debug
You're the implementer You're the PM
Single context window Many workers, shared memory
Session ends, context lost Workers persist, memory survives

The 8-Role Architecture

Gas Town organizes agents into 8 specialized roles across three levels: Town (global coordination), Rig (per-project operations), and Human (you, the overseer). Each role has a specific function, a recommended model, and clear responsibilities.

Town Level -- Chief
Mayor
Chief coordinator -- your concierge and chief-of-staff. Receives high-level directives, routes work to rigs, manages cross-rig coordination.
Model: Opus (complex coordination)
Town Level
Deacon
Health monitoring and agent coordination. Handles agent-to-agent handshakes, reports anomalies to Mayor, tracks work progress.
Model: Haiku (simple health checks)
Town Level
Dogs
Quality assurance and safety gates. Reviews work before acceptance, enforces standards, guards against regressions. Can reject and rollback.
Model: Sonnet (reasoning, not Opus-level)
Rig Level
Refinery
Task decomposition engine. Receives feature requests from Mayor, breaks them into atomic tasks, assigns to Polecats or Crew. Manages the merge queue.
Model: Opus (architecturally complex)
Rig Level
Witness
Per-rig observer. Watches all work in assigned rig, records history, provides state to new workers. Unsticks drifting agents.
Model: Haiku (observation and logging)
Rig Level
Polecat
Named, persistent agent workers with specializations. Execute assigned tasks, maintain context across sessions. Human names (alice, bob, carol) for mental model.
Model: Sonnet (main workhorse)
Rig Level
Crew
Short-lived, task-specific burst workers. No persistent state, cheap and disposable. Spawn, complete, disappear. Used for parallel bursts.
Model: Haiku (simple, isolated tasks)
Human Level
Overseer (You)
Product Manager and decision maker. Define features and priorities, review inbox decisions, approve major changes, provide human judgment. You have an identity in the system with your own inbox.
Model: Human brain (irreplaceable)
Tip

The naming convention matters: human names for Polecats (alice, bob, carol) help you track which persistent worker is doing what. Crew are unnamed and disposable -- numbered, not named.

Communication Flow

The Witness observes everything. Dogs run quality checks. The Deacon monitors health. It is a full production line, not a flat pool of agents.

Section 03 -- The Memory Layer

Beads: The Memory System

The "50 First Dates" Problem

Every new Claude session starts fresh. Agents suffer from "50 First Dates" syndrome -- they wake up with no memory of yesterday. This is the fundamental problem that prevents agent colonies from functioning. Without shared memory, each agent must reconstruct the entire project state from scratch, every time.

Challenge

Markdown plans fail at scale because they are not queryable, have no dependency tracking, are conflict-prone when multiple agents edit the same files, and create implicit task stacks with "intelligent wandering" instead of clear direction.

Beads solves this. It is a ~225,000 line Go project providing git-backed storage, universal context across all agents, hash-based IDs (like bd-a1b2) that prevent merge conflicts, explicit dependency tracking, and memory decay through compaction that summarizes old closed tasks.

Key Insight Why Git as Database?

Git as database provides durable, diffable, auditable storage with zero infrastructure. Everything is committed, so it survives crashes. Full history means a complete audit trail of all agent actions. Recovery is trivial -- roll back to any point in time. Multiple agents share the same store. No heavyweight message bus required.

Key Beads Commands

bd ready           # List tasks with zero blockers (what can be done NOW)
bd create "Title"  # Create new task
bd dep add X Y     # X is blocked by Y
bd show <id>       # Full task state and history

Memory Model Architecture

Layer Storage Contents
Hot Context RAM-cached Current task state, last N interactions, active file references
Warm Context SQLite + Git index Recent commit history, cross-agent message log, rig state snapshots
Cold Context Git history Full project history, previous run outcomes, archived decisions

Retrieval flows Hot -> Warm -> Cold, querying deeper layers only when needed.

Section 05 -- The Operating Principles

Core Principles

Gas Town is governed by a set of principles that emerge from the realities of managing AI agent colonies. These are not theoretical guidelines -- they are operational necessities discovered through the daily practice of running 10+ agents simultaneously on production code.

GUPP: The Universal Propulsion Principle

Essence

"If there is work on your hook, YOU MUST RUN IT."

GUPP solves the fundamental problem that "Claude Code ends." Work is "slung" to agent hooks (persistent git worktrees). When an agent starts, it checks its hook. If work exists, execution is immediate -- no confirmation, no waiting. Work survives crashes and restarts.

But Claude Code is "so miserably polite" that GUPP doesn't always work. Gas Town implements a "Gupp Nudge" -- agents get a startup poke with gt nudge roughly 30-60 seconds after starting.

Zero Framework Cognition

Claude makes all decisions. The orchestrator executes. No heuristics, regex, or parsing in the orchestrator. The framework provides infrastructure, not intelligence. This is a radical architectural choice: the framework is deliberately stupid so the AI can be smart.

Agents as Cattle, Not Pets

This is Kubernetes thinking applied to agents. Individual sessions are ephemeral. The work graph is persistent. The agent is not the session -- the agent is a Bead (an identity with a global address). Sessions come and go; the identity and its work history persist.

Nondeterministic Idempotence

Workflows can be interrupted and resumed at any time. The path is fully nondeterministic -- different runs take different routes. But the workflow eventually finishes as long as you keep throwing agents at it, because acceptance criteria are well-specified. Agents may make mistakes but self-correct through the quality gate system.

# Step 2: Verify installation

# Step 3: Initialize your town (creates ~/.gt)

The recommended startup sequence mirrors the role hierarchy -- town-level first, then rig-level, then workers:

# 1. Start town-level roles first
gt mayor up
gt deacon up
gt dogs up

# 2. Start rig-level infrastructure

# 3. Start named workers

Section 07 -- Troubleshooting

Common Failures and Recovery

Gas Town is honest about its failure modes. Some bugs get fixed 2-3 times. Some fixes get lost. The focus is throughput, not perfection. Here are the most common failures and how to recover.

Context Collapse
Agents stop making sense
Cause: Beads state corruption or no auto-compact. Recovery: gt town reset --soft then gt beads sync --force
Merge Conflicts
Conflicting commits
Cause: Multiple Polecats editing same files. Recovery: Use git worktrees for isolation and the Refinery's merge queue.
Runaway Crew
Cost spike, dozens spawned
Cause: Refinery misconfiguration or task explosion. Recovery: gt crew purge --all then set crew.max_per_rig 5
Mayor Deadlock
No work getting assigned
Cause: Mayor waiting for human decision. Recovery: Check inbox and approve/reject pending items.
Beads Desync
Stale context everywhere
Cause: Git conflicts in Beads repo. Recovery: gt beads sync --force, or if that fails, gt beads reset
Agent Won't Start
gt polecat up returns error
Cause: Claude account issue, rate limiting, or config problem. Recovery: Check config, try different account.

Emergency Recovery Commands

gt town reset --soft   # Keep config, clear state
gt town reset --hard   # Full reinstall
gt town down            # Kill all agents
gt pause --all          # Emergency cost stop
gt resume --all         # Resume when ready

Section 08 -- The Decision

When to Use Gas Town

Use Gas Town When
  • You're at Stage 7+ (10+ agents, comfortable with chaos)
  • Budget is not a concern ($50-200/day)
  • You want team-scale output as a solo developer
  • You have clear product vision but need implementation velocity
  • You're comfortable with imperfection (some bugs fixed 2-3 times)
  • You have tmux proficiency
Don't Use Gas Town When
  • You're below Stage 6 on the dev evolution scale
  • Budget is constrained
  • You need polish and stability over throughput
  • You need predictable, controlled outcomes
  • The project is simple enough for Ralph loops

Decision Talk to the Plan, Not the Agent

Because agents have limited context windows and experience "amnesia," the operating principle is to maintain persistent, readable plans in files. Both you and agents reference these plans. At session end, agents "Land the Plane" -- clean up git, update issues, and produce a prompt for the next session. The plan persists; the agent is transient.

This is the deepest insight in Gas Town's architecture: the durable unit is not the agent, not the session, not even the code. It is the plan. Everything else is execution infrastructure around the plan.
