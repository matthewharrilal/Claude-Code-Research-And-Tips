What is this? A vocabulary catalog of every designed visual element across the design system's 18 HTML explorations. Elements are classified into three categories: Instruments (reader orientation devices), Gestures (CSS-encoded reading register shifts), and Components (structured content containers). Each element was independently cataloged, buddy-reviewed for accuracy, and retro-synthesized for cross-file patterns. Use the sidebar filters to browse by category or exploration family.

Classification Decision Tree

1. Does it exist in the DOM? NO → PHANTOM. Exclude. 2. Is it standard infrastructure (::selection, :focus-visible, print, reduced-motion)? YES → INFRASTRUCTURE. Exclude. 3. Is it a CSS-encoded shift across 2+ properties that changes reading register? YES → Is it perceptible to an unaware reader? YES → GESTURE. Multiple sub-channels encoding ONE gradient? → Catalog as ONE gesture. NO → Below perceptual threshold. Exclude or note as micro-gesture. 4. Does it have defined content slots and deliver article content? YES → COMPONENT. Zero slots but dedicated class + visible? → Component [structural marker]. 5. Does removing it degrade reader's awareness of position, identity, or reading mode? YES → INSTRUMENT. Sub-element of larger I/C? → Does it pass Evidence DNA test? YES → Independent Instrument. NO → Merge into parent. NO → Not designed vocabulary. Exclude.

Meta-Cognitive Analysis

What the 7-page comparison reveals about the pipeline's fundamental gap — and why the fix is compositional and structural variety, not more mechanisms or better texture.

1. The Gap is Invention, Not Volume

Pipeline CSS volume MATCHES explorations (1,386 avg vs 1,077-1,393). Pipeline custom properties match (45-46 vs 52-55). Pipeline class counts are close (94-111 vs 107-149). The numbers say the pipeline is doing comparable work.

But explorations invent 5-9 novel CSS techniques each (scroll witness, fractal annotations, editorial drops, pattern echo grids, diagonal clip-paths, solid offset depth, territory grids, spiral layouts, hub-spoke grids). Pipeline builds invent 0.

The mechanism catalog acts as a ceiling — what's in the catalog gets deployed; what's not there never appears. The pipeline is a DEPLOYMENT engine for known techniques, not a DISCOVERY engine for new ones.

2. Category 3 Questions Don't Exist in the Pipeline

Explorations have 6 question categories. Categories 1-3 are exploration-only. The pipeline has Categories 4 (experimental) and 5 (opposition-derived). Category 3 — Phase 1 Multi-Axis Structural questions — is entirely missing.

This matters because Category 3 questions generate STRUCTURE:

  • "Does content have natural confidence gradients?" → 4 strata (OD-004)
  • "Can this content use one organizational pattern, or must it use multiple?" → 6 sections (OD-006)
  • "Does this content STATE a thesis or PROVE a thesis?" → Self-referential layout (OD-006)

Categories 4-5 generate TEXTURE (warm factory, voice eruptions, dark terra incognita). The pipeline has rich texture but thin structure because the structural questions were never asked.

3. The Specification-Question Gap

The pipeline provides: metaphor label + mechanism list + CSS specifications.

The explorations provided: metaphor question that IMPLIES the mechanism list.

The gap is the difference between telling the builder what to do and asking the builder a question whose answer naturally produces the right layout. When the pipeline says "deploy 4px/3px/1px border-weight gradient per Package S2-M02," the builder executes. When OD-004's question says "What if documentation was organized by confidence level?", the builder DISCOVERS that borders should encode certainty.

The specification produces 1 channel (border weight). The question produces 9 channels (border weight + color + background + padding + max-width + h3 size + line-height + font-size + content density) because the RULE requires all channels to participate.

4. Three Question Types Map to Different Pipeline Capabilities

Type A: Metaphor-as-Rule — "What if the page was a [physical metaphor]?" The metaphor implies a gradient with multiple dimensions. The pipeline can carry these (TC briefs already produce them). The gap: the pipeline ALSO sends a mechanism list that overrides the metaphor's implications.

Type B: Constraint-as-Generator — "What if the page couldn't [do X]?" The constraint forces structural variety. The pipeline CANNOT carry these because it has no mechanism for structural constraints — it only has mechanisms for deployment instructions.

Type C: Experience-as-Architecture — "What if reading felt like [experience]?" The experiential metaphor implies temporal rhythm and emotional arc. The pipeline partially carries these through the TC brief's opposition map, but they get compressed to deployment specs by the synthesizer.

5. I/G/C Survival is Selective: Containers Survive, Wayfinding Dies

The pipeline survival gradient reveals what the pipeline values vs. what explorations value:

  • Universal elements (10): 70-80% survival — backgrounds, borders, fonts, code blocks
  • Named instruments (6): 0-8% survival — scroll witnesses, mode signals, fractal annotations
  • Named gestures (4): 12-25% survival — drop caps, tension meters, register shifts
  • Named components (3): 0-17% survival — Q&A pairs, strata, dual lenses

The pipeline preserves what holds content (containers) but loses what creates the reading EXPERIENCE (instruments, gestures). A page without instruments is a page without wayfinding — the reader navigates blind.

6. The Fundamental Contrast: Compositional Flatness

Across all 7 pages, the explorations share one property that pipeline builds lack: each section is structurally different from the next. OD-006 uses 6 different organizational modes. AD-006 uses 7 different grid systems. AD-002 uses 5 acts with different scan patterns. Even simpler explorations like AD-005 alternate between hub grids and spoke deep-dives.

Pipeline builds use the SAME structural approach for every section: heading → prose → callout → code block → heading → prose. The zones have different backgrounds and border weights, but the internal STRUCTURE is identical. This is compositional flatness.

The fix is not more mechanisms or better texture. It is structural questions that generate compositional variety — questions that make each section's FORM respond to its CONTENT, rather than applying a uniform template with varying surface properties.

7. The Mediation Pattern: Knowledge Without Discovery

Explorations cite 15-40 external sources directly (NNG, Stripe, Gwern, Freytag, Kevin Lynch, Hofstadter). Pipeline builds cite 0 — all influence is mediated through the compositional core.

This mediation is a deliberate design choice: the pipeline standardizes quality by distilling external knowledge into a reusable vocabulary. But it creates a ceiling: the pipeline builder can only deploy what's already in the vocabulary. OD-006 invented the fractal annotation layer because it read about Hofstadterian strange loops. No pipeline builder has access to that conceptual source.

The question isn't whether to give pipeline builders live research access (probably not practical). The question is: how do you make the QUESTIONS rich enough to imply structural invention without requiring the builder to have read Hofstadter?

8. What "Naturally Derive" Means

The user's core insight: the pipeline should naturally derive compositional variety rather than being told to produce it. What does this look like concretely?

Currently: TC produces metaphor → Synthesizer compresses to mechanism list → Builder deploys mechanisms → Every section gets the same structure with different surface properties.

Ideal: TC produces metaphor + structural constraint → Synthesizer preserves the constraint as a QUESTION → Builder reads the question and DERIVES a different structure per section because the question requires it.

The key change: instead of telling the builder "deploy CRESCENDO typography: zone--overseer h2 at 2rem, zone--operations at 1.625rem," ask "What if EVERY visual channel compressed as you descend? Typography shrinks. Spacing tightens. Line-height compresses. Letter-spacing neutralizes. Test: cover the zone labels — can you FEEL which floor you're on from visual weight alone?"

The first produces 1 channel of gradient. The second produces 5-6 channels because the question implies ALL channels participate. The pipeline doesn't need more mechanisms — it needs questions whose answers are naturally multi-channel.

9. The Three Levels That Need Change

TC Level: Needs structural discovery questions, not just deployment questions. "Does this content have sections that demand different reading modes?" (Category 3) should be asked alongside "What metaphor fits?" (Category 1). The TC agent currently derives a metaphor and an opposition map — it should ALSO derive a structural variety mandate.

Synthesizer Level: Needs to preserve questions AS QUESTIONS, not compress them to mechanism lists. Section 10 (Compositional Questions) in the v1.1 plan addresses this — but it must carry Type B constraints ("can't use the same density mode twice") not just Type A metaphors.

Builder Level: Needs permission to invent beyond the catalog. The current prompt says "sample 2-4 mechanisms." The fix: "the catalog is a vocabulary, not a menu. The compositional question may require you to invent elements the catalog doesn't contain."

10. The Compositional Question Quality Test

A compositional question is generative when it satisfies ALL three criteria:

  • Implies a rule, not a property. "Use 4px borders" is a specification. "Border thickness encodes certainty" is a rule. Rules require multi-channel coordination.
  • Has entailments beyond the obvious. "Geological core sample" implies layers (obvious). It ALSO implies boundary markers, compression, fractal repetition, and evidence types (not obvious). The indirect entailments are where compositional richness lives.
  • Constrains without specifying. "Each section must be its own proof" constrains radically but specifies nothing. This is the opposite of a mechanism list.

If a question names specific CSS properties or mechanism names, it's a specification disguised as a question — rewrite it. The pipeline needs questions, not specifications that look like questions.
