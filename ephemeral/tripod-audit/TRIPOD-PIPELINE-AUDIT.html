S1: Findings Mapper — 24-25 HIGH findings per run

S1 reads 5 research streams (~337 findings) and curates the top 24-25 for this specific content. Each finding gets a content-specific CSS action. Both runs show genuine content-responsive curation — different findings elevated for different content. Molly promotes cartographic vocabulary (geological model, asymmetric spacing), Yegge promotes industrial vocabulary (role-card hierarchy, code-block differentiation).

Transfer chain: 100% of S1's HIGH findings appear in synthesized packages. 20/24 (Molly) and 24/25 (Yegge) deployed to HTML. Traceability gap: builders cite PACKAGE IDs not original S1 finding IDs, making audit harder.

Weakness: Builder-side citation inconsistency — Yegge has 60+ CITE comments, Molly has 3.

S2: Validation Mapper — Confidence ranking is the value

S2 traces provenance chains through the design system's validated explorations, producing confidence-ranked build applications. The value is not the mechanisms themselves but their RANKING — which chains are ELEVATED (direct validation), BOUNDED (conditional), or REINFORCED (convergent evidence).

Asymmetric propagation: Yegge's builder produced 60+ CSS comments citing S2 chains. Molly's builder produced 3. Same specialist quality, dramatically different builder uptake. This is a prompt enforcement gap, not a specialist gap.

S3: Case Study Mapper — Strongest overall chain (8.5/10)

S3 reads case study files (OD-004, DD-004, CD-001, DD-003, CD-006, OD-006) and extracts PROCESS intelligence — not what they built, but how they thought. Both runs demonstrate genuine structural reasoning and follow the anti-prescription mandate (extracting process, not copying CSS).

4-5 of 5 case study CSS patterns traceable to final HTML in both runs. Example: OD-004's border-weight-as-certainty pattern appears as .role-card--town (4px) / .role-card--rig (3px) / .role-card--worker (1px) in Yegge.

S4: Constraint Mapper — Zero soul violations + custom anti-patterns

S4 maps content-specific constraints: soul checklist, conventions with reasoning, risk-profiled anti-patterns, and creative territory. Both runs produced custom anti-patterns not in the registry — Yegge's "Uniform Role Cards" (derived from the 8-role section) is the standout.

HTML result: Zero border-radius violations, zero box-shadow, zero rgba(), zero gradients across all builds. The callout cacophony mitigation is explicitly documented in HTML comments.

Bug found: Yegge builder deployed 10 voice-eruptions when S4 budgeted 3-4. Different component name circumvented the budget.

S5: Protocol Analyst — The pipeline's structural backbone

S5 defines HOW the package is organized and HOW the builder consumes it. It's the only specialist that produces testable structural propositions. 10/10 propositions verified in CSS across both runs — every single one has HTML evidence.

Strongest trace: Molly's .zone-annotations .survey-container { padding-left: 80px } — the asymmetric margin IS S5's "written in the margin" effect. The 80px left padding creates the spatial annotation metaphor S5 prescribed.

Cross-run differentiation: Zero shared propositions between runs. Molly's propositions involve survey deepening and map-running-out. Yegge's involve tower descent and authority hierarchy. The protocol extracts genuinely different structural hypotheses from different metaphors.

Both runs demonstrate genuine synthesis, not concatenation. Multiple specialist findings are collapsed into single unified mechanisms with cross-referencing. Conflict resolution follows the correct hierarchy (TC brief > validated > specific > generic).

Content Specificity: 10/10

The swap test — could you give Molly's packages to Yegge's builder? No. They would be unusable. Every class name (.zone--cartouche vs .zone--overseer), every custom property (--zone-parchment vs --tower-surface), every anti-pattern risk profile is saturated with content-specific language.

Concerns

Identical structural counts: Both runs produce exactly 8 mechanisms, 3 case studies, 5 propositions — suggesting template-following rather than content-driven adaptation. The prompt says "5-8 mechanisms" and "2-4 case studies" but the synthesizer lands at the same number every time.

Line budget overrun: Both runs exceed the 600-line ceiling (739 and 788 lines). Consistent 23-32% overshoot.

III. 3-Pass Build Evolution

The 3-pass rotation is NOT redundant. Each pass has a genuinely distinct cognitive mode:

PassModeEvidence
Pass 1GENERATIVEBuild from nothing. 149-169 selectors, 8 mechanisms deployed, zone architecture created
Pass 2AUGMENTATIVE+20% CSS, 12 new classes (Molly), progressive disclosure (0→17 <details>)
Pass 3EVALUATIVEAnti-pattern fixes, accessibility (skip links, ARIA), dark zone contrast fix (51→199 RGB pts)

Would a single pass produce equivalent quality? No. The information budget argument is decisive: Pass 1 consumes Sections 0/1/2/9, Pass 2 consumes 3/4/5, Pass 3 consumes 6/7/8. A single builder consuming all 10 sections would exceed effective context utilization.

Continuity regression: Molly lost 7 <details> elements and the transition table during the fix cycle. The fix cycle builder should have a continuity checklist.

Fresh-eyes perception is genuine — zero pipeline vocabulary contamination across 10 auditors. Language is strongly experiential ("like being handed a classified document in a dimly lit room," "I hear a warm acoustic bass"). Convergence identifies real patterns verified against screenshots.

Calibration Concern

10/10 auditors across both runs said REFINE. While justified, this unanimity raises the question: is the question battery biased toward finding weaknesses? Questions like E-11 ("fighting content") and E-19 ("almost working") explicitly probe for problems. A well-built page should occasionally produce at least one SHIP from a fresh-eyes auditor.

Weaver override: Both Weavers moved from 5/5 REFINE to SHIP after fix cycle 1, with sound cost-benefit reasoning. But the override is not explicitly flagged — the Weaver should state "I am overriding 5/5 auditor REFINE."

VI. The Missing Layer — Compositional Questions

The audit's most significant discovery.

The original validated explorations (OD-004, CD-001, DD-006, etc.) were driven by compositional questions — "what if..." interrogatives about layout structure. 38 such questions were extracted from the knowledge base. These questions don't just inspire layouts — they dictate them. Each question constrains the solution space so tightly that the visual answer is nearly deterministic.

The Gap

Original explorations:
Content tensions → COMPOSITIONAL QUESTION → Structural constraints → Mechanism selection → CSS

Current pipeline:
Content tensions → Metaphor → ??? (missing) → Mechanism selection → CSS

Question propagation rate: ~10%. TC brief questions die at the specialist boundary. The pipeline gives builders the ANSWERS without the QUESTIONS — producing execution mode rather than derivation mode.

The research phase DOES bring questions into HTML through an indirect mechanism: the metaphor generates the zone system, which generates gradients. But approximately 30-40% of the explorations' question-driven approach is lost — specifically the meta-layer (self-aware visualizations, explicit hypothesis statements) and the generative question as a reader-facing content element.

Root Causes

  1. Prompts designed for knowledge compression, not question transfer. Questions are "overhead" in a compression frame.
  2. No question slot in any output format. Finding → CSS Action has no field for "what if..."
  3. TC brief questions die at the specialist boundary. Specialists use the brief as a lens, not a source of questions to carry forward.

IX. Verdict

THE PIPELINE WORKS — WITH ONE STRUCTURAL GAP

The /research-compose pipeline successfully orchestrates 23 Opus agents across 6 phases to produce fully designed HTML pages from content markdown. Knowledge chains are strong (all 5 specialists achieve DEEP absorption), the synthesizer produces genuine merges with 10/10 content specificity, the 3-pass rotation creates measurable progressive improvement, and the PA evaluation identifies real perceptual patterns.

The structural gap is the missing compositional questions layer. The original explorations that created the knowledge base were driven by "what if..." questions about layout. These questions generated the mechanisms, the case studies, the validated patterns. But the pipeline strips the questions during knowledge compression and delivers only the answers. The builder operates in execution mode (apply these 8 mechanisms) rather than derivation mode (explore these questions and discover your layout). This is the primary difference between the pipeline's "mechanistically correct but compositionally safe" output and the explorations' "question-driven, structurally imaginative" output.

Despite this gap, the research phase produces measurably more question-driven CSS (~50%) than the non-research pipeline (~32%). The metaphor-question fusion — where the metaphor generates the zone system which generates the gradients — is the mechanism through which questions indirectly reach the HTML. Adding an explicit question propagation layer would close the remaining 30-40% gap.
