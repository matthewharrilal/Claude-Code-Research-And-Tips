You Are Here: This diagram shows the complete Pipeline v2 architecture after resolving all 3 BLOCKING contradictions and 8 design decisions from the 17-agent validation audit. It shows how /build-page orchestrates the full journey from raw content to a verified HTML page, how it invokes the tension-composition and perceptual-auditing skills, and how verification works at every layer.

The One-Sentence Summary

A user says /build-page. An orchestrator classifies the tier, invokes tension-composition for content analysis, hands a recipe + build plan to an Opus builder, runs programmatic gates during the build, then launches a full perceptual audit with Playwright screenshots and fresh-eyes auditors. Two verification layers — numbers AND eyes — ensure the page is genuinely designed, not just technically compliant.

The Complete User Journey

What happens from the moment a user types /build-page to the moment they receive a verified HTML artifact.

User Journey — Sequential Flow
Step 1 — User Input
/build-page [content-source] [tier?]
User provides a content source (markdown file, research doc, etc.) and optionally specifies a tier. If no tier specified, the orchestrator auto-classifies based on content complexity (word count, element types, strategic importance).
Step 2 — Tier Classification
Orchestrator classifies and configures
Middle (default, 40-50% of pages): 1 Opus builder, 2-PA quick audit, self-checkpoints
Ceiling (20-30%): 1 Opus builder + planner, 4-PA standard audit, checkpoint approvals
Flagship (5-10%): 1 Opus builder + planner, 9-PA Mode 4 audit, mandatory approvals at 5 phases
Step 3 — Content Analysis (invokes /tension-composition)
TC Pipeline: Phases 0 → 3.5
TC reads the content source. Runs multi-axis questioning (FEEL / UNDERSTAND / DO / BECOME + discovered axes). Derives tensions. Collapses to metaphor (Ceiling+ only). Generates zone architecture, mechanism deployment table, section inventory, transition plan, isomorphism table (Ceiling+), and bridge prose. Outputs: _build-plan.md — the handoff document.
Gate — Handoff Validation
Orchestrator validates _build-plan.md
Binary checks: All required fields present? Zone count within 2-5? Adjacent backgrounds differ by >= 15 RGB? At least 2 grid layouts assigned? Per-category mechanism minimums met (1+ Spatial, 1+ Temporal, 1+ Material, 1+ Behavioral, 1+ Relational)? If FAIL: TC re-runs with specific fix instructions.
Step 4 — Builder Receives Package
Opus builder reads recipe + build plan + design system files
Builder receives: operational recipe (~650 lines), _build-plan.md (from TC), tokens.css, prohibitions.md, merged-components.css. For Ceiling+: also mechanism-catalog.md excerpts and isomorphism table. Builder reads the conviction preamble first (anti-scale model, IS/IS NOT axes, "What DESIGNED means"), then executes Phases 0-8 sequentially.
Step 5 — Build Execution (Phases 0-8)
Builder executes 9-phase recipe
Phase 0: Content analysis + zone architecture
Phase 1: HTML skeleton with semantic markup
Phase 2: CSS reset + soul enforcement + base typography
Phase 3: Zone backgrounds + warm palette
Phase 4: Structural borders + dividers + components
Phase 5: Typography zones
Phase 6: Element-level richness
Phase 7: Accessibility + responsive
Phase 8: Verification (programmatic + self-check)
Layer 1 — Programmatic Gates (during + after build)
12 binary checks with measured values
Runs inline during build AND as final verification. Uses getComputedStyle via Playwright to measure ACTUAL rendered values — not CSS source, but what the browser computes. Every gate reports: property, measured value, threshold, PASS/FAIL. Zero judgment. If FAIL: builder fixes and re-checks before proceeding.
Layer 2 — Perceptual Audit (invokes /perceptual-auditing)
Full PA with Playwright screenshots + fresh-eyes auditors
This is the heavy verification. Screenshots captured at 1440px, 768px, 480px. Animations disabled. Scroll-through at each viewport. For Flagship: 9 independent Opus auditors, each assigned specific PA questions, analyzing what they actually SEE in the screenshots. All 48 PA questions answered. Weaver synthesizes into verdict with PA-05 score. This catches what numbers cannot: "does this FEEL designed?"
Step 6 — Fix Integration (if needed)
Builder applies PA-identified fixes
If PA-05 < tier threshold (Middle: 3/4, Flagship: 3/4 with aspiration to 4/4): builder receives specific fix list from PA weaver. Applies fixes. Re-runs both verification layers. Maximum 3 fix cycles before escalation to user. Each cycle targets the TOP 3 blocking issues only.
Step 7 — Ship Decision
SHIP / FIX / ESCALATE
SHIP: PA-05 >= 3/4 AND all programmatic gates PASS AND 0 soul violations
FIX: PA-05 < threshold OR any gate FAIL (maximum 3 cycles)
ESCALATE: 3 fix cycles exhausted without reaching threshold — user decides

Key Insight — Skill Separation

Each skill has a clear boundary. TC analyzes content and outputs a plan. The recipe tells the builder exactly what CSS to write. PA judges whether the result looks designed. No skill does another's job. The orchestrator connects them but never builds, analyzes, or judges itself.

Two-Layer Verification — Numbers AND Eyes

This is the core architectural principle: programmatic gates catch threshold violations (is background delta >= 15 RGB?), but ONLY perceptual auditing catches gestalt failures (does this FEEL designed?). Both layers are essential. Neither is sufficient alone.

Layer 1: Programmatic Gates

When: DURING build (per-phase) + AFTER build (final check)
How: Playwright getComputedStyle on rendered page
Speed: ~30 seconds for all 12 gates

  • SC-01: Container 940-960px
  • SC-02: border-radius: 0, box-shadow: none
  • SC-03: Font trinity only
  • SC-04: Warm palette (R >= G >= B)
  • SC-05: No pure #000000 or #FFFFFF
  • SC-06: ARIA landmarks >= 3
  • SC-07: Skip-link present
  • SC-08: Component library classes
  • SC-09: Background delta >= 15 RGB
  • SC-10: Stacked gap <= 108px (Flagship)
  • SC-11: Font-size zones >= 2px delta
  • SC-12: Zone count 2-5

What it catches: Threshold violations, soul breaches, structural compliance
What it misses: Gestalt quality, spatial confidence, "does it feel designed?"

Layer 2: Perceptual Audit

When: AFTER build completes
How: Playwright screenshots + fresh-eyes Opus auditors
Time: ~15-45 min depending on tier

  • Screenshots at 1440px, 768px, 480px
  • Animations disabled (opacity: 1, animation: none)
  • Full scroll-through captured
  • Auditors SEE screenshots (not code)
  • 48 PA questions across 5 dimensions
  • PA-05: "Does this feel DESIGNED?"
  • Spatial confidence assessment
  • Zone differentiation by eye
  • Component variety evaluation
  • Screenful diversity check
  • Metaphor structural presence (Ceiling+)
  • Fresh-eyes with ZERO build context

What it catches: Gestalt failures, whitespace voids, monotony, "technically correct but lifeless"
What it misses: Nothing the numbers catch — it sees everything, just slower

Why Both Layers Are Non-Negotiable

The flagship experiment PASSED all programmatic gates. Gate 4 confirmed 6 CSS channels with 4+ shifts per boundary. Then fresh-eyes PA found the shifts were 1-8 RGB points apart — imperceptible. The page was technically rich and visually flat. Programmatic gates check MAGNITUDE. Perceptual audits check PERCEPTION. A page can have the right numbers and still look wrong.

Flagship 4-Wave Architecture

Flagship builds use a 4-wave agent architecture. Each wave completes before the next begins. This prevents the "builder can't ask planner" failure that caused the missing footer in the flagship experiment.

Wave 0 — Content Analysis

  • TC planner (Opus) runs Phases 0-3.5
  • Outputs _build-plan.md handoff
  • Orchestrator validates handoff
  • Duration: ~30-60 min

Wave 1 — Build Execution

  • 1 Opus builder executes Phases 0-8
  • Mandatory checkpoints at 5 phases
  • Planner reviews + approves each
  • Programmatic gates inline
  • Duration: ~90-150 min

Wave 2 — Verification

  • Lead captures all screenshots
  • 9 Opus PA auditors (parallel)
  • All 48 PA questions answered
  • Weaver synthesizes verdict
  • Duration: ~30-45 min

Wave 3 — Fix Integration

  • Builder applies top 3 blocking fixes
  • Re-runs programmatic gates
  • Quick PA re-check (2 auditors)
  • Ship decision
  • Duration: ~30-60 min
Screenshot Pre-Capture Pattern

The team lead captures ALL screenshots BEFORE spawning PA auditors. Auditors read saved images via the Read tool. This eliminates Playwright contention, enables 9+ parallel auditors, and ensures every auditor sees the same page state. Proven in Mode 4 PA: 102 screenshots, 9 agents parallel, zero contention.

Critical Design Decision

PA enrichment is the HIGHEST leverage enrichment point. When PA learns to see something new (e.g., "sub-perceptual micro-typography"), every future build gets checked for it. The recipe doesn't need to change — the verification catches what the recipe misses. This is why the heaviest PA invocation matters: more eyes, more questions, more dimensions = more issues caught = more enrichment opportunities.

Resolved Decisions (My Judgment)

You left these to my judgment. Here's what I decided and why.

Decision 1 — Background Threshold

15 RGB. All template colors already satisfy this. The recipe's own examples pass. 10 RGB is imperceptible without a color picker — the exact failure mode we diagnosed. Updating File 42.

Decision 2 — Stacked Gap

108px for Flagship, 120px for Middle/Ceiling. File 42's arithmetic proves worst-case breathing = 108px. Any gap exceeding this means the builder used non-recipe values. 192px gate was too loose — removed.

Decision 3 — Anti-Scale Model

Goes in recipe Phase 8G AND /build-page preamble. The formula Richness = semantic_density x restraint x spatial_confidence correctly predicted the ceiling failure. It belongs in Phase 8G as a verification lens: "If spatial_confidence = 0, the entire product = 0."

Decision 4 — Conviction Content

20-line Pipeline Philosophy in /build-page preamble. Contains: anti-scale model, IS/IS NOT axes (Warm not Soft, Austere not Cold, Authoritative not Aggressive, Restrained not Empty), "What DESIGNED means" (4 points), kill criteria. Separated from routing logic.

Decision 5 — Kill Criteria

Added to /build-page orchestrator. Three criteria: (1) 3 consecutive PA-05 < 3/4 = recipe failure, investigate root cause. (2) Any soul violation = structural leak, fix immediately. (3) All gates pass but PA-05 < 3/4 = recipe captures minimum, not quality — escalate to TC for metaphor-driven composition.

Decision 6 — Recipe Length

Extract to ~650 lines. Preserving: border alternation pattern (40-60% rule + specific selectors), "adapt selectors to YOUR section count" notes, stacking arithmetic proof, and all perception check details. Conviction layer moves to /build-page preamble. Analysis sections move to reference file.

Decision 7 — Letter-Spacing Threshold

0.03em (0.5px at 16px base). Resolves the 0.025em vs 0.5px discrepancy. 0.03em is more conservative and aligns with the architecture's research-derived value. Recipe updated.

Are You Missing Anything?

No. Your two instincts are exactly right:

Instinct 1: /build-page as orchestrator is correct

The form-definer argued against it ("the USER is the orchestrator"), but after 17 agents validated the architecture, the thin orchestrator pattern is the right call. It's ~350 lines of routing, not a complexity ratchet. It handles tier classification, skill invocation, and the fix loop — things the user shouldn't have to manage manually every time.

Instinct 2: Heavy PA is non-negotiable

The propagation-auditor recommended programmatic verification to replace self-assessment perception checks. That's correct for the BUILDER's inline checks. But you correctly identified that this must NOT replace actual perceptual auditing. The architecture is both: programmatic gates DURING build (fast, catches thresholds) + full PA AFTER build (heavy, catches gestalt). Especially with future PA enrichments — more questions, more dimensions, deeper analysis — the perceptual layer is where quality improvements compound.

One thing to keep in mind

The paradigm-auditor's "compound microscope" framing sets the right expectation: PV2 reliably produces 3/4 COMPOSED pages from arbitrary content. It probably cannot reach CD-006's 39/40 emergent brilliance, because pre-computed values cap quality at ~3.5/4. The telescope's rare genius requires creative freedom the microscope constrains by design. If you ever want to chase 4/4 DESIGNED, the recipe is the floor — TC's metaphor-driven composition is the ceiling. The pipeline supports both: recipe gets you to 3/4, then TC + Opus creative judgment pushes toward 4/4 for Flagship pages.
