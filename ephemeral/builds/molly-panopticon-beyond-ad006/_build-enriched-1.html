Part I — The Philosophy of Self-Legibility

The Apparatus of Sight

Pre-modern states were essentially blind to their citizens. They couldn't see population distributions, land ownership, economic activity, or individual behavior in any systematic way. This blindness limited their power.

To overcome this, states built what Cantillon calls the apparatus of sight: censuses that counted people, surname systems that tracked lineages, cadastral maps that documented property, and registration systems that monitored movement. These tools made populations legible to power.

The modern twist? Corporations have perfected this apparatus far beyond what states ever achieved. Every digital interaction generates behavioral data. Every purchase, click, scroll, pause, and abandonment creates a signal.

The Asymmetry
What They See
  • Population distributions
  • Behavioral patterns
  • Purchase histories
  • Emotional states
  • Productivity models
  • Predictive profiles

Companies like Google, Meta, and Amazon have assembled portraits of individuals more detailed than any census taker could dream of.

What You See

No dashboard of your own patterns.

No alert system for behavioral drift.

No synthesis of scattered data.

The Asymmetry Is Total

They can see you. You cannot see yourself the same way they do. You have no dashboard of your own patterns, no alert system for your own behavioral drift, no synthesis of your own scattered data streams.

The Reversal

What Self-Legibility Actually Means

Self-legibility is the act of building your own surveillance infrastructure—not to control yourself punitively, but to see yourself with the same clarity that institutions achieve. It's reversing the panopticon: instead of an external observer watching you from a tower, you ARE the observer. You build your own watchtower over your own life.

When a bank knows your spending patterns better than you do, when an advertiser can predict your emotional states better than you can articulate them, when an employer can model your productivity patterns more accurately than you perceive them—you're at a structural disadvantage.

Cantillon's response is radical: build the same apparatus for yourself.

Domain Design Principles

Principle 01
Clear Boundaries

A domain should have clear entry and exit criteria. If you frequently wonder "which domain handles this?", the taxonomy is wrong.

Principle 02
Different Optimization Functions

Each domain optimizes for something different. These objectives can conflict. Isolation prevents one domain's optimization from corrupting another's.

Principle 03
Different Time Horizons

~/trades operates daily. ~/health operates monthly. ~/nox operates on product cycles. Matching the domain to its natural rhythm matters.

Principle 04
Different Data Sources

Each domain has its own feeds. Brokerage data for trades, WHOOP for health, Amplitude for metrics. No shared input streams.

Part III — How Isolation Works

Filesystem as Memory

Each Claude instance operates in its own directory. The directory IS the memory. Files persist between sessions. The agent can read what it wrote yesterday. This is fundamentally different from conversation-based memory. Conversations are ephemeral. Files are durable.

Part IV — Implementation Details

Running 8 Parallel Instances

The basic mechanism is simple: 8 terminal sessions, each running Claude Code in its corresponding directory. For management, tmux provides the natural interface.

launch-panopticon.sh
#!/bin/bash DOMAINS="nox metrics email growth trades health writing personal" for domain in $DOMAINS; do cd ~/$domain tmux new-session -d -s $domain "claude" done echo "All domains launched."

Automation Layer

Beyond manual interaction, automation drives scheduled operations. Cron orchestrates overnight data pulls, morning briefs, and cross-domain synthesis.

crontab -e
# Keep system awake during long runs caffeinate -i # Morning synthesis (6 AM) 0 6 * * * cd ~/trades && claude "Generate morning brief" 5 6 * * * cd ~/health && claude "Analyze overnight recovery" 10 6 * * * cd ~/correlation && claude "Synthesize daily brief"

Domain Configuration

Each domain gets its own CLAUDE.md file with tailored rules. The configuration IS the identity of each domain.

~/trades ~/health ~/email
# Domain: Personal Finance & Trading ## Data Sources - Brokerage exports in ./data/ - Congressional disclosures from House API - Polymarket odds via MCP - X sentiment from firehose ## Risk Rules - NEVER execute actual trades - Flag any position >5% of portfolio - Always note data freshness
# Domain: Fitness & Wellness ## Data Sources - WHOOP exports in ./data/whoop/ - Apple Health via shortcuts - Workout logs in ./workouts/ ## Rules - Monthly/quarterly optimization cycles - Flag recovery score < 33% - Never override subjective feeling with metrics
# Domain: Communications ## Data Sources - Mail server via IMAP - Contact metadata in ./contacts/ ## Rules - Auto-draft replies, NEVER auto-send - Priority triage: urgent / today / this week - Flag threads with no response > 48h
Implementation Note

The implementation is deliberately simple: directories, cron, tmux, bash scripts. No custom infrastructure. No complex orchestration layer. Simple systems are debuggable. When ~/trades stops working, you can cd ~/trades and investigate.

Part VII — The Technical Critique

The Sharpest Objection

@flutterwhat Technical Critic

"Swarm of eight instances in parallel" is a cute way to say "I made eight folders and accidentally reinvented ~/bin plus crontab -e with worse failure modes and telemetry going to Anthropic."

A valid technical observation. The underlying technology is not novel: directories (very old), cron jobs (very old), terminal multiplexing (very old). What Cantillon built isn't a new technology. It's a new organization of existing technologies, informed by a specific philosophy.

Unix philosophy says "do one thing well." But Unix doesn't tell you what the "one thing" should be for your life. The 8-domain architecture answers that question.

Part IX — Anti-Patterns and Gotchas

What Can Go Wrong

Ignoring the Meta-Level Critical

Over-optimizing domain metrics. WHOOP says recovered but you feel like death. Trusting the brief over lived experience.

Solution: Maintain a meta-level outside the system. Periodic human review. Be ready to delete any domain.

Over-Isolation: Can't Coordinate High Risk

Domains become silos with no awareness of each other. Duplicate work, conflicting decisions, no synthesized view.

Solution: Implement the correlation engine. Schedule cross-domain synthesis. Create explicit handoff protocols.

Under-Isolation: Context Bleed High Risk

Single Claude instance handling everything. Long context with mixed domain information. Privacy breaches.

Solution: Strict directory separation. Separate tmux sessions. Domain-specific CLAUDE.md files.

Domain Sprawl Medium Risk

Too many micro-domains. Overhead of managing 15+ instances. Unclear boundaries.

Solution: Start with 3-5 core domains. Use subagents within domains instead of new domains.
