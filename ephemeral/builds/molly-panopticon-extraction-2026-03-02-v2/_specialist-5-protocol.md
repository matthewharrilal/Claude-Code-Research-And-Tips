## Consumption Protocol

The builder absorbs the research package in 7 steps. The sequence follows cognitive logic: boundaries before direction, direction before tools, tools before structure, structure before enrichment, enrichment before defense.

1. **READ** soul checklist (Section 0) — Establish non-negotiable boundaries. The builder must know what is IMPOSSIBLE before anything else. For the Panopticon page: border-radius: 0 means observatory instruments have machined edges, not soft curves. box-shadow: none means the watchtower casts no fake shadows. These are physics, not aesthetics.

2. **INTERNALIZE** the TC brief (Section 1) — Absorb the observatory metaphor BEFORE reading mechanism selections. If mechanisms come first, the builder composes FROM tools. If the metaphor comes first, the builder composes FROM conviction — eight survey instruments pointed at the self, the panopticon inverted from confinement to sovereignty — and selects tools to serve it. The 5 creative waypoints (Violent Gap moment, Eight Instruments grid, Implementation Simplicity, Meta-Level, Closing Constellation) are the compositional anchors.

3. **STUDY** mechanism selections (Section 2) — Not a shopping list. Each mechanism was chosen FOR the observatory metaphor and THIS content's 6-zone architecture. Read the rationale for each. Understand WHY Typography Compression serves Zone 1's philosopher voice and WHY Border-Weight Gradient encodes domain isolation in Zone 2's instrument panels.

4. **MAP** content to zones (Section 4) — The 6-zone observatory arc (Threshold → Instrument Floor → Observation Log → Observer's Journal → Survey Manual → Night Sky) defines spatial allocation. The builder assigns content sections to zones BEFORE writing CSS. This prevents orphan sections — content that exists but belongs to no zone.

5. **DEPLOY** findings zone-by-zone (Section 3) — Apply research findings to specific zones, not generically. A finding about "typographic temperature shift" applies to the Zone 2→Zone 4 transition (architect voice to philosopher voice), not everywhere. Zone-specific deployment prevents carpet-bombing.

6. **VERIFY** against anti-patterns (Section 6) — Defensive check AFTER building, not during. The builder should not read anti-patterns before Zone 4 is composed — premature awareness of anti-patterns causes avoidance behavior that flattens creative output. Check after, fix surgically.

7. **TEST** compositional questions (Section 10) — The 3 compositional questions are integration tests. Can each domain panel's spatial rhythm alone identify its domain? Does the typographic temperature shift between architect and philosopher register as a voice change? Do the closing quotes feel like a field of observations, not a bulleted list? If these fail, the build needs structural revision, not cosmetic fixes.

## Agent-0C Effective Patterns

### Pattern 1: Invariant 8-Section Template Structure

All 6 packages share an identical 8-section scaffold: (0) Soul Checklist, (1) Common References, (2) Your Combination, (3) Components to Use, (4) R-5 Findings to Test, (5) Density Pattern, (6) Structural Constraints, (7) Anti-Patterns to Avoid, (8) Scoring Criteria. The ordering is deliberate: soul boundaries → shared context → specific assignment → tools → research → spatial map → hard constraints → defense → success criteria. This matches the consumption protocol's cognitive sequence. The template structure IS the consumption protocol made structural.

**Why effective:** Builders know EXACTLY where to find any piece of information. Section 2 is always "Your Combination." Section 5 is always the density diagram. This predictability reduces cognitive load and prevents the builder from scanning the entire document hunting for a specific instruction.

**Recommendation for synthesizer:** Adopt the 8-section structure but extend to 11 sections (0-10) to accommodate the tripod pipeline's additional inputs (TC brief, zone architecture, compositional questions). Map the observatory metaphor into the template: Section 1 becomes "Observatory Context" (TC brief + content map), Section 4 becomes "Zone Architecture" (the 6-zone observatory arc).

### Pattern 2: Recipe Format with Sequenced Verbs

Agent-0C used action verbs in imperative sequence: "Place Tip Callout AFTER every heavy component," "Follow this recipe structure for the main teaching section," "Use CRESCENDO to escalate: start with light components, build to medium, climax with heavy composite." These are recipes — sequenced instructions with specific CSS/structural actions. Contrast with checklist format: "Verify zero violations," "Must include footer." The research packages never say "verify" or "must"; they say "place," "follow," "use," "start with," "build to."

**Why effective:** Recipe verbs produce creative output. Checklist verbs produce compliance. A builder who reads "Place Tip Callout after Core Abstraction" writes CSS for that placement. A builder who reads "Verify tip callout exists" scans for presence without composing placement. The verb determines the cognitive mode.

**Recommendation for synthesizer:** Write ALL builder instructions as recipes, not checklists. For the Panopticon page: "OPEN Zone 1 with Instrument Serif at display scale — the philosopher's entrance. COMPRESS Zone 2 to Inter body with JetBrains Mono for code — the architect's precision. RELEASE Zone 4 to wider margins and lighter borders — the observer stepping onto the balcony." Never "Verify Zone 1 uses serif."

### Pattern 3: ASCII Density Diagrams

Every package includes a content-specific ASCII density diagram. cd-001 uses CRESCENDO (`LOW .... / MED ........ / HIGH ............#### / PEAK ################`). cd-002 uses PULSE (`DENSE ####    ####    #### / SPARSE    ....    ....    ....`). cd-003 uses ISLANDS (`OCEAN ....        ....        .... / ISLAND       ####        ####`). cd-006 specifies FRACTAL at 5 scales. Each diagram is followed by pixel-level breathing zone specifications: "Between Rising and Building: 24px (tightening — CRESCENDO compression)."

**Why effective:** Builders are spatial-visual thinkers. A 4-line ASCII diagram transfers the entire density intention faster than 200 words of prose. The diagram encodes TWO variables simultaneously: vertical position (scroll progression) and horizontal density (information load). Each density zone is then anchored to specific pixel values, eliminating interpretation ambiguity.

**Recommendation for synthesizer:** Include a custom ASCII density diagram for the Panopticon's observatory arc. The observatory metaphor suggests a compound density: wide opening (Threshold) → compressed instrument floor → moderate observation log → breathing philosophical turn → compressed survey manual → wide release (Night Sky). Map this arc to the TC brief's spatial arc (64-80px → 40-48px → 48px → 56-64px → 32-40px → 64-80px) in the diagram.

### Pattern 4: Component Tables with Role/Weight/Velocity per THIS Page

Agent-0C's component tables are never generic inventories. Each table has 4 columns: Component | Role in This Page | Weight | Velocity. The "Role in This Page" column connects the component to THIS content's specific needs. cd-001: "Code Snippet | Primary content — implementation examples | Heavy | Slow." cd-003: "File Tree | Primary — structural map | Medium | Medium." The same component (Code Snippet) has different ROLES in different pages.

**Why effective:** Generic component lists produce generic deployments. A builder who reads "Code Snippet: Heavy, Slow" places code blocks mechanically. A builder who reads "Code Snippet: Primary content — implementation examples, the densest section of the CRESCENDO" places code blocks as the climax of a density progression. The role column transforms inventory into intent.

**Recommendation for synthesizer:** Build the component table with observatory-specific roles. For the Panopticon page, content is prose-heavy with code blocks, tables, blockquotes, and directory trees. The component table should map these to observatory roles: "Code Block | Instrument specification — old technology (bash, tmux, cron) rendered as brass fittings | Dense | Slow" or "Blockquote | Eyepiece moment — the reader sees through Cantillon's lens | Medium | Moderate."

### Pattern 5: Content-Specific Risk Profiling

Anti-pattern tables rate risk as HIGH/MEDIUM/LOW specifically for THIS page, not generically. cd-001 rates "Code Wall" as HIGH because it has 3 heavy components, but "Anxiety Ending" as LOW. cd-003 rates "Callout Cacophony" as HIGHEST because it uses 4 callout types. cd-004 rates "Semi-Transparent Backgrounds" as CRITICAL because "ambient" intuitively tempts opacity tricks. The same anti-pattern has different risk levels in different packages.

**Why effective:** Generic risk lists produce uniform caution. Content-specific risk profiling concentrates the builder's defensive attention where it matters most. A builder working on the Panopticon page should worry most about the observatory metaphor encouraging calmness/stillness when the content has emotional voltage (the TC brief explicitly warns of this), NOT about callout cacophony (the content has few advisory elements).

**Recommendation for synthesizer:** Profile risks specifically for the Panopticon content. Highest risk: visual stillness (observatory metaphor suppressing emotional weather), code block monotony (Parts IV, VIII, XII are code-heavy), Zone 5 density collapse (the survey manual section risks becoming a flat reference dump). Lower risk: callout overload (the content has few advisory moments).

### Pattern 6: Scalable Template — What Varies vs. What Stays Constant

Comparing cd-001 (5 components, 11 R-5 findings, 249 lines) to cd-006 (11 components, ALL 39 R-5 findings, 341 lines, with extra sections 0.5 Crown Jewel Status and 8 Wave 2 Note), the INVARIANT structure is the 8-section template and the soul checklist. What SCALES: component count (5→11), R-5 coverage (11→39), breathing zone detail (5 zones → per-scale specifications), scoring target (35-37 → 39-40), and additional sections for crown jewel status. The template absorbs complexity by expanding Section 3 (Components) and Section 4 (R-5 Findings) while keeping Sections 0, 1, 6, and 7 stable.

**Why effective:** The builder always knows the shape of the document regardless of its complexity. A Wave 1 builder and the crown jewel builder read the SAME template structure. Complexity lives inside sections, not in structural surprises.

**Recommendation for synthesizer:** The Panopticon package is medium-complexity (prose-heavy, moderate component count, 6-zone architecture). Size it between cd-003 and cd-004 in density. Keep the 11-section template constant; scale internal detail based on zone complexity (Zone 2 — the 8-domain instrument floor — gets the most detail because it solves the hardest spatial problem).

## Anti-Orphaning Principle

Every CSS property in the final HTML must trace to a source in the research package. No orphan CSS — properties that exist without research backing. The builder maintains traceability through inline CSS comments using a standardized citation format.

**Why this matters for the Panopticon page:** The comparison report (Phase 4) maps what the package instructed → what the builder built → what the PA found. Orphan CSS breaks this chain. If the builder writes `background: #FEF9F5` for Zone 1, that must trace to TC-BRIEF: Calibration — "Zone 1 opens at #FEF9F5 (warm cream — the dome interior lit by last light)." If the builder writes `letter-spacing: -0.02em` on headings, that must trace to a specific mechanism or finding, not emerge from habit.

**Citation format (standardized for comparison report parsing):**

```
/* PACKAGE S[N]-M[NN]: [mechanism name] — [what it does] */
  → For mechanism deployments from Section 2

/* FINDING [ID]: [finding title] — [CSS action taken] */
  → For research finding deployments from Section 3

/* CASE-STUDY [ID]: [technique name] — adapted from [source] */
  → For case study technique integrations from Section 5

/* SOUL: [constraint name] */
  → For soul constraint compliance

/* TC-BRIEF: [section] — [conviction direction] */
  → For CSS driven by TC conviction brief direction

/* QUESTION Q[N]: [abbreviated question] — [what this CSS explores] */
  → For CSS exploring compositional questions from Section 10
```

**Rule:** If a CSS property has no identifiable source from the package, question whether it belongs. The builder should be able to delete any uncommented CSS line and explain what research backing was lost.
