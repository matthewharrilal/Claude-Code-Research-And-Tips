Operations Stratum · Architecture

3. The 8 Roles Architecture

Gas Town runs on 8 named roles organized into three tiers. Each role has a specific function, a recommended model tier, and a set of CLI commands. This is the full machine — the factory floor revealed.

Info

Town-level roles run once per Gas Town instance (global coordination). Rig-level roles run per-project — you can have multiple rigs. The Overseer is you: the human boss who manages through an inbox, not through code.

Hierarchy Overview

Essence

Eight roles, three tiers, one factory. The architecture is the chaos management — every role exists because a specific failure mode demanded it.

Role Deep Dives

Town-Level Roles — Global Coordination
1. Mayor Opus 4.5
Chief coordinator — your concierge and chief-of-staff

Responsibilities

  • Receives high-level directives from Overseer
  • Routes work to appropriate rigs
  • Manages cross-rig coordination
  • Handles town-level decisions
  • Primary AI coordinator for the entire town

Commands

gt mayor up # Start mayor gt mayor status # Check mayor state gt mayor down # Stop mayor
2. Deacon Haiku
Health monitoring and agent coordination

Responsibilities

  • Monitors agent health across the town
  • Handles agent-to-agent handshakes
  • Reports anomalies to Mayor
  • Tracks work progress
  • Manages handoff protocols and worker recycling

Commands

gt deacon up # Start deacon gt deacon status # View monitoring state
3. Dogs Sonnet
Quality assurance and safety gates — town-level grunts

Responsibilities

  • Reviews work before acceptance
  • Enforces quality standards
  • Guards against regressions
  • Can reject/rollback work
  • Stale branch cleanup

Commands

gt dogs up # Start dogs gt dogs bark # Trigger quality check
Rig-Level Roles — Per-Project Workers
4. Refinery Opus 4.5
Breaks down large tasks into agent-sized work

Responsibilities

  • Receives feature requests from Mayor
  • Decomposes into smaller, atomic tasks
  • Assigns to Polecats or Crew
  • Tracks decomposition progress
  • Manages the merge queue with sequential rebasing

Commands

gt refinery up <rig> # Start refinery for rig gt refinery decompose <task> # Manually trigger decomposition
5. Polecat Sonnet
Named, persistent agent workers with specializations

Responsibilities

  • Execute assigned tasks
  • Maintain context across sessions
  • Specialize in rig-specific knowledge
  • Report completion to Refinery
  • Ephemeral but can be persistent workspaces

Commands

gt polecat up <rig> alice # Start polecat named "alice" gt polecat up <rig> bob # Start polecat named "bob" gt polecat list <rig> # List polecats for rig gt polecat assign alice <task> # Assign task to alice
Naming convention: Human names (alice, bob, carol) help with mental model
6. Witness Haiku
Observes and records rig activity — watches over polecats

Responsibilities

  • Watches all work in assigned rig
  • Records history and context
  • Provides state to new workers
  • Reports anomalies to Deacon
  • Unsticks drifting agents

Commands

gt witness up <rig> # Start witness for rig gt witness report # Get current state report
7. Crew Haiku
Short-lived, task-specific burst workers

Responsibilities

  • Execute single tasks
  • No persistent state
  • Cheap and disposable
  • Used for parallel bursts
  • Spawn, complete, disappear

Commands

gt crew spawn <rig> --task "fix this bug" gt crew count <rig> # Count active crew gt crew purge <rig> # Kill all crew
Human Level — You
8. Overseer (YOU)
Product Manager and decision maker

Responsibilities

  • Define features and priorities
  • Review inbox decisions
  • Approve major changes
  • Provide human judgment
  • Has identity in the system with own inbox

Interface

  • Human inbox for decisions
  • tmux panes for observation
  • gt CLI for commands
  • Can send and receive town mail

"That's you, Human. As the Overseer, you have an identity in the system, and your own inbox, and you can send and receive town mail."

Tip

Model recommendations are about cost optimization, not capability gates. You can run all roles on Opus 4.5 — but Haiku handles observation and simple tasks at a fraction of the cost. Save your Opus budget for the coordinators (Mayor) and decomposers (Refinery) where complex reasoning actually matters.

Role Communication Flow

A feature request enters the system and flows through the roles. This is the machine running — not just its parts on display, but the parts in motion:

Feature Request Flow: "Build user authentication"
Overseer
"Build user authentication" — filed as feature request
Mayor
Routes to rig — selects appropriate project, sends status reports back
Refinery
Decomposes into tasks:
— task 1: DB schema — task 2: API routes — task 3: Frontend forms
Polecat: alice
Executes task 1 — observed by Witness throughout
Polecat: bob
Executes task 2 — in parallel
Crew: temp-1
Executes task 3 — ephemeral burst worker
Witness
Observes all activity — on anomaly, reports to Deacon
Dogs
Quality check — reviews work, enforces standards, can reject/rollback
Complete
Work passes gates — merged, delivered, done
Gotcha

The merge queue is sequential, not parallel. While Polecats and Crew execute tasks in parallel, merging happens one branch at a time through sequential rebasing. This is the architecture managing chaos — parallel execution, serial integration.

Challenge

Trace a real feature you've built recently through the 8-role flow above. Where would the decomposition happen? Which tasks would be Polecat-persistent vs Crew-ephemeral? If you can map your own work onto this architecture, you're ready for the next section.
