Kanban Loop

A bash loop that runs Claude Code repeatedly with fresh context until a completion condition is met. Each iteration gets a clean slate with access to git history and file-based memory. Enables overnight feature shipping.

Implementation
bash
#!/bin/bash
MAX_ITERATIONS=${1:-10}
for (( i=1; i<=$MAX_ITERATIONS; i++ )); do
  echo "Iteration $i / $MAX_ITERATIONS"
  claude "Review plans/prd.json...
    Mark PRD items as passes: true.
    Make a git commit.
    If complete, output PROMISE COMPLETE HERE." \
    | tee output.txt
  if grep -q "PROMISE COMPLETE HERE" output.txt; then
    break
  fi
done
Key Principles
  • Fresh context per iteration -- avoids context rot
  • External memory -- git history + progress.txt
  • Verifiable completion -- tests must pass
  • Small task sizing -- one story per context window

Parallel Strategies & Isolation

Four isolation strategies for running agents in parallel, from simple directory separation to full container isolation.

from S01 Panopticon enables S04-05 deepened in S14-21
Domain-Isolated Instances

From the Panopticon pattern: run separate Claude Code processes in different directories. Filesystem isolation prevents context pollution.

bash
# Terminal 1: Frontend specialist
cd ~/project/frontend && claude

# Terminal 2: Backend specialist
cd ~/project/backend && claude

# Terminal 3: Documentation specialist
cd ~/project/docs && claude
Branch-Based Parallel Work

Each agent works on its own branch. Human reviews and resolves conflicts during PR. Simple, effective, battle-tested.

Isolation intensity scales with agent count. Directory separation works for 2-3 agents. Branch-based works for small teams. But at 4+ parallel agents, you need worktrees or containers -- anything less and merge conflicts will consume more time than the parallelism saves.
Git Worktrees

Git worktrees create isolated working directories sharing the same repo. The gold standard for multi-agent isolation -- each agent gets its own filesystem checkout without full clones.

bash
~/project/
  ├── .git/           # Shared git database
  └── src/            # Main working directory

~/project-worktrees/
  ├── feature-a/      # Agent 1's isolated directory
  ├── feature-b/      # Agent 2's isolated directory
  └── bugfix-c/       # Agent 3's isolated directory
Container-Isolated Parallelism

For heavy parallel work, run Claude Code instances in isolated Docker containers (AI-Maestro pattern). No branch/file conflicts, per-container firewalls, persistent caches.

Subagent Coordination & Task Management

How orchestrators delegate to specialized workers, and the task management primitives that enable structured coordination.

implements S07 Hub enabled by S02-03 specializes into S08
"You do not write code. You do not run commands. You do not explore codebases. You are the conductor. Your agents play the instruments."
The conductor/instrument split is fundamental. Orchestrators manage the task graph; workers execute. Mixing roles leads to context pollution and lost coordination state.
Orchestrator Tools
  • Read -- references, guides (1-2 files max)
  • TaskCreate, TaskUpdate, TaskGet, TaskList
  • Task -- spawn workers
Worker Tools
  • Read, Write, Edit, Bash
  • Glob, Grep, WebFetch, WebSearch
  • Cannot spawn sub-agents or manage task graph

Agent-to-Agent Handoffs

Four communication mechanisms for agents that need to pass context, state, and instructions to each other.

File-Based Handoffs
Agents communicate through the filesystem. Structured JSON files in shared directories. The Panopticon pattern -- each agent reads/writes to its own directory as persistent memory.
Sync: Polling | Latency: ~seconds | Complexity: Low
Git-Based Handoffs
Progress persists in files, git history provides context. Append-only progress.txt, PRD status tracking, commit history as memory. Each iteration gets a clean slate.
Sync: Commit/Pull | Latency: ~seconds | Complexity: Low
Docker Volume Handoffs
Agents write output JSON to shared Docker volumes. Not real-time, but lightweight and requires no external dependencies. AI-Maestro uses this for inter-container messaging.
Sync: Shared Volume | Latency: ~ms | Complexity: Medium
MCP Agent Mail
Email-like interfaces for agent collaboration. No training needed, natural "agent village" behavior, async by default. From Steve Yegge's Gas Town -- agents have inboxes.
Sync: Async Queue | Latency: ~ms | Complexity: Medium

Conflict Resolution

Five strategies for managing the inevitable conflicts when multiple agents work in shared codebases.

resolves S02-03 resolves S01 Ralph deepened in S14-21
Conflicts are inevitable, not optional. The question is never "will agents conflict?" but "how quickly will conflicts be detected and resolved?" These five strategies trade off simplicity against granularity.
Branch Isolation

"Parallel agents cause merge conflicts -- use serial." The primary strategy: each agent works on its own branch. Human merges.

File-Level Locking

Acquire a lock file before editing, release after. Simple but effective for preventing concurrent modification.

bash
if [ ! -f "$FILE.lock" ]; then
  touch "$FILE.lock"
  # do work
  rm "$FILE.lock"
else
  echo "File locked, waiting..."
fi
Directory Ownership

Each agent owns specific directories. agent-frontend/ is exclusive to the frontend agent. shared/ is read-only for all, write via handoff protocol.

Transactional File Operations

Write to temp files, then atomic rename: echo "content" > file.tmp && mv file.tmp file.actual

Git-Based Conflict Detection

Check for conflicts before starting work with git merge-base --is-ancestor. Detect early, resolve before divergence compounds.

Advanced Worktree Management extends S03

Production-grade worktree lifecycle with automatic cleanup, agent spawning, and shared state synchronization.

bash
create_worktree() {
  local branch=$1
  local worktree_path="$WORKTREE_BASE/$branch"
  git worktree add -b "$branch" "$worktree_path" main
  cp .claude/CLAUDE.md "$worktree_path/.claude/"
  date +%s > "$worktree_path/.worktree-created"
  echo "$worktree_path"
}

cleanup_stale_worktrees() {
  local cutoff=$(($(date +%s) - STALE_HOURS * 3600))
  for wt in $(git worktree list --porcelain | grep "^worktree" | cut -d' ' -f2); do
    local created=$(cat "$wt/.worktree-created" 2>/dev/null)
    if [ -n "$created" ] && [ $created -lt $cutoff ]; then
      git worktree remove --force "$wt"
    fi
  done
}

Message Queue Pattern extends S06

Beyond file-based handoffs: FIFO pipes for lightweight real-time message passing between agents.

bash
AGENT_INBOX="/tmp/agent-$AGENT_ID-inbox"
mkfifo "$AGENT_INBOX" 2>/dev/null

send_message() {
  local target_agent=$1
  local message=$2
  echo "{\"from\":\"$AGENT_ID\",\"ts\":$(date +%s),\"msg\":\"$message\"}" \
    > "/tmp/agent-$target_agent-inbox"
}

check_messages() {
  if read -t 0.1 message < "$AGENT_INBOX"; then
    echo "Received: $message"
  fi
}

Circuit Breaker Pattern extends S10

Prevent cascading failures across agent pools. States: CLOSED (normal) -> OPEN (failing fast) -> HALF_OPEN (testing recovery).

bash
FAILURE_THRESHOLD=3
RECOVERY_TIMEOUT=300  # seconds

call_agent() {
  local agent_type=$1
  case ${circuit_state[$agent_type]:-CLOSED} in
    OPEN)
      if [ $((now - last)) -gt $RECOVERY_TIMEOUT ]; then
        circuit_state[$agent_type]="HALF_OPEN"
      else
        echo "Circuit OPEN - failing fast"; return 1
      fi ;;
  esac
  # ... attempt call, track failures
}

Testing Multi-Agent Systems new

Chaos Testing Scenarios

  • Agent Crash Recovery: Kill agent mid-task. Verify orchestrator detects within 60s, requeues task, no data loss.
  • Network Partition: Drop API traffic. Verify backoff mode. Restore. Verify resume without duplicate work.
  • Conflicting Edits: Assign same file to two agents. Verify conflict detection before merge.
  • Resource Exhaustion: Queue more tasks than agents can handle. Verify priority ordering maintained.
