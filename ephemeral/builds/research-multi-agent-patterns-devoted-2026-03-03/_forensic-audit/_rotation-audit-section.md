## 5. Primacy Rotation Compliance {#rotation-audit}
*Agent: rotation-auditor | Tasks #3 + #9 | 2026-03-03*

### Summary Verdict

**The primacy rotation protocol was not followed.** The builder used a batch-read-then-apply model in every research wave instead of the prescribed one-file-at-primacy rotation. No files received formal primacy position, and the universal mechanisms (neglect threshold, diagnostic override, per-file phase tracking) were not used.

**However:** The builder's alternative approach produced genuine spatial inventions — the topology color system at 10 scales, the network map, velocity callouts, CRESCENDO density, and fractal persistence. These demonstrate real design intelligence that emerged through a different absorption model than the one prescribed. The question is not "did it fail?" but "what was gained and lost by using batch-read instead of rotation?"

---

### 5.1 What the Skill Prescribes

**Wave 2 rotation (SKILL.md line 610):**
```
R1 → R4 → R3 → R5 → R1(2nd) → R2 → R3(2nd) → R4(2nd) → R5(2nd) → R2(2nd) → SYNTHESIS → SYNTHESIS(2nd) → diagnostic → depth passes
```
Minimum 12 passes. Each file gets 2+ primacy turns.

**Wave 3 rotation (SKILL.md lines 736-751):**
Tiered rotation — crown jewels (2-3 turns), tier 2 (1-2 turns), tier 3 (0-1 turns). Minimum 12 passes.

**Universal mechanisms:** 4-pass neglect threshold, diagnostic override, per-file phase tracking (STRUCTURAL/INTEGRATION/DEPTH), delta profile, dimension permeation tracking.

---

### 5.2 What Actually Happened

#### The Batch-Read Pattern

In every research wave, the builder read multiple files simultaneously in a single pass, then applied insights in 1-2 subsequent build passes:

| Wave | Prescribed | Actual | Research Model |
|------|-----------|--------|----------------|
| Wave 1 (Soul) | 9 positions, 10 passes | ~5 passes | Batch-read 3 files in pass 1.1 |
| Wave 2 (R1-R5) | 14 positions, 12 passes | 2 passes | Batch-read 5 files in pass 2.0 |
| Wave 3 (Case Studies) | 12 positions, 12 passes | 4 passes | Batch-read 6 files in pass 3.0 |
| Wave 4 (Synthesis) | 8 positions, 8 passes | 4 passes | No new files read (internal assessment) |
| Wave 5 (Combination) | 9 positions, 8 passes | 2 passes | No new files read (adversarial tests) |

**No file received primacy position in any wave.** The skill states "Do not read all files equally in every pass — that defeats the purpose of rotation" (line 445). The builder consistently chose the batch approach instead.

#### What Was Gained by Batch-Read

The batch approach has a genuine advantage the skill doesn't acknowledge: **cross-file synthesis in a single cognitive moment.** When the builder read R1 + R3 + R5 + SYNTHESIS + mechanisms simultaneously in pass 2.0, it could immediately see connections between CRESCENDO (R3), density rhythm (R1), and anti-stacking (R5). The resulting CRESCENDO implementation — with padding compression from 56px to 44px, zone background alternation, and velocity callouts — integrates multiple research dimensions in a way that suggests the batch read enabled cross-pollination.

Similarly, reading DD-006 + CD-001 + OD-006 + DD-003 + CD-006 in one pass let the builder see that fractal persistence (DD-006) + velocity interleaving (CD-001) + phase transitions (OD-006) could work as a unified system, which became the topology color thread at 10 scales.

#### What Was Lost

1. **R2 and R4 were never read.** R2 (27 findings on creative layouts, editorial courage) and R4 (192 findings, 57% of all research) were skipped entirely. This is the most consequential gap — not the batch vs. rotation format, but the outright omission of 65% of the research corpus. The rotation schedule would have ensured R2 reached position 6 and R4 reached positions 2 and 8. The batch approach had no mechanism to guarantee coverage of files the builder didn't choose to read.

2. **No re-reading after building.** The rotation model's deepest argument is that research "changes meaning" after building. The builder never re-read any research file. We cannot know what findings would have become relevant after the page took shape — this is an unknowable counterfactual, but the skill's theory is plausible.

3. **No depth progression per file.** Files that were read went through one engagement: extract key insights, apply them. The STRUCTURAL → INTEGRATION → DEPTH progression — where the same file yields different insights at different page-maturity stages — did not occur. DD-006 is the partial exception: its fractal principle was actively deepened across passes 3.1, 3.3, and 4.3, showing real integration work even without the formal rotation.

#### What This Tells Us

The batch-read vs. rotation question is ultimately about two different things:

- **Coverage guarantee:** The rotation ensures every file gets attention. Batch-read relies on the builder's judgment about which files to read. The builder's judgment missed R2 and R4 entirely — a coverage failure that rotation would have prevented.
- **Depth vs. breadth:** Rotation optimizes for depth per file (10+ cycles). Batch optimizes for cross-file synthesis. Both have theoretical merit. This build tested batch-read and got genuine results — but with significant coverage gaps.

The honest assessment: **the rotation protocol's main value may be as a coverage guarantee rather than a depth mechanism.** The builder demonstrated that real spatial invention can emerge from batch reading, but the batch model has no safety net for files the builder doesn't select.

### 7.3 Honest Depth Assessment: What the Numbers Miss

The cycle counts above measure formal compliance. But some files achieved more depth than the numbers suggest, and the strict cycle-counting framework may undervalue what actually happened:

**DD-006 (fractal) — deeper than "3-4 cycles" implies.** The fractal persistence principle was not just "applied" — it became the page's organizing logic. Topology colors at 10 distinct scales (legend dots, topo-tags, card borders, section borders, xref indicators, handoff tops, story borders, selection dots, grid cell borders, network map labels) with 58 instances is a genuine spatial invention. The builder didn't just extract "fractal = repeat at scales" — it TRANSLATED the principle into a content-specific color encoding system. This suggests the builder achieved real depth with this file's dimension even without the prescribed 10+ formal cycles.

**CD-001 (velocity) — integrated, not just applied.** 8 velocity callouts placed at specific content-driven positions (between heavy blocks) with content that reflects the section's argument ("Three philosophies, one ecosystem", "The conductor/instrument split is fundamental"). This is not rote mechanism deployment — it is content-sensitive interleaving.

**The CRESCENDO pattern — emerged from cross-file synthesis.** The padding progression (56 → 52 → 44 → 80/64 → 64) was derived from R1's density rhythm + R3's CRESCENDO + the content's own velocity arc. This compound insight may have benefited from batch reading (seeing all three simultaneously) rather than the rotation model (encountering them in isolated passes).

### 7.4 What 10+ Cycles Would Likely Have Added

Being honest about the counterfactual:

**For files that were read (likely impact: MODERATE):**
- More findings per file would have been applied — the "one dominant idea" pattern would break at cycle 5-6 as the page matures
- DD-006's fractal principle might have deepened to include spacing compression gradients and border-weight hierarchies (mentioned in DD-006 but not extracted)
- R3's full 6-dimension density model (temporal, depth, semantic, cognitive, interaction, spatial) might have been engaged instead of just spatial density
- Later cycles would have revealed compound interactions between dimensions already built

**For files that were never read (likely impact: HIGH):**
- R4's 192 axis findings would have introduced non-default reading paths — the page currently has no Z/F/Gutenberg patterns
- R2's editorial techniques would have added broken grids, asymmetry, pull quotes — spatial courage the page lacks
- The 10+ cycle model is moot for unread files — the primary gap is coverage, not depth

**For the overall build (speculative):**
- The "changed-meaning effect" — re-reading research after building — is the skill's strongest theoretical argument for 10+ cycles. We genuinely cannot know what re-reading R3 after building the topology system would have revealed. The skill's claim is plausible but untested.

### 7.7 Root Cause and Honest Assessment

The builder and the skill use fundamentally different absorption models:

**Skill's model:** Read ONE file → build → look → refine → repeat 10+. Depth through repetition.
**Builder's model:** Read MANY files → synthesize → build integrated solutions → iterate on the build. Depth through synthesis.

The builder's model is not inherently inferior — it produced genuine spatial inventions that integrate multiple research dimensions simultaneously. The topology color system draws on DD-006 (fractal), CD-001 (velocity interleaving), and OD-006 (phase transitions) in a unified way that might not have emerged from isolated primacy turns.

**What the builder's model cannot do:** Guarantee coverage. The rotation schedule ensures every file gets attention. The builder's self-directed model skipped R2, R4, and half the case studies — a 65% corpus gap that no amount of depth on the remaining files can compensate for.

**What the skill's model hasn't proven:** That 10+ formal cycles per file produces qualitatively different output than 3-6 cycles with cross-file synthesis. The skill argues this theoretically (the "changed-meaning effect," compound discovery, perceptual resolution deepening) but this is a first run — we don't have comparison data.

**The practical question for skill revision:** Is the rotation's primary value (a) depth per file, or (b) coverage guarantee? This build suggests (b) is more critical. The builder achieved meaningful depth on files it engaged with, but failed catastrophically on coverage. A revised skill might prioritize coverage assurance (ensuring all files are read at least once) over the precise rotation schedule.
