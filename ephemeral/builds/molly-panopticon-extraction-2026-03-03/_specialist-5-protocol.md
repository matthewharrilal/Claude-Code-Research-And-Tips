## Consumption Protocol

The builder should absorb the research package in this sequence. Each step has a cognitive rationale -- the ordering is not arbitrary.

1. **READ** soul checklist (Package Section 0) -- Establish what is IMPOSSIBLE. Soul constraints are physics: border-radius: 0, box-shadow: none, the typography trinity, the 3-category border system. The builder must internalize the boundary conditions before encountering creative invitations, because creative work within unknown boundaries produces violations.

2. **INTERNALIZE** build context and metaphor (Package Section 1) -- Absorb the cartographic survey station metaphor from the TC brief's World-Description BEFORE reading mechanism selections. For this Panopticon content, the surveyor's grid, contour lines, datum points, and projection shifts are the creative vocabulary. Mechanisms read first = building FROM tools. Metaphor read first = building FROM conviction, selecting tools to serve it.

3. **STUDY** mechanism selections (Package Section 2) -- Each mechanism was selected FOR this content's cartographic metaphor and its 5-zone architecture (Datum Point / Survey Grid / Field Notes / Contour Map / Projection). Understand WHY each was chosen -- not as a shopping list but as a reasoned response to the content's tensions (recursive self-observation, bilateral symmetry, parallel isolation, empowerment-plus-warning).

4. **MAP** content to zones (Package Section 4) -- Use the zone architecture to assign content sections (Parts I-XVII) to spatial zones. For this article: Part I maps to the Datum Point (sparse, establishing), Parts II-III to the Survey Grid (moderate, structured), Parts IV-V to Field Notes (dense, technical), Parts VI-X to the Contour Map (dense-with-variation, dark philosophical terrain), Parts XI-XVII to the Projection (opening back out). The density diagram is the spatial blueprint.

5. **DEPLOY** findings zone-by-zone (Package Section 3) -- Apply research findings to specific zones, not globally. A finding about typography compression applies to the Contour Map zone where philosophical arguments tighten, not to the Datum Point where spacing is generous. Zone-specific deployment prevents "flat application" where every finding applies everywhere equally.

6. **VERIFY** against anti-patterns (Package Section 6) -- Defensive scan. Check the 8-domain grid section against parallelism-without-cacophony. Check the dark Contour Map zone against void patterns. Check code blocks in the Field Notes zone against code wall anti-pattern. This step is forensic, not creative.

7. **EVALUATE** structural propositions (Package Section 11, if present) -- If the package contains structural propositions (e.g., a bento grid for the 8-domain taxonomy, bilateral layout for the institutional-vs-personal gaze), decide ADOPT/MODIFY/REJECT for each. Adopted structural sections follow the Standard/Structural build path. For structural sections, the reading path is determined by the spatial form, not by section order -- grid sections are scanned, not scrolled.

8. **TEST** against compositional questions (Package Section 10) -- The TC brief's 3 compositional questions are builder invitations: What if domain cards encoded temporal urgency through border weight? What if zone boundaries performed projection shifts? What if the dark-zone descent was gradual, not abrupt? These questions generate multi-channel CSS explorations that mechanisms alone would not produce.

## Agent-0C Effective Patterns

### Pattern 1: Invariant 8-Section Template

All 6 packages share an identical 8-section skeleton: (0) Soul Checklist, (1) Common References, (2) Your Combination, (3) Components to Use, (4) R-5 Findings to Test, (5) Density Pattern, (6) Structural Constraints, (7) Anti-Patterns, (8) Scoring Criteria. The structure is invariant; the content within each section is fully customized per builder. This template succeeds because it creates a predictable consumption path -- the builder always knows WHERE to find something (mechanism selections are always in Section 2, anti-patterns always in Section 7) without the cognitive overhead of learning a new document structure.

**RECOMMENDATION FOR SYNTHESIZER:** Adopt the numbered-section template. Extend it from 8 to 12 sections (0-11) to accommodate the tripod pipeline's additional layers (case study integrations, compositional questions, structural propositions), but preserve the invariant ordering. The builder's cognitive overhead is in the CONTENT, not the STRUCTURE.

### Pattern 2: Recipe Verbs Over Checklist Verbs

Agent-0C used sequenced action verbs throughout: "Place Tip before Code with 'In line X below...' reference" (cd-001 Section 4), "Use CRESCENDO to escalate: start with light components, build to medium, climax with heavy composite" (cd-001 Section 3). This is RECIPE format -- sequenced steps with concrete CSS actions. Compare to checklist format: "Verify that no two SLOW components are adjacent" -- same rule, passive enforcement. The recipe format produced richer creative output because it tells the builder what TO DO, not just what to check.

**RECOMMENDATION FOR SYNTHESIZER:** Write all builder-facing instructions as recipes with sequenced verbs (READ, DEPLOY, PLACE, BUFFER, ESCALATE, COMPRESS). The builder's orientation is "what do I do NEXT?", not "am I compliant?"

The diagram is the conceptual map; the pixel values are the execution spec.

**RECOMMENDATION FOR SYNTHESIZER:** Include an ASCII density diagram in Package Section 4 (Zone Architecture) that maps the Panopticon content's 5-zone structure: Datum Point (sparse) --> Survey Grid (moderate) --> Field Notes (dense) --> Contour Map (dense-with-variation) --> Projection (moderate-to-sparse). Follow the diagram with pixel-level spacing per zone transition, sourced from the TC brief's spatial arc (64-80px, 40-48px, 32-40px, 32-48px, 48-64px).

### Pattern 4: Component Tables with Content-Specific Role Column

Every package included a component table with four columns: Component | Role in THIS Page | Weight | Velocity. The critical column is "Role in This Page" -- it connects the component to THIS content rather than listing it generically. Compare cd-001: "Code Snippet | Primary content -- implementation examples" versus cd-004: "Essence Callout | AMBIENT -- philosophical background voice." Same component type, radically different roles, because the content differs.

For the Panopticon content, the component table must map to the cartographic metaphor: Cantillon's key quotes are not generic blockquotes -- they are "terrain formations the surveyor has discovered" (from TC brief). Code blocks are not generic code -- they are "field instruments" (bash scripts, cron entries). The role column connects each component to its metaphorical function.

**RECOMMENDATION FOR SYNTHESIZER:** Build the component table with a Role column that references the cartographic survey station metaphor. The role should specify WHERE in the 5-zone architecture each component primarily appears and WHAT metaphorical function it serves.

### Pattern 5: Content-Specific Anti-Pattern Risk Profiling

Agent-0C rated every anti-pattern's risk level specifically for THIS builder's content, not generically. cd-001 rated "Code Wall" as HIGH because it uses 3 heavy components, while cd-005 rated it LOW because its focus is callout family DNA. cd-004 rated "Semi-Transparent Backgrounds" as CRITICAL because the "ambient Essence" concept might tempt opacity tricks, while cd-001 rated it ZERO TOLERANCE (standard).

For the Panopticon content: Code Wall risk is MEDIUM (the Field Notes zone has multiple bash code blocks but they are short). Whitespace void risk is HIGH (the Datum Point and Projection zones have generous spacing that could become voided). Dark-zone legibility risk is HIGH (the Contour Map zone uses near-black background). These risk ratings must be DERIVED from the content analysis, not applied from a generic list.

**RECOMMENDATION FOR SYNTHESIZER:** Rate each anti-pattern's risk level specifically for the Panopticon content's 5-zone architecture, with mitigation strategies that reference specific zones and specific content sections.

### Pattern 6: Scalable Investment -- What Grows, What Stays Constant

Across the 6 packages, the INVARIANT sections were: Section 0 (Soul Checklist -- identical in all 6), Section 1 (Common References -- identical), Section 6 (Structural Constraints -- nearly identical), and the transition grammar table. The VARIABLE sections were: Section 2 (combination triple -- unique), Section 3 (component table -- unique count: 5 components in cd-001, 11 in cd-006), Section 4 (R-5 findings -- 11 tested in cd-001, all 39 in cd-006), Section 5 (density pattern -- different diagram per package), Section 7 (anti-pattern risk -- different ratings). cd-006 received TWO extra sections that no other package had: Section 0.5 (Crown Jewel Status) and Section 8 (Wave 2 Note). The crown jewel escalation is a structural pattern: the highest-investment build gets the most customized package.

**RECOMMENDATION FOR SYNTHESIZER:** For the tripod pipeline, the soul checklist, structural constraints, and citation format should be INVARIANT across all builds. The metaphor, zone architecture, mechanism selections, compositional questions, and anti-pattern risk profiles should be FULLY CUSTOMIZED per content. Investment scales with content complexity.

### Pattern 7: Compatibility Notes from the 11x11 Matrix

Each package included a compatibility notes table showing specific component pairs WITH ratings (Yes/OK/No) AND contextual notes for THIS content. cd-001: "Reasoning + Core Abstraction | Yes | OK with buffer -- both framed, need connective tissue (G2)." cd-004: "Essence + Essence | No | AVOID -- axiom fatigue (T1 jarring collision)." These notes make the abstract matrix ACTIONABLE by connecting ratings to specific composition challenges in the builder's assignment.

**RECOMMENDATION FOR SYNTHESIZER:** When specifying mechanism interactions for the Panopticon build, include compatibility notes for mechanism pairs that will appear within the same zone. Example: in the Contour Map zone, the dark background and Cantillon's key quotes (Instrument Serif) must co-exist with dialectical exchanges and the "meta-level" benchmark -- specify how these elements interact, not just that they appear.

## Anti-Orphaning Principle

Every CSS property in the final HTML must trace to a source in the research package. No orphan CSS -- CSS that exists without research backing. If the builder writes `background: #1A1A1A` for the Contour Map zone, that property traces to TC-BRIEF: Calibration -- "Contour Map zone: #1A1A1A (near-black bedrock)." If the builder writes `letter-spacing: -0.02em` on a heading, that property traces to a mechanism (Typography Compression) or a soul constraint (from identity.md). If a CSS rule has no identifiable source, the builder should question whether it belongs.

**STANDARDIZED CITATION FORMAT (all builders must use for comparison report parsing):**

```
/* PACKAGE S[N]-M[NN]: [mechanism name] — [what it does] */
  → For mechanism deployments from Package Section 2

/* FINDING [ID]: [finding title] — [CSS action taken] */
  → For research finding deployments from Package Section 3

/* CASE-STUDY [ID]: [technique name] — adapted from [source] */
  → For case study technique integrations from Package Section 5

/* SOUL: [constraint name] */
  → For soul constraint compliance (identity.md / tokens.css)

/* TC-BRIEF: [section] — [conviction direction] */
  → For CSS driven by TC conviction brief direction

/* QUESTION Q[N]: [abbreviated question] — [what this CSS explores] */
  → For CSS generated from Package Section 10 compositional questions

/* STRUCTURAL: [content-logic] — [spatial form chosen] */
  → For structural invention CSS from Section 11 propositions
```

This format enables automated comparison report extraction -- the weaver can grep for citation prefixes and count compliance per source category. For the Panopticon build, `TC-BRIEF: Calibration` citations will be frequent (the brief's 5-zone background arc drives many CSS values), and `SOUL: border-radius: 0` will appear on every bordered element.

## Template Recommendation for Synthesizer

The 12 sections (0-11) should be organized across 3 files to match the tripod pipeline's 3-pass builder architecture:

### File 1: Pass 1 Package (~Sections 0, 1, 2, 4, 11)
- **Section 0: Soul Checklist** -- INVARIANT. The hard boundaries.
- **Section 1: Build Context** -- Metaphor summary from TC brief, content map, emotional arc, zone names.
- **Section 2: Mechanism Selections** -- WHY each mechanism was chosen for THIS content and THIS metaphor. Include the cartographic isomorphism table from the TC brief.
- **Section 4: Zone Architecture** -- ASCII density diagram, pixel-level spacing per zone, background arc, type arc. Zone-to-content mapping table.
- **Section 11: Structural Propositions** -- If present: bento grid for 8-domain taxonomy, bilateral layout for institutional-vs-personal gaze, etc. Each proposition includes blueprint CSS and ADOPT/MODIFY/REJECT instructions.

### File 2: Pass 2 Package (~Sections 3, 5, 6, 7)
- **Section 3: Findings Deployment** -- Zone-specific finding applications, not global lists. Grouped by zone: "In the Survey Grid zone, deploy [findings X, Y, Z] because [rationale]."
- **Section 5: Case Study Integrations** -- Relevant case study techniques with adapted CSS. Anti-prescription format: "This technique was used in [source] for [purpose]; adapt for [this content's purpose]."
- **Section 6: Structural Constraints** -- Container width, responsive breakpoints, border system, callout limits. MOSTLY INVARIANT.
- **Section 7: Anti-Pattern Risk Profile** -- Content-specific risk ratings with zone-specific mitigations.

### File 3: Pass 3 Package (~Sections 8, 9, 10)
- **Section 8: Scoring Criteria** -- Per-zone scoring expectations. What 4/4 looks like for THIS content.
- **Section 9: Consumption Protocol** -- THIS DOCUMENT's consumption protocol, condensed to a numbered list the builder follows on first read.
- **Section 10: Compositional Questions** -- The TC brief's 3+ compositional questions, carried VERBATIM, with implied CSS channels listed below each. These are creative invitations, not requirements.
