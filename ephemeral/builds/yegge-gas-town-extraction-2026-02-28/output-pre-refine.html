Section 03 — The Mental Model

The Gas Town Mental Model

Imagine a small frontier town where everyone has a specialized role: the Mayor makes strategic decisions, various shop owners handle specific domains, the Town Clerk maintains records and context. Each shop has its own expertise but they coordinate.

The name references frontier utility — a gas station town exists because it does something useful. Not fancy, not glamorous — functional. The whole point is pragmatic productivity, not architectural beauty.

The insight is not "use agents." Everyone knows that. The insight is "organize agents into a TOWN" — with roles, memory, coordination, and progression.

Section 04 — Architecture

The 8 Roles Architecture

Gas Town defines 8 specialized roles. Each can be filled by a different Claude Code session. Together they form a coordinated factory.

Role 01 — Commander
The Mayor (Orchestrator)
The human developer or main Claude session directing all operations.
  • Makes strategic decisions about priorities
  • Delegates tasks to specialist roles
  • Reviews and approves major deliverables
  • Maintains the "big picture"
Role 02
The Architect
Designs system structure and interfaces.
  • Creates component hierarchies
  • Defines API contracts
  • Writes ADRs
Role 03
The Builder
Implements the actual code.
  • Writes production code
  • Follows architect's specs
  • Creates PRs
Role 04
The Researcher
Gathers information and analyzes options.
  • Evaluates libraries and tools
  • Reads documentation
  • Produces research summaries
Role 05
The Critic
Reviews code quality and identifies issues.
  • Code reviews with feedback
  • Security analysis
  • Edge case identification
Role 06
The Tester
Creates and runs test suites.
  • Writes unit and integration tests
  • Runs test suites
  • Reports coverage
Role 07
The Documenter
Creates and maintains documentation.
  • API documentation
  • User guides
  • README maintenance
Role 08 — Critical Infrastructure
The Town Clerk
Maintains shared memory and context — the institutional backbone.
  • Manages "beads" (memory artifacts)
  • Maintains context files
  • Tracks project state
  • Ensures continuity between sessions

Section 05 — Communications Infrastructure

Beads: The Memory System

Beads are structured memory artifacts that persist context between agent sessions. They are the Gas Town equivalent of institutional memory. Without them, multi-agent is just parallel single-agent with extra steps.

Core Truth
The secret sauce of Gas Town is not the agents. It is the BEADS. Without memory, you are just running Claude in multiple terminals and calling it orchestration.

DB-Choice: PostgreSQL over MongoDB

Context

E-commerce platform with complex relationships, ACID compliance needed for transactions, analytics queries on historical data.

Options Considered

  1. PostgreSQL — Relational, ACID, mature ecosystem
  2. MongoDB — Flexible schema, good for rapid iteration

Section 06 — Strategic Context

The 6 Waves of AI

Yegge frames the evolution of AI-assisted development in 6 waves. Gas Town lives at Wave 5. Most developers are still on Wave 2.

Wave 1 Autocomplete 2021-2022 Copilot-style suggestions. Single-line completion. You drive everything.
Wave 2 Chat 2022-2023 ChatGPT, Claude chat. Question-and-answer. Copy-paste workflow.
Wave 3 Inline Editing 2023-2024 Cursor, Windsurf, Claude Code. AI edits files directly.
Wave 4 Agent Loops 2024-2025 PRD-driven autonomous loops. Agent reads spec, implements, tests, iterates.
Wave 5 Multi-Agent 2025 Multiple specialized agents. Orchestration and delegation. Gas Town lives here.
Wave 6 Autonomous Future Self-directing agent systems. Minimal human oversight. 
Wave 1: AI completes YOUR code
Wave 2: AI answers YOUR questions
Wave 3: AI edits YOUR files
Wave 4: AI follows YOUR spec
Wave 5: AI coordinates WITH you     ← Gas Town
Wave 6: AI develops FOR you
Section 07 — Doctrine

Core Principles

Principle 01
Roles Over Tools
"Do not think about what tools to use. Think about what ROLES you need."
Instead of "I will use Claude to write tests," think: "I need a Tester role that owns quality assurance."
Principle 02
Memory is Architecture
"Without persistent memory, multi-agent is just parallel single-agent with extra steps."
Beads are not optional — they are what makes Gas Town more than "run Claude in multiple terminals."
Principle 03
Progressive Delegation
Start with 2-3 roles. Add complexity as patterns stabilize. Day 1: Mayor + Builder. Week 1: add Architect + Tester. Month 1: full 8-role deployment.
Principle 04
The Mayor Never Codes
"The moment you start coding, you have abandoned your role as Mayor."
The Mayor's job is strategic: priorities, architecture decisions, quality standards. Coding is the Builder's job.
Principle 05
Interfaces Before Implementation
Each role communicates through defined interfaces: Architect → Builder via ADRs. Builder → Critic via Pull Requests. Researcher → Architect via Research summaries. Tester → Builder via Test reports.
Principle 06
Fail Fast, Delegate Faster
"If an agent gets stuck for more than 5 minutes on the wrong approach, kill it and start fresh. Context is cheap."
Do not debug agent confusion. Restart with clearer instructions.

Section 08 — Field Operations

Implementation Guide

Starting Gas Town (Minimum Viable Setup)

Step 1: Create Project Structure

Project Structure
your-project/ ├── .claude/ │ ├── CLAUDE.md # Project-level instructions │ ├── commands/ │ │ ├── architect.md # Architect role prompt │ │ ├── builder.md # Builder role prompt │ │ ├── tester.md # Tester role prompt │ │ └── researcher.md # Researcher role prompt │ └── beads/ │ ├── context/ # Context beads │ ├── decisions/ # Decision beads │ └── progress/ # Progress beads ├── src/ └── tests/

Step 2: Write Role Prompts

Each role gets a slash command that sets context. Here is an example for the Architect role:

.claude/commands/architect.md
You are the Architect for this project. Your responsibilities: 1. Design system structure and interfaces 2. Write Architecture Decision Records in .claude/beads/decisions/ 3. Define component hierarchies and API contracts 4. Review Builder's implementations for architectural compliance Your constraints: - Do NOT write implementation code (that's the Builder) - Do NOT write tests (that's the Tester) - ALWAYS document decisions as Decision Beads - FOCUS on interfaces, not implementations Current project state: Read .claude/beads/context/ for latest context.

Step 3: Establish Memory Beads

Create initial context beads to establish the project baseline:

.claude/beads/context/project-overview.md
# Context Bead: Project-Overview ## Created: [date] ## Status: ACTIVE ### Project Description [What you're building] ### Tech Stack [Technologies chosen] ### Architecture [High-level architecture] ### Current Phase [Where you are in development]

Step 4: Run the Town

# Terminal 1: Mayor (you, making decisions)
claude

# Terminal 2: Architect (designing)
claude --command /architect

# Terminal 3: Builder (implementing)
claude --command /builder

# Terminal 4: Tester (testing)
claude --command /tester

Scaling Gas Town

Scaling Pattern

For large tasks, spawn multiple Builders with domain specialization. Use git worktrees for parallel operations:

# Each builder gets their own worktree
git worktree add ../project-builder1 feature/auth
git worktree add ../project-builder2 feature/api
git worktree add ../project-builder3 feature/database

Section 10 — Troubleshooting

Troubleshooting

Problem: Agents Do Not Read Beads
Cause: Bead paths not in CLAUDE.md or role prompts
Fix: Add explicit bead reading instructions to each role prompt — "BEFORE starting any task: Read all files in .claude/beads/context/"
Problem: Roles Overlap
Cause: Unclear role boundaries
Fix: Make each role prompt include "You do NOT do X" constraints. Explicit negative boundaries prevent territorial confusion.
Problem: Memory Gets Stale
Cause: No one updating beads
Fix: Add bead maintenance to each role: "After completing any task: Update relevant context beads, create decision beads for choices made, update progress bead."
Problem: Too Many Cooks
Cause: Scaling too fast
Fix: Start with Mayor + Builder. Add Architect when architecture matters. Add Tester when code exists. Add others as needed.
Section 11 — Operational Assessment

When to Use Gas Town

Gas Town Shines When
  • Projects have 5+ interconnected files — coordination matters
  • Multiple concerns run in parallel — frontend + backend + tests + docs
  • Context exceeds single-session capacity — too much for one chat
  • Quality requires different perspectives — builder vs critic vs tester
  • Projects span multiple sessions — beads maintain continuity
Gas Town is Overkill When
  • Single-file scripts — just use Claude directly
  • Quick bug fixes — one session is fine
  • Exploratory prototyping — too much structure kills exploration
  • You are learning the codebase — explore first, organize later
The Sweet Spot
Gas Town works best for projects that take 2-20 hours of developer time. Too small and the overhead kills you. Too large and you need actual human team coordination.
