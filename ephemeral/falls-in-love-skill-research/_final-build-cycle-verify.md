## Verification 3: Remaining Gaps

After reading all 1001 lines, these are the top 3 remaining gaps that would most impact a $150-300, 6-10 hour build:

### Gap 1: No Context Window Transition Protocol (HIGHEST IMPACT)

**What's missing:** The skill acknowledges context window limits (lines 943-975) and says the build spans "8-12 context windows." It describes what carries forward (HTML, notebook, retrospectives) and gives a reading order for new windows. But it does NOT address the operational mechanics of HOW the transition happens.

**Specifically missing:**
- Does the user manually start a new conversation and paste the skill? Or does the agent use `/compact`? Or `/clear`? Or does the user invoke the skill again?
- At what point should the agent signal "I am approaching context limit, prepare for transition"?
- Should the agent write a specific HANDOFF file (beyond the notebook) that includes: current wave, current pass number, current rotation position, what was just done, what needs to happen next?
- How does the agent resume mid-wave? If it is at Wave 3 pass 7 when the context window fills, does the next window start at Wave 3 pass 7 or Wave 3 pass 1?

**Where it should go:** Lines 943-975 (Context Window Management section). A subsection called "Transition Protocol" with: (a) the signal the agent gives when approaching limit, (b) the handoff file format, (c) the resume procedure.

**Why it matters:** A $150-300 build spanning 8-12 context windows WILL hit this problem 7-11 times. Without a protocol, each transition is ad-hoc, and the user must figure out how to restart. This is the single largest operational gap.

### Gap 2: No Explicit "How to Start" / Invocation Protocol

**What's missing:** The skill describes what the agent receives (lines 153-170) and the process (lines 180+). But it never says what the user actually TYPES to start the build, or what the agent does in the FIRST 5 minutes.

**Specifically missing:**
- What is the expected invocation? `/falls-in-love [path-to-content.md]`? Or does the user paste the content? Or point to a URL?
- Where does the output HTML file go? The skill says "One HTML file" (line 174) but never specifies the output path. Should it go in `ephemeral/`? A user-specified directory? Does the agent create an output folder?
- The builder's notebook — where does it live? What is its filename? Is it created automatically?
- Does the agent read ALL input files (identity.md, vocabulary.md, tokens.css, components.css, content) in the first pass, or does it follow the wave schedule strictly (content-only in Wave 0)?

**Where it should go:** Between the Input/Output section (lines 153-177) and The Process section (line 180). A section called "Getting Started" or "First 5 Minutes" with: (a) invocation syntax, (b) output directory creation, (c) initial file reads, (d) notebook creation.

**Why it matters:** Without this, the user's first experience is confusion about how to actually start, and the agent may read design system files during Wave 0 (when the skill explicitly says "No design system. No research. No vocabulary.").

### Gap 3: No Guidance on What to Do When Content Lacks Spatial Potential

**What's missing:** The skill mentions this possibility in several places:
- Line 30: "NOT applicable to all content. Some content is genuinely linear."
- Line 48: "Content with internal structural tension generates the state. Flat procedural content does not."
- Lines 291-292: "If you cannot build a loss-detection table for the content, the content may not have spatial potential."

But it never says what the agent should DO if it determines (say at Wave 0 pass 4) that the content genuinely lacks spatial potential. Does it:
- Tell the user and stop?
- Fall back to a standard build?
- Continue anyway at reduced depth?
- Build at middle-tier quality without the devoted process?

**Where it should go:** At the end of Wave 0 (after line 429). A brief "Early Exit" or "Spatial Potential Assessment" protocol: if the loss-detection table yields fewer than N entries by pass 6, the agent reports to the user with its assessment and asks whether to continue with the devoted process or switch to a standard build.

**Why it matters:** At $150-300 per build, spending 6-10 hours on content that the skill itself says is "genuinely linear" is an expensive mistake. An early detection and off-ramp saves the user significant cost.

### Honorable Mentions (Not Top 3 But Worth Noting)

- **No error recovery for Playwright failures.** The screenshot protocol (lines 825-848) is clean but does not address what happens when screenshots come back blank (which the user's MEMORY.md documents as a recurring issue). A retry-with-verification step would help.
- **Wave 3 case study file references may be stale.** The skill references specific case study files (DD-003-islands.md, etc.) and visual companions. If these files have been renamed or restructured since the skill was written, the agent will fail silently. No verification step exists.
- **No cost tracking or progress reporting.** Over 6-10 hours, the user has no visibility into progress. A brief mid-wave status ("Wave 3, pass 8 of ~14, approximately 45% through total build") would help the user understand where the build stands.
