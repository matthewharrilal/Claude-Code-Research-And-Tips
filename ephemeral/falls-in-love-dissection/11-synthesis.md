## PART 1: THE COMPLETE MECHANISM

### 1.1 The Unified Model in One Page

Ten reports, three thousand lines. Here is what they all describe, from different angles, as one phenomenon:

**An Opus-class model, given sufficient unmediated exposure to structured content under a hypothesis-driven frame, forms a persistent mapping from content semantics to spatial geometry. This mapping -- the content topology -- then dominates all subsequent CSS decisions, producing output characterized by content-semantic naming, asymmetric proportions encoding hierarchy, self-referential components, layered z-composition, non-default reading directions, and structural diversity. The topology forms THROUGH building, not before it. It requires a phase transition at 30-60 minutes of sustained engagement where the builder's relationship to the content shifts from "material to be placed" to "argument to be embodied." The resulting output is categorically different from procedural output across every measured dimension (10/10 vs 0/10 on falsifiable criteria). Beyond the known ceiling, the frontier lies in temporal spatial forms (scroll-driven layout evolution), container-aware components, cross-section spatial dialogue, and CSS capabilities the explorations never used.**

That is the mechanism. The rest of this document unpacks it.

### 1.2 The Three Faces Are One Phenomenon

The CLAUDE.md framed three inseparable questions: Discovery, Sustained Engagement, Visual Maturity. After 10 reports, we can see they are not three things but three temporal views of ONE process:

| Time | What is happening | Which "question" this answers |
|------|-------------------|-------------------------------|
| 0-15 min | Builder scans content, forms surface impressions | None yet -- this is pre-engagement |
| 15-30 min | Builder re-reads with spatial vocabulary available, forms multi-hop hypotheses | **Discovery** begins (structural recognition) |
| 30-60 min | Builder writes CSS that forces content-spatial integration; topology crystallizes | **Discovery** peaks (invention), **Engagement** crosses threshold, **Maturity** components 1-4 activate |
| 60-120 min | Builder evaluates own output, discovers emergent visual grammar, achieves cross-section coherence | **Engagement** deepens, **Maturity** components 5-8 activate |
| 120-180 min | Evaluation feedback + iterative correction; perceptual polish | **Maturity** components 9-11 activate |
| 180+ min | System-level integration; findings extraction | All three complete |

**Discovery is what the builder FINDS. Engagement is HOW LONG the builder stays. Maturity is what the OUTPUT looks like.** They are cause, duration, and effect of the same process. You cannot have discovery without engagement (you need time to form multi-hop associations). You cannot have maturity without discovery (maturity requires content-driven spatial decisions). You cannot have engagement without the content activating spatial reasoning (engagement is not just time -- it is time spent forming and testing topology).

### 1.3 The Invention Taxonomy Maps to the Depth Curve

Report #7 cataloged 18 genuine inventions across 5 explorations and classified them into 5 types. Report #8 mapped 6 phases of depth accumulation. These are not independent findings. The invention types emerge at specific depth phases:

| Invention Type | Definition | Depth Phase Required | Example |
|---------------|-----------|---------------------|---------|
| **Type A: Content-topology** | Spatial form that embodies content meaning | Phase 2-3 (hypotheses to topology) | Confidence-mapped border colors, golden ratio for progressive depth |
| **Type B: Self-referential** | Component that references the page's own structure | Phase 3 (topology formation) | Pattern echo grid, wave density bar chart, scroll witness |
| **Type C: Compositional** | Multiple CSS features compose into emergent capability | Phase 3-4 (topology to coherence) | Fractal annotation (grid + writing-mode + sticky + border = marginalia) |
| **Type D: Proportional** | Non-trivial mathematical relationships encoding meaning | Phase 2-3 | Golden ratio columns (1.618fr), 3:1 asymmetric splits |
| **Type E: Vocabulary expansion** | CSS technique used in a structurally novel way | Phase 4-5 (coherence to polish) | Scroll-driven progress markers (@property + animation-timeline + scaleY) |

**The key relationship:** Type A and Type D inventions can emerge at Phase 2 (hypotheses), but they only become NECESSARY -- meaning removing them loses content meaning -- at Phase 3 (topology). Before Phase 3, the builder might use golden ratio "because it looks nice." After Phase 3, the builder uses golden ratio "because progressive depth demands proportional narrowing." The same technique, but the REASON changes at the phase transition.

Type B inventions (self-referential) require Phase 3 because they demand the recursive insight that the page is about what the page does. The pattern echo grid requires understanding that a page about axis patterns IS a compound axis pattern. The wave bar chart requires understanding that the page has a density rhythm that can be visualized. These are meta-cognitive insights that only emerge after deep content engagement.

Type C inventions (compositional) require Phase 3-4 because they require multiple CSS features to be combined in service of a single content-driven purpose. The fractal annotation margin combines grid, writing-mode, sticky positioning, border, opacity transition, and rotation into a single emergent system. This composition requires both deep content understanding (what the marginalia serves) and accumulated CSS vocabulary (knowing all 6 features and how they compose). Phase 4's cross-section coherence is what produces this kind of synthesis.

| 4. Spatial invention | Layouts that exist only for this content | Phase 3 | Requires the "no existing pattern fits" insight |

The implication for architecture: a single 40-minute build pass can produce components 1-6. Components 7-11 require evaluation feedback and additional building passes. The three-pass architecture (BUILD-1 -> EVALUATE -> BUILD-2 -> EVALUATE -> BUILD-3) is not arbitrary -- it maps to the maturity component emergence sequence.

### 1.5 The Four Critical Questions Answered

**Q1: What is the MINIMUM cognitive state that produces genuine spatial invention?**

The minimum is Phase 2→3 transition: the builder must have read the content deeply enough to form multi-hop associations (Phase 2) AND have begun writing CSS that forces those associations to crystallize into persistent spatial commitments (Phase 3). The specific minimum conditions:

1. The builder has read the full article at least once without procedural framing
2. The builder has spatial vocabulary available (curated research, not mechanism names)
3. The builder has begun writing CSS (topology forms THROUGH building)
4. The builder has spent at least 30 minutes in combined reading + building

Conditions 1 and 2 are achievable in the first 15 minutes. Condition 3 begins at minute 15-20. Condition 4 sets the floor. The minimum viable "falls in love" is therefore approximately 30 minutes of content reading + CSS writing, with spatial vocabulary available.

This is the MINIMUM. AD-006-level output requires 60-120 minutes (Phase 3-4). Beyond-AD-006 requires 120+ minutes (Phase 4-5).

**Q2: What is the RELATIONSHIP between discovery, engagement, and maturity?**

They are not sequential but CONCURRENT AND MUTUALLY REINFORCING:

```
Discovery ←──────→ Engagement
    ↑                    ↑
    │                    │
    └──── Maturity ──────┘
```

- Discovery drives engagement: finding a spatial insight motivates deeper exploration
- Engagement enables discovery: spending more time creates opportunities for multi-hop associations
- Both produce maturity: discovery creates structural foundation, engagement deepens it
- Maturity enables further discovery: achieving visual grammar coherence (component 7) reveals new spatial possibilities that were invisible at lower maturity levels

The relationship is a positive feedback loop. This is why the phase transition at 30-60 minutes is so dramatic: once the loop begins, each element accelerates the others. Before the loop starts (Phase 0-1), the builder is doing procedural work. After the loop starts (Phase 3+), the builder's output quality accelerates non-linearly.

**Q3: What does "beyond AD-006" ACTUALLY require?**

Report #10 identified 11 CSS capabilities the explorations never used. But capabilities are not the bottleneck -- cognitive state is. The beyond-AD-006 frontier requires:

1. **Phase 4+ engagement** (120+ minutes): AD-006 represents Phase 3-4 output (~2-3 hours total build lifecycle). Beyond AD-006 requires Phase 4-5, which adds cross-section dialogue, reader-state-aware layout, and multi-scale responsive strategies.

2. **Temporal spatial thinking**: The explorations treat layout as STATIC. Beyond AD-006 requires thinking of layout as TEMPORAL -- spatial forms that evolve as the reader scrolls (scroll-driven animations), components that respond to their container (container queries), and cross-section spatial relationships (subgrid, anchor positioning).

3. **The specific capabilities**: Container queries, scroll-driven animations, subgrid, clip-path compositions, anchor positioning, color-mix(), and advanced writing-mode. These are available today (90%+ browser support for most) but require a mental model shift from "sections are independent rectangles" to "sections are temporal episodes in a spatial narrative."

4. **Vocabulary for the builder**: The curated research package must include beyond-ceiling examples. Not mandated -- available. The builder who reaches Phase 4 will naturally reach for these techniques if they exist in the vocabulary. The builder who never reaches Phase 3 will not use them regardless.

The cognitive state that produces beyond-AD-006 is not a different KIND of state from the "falls in love" state. It is the same state at greater depth -- more time, more iteration, more vocabulary. The difference is quantitative (more engagement) and qualitative (temporal rather than static spatial thinking).

**Q4: Is the "falls in love" state BINARY (on/off) or CONTINUOUS (degrees)?**

**Both.** The depth curve (Report #8) shows continuous improvement. But there is a STEP FUNCTION at Phase 2→3 (the 30-60 minute mark) where output quality jumps from 0-2/10 to 7+/10 on falsifiable criteria. The resolution:

- The CAPABILITY to produce spatially inventive output is BINARY. Before the phase transition, the builder cannot produce content-topology coupling. After it, the builder can. This is the "on/off" switch.
- The QUALITY of spatially inventive output is CONTINUOUS. After the phase transition, more time produces higher quality: better coherence (Phase 4), better polish (Phase 5), better system integration (Phase 6). This is the gradient.

The analogy is water freezing. The phase transition (liquid to solid) is binary. The quality of the ice crystal (clarity, structure, beauty) is continuous and improves with controlled conditions. The "falls in love" phase transition ENABLES spatial invention. Subsequent engagement DEEPENS it.

The practical implication: the architecture must GUARANTEE reaching Phase 3 (the binary threshold) and then MAXIMIZE time in Phase 4-5 (the continuous gradient). Reaching Phase 3 unreliably is worse than not reaching it at all, because a build that spends $150 and lands at Phase 2 (5-6/10) is a worse outcome than a $30 pipeline build at Phase 0 (0/10 but expected).

### 1.6 The 10 Falsifiable Criteria Re-Evaluated

Report #2 proposed 10 criteria that distinguish "in love" output from procedural output. After 8 more reports of deeper analysis, how do these hold up?

| Criterion | Original Assessment | Post-Synthesis Assessment |
|-----------|-------------------|--------------------------|
| 1. Named grid areas (content vocab) | Valid | **VALID but insufficient.** A builder can add `grid-template-areas: "left right"` to satisfy the criterion without content engagement. The criterion detects presence but not AUTHENTICITY. |
| 2. Asymmetric grid proportions | Valid | **VALID.** Hard to fake. `1.618fr` or `3fr 1fr` requires a specific content-driven rationale. |
| 3. Layout loss = content loss | Valid | **THE STRONGEST CRITERION.** Cannot be gamed. If removing the layout does not damage the content's argument, the layout is decorative regardless of its complexity. |
| 4. Self-referential components | Valid | **VALID and diagnostic.** Self-referential components require Phase 3 engagement. They cannot emerge from procedural building because they require the builder to notice that the page is about what it does. |
| 5. Position relative/absolute pairs | Valid | **VALID but low-specificity.** Positioned composition can exist for decoration (parallax effects) or for content meaning (fractal annotations). The criterion detects z-axis usage but not z-axis MEANING. |
| 6. Content-mapped color semantics | Valid | **VALID.** Color-as-data (confidence gradients) cannot be faked by adding accent colors randomly. Requires understanding the content's epistemological structure. |
| 7. Distinct spatial organizations | Valid | **VALID with a needed ceiling.** Report #6 (FM-7) warns that evaluators reward diversity without penalizing excess. Add: "distinct spatial organizations must each be JUSTIFIED by the content they serve." |
| 8. Transition/animation density | Valid | **WEAKEST CRITERION.** Transitions and animations can be added mechanically without content engagement. A builder can add 6 hover states that serve no content purpose. Consider replacing with "content-driven temporal responsiveness" (scroll progress, section-appropriate animation speeds). |
| 9. CSS comments = content reasoning | Valid | **VALID and valuable for diagnosis.** Content-reasoning comments are a reliable signal of engagement because the builder must articulate WHY a spatial choice serves the content. Cannot be faked without actually understanding the content. |
| 10. Non-default reading directions | Valid | **VALID.** `writing-mode: vertical-lr` or `direction: rtl` for content-driven purposes requires understanding what the reading direction MEANS for this content. |

**Net assessment:** Criteria 3, 4, 6, and 9 are the most reliable. Criteria 1 and 8 are the most gameable. Criteria 2, 5, 7, and 10 are valid but need refinement.

**The meta-insight from Report #6 (FM-7):** These criteria should be used for EVALUATION, not communicated to the BUILDER. When the builder sees the criteria, they become targets to optimize. When evaluators use the criteria privately, they remain diagnostic instruments. Report #8's proposed modification -- qualitative feedback, not scores -- is correct.

### 1.8 The Mechanism in Its Full Form

Synthesizing all 10 reports, the complete mechanism is:

**PRECONDITIONS:**
1. Opus-class model (Sonnet may comply without inventing; Opus invents within compliance)
2. Structured content with internal argument, tension, or recursion (linear/procedural content does not activate the mechanism)
3. Hypothesis-driven framing ("What spatial form does this content demand?" not "Place this content in zones")
4. Direct, unmediated access to the article before any procedural framing
5. Spatial vocabulary available as reference (curated research with CSS examples, not mechanism names)
6. Absence of procedural interruption during the creative phase (no deployment quotas, no reading-order mandates, no "build vertical first")

**THE PROCESS:**
1. Content scanning (0-5 min): Topic identification, section count, surface features
2. Structural recognition (5-15 min): Argument structure, emotional arc, content types per section
3. Content-form hypotheses (15-30 min): Multi-hop associations from content to spatial form. Spatial vocabulary from curated research enables projection.
4. **PHASE TRANSITION (30-60 min):** Content-topology formation. The builder writes CSS that forces hypotheses to crystallize into persistent spatial commitments. Grid area names reference content. Proportions encode hierarchy. Self-referential components emerge. The topology becomes the dominant decision frame. **This is the "falls in love" state.**
5. Compositional deepening (60-120 min): Evaluation feedback + continued building. Cross-section visual grammar emerges. Border categories, spacing systems, typographic authority become consistent. Responsive intelligence develops.
6. Iterative refinement (120-180 min): Perceptual calibration. Micro-responsive adjustments. Polish.
7. System integration (180+ min): Findings extraction. Convention codification. Vocabulary expansion.

**THE OUTPUTS:**
- Falsifiable criteria: 0/10 before phase transition, 7+/10 after, 9+/10 with evaluation cycles
- Visual maturity components: 0-2/11 before phase transition, 4-6/11 immediately after, 8-11/11 with sustained engagement
- Invention count: 0 before phase transition, 3-6 at Phase 3, 8-12 at Phase 4, 12-18 at Phase 5

**THE KILLERS (in order of destructiveness):**
1. "Build vertical first" mandate (prevents topology-foundational building)
2. Structural proposition menu from upstream agent (replaces builder's own spatial reasoning)
3. Maximum 2 structural sections cap (truncates any topology richer than 2 ideas)
4. Mandated reading order with package first (preempts independent content engagement)
5. Pass 1 builder having narrowest information access (impoverishes spatial vocabulary)
6. Deployment quotas and citation-at-write-time (consume attention budget)
7. Cultural signals framing structure as risky (self-censorship)

## PART 2: ARCHITECTURE IMPLICATIONS

### 2.1 The Central Finding: Topology Forms THROUGH Building

Report #8's most important finding: "Phase 3 cannot be reached by reading alone. It requires CSS." The content-topology crystallizes in the act of writing `grid-template-areas: "spiral-wide spiral-wide spiral-wide"` -- the naming forces the builder to DECIDE what each area means, and that decision IS the topology.

**Implication for the immersion phase:** The immersion agent (Phase 3 of the 11-phase architecture) forms hypotheses, not topology. It reaches Phase 2 on the depth curve. Only the builder reaches Phase 3, because only the builder writes CSS. The immersion agent produces USEFUL input (spatial hypotheses, content analysis, structural observations) but it does not and cannot produce the "falls in love" state.

**Should the immersion agent be eliminated?** No. But its role must be correctly understood:

- The immersion agent is a VOCABULARY AMPLIFIER, not a topology generator. It reads the content deeply and translates content structure into spatial possibilities that the builder can consider. It expands the builder's hypothesis space.
- The immersion agent is a TIME SAVER, not a time substitute. Without it, the builder spends the first 15-20 minutes doing work the immersion agent already did. With it, the builder starts at Phase 2 (hypotheses already formed by immersion agent's document) and can reach Phase 3 faster.
- The immersion agent is a FRESH-EYES content analyst. The builder will form its own content understanding during building, but the immersion agent's analysis may notice structural properties the builder misses because the builder is simultaneously managing CSS complexity.

The immersion agent should NOT be framed as "the agent that falls in love so the builder doesn't have to." It should be framed as "the agent that gives the builder a 15-minute head start on content understanding."

### 2.2 The Telephone Problem and Its Resolution

Report #6 (FM-3) and Report #8 converge on the same finding: the topology forms in the agent that writes CSS. Transferring topology via document is lossy -- the document describes hypotheses (Phase 2), not topology (Phase 3). The builder who reads the topology document is an IMPLEMENTER of someone else's hypotheses, not an EXPLORER forming their own topology.

Report #6's adversarial review proposed M-1A: merge Phases 3+4+5 into a single agent. This eliminates the transfer problem entirely. One agent reads the content, forms hypotheses, and builds -- all in one context window.

**But there is a cost:** The merged agent must spend its first 20-30 minutes on immersion/discovery before writing any CSS. During those 20-30 minutes, it is doing work that a separate immersion agent could have done in parallel with the curation agents. The merged approach adds 20-30 minutes to the critical path.

**The resolution: a hybrid.** The immersion agent runs in Phase 3 as proposed. Its output goes to the builder. But the builder's prompt is reframed:

```
The immersion agent's document is a SECOND OPINION, not a blueprint.

YOUR FIRST TASK: Read the article yourself. Form your OWN spatial
impressions. Write them down in 3-5 bullet points.

YOUR SECOND TASK: Read the immersion agent's topology document.
Where does it align with your impressions? Where does it diverge?
Where does it see something you missed?

YOUR THIRD TASK: Build from YOUR OWN understanding, informed by
the immersion agent's analysis where it is compelling.
```

This preserves the immersion agent's value (vocabulary amplification, fresh-eyes analysis) while ensuring the builder forms its OWN topology through building. The builder is not implementing someone else's plan. It is building from its own understanding, with the benefit of an additional perspective.

### 2.3 What Gets the Builder to Phase 3 Fastest?

Report #8 identifies the activities that produce the fastest phase transitions:

- **Phase 0→1 (scanning→recognition):** Reading the article ONCE, FULLY, without procedural framing. Content FIRST.
- **Phase 1→2 (recognition→hypotheses):** Reading the article a SECOND time with spatial vocabulary available.
- **Phase 2→3 (hypotheses→topology -- THE critical transition):** BUILDING. Not planning, not reading more -- writing CSS.

**The fastest path to Phase 3:**
1. Builder reads the article (10 minutes). Unframed. No package, no brief, no constraints.
2. Builder reads the immersion agent's spatial analysis + curated structural research (5 minutes). This provides vocabulary and a second perspective.
3. Builder begins writing CSS immediately (minute 15). Not a transition table. Not a skeleton. CSS for the FIRST section that the builder feels strongest about.
4. Builder continues building section by section, re-reading the article as each section demands (minutes 15-40).
5. By minute 30-40, the topology has crystallized through the accumulated decisions.

**What this means for the architecture:** The BUILD-1 prompt should NOT ask the builder to produce a plan before writing CSS. It should ask the builder to START WRITING CSS after reading the article. The planning happens IN the CSS -- grid area names, proportional choices, component designs. These ARE the topology. Asking the builder to plan first is asking it to produce Phase 2 hypotheses on paper and then implement them, skipping the Phase 2→3 transition that only happens through building.

### 2.4 Which Maturity Components Need Explicit Support?

Of the 11 atomic maturity components (Report #9), which emerge naturally from content engagement, and which need explicit scaffolding?

**Emerge naturally from Phase 3 engagement (no explicit support needed):**
- Component 1: Proportional confidence -- emerges when the builder understands content hierarchy
- Component 4: Spatial invention -- emerges when no existing pattern fits the content
- Component 6: Typographic authority -- emerges when the builder hears the content's voice
- Component 8: Content-semantic naming -- emerges when the builder understands content enough to name it

**Emerge with vocabulary support (curated research must include examples):**
- Component 2: Z-axis mastery -- the builder must KNOW that `position: sticky` and layered z-index exist and can serve content meaning. Include 2-3 exploration examples in curated research.
- Component 3: Temporal responsiveness -- the builder must KNOW that scroll-driven animations, hover transitions, and @keyframes can make the page alive. Include examples.
- Component 10: Compositional layering -- the builder must have SEEN examples of 3+ CSS features composing into emergent effects (grid + writing-mode + sticky = marginalia). Include the fractal annotation example.

**Require evaluation feedback (cannot emerge in first build pass):**

- Component 11: Micro-perceptual calibration -- requires evaluator feedback on specific thresholds (15 RGB delta, 0.025em tracking)

**Architecture implication:** The curated research package must include examples of components 2, 3, and 10. The evaluation agents must specifically assess components 5, 7, 9, and 11. Components 1, 4, 6, and 8 will emerge naturally if the preconditions are met.

### 2.5 Beyond-AD-006 Techniques: Available Without Mandated

Report #10 identified 11 CSS capabilities the explorations never used. The question is how to make these available to the builder without creating a new constraint system.

**The answer: include them in the curated research as EXAMPLES, not as REQUIREMENTS.**

The textural researcher (Phase 2, Agent 2B) should include a section titled "FRONTIER TECHNIQUES" with 3-5 specific beyond-ceiling CSS patterns:

1. **Container queries** -- one example showing a component that renders differently based on its container width (compact in sidebar, expanded in main column)
2. **Scroll-driven animations** -- one example showing scroll-position-driven layout evolution (margins narrowing, colors deepening, borders thickening as you scroll deeper)
3. **Subgrid** -- one example showing nested elements aligning with a parent grid's tracks
4. **clip-path** -- one example showing non-rectangular section boundaries encoding content meaning

These are presented as vocabulary, not mandates. The builder who reaches Phase 3-4 may naturally reach for them when the content demands it. The builder who does not reach Phase 3 will ignore them, which is correct -- beyond-ceiling techniques deployed without content engagement produce decoration, not design.

**Critical constraint: do NOT include frontier techniques in evaluation criteria.** If evaluators check for container queries, the builder will add them to satisfy the check. Frontier techniques must emerge from content engagement, not from compliance pressure.

### 2.6 FM-14 Texture Regression: How to Ensure Both Structure AND Texture Deepen

Report #6 identified FM-14: the risk that structural invention consumes the entire CSS budget, leaving texture flat. Report #9's maturity decomposition reveals why: structural maturity (components 1-4) and textural maturity (components 5-8) emerge at DIFFERENT phases and require DIFFERENT inputs.

**The problem:** A builder in the "falls in love" state naturally prioritizes STRUCTURE (content-topology coupling, spatial invention) over TEXTURE (border systems, spacing rhythms, typographic variation). AD-006 spent 716 lines of creative CSS on spatial structure. If the builder is freed from all constraints, it may spend 100% of its creative attention on structure and produce a page with magnificent spatial logic and generic borders, uniform spacing, and flat typography.

**The solution: the textural research package is the texture floor.**

The curated textural research (Phase 2, Agent 2B) provides specific CSS values for:
- Border treatments mapped to the TC metaphor (e.g., "Use 3px solid for structural boundaries, 1px solid for internal separators")
- Spacing rhythms mapped to content density (e.g., "64px between conceptual shifts, 32px within groups, 16px between related elements")
- Color semantics mapped to content meaning (e.g., "Blue borders for validated claims, amber for exploratory, purple for speculative")
- Typographic variation mapped to content register (e.g., "Display serif for wisdom/philosophy, mono for technical/code, sans for standard prose")

The builder receives this as VOCABULARY, not as mandates. But the textural research is rich enough that a builder engaging with it will naturally integrate textural decisions into structural building. The key is that textural vocabulary must be as implementation-specific as structural vocabulary -- not "use border-weight gradient" but "use `border-left: 4px solid var(--accent-blue)` for foundational claims."

**The evaluation agents also address texture:** The visual maturity evaluator (Phase 6, Agent 6B) specifically assesses components 5-8 (semantic restraint, typographic authority, border architecture, content-semantic naming). If the BUILD-1 output has strong structure but weak texture, the evaluation feedback directs BUILD-2 to deepen texture alongside structure.

The Report #6 adversarial review's M-2B (minimal breadth floor) is also valuable: three lines in the BUILD-1 prompt requiring at least 3 distinct zone backgrounds, 2 meaningful border treatments, and 2 hover/transition states. These are floors that prevent texture blindness without constraining structural creativity.

## PART 3: THE REVISED ARCHITECTURE

### 3.1 Should Phases 3-5 Be Merged?

Report #6's strongest recommendation is M-1A: merge Immerse + Discover + Build into a single agent. The arguments:

**FOR merging:**
- Eliminates the telephone problem (FM-3)
- Topology forms in the agent that writes CSS
- Reduces cost by $14 (no separate immersion and discovery agents)
- Mirrors the exploration era (one builder, extended engagement)

**AGAINST merging:**
- The single agent must spend 20-30 minutes on immersion before writing CSS, adding to the critical path
- The agent's context window must hold: article + curated research + convention spec + immersion thinking + CSS output. For long articles, this may exceed the window.
- No external challenge to the topology (Phase 4's discovery agent provides a second opinion)

**The recommendation: MERGE Phases 3+5 (Immerse + Build), KEEP Phase 4 (Discover) as a pre-builder checkpoint.**

The revised flow:
```
Phase 2: CURATE (3 agents parallel)          [25 min, $12]
Phase 3: DISCOVER (topology pre-check)       [15 min, $5]
Phase 4: BUILD-1 (immerse + build, merged)   [50 min, $22]
```

The DISCOVER agent (formerly Phase 4) reads the article and curated research and produces a SHORT (50-100 line) structural analysis:
- 3 spatial observations about the content's argument structure
- 2 potential self-referential opportunities
- 1 "what is unique about this article that no standard layout captures"

This document is ONE of the builder's inputs. It is a spark, not a blueprint. The builder forms its OWN topology through building, informed by the structural analysis.

The BUILD-1 agent (formerly Phase 5, now incorporating Phase 3's immersion) spends its first 15 minutes reading the article and the structural analysis, then builds for 35 minutes. Total: 50 minutes. This is sufficient to reach Phase 3 (the phase transition) and begin Phase 4 (coherence).

**Cost savings:** $14 -> $5 for discovery. $18 -> $22 for merged build. Net: -$5. But more importantly: the topology forms in the builder.

### 3.2 How Many Build Passes Are Actually Needed?

The 11-phase architecture proposed 3 build passes (BUILD-1, BUILD-2, BUILD-3). Report #8's depth curve suggests:

- BUILD-1 (50 min): Reaches Phase 3-4. Produces structural foundation + initial coherence. Quality: 6-8/10.
- EVALUATE-1: Identifies weakest maturity components, compliance violations, texture gaps.
- BUILD-2 (30 min): Deepens from Phase 4 into Phase 5. Addresses evaluation feedback. Quality: 8-9/10.
- EVALUATE-2: Full PA with prescriptive feedback.
- BUILD-3 (20 min): Polish and hardening. Quality: 9-10/10.

**Are all three passes needed?** Report #6's cost analysis shows the first build-evaluate-deepen cycle (BUILD-1 + EVAL-1 + BUILD-2) captures approximately 80% of the quality improvement. BUILD-3 adds polish but is in diminishing-returns territory.

**Recommendation: 2 mandatory passes, 1 optional.**

```
Phase 4: BUILD-1 (immerse + build)      [50 min, $22]  -- reaches Phase 3-4
Phase 5: EVALUATE-1 (3 agents)          [20 min, $12]  -- prescriptive feedback
Phase 6: BUILD-2 (deepen)               [30 min, $14]  -- reaches Phase 4-5
Phase 7: EVALUATE-2 (weaver + PA)       [25 min, $18]  -- final assessment
Phase 8: BUILD-3 (optional polish)      [20 min, $10]  -- only if EVAL-2 identifies fixable issues
Phase 9: FINAL CHECK (1 agent)          [10 min, $4]   -- compliance verification
```

Total mandatory: $87 (Phases 0-7 + 9)
With optional BUILD-3: $97
With content triage routing simple articles to a reformed pipeline ($30): effective average drops further.

### 3.3 What Does the Builder's Prompt Look Like?

Given the complete mechanism, the BUILD-1 prompt should:

1. **Put the article first.** Before any other input.
2. **Ask for personal spatial impressions before showing the structural analysis.** The builder forms its own Phase 1-2 understanding.
3. **Provide spatial vocabulary as reference, not instructions.** Curated research is available for consultation, not mandated for deployment.
4. **Start building early.** No transition tables. No skeleton. CSS for the first section that inspires the builder.
5. **Include a minimal texture floor.** 3 zone backgrounds, 2 border treatments, 2 hover states.
6. **Include soul constraints as absolutes.** 5 lines, binary, non-negotiable.
7. **Include frontier techniques as available vocabulary.** Not mandated. Present for the builder who reaches Phase 4 and naturally reaches for them.
8. **Provide space for genuine reflection.** Not a template. 3 open-ended questions.

The prompt structure (approximately 80-100 lines):

```
SECTION 1: FRAMING (5 lines)
"You are building a designed page for this article. You have $22 of
Opus processing time. Spend it wisely: read deeply, then build boldly."

SECTION 2: THE ARTICLE (sent as content.md attachment)
"Read this article fully before doing anything else. Form your own
impressions: What is it about? What does it argue? What surprised you?"

SECTION 3: YOUR SPATIAL IMPRESSIONS (5 lines)
"Before reading any other file, write 3-5 bullet points about what
spatial form this content suggests to you. Be specific: which sections
demand which kinds of spatial treatment?"

SECTION 4: THE STRUCTURAL ANALYSIS (sent as _structural-analysis.md)
"Read this second opinion. Where does it align with your impressions?
Where does it see something you missed? Update your impressions if
compelling."

SECTION 5: AVAILABLE VOCABULARY (sent as _curated-structural.md,
_curated-textural.md)
"These are spatial and textural techniques available to you. Use what
the content demands. Ignore what it doesn't."

SECTION 6: FRONTIER TECHNIQUES (10 lines within textural research)
"Container queries, scroll-driven animations, subgrid, clip-path are
available if the content demands temporal or non-rectangular spatial
forms. Use them only if genuinely warranted."

SECTION 7: ABSOLUTES (10 lines)
"border-radius: 0. box-shadow: none. No gradients. No semi-transparent
backgrounds. Container: 940-960px. Responsive at 1440px and 768px.
Skip link. ARIA landmarks."

SECTION 8: TEXTURE FLOOR (5 lines)
"At minimum: 3 distinct zone backgrounds (delta >= 15 RGB), 2 meaningful
border treatments, 2 hover/transition states."

SECTION 9: BUILD (5 lines)
"Start building the section you feel strongest about. Build zone by zone.
Name grid areas after content, not position. Name classes after meaning,
not appearance."

SECTION 10: REFLECTION (5 lines)
"After building, write to _build-1-reflection.md:
- What spatial decisions surprised you?
- Where did you deviate from the structural analysis?
- What would you do differently with 30 more minutes?"
```

Total: ~80 lines. Content engagement: ~40% (article + impressions + structural analysis). Procedural: ~30% (absolutes + texture floor + output instructions). Creative vocabulary: ~30% (curated research + frontier techniques).

Compare to the current pipeline's 1,410 lines at 87.6% procedural. The ratio is inverted.

### 3.4 How Do Beyond-AD-006 Techniques Become Available?

Three mechanisms, layered:

**Layer 1: Curated research includes examples.** The textural researcher (Phase 2, Agent 2B) includes a "FRONTIER TECHNIQUES" section with copy-paste CSS examples of container queries, scroll-driven animations, subgrid, and clip-path. These are VOCABULARY, not mandates.

**Layer 2: Structural analysis may suggest them.** The discovery agent (Phase 3) may note: "This article's argument structure converges over its length -- the content physically narrows. Scroll-driven animations could make this narrowing visible as the reader scrolls." This is a SUGGESTION the builder can adopt or ignore.

**Layer 3: Evaluation may recommend them.** If BUILD-1 produces strong static spatial structure but the evaluator notices that the content has temporal qualities (building tension, progressive convergence), the evaluator's prescriptive feedback can say: "Consider scroll-driven animations for the convergence section -- the narrowing would be more powerful as a progressive effect than a static layout change." This gives the builder both the TECHNIQUE and the RATIONALE.

**What this avoids:** Mandating frontier techniques ("use at least 1 container query") which would produce compliance-driven decoration. The builder uses frontier techniques only when the content demands them, which is the only condition under which they produce genuine beyond-AD-006 output.

### 3.5 Evaluation Without Criteria Gaming (FM-7)

Report #6 warns that evaluation criteria become the new constraints. Report #8 proposes qualitative feedback, not scores. The complete resolution:

**Evaluators use the 10 falsifiable criteria INTERNALLY** to structure their analysis. They do NOT share scores with the builder. Instead, they translate criteria-based observations into NARRATIVE PRESCRIPTIONS:

Instead of: "Criterion 3: 1/10 (layout loss = content loss fails). FAIL."
The builder receives: "Section 4 compares two paradigms using a symmetric 1fr 1fr grid. The article presents the new paradigm as winning -- the old paradigm is being displaced. An asymmetric grid (2fr 1fr or 3fr 1fr) with the new paradigm in the wider column would embody this argument. Currently, removing the grid loses nothing because both columns are treated equally."

This feedback:
1. Identifies a SPECIFIC section and SPECIFIC CSS
2. Explains the CONTENT REASON for the recommended change
3. Proposes a SPECIFIC spatial alternative
4. Avoids numbers that the builder could optimize

The builder reads ANALYSIS, not a scorecard. The analysis forces content re-engagement ("the article presents the new paradigm as winning") which feeds the topology rather than the checklist.

## PART 4: WHAT WE STILL DON'T KNOW

After 10 reports and 5,000+ lines of research, the following questions remain empirically unanswered:

### 4.1 Does the Phase Transition Actually Occur in a Single-Session Pipeline Build?

All evidence for the phase transition comes from the exploration era, where builders had accumulated context (88+ findings, 24 prior explorations, weeks of project history). The maximum-effort architecture gives the builder NONE of this accumulated context. The curated research provides vocabulary, but vocabulary is not experience.

**What we need to test:** Run the revised architecture on 3 articles of varying complexity. Measure falsifiable criteria on each BUILD-1 output. If BUILD-1 scores 4+/10, the phase transition is occurring. If BUILD-1 scores 0-2/10 despite 50 minutes of engagement, the phase transition may require accumulated context that a single session cannot provide.

### 4.2 Is Opus Necessary, or Does It Just Help?

Report #2 lists "Opus-class model" as a precondition. But this is unconfirmed. The strongest evidence is that exploration crown jewels were built by unspecified agents in an Opus-rich environment. No controlled experiment has compared Opus vs Sonnet builders with identical inputs and conditions.

**What we need to test:** Run the revised architecture twice on the same article: once with an Opus builder, once with a Sonnet builder. All other inputs identical. Compare falsifiable criteria scores. If Opus scores 7/10 and Sonnet scores 5/10, Opus matters but Sonnet is viable. If Opus scores 7/10 and Sonnet scores 1/10, Opus is essential.

### 4.3 How Much Does the $0 Reform Contribute?

Report #6's "progressive testing protocol" asks: what if simply removing the 42 killed constraints from the current $30 pipeline produces 3-5/10 without spending additional money? If constraint removal alone captures half the quality gap, the remaining $60 of architecture buys 4 points. If constraint removal captures nothing, the full architecture is justified.

**What we need to test:** Run the current pipeline with 3 changes: (1) builder reads article first, (2) no "build vertical first," (3) no structural cap. Score on falsifiable criteria. This is a zero-cost experiment that bounds the value of the full architecture.

### 4.4 Does Frontier Vocabulary Actually Get Used?

The architecture makes frontier techniques (container queries, scroll-driven animations) available as vocabulary. But will a builder who reaches Phase 3-4 actually reach for them? The exploration-era builders had 337 accumulated findings and never used container queries. It may be that frontier techniques require not just vocabulary but EXEMPLIFICATION -- the builder must see a completed page that uses them before it can imagine using them.

**What we need to test:** Run the architecture with and without the frontier techniques section in the curated research. Compare whether the "with frontier" builds use any frontier techniques. If not, the vocabulary alone is insufficient and we need exemplification (complete HTML reference files that demonstrate frontier techniques).

### 4.5 What is the Variance?

Report #6 raises the reproducibility question (FM-9). The phase transition may be deterministic given sufficient conditions, but "sufficient conditions" include the builder's random seed, the content's structural richness, the curated research's quality, and the discovery agent's observations.

**What we need to test:** Run the architecture 3 times on the same article. Measure falsifiable criteria variance. If scores range from 5-8/10, variance is acceptable. If scores range from 2-9/10, the architecture is unreliable and needs stabilization mechanisms.

### 4.6 Does the Evaluation-Iteration Cycle Actually Deepen Quality?

The architecture assumes that BUILD-2 (informed by EVALUATE-1 feedback) produces deeper output than BUILD-1. But the evaluation may produce cosmetic patching (FM-8B) rather than genuine structural deepening. The builder may add named grid areas to satisfy the coupling evaluator without re-engaging with the content.

**What we need to test:** Compare BUILD-1 and BUILD-2 outputs on 3 articles. Do the differences represent STRUCTURAL deepening (new spatial organizations, new self-referential components) or SURFACE patching (renamed grid areas, added hover states)? If the latter, the evaluation feedback format needs revision.

### 4.7 Is Content Complexity Triage Reliable?

The architecture routes Tier C content (linear/procedural) to the reformed pipeline. But the TC agent may misjudge complexity. A seemingly simple tutorial about setting up a development environment might have hidden structural richness (comparison of approaches, progressive deepening of configuration, a recurring metaphor of building a house). If the triage agent misclassifies, the article gets the wrong build path.

**What we need to test:** Have the triage agent classify 10 diverse articles. Have a human classify the same 10. Compare agreement rate. If agreement < 80%, the triage criteria need refinement.

### 4.8 The Biggest Unknown: What Can Only Be Learned by Building

The research has been analysis of PAST builds (exploration-era output). The revised architecture is a PREDICTION about how to produce FUTURE output at exploration quality. Predictions built on analysis of past output are inherently limited by the assumption that the same conditions produce the same results.

The conditions are NOT the same:
- Single session vs multi-week project
- No accumulated context vs 337 findings + 88 stage findings
- No human direction vs active human curation
- Automated evaluation vs human judgment

The revised architecture compensates for these differences (curated research replaces accumulated context, evaluation agents replace human judgment, content triage replaces human direction). But these are SUBSTITUTES, not equivalents.

**The only way to validate the architecture is to run it.** Three builds, three different articles (one Tier A, one Tier B, one Tier C), with falsifiable criteria scoring on each pass (BUILD-1, BUILD-2, BUILD-3). The results will answer most of the above questions simultaneously.

## PART 5: THE CORE INSIGHT

After 10 reports, the single most important finding is this:

**The gap between pipeline output (0/10) and exploration output (10/10) is not a gap in CSS technique, prompt design, or agent architecture. It is a gap in COGNITIVE STATE. The builder must form a persistent, content-driven spatial topology that dominates all CSS decisions. This topology forms THROUGH the act of writing CSS after sustained content engagement. No amount of upstream processing, constraint tuning, or evaluation machinery can substitute for this state forming in the agent that writes the code.**

The architecture's job is not to produce good output. It is to produce the CONDITIONS under which the builder produces good output: content-first reading, spatial vocabulary, absence of procedural noise, sufficient time, and evaluation feedback that drives deeper content engagement rather than checklist optimization.

Everything else -- the curated research, the structural analysis, the convention spec, the evaluation agents, the iterative deepening -- is support infrastructure. The builder is the central agent. The builder's cognitive state is the product. The CSS is the by-product of the cognitive state.

If the builder falls in love with the content, the architecture succeeds at any cost. If the builder does not, the architecture fails at any cost.

The architecture's only job is to make falling in love reliable.
