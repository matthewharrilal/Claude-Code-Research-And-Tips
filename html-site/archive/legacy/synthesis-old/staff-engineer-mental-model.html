The 8 Staff Engineer Principles

PRINCIPLE 1
Think in Waves, Not Tools
Don't ask "how do I use Claude Code better?" Ask "am I ready for Wave 5 (agent clusters)?" The competitive advantage isn't tool mastery - it's being ahead on the wave progression.
PRINCIPLE 2
Infrastructure Over Wrappers
"Almost everyone starts with LangChain or similar frameworks. Almost everyone eventually strips them out." Production is FastAPI and direct API calls. Frameworks are for demos.
PRINCIPLE 3
Agents Are Junior Developers
Think of AI coding as having an extremely persistent but context-limited junior developer who never sleeps. They need supervision, clear tasks, verification. They will make mistakes. They don't remember yesterday.
PRINCIPLE 4
The Trust Gradient
"Trust is the number one thing keeping the industry from rocketing forward. You can't trust anything - you have to validate and verify." Never trust: file/database operations, deployments. Verify always: code changes, API calls. Trust with monitoring: reads, analysis.
PRINCIPLE 5
Sandbox Everything
"Never give an autonomous AI agent access to your primary operating system or critical file storage." Docker containers, git worktrees, budget caps, circuit breakers, timeouts.
PRINCIPLE 6
The Economics of Agents
Each agent costs approximately $10-12/hour in tokens. Each agent provides roughly 1 developer of productivity. 5 agents = $50/hour = 5x productivity. Budget accordingly: $50k+/year in LLM spend per developer.
PRINCIPLE 7
Planning is Non-Negotiable
From Anthropic and every advanced practitioner: Planning Before Implementation. Explore, Plan, Code, Commit. PRD-driven development. Acceptance criteria before execution.
PRINCIPLE 8
The Orchestrator/Worker Separation (Iron Law)
ORCHESTRATORS COORDINATE. WORKERS EXECUTE. NEVER MIX. This is the Iron Law. Violate it and your system fails. Centralized orchestration reduces error amplification from 17.2x to 4.4x.

3 Framework Decisions

WHY ADOPT THE SIX WAVES FRAMEWORK?
AI development productivity is non-linear. Each wave represents a 5x productivity multiplier. Being one wave ahead of mainstream provides competitive advantage that compounds over time. Ignoring waves means optimizing for yesterday's paradigm.
What this means for you
Identify your current wave and the industry mainstream. If you're at Wave 3 while Wave 4 is emerging, you're already behind. Plan transitions deliberately - skipping waves doesn't work because Wave 5 requires Wave 4 mastery.
WHY THE IRON LAW (ORCHESTRATOR/WORKER SEPARATION)?
Research shows error amplification in multi-agent systems: independent agents have 17.2x error amplification, while centralized orchestration reduces this to 4.4x. When orchestrators also code, context pollution causes quality degradation across the entire system.
What this means for you
Design your architecture with strict role separation. Orchestrators: coordinate only, never touch code. Workers: execute only, never spawn sub-agents. This isn't bureaucracy - it's error containment.
WHY THE $10-12/HOUR ECONOMICS MODEL?
Without cost awareness, multi-agent systems can spiral out of control. The $47,000 recursive loop incident (11 days unnoticed) demonstrates the risk. Having a mental model for agent economics enables budget planning and prevents surprises.
What this means for you
Budget = agents x hours x $10. For 5 agents running 8 hours, that's $400/day, $100k+/year. This isn't optional - it's the cost of Wave 5+ productivity. Build it into project planning.

7 Paradigm Shift Tensions

These are fundamental transitions in how we work, not bugs to fix. From Yegge and Kim's "Vibe Coding": "This is one of the most challenging transitions the industry has ever faced."

Role Change: Writer to Orchestrator
From: Writing code line-by-line
To: Orchestrating agents, validating output
Why It's Hard: Identity is tied to coding skill. "I'm a developer" becomes "I'm a coordinator."
Coping: Embrace new expertise. Orchestration IS the new skill. Architecture matters more than ever.
Accountability Gap
From: Write it, own it - direct accountability
To: Don't write it, still own it - indirect accountability
Why It's Hard: Responsibility without control. You're accountable for code you didn't write.
Coping: Build verification systems. Automate trust but verify always. The verification IS your contribution.
Trust Inversion
From: Trust verified code (tests pass = done)
To: Verify before trusting (tests pass = start verifying)
Why It's Hard: Verification is additional work. "It works" isn't enough anymore.
Coping: Automate verification. Invest in tooling that makes verification fast enough to be practical.
Speed Paradox
From: Faster development = less review needed
To: Faster development = same or more review needed
Why It's Hard: Review doesn't scale linearly with output. 10x output still needs review.
Coping: Prioritize critical paths. Not all code needs same review depth. Build trust incrementally.
Mental Model Shift
From: Tool mastery (learn the IDE, learn the framework)
To: Wave positioning (understand where development is heading)
Why It's Hard: Different skill entirely. Strategic thinking, not tactical execution.
Coping: Study this document. Follow frontier practitioners. Read Yegge, watch the research.
Team Dynamics Shift
From: Individual skill differentiation
To: Team wave-level competency matters more
Why It's Hard: One person can't carry a team. Everyone needs shared mental models.
Coping: Build shared vocabulary. Team training, shared CLAUDE.md, common frameworks.
