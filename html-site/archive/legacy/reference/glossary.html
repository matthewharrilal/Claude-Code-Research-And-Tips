Patterns & Workflows
Ralph Wiggum Loop

Named after the Simpsons character, a continuous iteration pattern for autonomous development:

while :; do cat PROMPT.md | claude-code ; done

Core philosophy: Fresh context beats extended sessions. Each iteration starts clean, preventing context rot.

Origin: Matt Pocock (popularized), Ryan Carson (refinement)

See also: Pocock & Carson, Ralph Complete Guide

Gas Town

Steve Yegge's factory-scale agent architecture with 8 specialized roles working in concert. Named after the Mad Max location. Designed for continuous development at scale.

The 8 roles: Overseer, Mayor, Deacon, Dogs, Refinery, Witness, Polecat, Crew.

Origin: Steve Yegge (2025)

See also: Steve Yegge, Gas Town Guide

CC Mirror

Hidden multi-agent orchestration enablement pattern. A Claude Code instance monitors and coordinates other Claude Code instances, creating a "mirror" that reflects and manages the development environment.

Origin: Community pattern, documented in Wave 2.5

See also: Multi-Agent Patterns

Personal Panopticon

Architecture of 8 parallel Claude instances managing different life domains (trades, health, email, etc.). Each operates independently with its own directory as persistent memory.

Philosophy: "The filesystem IS the memory" - no shared context between domains.

Origin: Molly Cantillon

See also: Molly Cantillon

Hub-Spoke Architecture

Multi-agent pattern where a central coordinator (hub) dispatches tasks to specialized workers (spokes). The hub maintains state and routes tasks; spokes execute independently.

Example: Mayor in Gas Town coordinates Dogs.

See also: Gas Town, Orchestration Taxonomy

Techniques & Practices
Ultrathink

Extended thinking keyword that triggers Claude's deepest reasoning mode with maximum token budget (31,999 tokens).

Thinking budget keywords:

  • think - 4,000 tokens (standard planning)
  • think hard / megathink - 10,000 tokens (complex refactoring)
  • ultrathink / think harder - 31,999 tokens (architecture decisions)

Origin: Boris Cherny (documented), Community (coined "ultrathink")

60% Rule

Never fill more than 60% of the context window. Reserve 40% for Claude's reasoning and response. This prevents quality degradation as conversations grow.

/context command shows current usage - check frequently.

Origin: Ashley Ha

See also: Ashley Ha, Context Budget

Plan Mode

Two-phase workflow: Start in Plan Mode (Shift+Tab twice), iterate on architecture, then switch to auto-accept for execution. Prevents Claude from making 40+ unwanted changes.

Impact: 2-3x success rate improvement for complex tasks.

Origin: Boris Cherny (built into Claude Code)

See also: Boris Cherny

Verification Loop

Giving Claude a way to verify its own work. The single most important technique for quality output (2-3x improvement).

Methods: Chrome extension for UI testing, bash scripts for automated checks, mobile simulators, background verification agents.

Origin: Boris Cherny (core principle)

Context Budget

Treating context like a finite resource to be budgeted, not a bucket to be filled. Key practice for sustainable Claude Code productivity.

Estimate: ~4 characters = 1 token. 200k window = ~800k characters max.

See also: 60% Rule, Context Window

thoughts/ Directory Pattern

Externalized reasoning directory for persistent memory across sessions. Contains research files, plans, progress logs, and decision records.

Convention: research-*.md, plan-*.md, progress-*.md, complete-*.md

Origin: Ashley Ha (4-phase workflow)

See also: Ashley Ha

Architecture & Infrastructure
Context Window

The total amount of text (measured in tokens) that Claude can process in a single session.

Current sizes: Claude 3.5/4 Sonnet ~200,000 tokens, Claude 4 Opus ~200,000 tokens.

Rule of thumb: 4 characters â‰ˆ 1 token

MCP (Model Context Protocol)

Anthropic's protocol for connecting Claude to external tools and data sources. Enables Claude to interact with databases, APIs, file systems, Slack, etc.

Configuration: ~/.claude/settings.json or project-level .claude/settings.json

See also: Chrome MCP

Subagent

A Claude instance spawned by a parent Claude to handle a specific task. Subagents have isolated context and return results to the parent.

Location: .claude/agents/ (Markdown files defining agent behavior)

Key benefit: Context isolation - subagent context doesn't pollute main session.

Origin: Boris Cherny (built into Claude Code)

CLAUDE.md

Project configuration file read automatically by Claude Code. Contains build commands, code style, patterns, anti-patterns, and session protocols.

Location: Git repository root

Best practice: Keep under 2,000 tokens. Team-shared and version-controlled.

Origin: Boris Cherny (designed into Claude Code)

See also: Boris Cherny, Shrivu Shankar

Hooks

Automated actions triggered by Claude Code events. Ensure deterministic operations always happen (Claude might forget to format; hooks never do).

Types: PreToolUse, PostToolUse, Notification

Example: Run Black formatter after every Edit/Write operation.

Origin: Boris Cherny (built into Claude Code)

Gas Town Roles
Overseer

Role: Human supervision layer. YOU are the Overseer.

Responsibility: Approval decisions, strategic direction, final quality gate.

See also: Gas Town

Mayor

Role: Strategic planning agent in Gas Town.

Responsibility: High-level coordination, task prioritization, work distribution to Dogs.

Autonomy: Medium - proposes plans, waits for Overseer approval on major decisions.

See also: Gas Town

Deacon

Role: Code review patrol agent in Gas Town.

Responsibility: Continuous code quality monitoring, catching issues before they propagate.

Warning: Bug in Gas Town v0.3.x - Deacon may have false positives. Monitor carefully.

See also: Gas Town

Dogs

Role: Implementation workers in Gas Town.

Responsibility: Execute coding tasks assigned by Mayor. Multiple Dogs can work in parallel on different features.

Autonomy: High within task scope - work independently until complete.

See also: Gas Town, Mayor
