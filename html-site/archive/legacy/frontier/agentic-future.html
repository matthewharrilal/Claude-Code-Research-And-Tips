Live Context Orchestration

Your CLAUDE.md writes itself at session start

Every session starts with stale context. "What's the current state?" becomes your first question, every time. Live Context Orchestration fixes this with a SessionStart hook that generates fresh context automatically.

Key Insight: The agent starts already knowing what's broken, what tests are failing, what's uncommitted. Zero re-explanation needed.

Effort: Low - Simple shell script hook
Value: Medium - Immediate productivity gains
Risk: Low - Worst case, slow session start if tests run slowly

The hook queries git status, failing tests, TODO comments, and injects them into .claude/CONTEXT.md which your CLAUDE.md references.

See Implementation Sketch 
#!/bin/bash
# ~/.claude/hooks/live-context.sh (SessionStart hook)

CONTEXT_FILE=".claude/CONTEXT.md"

cat > "$CONTEXT_FILE" << EOF
## Current Session Context (Auto-Generated)
Generated: $(date)

### Git Status
\`\`\`
$(git status --short 2>/dev/null || echo "Not a git repository")
\`\`\`

### Failing Tests
\`\`\`
$(npm test 2>&1 | grep -E "(FAIL|Error)" | head -20)
\`\`\`

### Unfinished PRD Tasks
$(jq -r '.userStories[] | select(.passes == false) | "- [ ] \(.id): \(.title)"' prd.json 2>/dev/null)
EOF

Always-On Verifier

Every file edit triggers automatic verification

Quality discipline shouldn't require thinking. With a PostToolUse hook on Edit/Write operations, every change triggers typecheck and lint automatically. Slow tests run in background with notifications.

Key Insight: Fail fast. Catch errors before they compound. The agent can't accidentally skip verification because it's automatic.

Effort: Low - One-time hook configuration
Value: Medium-High - Faster feedback loops
Risk: Low - May slow agent if tests are slow (use async for slow checks)

Understanding Phase 1

At this point, you should understand:

  • Hooks can inject context automatically (Live Context)
  • Hooks can enforce quality automatically (Always-On Verifier)
  • Both are low-effort, single-afternoon implementations

If this makes sense: You're ready for Phase 2, which builds on these primitives with memory.

Memory-Augmented Ralph

Fresh context + targeted memory injection

Ralph's fresh-context iterations are powerful, but they also mean re-learning lessons each time. Memory-Augmented Ralph queries Claude-Mem before each iteration to inject only relevant memories.

Key Insight: "LEARNING: Config uses tabs not spaces" gets captured during iteration N and injected into iteration N+5 when relevant. No context rot, but also no repeating mistakes.

Effort: Medium - Claude-Mem integration + prompt modification
Value: High - Best of both worlds (fresh + memory)
Risk: Medium - Irrelevant memories could add noise

Pre-prompt hook queries: "$CURRENT_STORY" OR "gotcha" filtered by type=learning. Small, targeted injection keeps context clean.

Next

Adversarial Acceptance Testing

An agent that actively tries to break your implementation

Standard acceptance: "Does it work?" Adversarial acceptance: "What would make it fail?"

After implementation, a specialized agent attacks your acceptance criteria: What if the email is 10MB? What if the password is empty? What about concurrent requests? It finds the edge cases you didn't specify.

Key Insight: Shift adversarial thinking from human reviewers to automated agents. Catches bugs before shipping, improves criteria quality for next time.

Effort: Medium - Skill file + integration with Ralph
Value: Very High - Catches real bugs before production
Risk: Medium - False positives, overly aggressive attacks

Output: JSON with attacks, severity, and verdict (BLOCKED/PASSED). If BLOCKED, story stays incomplete and gets re-queued.

Understanding Phase 2

At this point, you should understand:

  • Fresh context and persistent memory aren't opposites - they compose
  • Targeted injection keeps context clean while preventing repeated mistakes
  • Adversarial agents provide defense-in-depth beyond standard testing

If this makes sense: You're ready for Phase 3, where agents begin orchestrating other agents.

Ralph Colonies

Each agent in a CC Mirror swarm runs its own Ralph loop

What if you could have parallel features, each with fresh-context protection? Ralph Colonies run multiple Ralph loops simultaneously, coordinated by a meta-orchestrator. Each colony owns a branch, works independently, and merges when complete.

Key Insight: Parallel + fresh context benefits combined. One colony's failure doesn't poison others. 3-5x development speed for large features.

Effort: High - Multi-layer coordination
Value: Very High - Factory-scale development
Risk: High - Merge conflicts, coordination drift

Architecture: Meta-orchestrator spawns colony workers, each in its own git worktree. Colonies communicate via HANDOFF.json files. When all colonies complete, coordinated merge to main.

See Architecture Diagram 
Meta-Orchestrator (CC Mirror)
    |
    +-- Ralph Colony A (auth-schema branch)
    |       |-- prd-colony-a.json
    |       |-- progress-colony-a.txt

    |       +-- ralph.sh iterations 1..N
    |
    +-- Ralph Colony C (auth-ui branch)
            |-- Uses API mocks
            +-- Integrates when A + B complete
Later

Learned Skills

Skills that write themselves from observed patterns

Instead of manually writing skill files, what if they emerged from what actually worked? A pattern miner runs weekly, identifies repeated successful workflows, and generates skill files automatically.

Key Insight: "This is how we actually do X here" becomes codified without manual documentation. Self-improving skill library.

Effort: High - Pattern mining is non-trivial
Value: Very High - Skills improve themselves
Risk: High - Learning bad patterns, overfitting to project quirks

Quality filters: Minimum 3 successful occurrences, no failures with same approach, 75% confidence threshold, human review before activation.

Later

Specialist Skill Squads

Permanent teams with deep domain expertise

Database Squad handles all schema work. API Squad handles all endpoints. Frontend Squad handles all UI. Each squad has pre-loaded skills and works in its own worktree. Dependencies are managed through handoff protocols.

Key Insight: Deep specialization + consistent patterns. The Database Squad always uses the same migration safety checks. The API Squad always follows the same error handling patterns.

Effort: Very High - Full team model
Value: Very High - Team-like development
Risk: High - Handoff gaps, interface mismatches

Understanding Phase 3

At this point, you should understand:

  • Multi-agent systems multiply capabilities but also complexity
  • Coordination primitives (handoffs, worktrees, branches) become critical
  • These are experiments for practitioners with solid fundamentals

The Bigger Picture

These seven innovations represent the frontier of agentic development. They're not random experiments - they follow a pattern:

Five Recurring Themes

  1. Fresh context + targeted memory - Not opposites, they compose
  2. Parallel + isolated - Scale without cross-contamination
  3. Adversarial verification - Attack your own work before shipping
  4. Self-improving systems - Patterns that learn from practice
  5. Composable primitives - Small pieces that combine into larger wholes

The future isn't one monolithic AI system. It's compositions of specialized agents, each doing one thing well, coordinated through simple protocols. The path from today to that future runs through the innovations on this page.

Start simple. Compound over time.
