The 13-Point Workflow

Phase 1

Session Management & Model Selection

1

Parallel Sessions

  • 5 parallel terminal Claudes + notifications
  • 5-10 web sessions + handoff via --teleport
  • Mobile starts for on-the-go ideation
Key Insight: Parallelism is a core productivity multiplier. Boris doesn't wait for one task to finish before starting another.
2

Model Choice

  • Always Opus 4.5 + thinking mode
  • Best tool use capabilities
  • Less steering required
  • Slower but wins long-term
Key Insight: Invest in quality upfront. Opus + thinking mode means less back-and-forth, fewer corrections, better first attempts.
Checkpoint: Session Setup

You should have: Terminal notifications configured. Multiple terminal tabs ready. Understanding that Opus + thinking mode is the default.

Phase 2

Knowledge Management

3

Shared Knowledge (CLAUDE.md)

  • Single shared CLAUDE.md in git
  • Updated frequently
  • Enforced via PR tags / GitHub action
  • Compounds knowledge across team/time
Key Insight: CLAUDE.md is where the magic happens. It's not just documentation - it's a compounding asset that makes every future session smarter.
Phase 3

Workflow Pattern

4

Plan Mode Workflow

  • Start PRs in Plan mode - refine the approach
  • Switch to auto-accept for 1-shot execution
  • Plan deeply, execute shallowly
Key Insight: This is the rhythm. Don't rush into execution. Invest time in planning, then let the AI execute with minimal interruption.
Checkpoint: Core Pattern

You should internalize: "Plan deeply, execute shallowly." Start in plan mode. Refine. Then switch to auto-accept for clean execution.

Phase 4

Automation Layers

5

Slash Commands

  • For inner loops (repeated workflows)
  • Example: /commit-push-pr with inline bash pre-compute
  • Git-checked in .claude/commands/
Example 
# .claude/commands/commit-push-pr.md
Run git status, stage changes, create commit message,
push to remote, create PR with gh cli
6

Subagents

  • For outer workflows
  • Examples: code-simplifier, verify-app
  • Markdown files in .claude/agents/
Key Distinction: Slash commands = inner loops (frequent, quick). Subagents = outer workflows (complex, multi-step).
7

Hooks

  • PostToolUse hook for final formatting
  • Example: Run black $CLAUDE_FILE_PATHS after Edit|Write
settings.json 
{
  "hooks": {
    "PostToolUse": {
      "matcher": "Edit|Write",
      "command": "black $CLAUDE_FILE_PATHS"
    }
  }
}
Phase 5

Security & Permissions

8

Permissions

  • /permissions whitelist instead of dangerous skip
  • Avoid reckless permission grants
  • Example: /permissions add Bash:npm*
Don't skip permissions entirely. Whitelist specific commands you trust. The extra friction is intentional safety.
Checkpoint: Automation Setup

You should have: At least one slash command created. Understanding of when to use subagents vs commands. Hooks configured for your formatter.

Phase 6

Tool Integrations

9

MCP Integrations

Claude uses Boris's tools:

  • Slack MCP - team communication
  • BigQuery - data analysis
  • Sentry - error tracking
Key Insight: MCP extends Claude's reach into your actual work environment. Connect the tools you use daily.
10

Long Tasks

  • Background verification agent
  • Stop hook
  • ralph-wiggum plugin + sandbox skip
Key Insight: For multi-hour autonomous work, you need infrastructure: background agents, state persistence, and loop mechanisms.
Phase 7 - CRITICAL

Verification Infrastructure

11

Verification (MOST IMPORTANT)

"Verification feedback loop creates 2-3x quality improvement."
  • Chrome extension UI tests
  • Bash scripts
  • Mobile simulators
  • Non-negotiable for production quality
Tool Purpose
Chrome extension UI tests for web apps
Bash scripts Automated checks
Mobile simulators iOS/Android verification
Background agent Continuous verification
"Generic bash tests insufficient for UI/mobile" - domain-specific verification is required. Don't rely on generic test scripts for visual or mobile work.
Checkpoint: Verification Setup

Critical question: How does Claude verify its work in YOUR project? If you don't have an answer, you're missing the single biggest quality multiplier.

You should have: At least one verification script. Understanding that 2-3x quality comes from verification, not better prompts.

The 5 Key Principles

1. Vanilla First - Customize only where it hurts. Start simple, add complexity only when needed.
2. Feedback Loops > Perfect First Drafts - Iterate quickly. Verification beats perfection.
3. Shared, Git-Checked Knowledge Compounds - CLAUDE.md + commands + agents = team knowledge that grows over time.
4. Plan Deeply, Execute Shallowly - Invest in planning. Let execution flow with minimal interruption.
5. Verification is Non-Negotiable - 2-3x quality improvement. This is where most of the magic happens.

The Mental Model

The Compounding System

Think of Claude Code as a compounding system - not a one-shot tool, but an evolving teammate. Invest in memory (CLAUDE.md), automation (slash/subagents/hooks), verification loops, and parallelism to create exponential returns over time.

Caveats & Gotchas

CLAUDE.md bloat

Issue: CLAUDE.md grows too large, becomes unwieldy.

Mitigation: Keep minimal, periodically prune. Focus on high-value context only.

Large contexts

Issue: Compaction can lose nuance.

Mitigation: Manual reflection. Don't rely entirely on auto-summarization.

Heavy parallelism

Issue: Managing 5-10 sessions requires discipline.

Mitigation: Notifications + clear task boundaries. Don't context-switch randomly.

Verification specificity

Issue: Generic bash tests insufficient for UI/mobile.

Mitigation: Domain-specific tests needed. Invest in the right verification tools for your domain.

Permission temptation

Issue: Temptation to skip all permissions for speed.

Mitigation: Whitelist > skip. The friction is intentional.

To implement Boris's workflow today:

  2. Write one verification script for your domain
  3. Create one slash command for your most common workflow
  4. Use Opus + thinking mode
