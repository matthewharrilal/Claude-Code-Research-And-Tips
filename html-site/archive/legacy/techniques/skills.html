Skills and Custom Commands

You Are Here

Skills are Markdown files that extend Claude Code's capabilities. They act as recipes, templates, or instruction sets that Claude references on demand. Think of them as hiring specialists with curated context. This guide covers creating, triggering, and composing skills for maximum leverage.

What Are Skills?

You find yourself explaining the same context, style guides, or workflows to Claude repeatedly. Each session starts from scratch with no memory of your preferences.

Skills are Markdown files (.md) that extend Claude Code's capabilities. They're stored in .claude/ directories and load on demand based on query relevance or explicit invocation.

"View .md files as building blocks for an AI-enhanced personal OS - like hiring specialists that reference your curated context on demand." - @elliot_garreffa
Key Properties of Skills
  • Plain text Markdown format
  • Stored in .claude/skills/ (project) or ~/.claude/skills/ (global)
  • Load on demand based on query relevance or explicit invocation
  • Can be project-specific or global

Skills vs Other Tools

Feature Slash Commands Skills Subagents Hooks
What Quick, atomic actions Instruction sets/recipes Mini-Claudes with isolated context Event-driven shell commands
Where .claude/commands/ .claude/skills/ .claude/agents/ .claude/settings.json
Trigger Manual (/command) Query relevance or manual Task tool or /agent Lifecycle events
Context Main conversation Main conversation Isolated context window N/A (shell only)
Best For Repetitive inner-loop tasks Complex workflows, personas Multi-step reasoning, parallel work Auto-formatting, linting, notifications
Skills vs Plugins: Skills are pure Markdown instructions (lightweight, portable). Plugins are full packages with code, hooks, commands, and skills (heavier, more powerful). Example: Claude-Mem is a plugin. A brand-voice skill is just a Markdown file.

Creating Custom Skills

Basic Skill Structure

Location: .claude/skills/ or ~/.claude/skills/ (global)

Minimal Skill Example

Create a file at .claude/skills/brand-voice.md:

# Brand Voice Skill

## Purpose
Ensure all written content matches our brand voice.

## Voice Characteristics
- Tone: Confident but approachable
- Vocabulary: Technical but accessible
- Avoid: Jargon without explanation, passive voice
- Prefer: Active voice, concrete examples

## Application Rules
1. Always start with the user's problem, not our solution
2. Use "you" more than "we"
3. Include at least one specific example per concept
4. End with a clear call-to-action

Advanced Skill Structure

For complex workflows like orchestration, skills can include ASCII art mental models, tool ownership declarations, and worker preambles:

# Orchestration Skill

## Mental Model
+---------------------------------------------------------------+
|                                                               |
|   You are the Conductor on the trading floor of agents        |
|                                                               |
|   Fast. Decisive. Commanding a symphony of parallel work.     |
|   Users bring dreams. You make them real.                     |
|                                                               |
+---------------------------------------------------------------+

## The Iron Law
YOU DO NOT WRITE CODE. YOU DO NOT RUN COMMANDS.
YOU DO NOT EXPLORE CODEBASES.
You are the CONDUCTOR. Your agents play the instruments.

## Tool Ownership

### Orchestrator Uses Directly
- Read (references, guides, agent outputs - 1-2 files max)
- TaskCreate, TaskUpdate, TaskGet, TaskList
- AskUserQuestion
- Task (to spawn workers)

### Workers Use Directly
- Read, Write, Edit, Bash
- Glob, Grep, WebFetch, WebSearch, LSP
- CAN'T: Spawn sub-agents or manage task graph
Token Cost Consideration

Complex skills can cost ~5,000+ tokens. However, the upfront cost saves tokens via delegation and focused work. Split very large skills into composable pieces.

File Naming Conventions

Recommended Structure
.claude/skills/brand-voice.md Simple skills (single file)
.claude/skills/orchestration/SKILL.md Complex skills with subfolder
.claude/commands/today.md Slash commands
.claude/agents/adversarial-critic.md Subagent definitions

Skill Triggers and Invocation

Manual Invocation

Three Ways to Load Skills
1

Direct Load Request

"Load the orchestration skill and initiate"
2

Slash Command Style

/skill brand-voice
3

Via Plugin System

/plugin install myskill@marketplace

Automatic Activation

Skills can auto-trigger based on query relevance. Define trigger conditions in your CLAUDE.md:

# Skills

## Available Skills
- crypto-sovereignty-analyst: Use when analyzing financial history, authoritarianism
- essence: Use when user wants to "extract essence", "understand the soul of"
- youtube-research: Use when user provides YouTube URL or video ID

Ambiguous Queries: Skills may not auto-trigger on ambiguous queries. For reliable triggering, use explicit invocation or add clear trigger phrases to skill descriptions.

Passing Arguments to Skills

# In Skill Invocation
/skill research --topic "quantum computing" --depth deep

# In Skill Definition (MDFlow style)
---
model: sonnet
---
Write a short poem about {argument}.

Best Practices and Anti-Patterns

Skill Design Principles

  1. Single Responsibility - One skill, one purpose. Don't cram multiple workflows.
  2. Clear Mental Models - Use ASCII art, metaphors. Make the role viscerally clear.
  3. Explicit Tool Ownership - State which tools the skill uses to prevent conflicts.
  4. Progressive Disclosure - Start with high-level purpose, detail rules later.
  5. Verifiable Outputs - Include acceptance criteria. Define "done" explicitly.

Anti-Patterns to Avoid

  • Context Pollution: Don't cram complex logic into slash commands - it pollutes main context
  • Skill Bloat: ~5,000+ tokens becomes expensive. Split into composable pieces.
  • Vague Triggers: Ambiguous trigger phrases cause misfires. Be specific.
  • Missing Tool Boundaries: Workers spawning sub-agents creates recursion. Define what agents CAN'T do.

Troubleshooting Skills

Skill doesn't trigger

Cause: Ambiguous query or trigger phrases

Fix: Use explicit invocation: "Load the X skill" or add clearer trigger phrases

Wrong behavior

Cause: Conflicting instructions

Fix: Simplify the skill, use clear priorities, check for conflicts with other loaded skills

Context overflow

Cause: Skill too large

Fix: Split into smaller skills. Audit token cost: "Calculate total token cost of loaded skills"

Tool conflicts in multi-agent

Cause: Multiple skills using same tools without boundaries

Fix: Define tool ownership explicitly in each skill

Recursion/infinite loops

Cause: Workers spawning sub-agents

Fix: Add explicit "CAN'T" rules in worker preamble

Debugging Workflow
  1. Check loading: "What skills are currently loaded?"
  2. Test isolation: Load skill alone in fresh session
  3. Review triggers: Are they specific enough?
  4. Token audit: How many tokens is the skill?
  5. Use Claude HUD: Monitor context usage in real-time
