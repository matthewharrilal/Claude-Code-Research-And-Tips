The Hive-Mind Swarm Model

Philosophy

The hive-mind metaphor is central to Claude-Flow's design. Unlike Gas Town's "agent factory" or CC Mirror's "hub-and-spoke," Claude-Flow models a biological swarm with emergent coordination.

Traditional Multi-Agent vs. Hive-Mind Swarm =================== =============== Human Human | | Orchestrator QUEEN | | +---------+ +------+------+ Worker Worker Worker Worker Worker Worker Worker Worker Worker Worker Worker Fixed hierarchy Dynamic, emergent Sequential dispatch Parallel swarming Explicit coordination Implicit coordination

Queen Agent Role

The Queen is not just an orchestrator but a meta-coordinator:

  • Task Distribution - Breaks down goals into worker-sized chunks
  • Resource Allocation - Assigns workers based on capacity and specialization
  • Swarm Health - Monitors worker status, handles failures
  • Emergent Behavior - Allows workers to self-organize within constraints

Worker Autonomy

"A swarm uses independently operating agents that operate with a common direction via an orchestrator. Each agent is autonomous--they can make their own determination how to solve problems."

Workers in Claude-Flow have more autonomy than in Gas Town or CC Mirror:

  • Can choose implementation approach
  • Can request resources from shared memory
  • Can signal need for peer consultation
  • Still constrained to their assigned domain

The Four-Agent Archetype

PLANNER

Breaks down requirements, creates plan, sequences dependencies

CODER

Implements features, writes code, runs tests, documents

CRITIC

Reviews work, finds issues, challenges assumptions, security check

INTEGRATOR

Merges results, resolves conflicts, validates integration, final QA

Dynamic Specialization

Unlike Gas Town's fixed roles, Claude-Flow allows dynamic role creation:

Queen: "This task requires a database specialist" -> Spawns: Database Agent (loaded with Prisma skills) Queen: "Now we need security review" -> Spawns: Security Agent (loaded with OWASP patterns) Queen: "Integration needs performance testing" -> Spawns: Performance Agent (loaded with benchmarking tools)

Memory Systems: AgentDB & ReasoningBank

Reuven's Key Insight: "For swarm memory, you don't need complex vector similarity searches. What you really need is a common simple SQLite storage environment that allows all agents to save their current state and access previous states."

AgentDB: State Persistence

A SQLite-based state management layer:

-- Agent State Table CREATE TABLE agent_state ( agent_id TEXT PRIMARY KEY, agent_type TEXT NOT NULL, current_task TEXT, status TEXT, last_updated TIMESTAMP, context_snapshot BLOB ); -- Inter-Agent Messages CREATE TABLE agent_messages ( message_id TEXT PRIMARY KEY, from_agent TEXT, to_agent TEXT, message_type TEXT, content TEXT, timestamp TIMESTAMP, acknowledged BOOLEAN DEFAULT FALSE );

Why SQLite:

  • Zero infrastructure dependencies
  • ACID transactions for state consistency
  • Fast reads for coordination
  • Easy backup/restore
  • Works in containerized environments

ReasoningBank: Semantic Memory

While rejecting complex vector setups for coordination, Claude-Flow does use vector search for semantic memory:

Agent Output -> Embedding -> Vector Store | Query: "How did we solve X?" -> Similarity Search | Top-K Results -> Agent Context

Performance claims: 96x to 164x faster vector search than alternatives.

The Hybrid Approach

Layer Technology Purpose
Coordination AgentDB (SQLite) Who is doing what? Task status? Messages? Resources?
Semantic ReasoningBank (Vectors) Similar problems? Applicable patterns? Relevant context?

Troubleshooting

Error: "Queen agent won't start"

Cause: Missing API key, incorrect configuration, or port conflict

  1. Verify API key: echo $ANTHROPIC_API_KEY
  2. Check config: cat claude-flow.json | jq .
  3. Check port conflicts: lsof -i :3000
  4. Restart with verbose: DEBUG=claude-flow:* npm run start:queen

Prevention: Always run npm run verify-api before starting.

Error: "Workers not spawning"

Cause: Resource limits, memory exhaustion, or AgentDB lock

# Check system resources free -h # Linux vm_stat # macOS # Verify AgentDB isn't locked lsof ./data/agents.db # Reset worker pool npm run workers:reset # Try spawning single worker manually npm run start:worker -- --type=coder --verbose

Prevention: Monitor resources; keep worker count under maxWorkers config.

Error: "AgentDB connection failed" (SQLITE_BUSY)

Cause: SQLite concurrent write contention (common at 20+ workers)

# Enable WAL mode for better concurrency sqlite3 ./data/agents.db "PRAGMA journal_mode=WAL;" # Kill orphaned connections pkill -f "sqlite.*agents.db" # If corrupted, reinitialize mv ./data/agents.db ./data/agents.db.backup npm run init-db

Prevention: Keep worker count under 50; use WAL mode from start.

Error: "Task stuck in PROCESSING state"

Cause: Worker crashed, infinite loop, or deadlock

# Check worker logs tail -100 ./logs/worker-*.log # Query stuck tasks sqlite3 ./data/agents.db "SELECT * FROM task_history WHERE status='processing' AND started_at < datetime('now', '-30 minutes');" # Force-complete stuck task npm run task:force-complete -- --task-id=TASK_ID # Reset worker sqlite3 ./data/agents.db "UPDATE agent_state SET status='idle', current_task=NULL WHERE current_task='TASK_ID';"

Prevention: Set reasonable task timeouts; enable worker heartbeat.

Emergency Recovery

# Kill all Claude-Flow processes pkill -f claude-flow pkill -f "node.*queen" pkill -f "node.*worker" # Reset AgentDB state sqlite3 ./data/agents.db "UPDATE agent_state SET status='idle', current_task=NULL;" sqlite3 ./data/agents.db "UPDATE task_history SET status='failed' WHERE status='processing';" # Clear lock files rm -f ./data/*.lock # Restart Queen (workers auto-recover) npm run start:queen
