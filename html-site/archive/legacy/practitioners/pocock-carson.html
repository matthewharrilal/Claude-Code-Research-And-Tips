Core Philosophy

1"Let It Cook"

The phrase isn't about being lazy - it's about removing yourself as the bottleneck. If you're watching the loop run, you're wasting the most valuable resource: your attention. Start the loop, trust the system, return to harvest results.

"If you're babysitting the loop, you're doing it wrong." - Ryan Carson

2Automation Over Intervention

Pre-commit hooks aren't optional - they're the entire point. The machine should verify itself. Tests, lints, and type checks act as automated gatekeepers that catch problems before they compound.

"The secret is the pre-commit hooks. They're not optional - they're the whole point." - Ryan Carson

3Small Files, Tight Loops

Keep iteration cycles fast and focused. A 50-line PROMPT.md that runs in 30 seconds beats a 500-line specification that takes 10 minutes. The goal is rapid feedback, not comprehensive planning.

4The Machine Should Verify Itself

Human review happens after the loop completes, not during. Tests and lints are the quality gates. If a loop iteration fails verification, it tries again automatically. You review the successful result, not every attempt.

"Ralph isn't about being lazy. It's about removing yourself as the bottleneck." - Matt Pocock

Key Techniques

1The 3-File System

The foundation of reproducible Ralph loops. Each file serves a distinct purpose:

PROMPT.md What to do this iteration Before each loop

Quick Win

Create these 3 files in your project root right now. Even empty files establish the pattern.

You should see: ls PROMPT.md LOG.md CONTEXT.md should list all three files. If any are missing, create them with touch PROMPT.md LOG.md CONTEXT.md.

2Pre-Commit Hook Integration

The verification layer that makes autonomous operation possible:

# .husky/pre-commit (example using Husky) #!/bin/sh . "$(dirname "$0")/_/husky.sh" npm run typecheck npm run lint npm test

Why this matters:

  • Automated test runs before every commit
  • Lint failures block the loop from continuing
  • Type checking as quality gate
  • Claude sees failures and self-corrects on next iteration
Critical

Without pre-commit hooks, Ralph becomes "expensive autocomplete." The hooks are what transform it into autonomous development.

3"Let It Cook" Discipline

A mindset more than a technique:

  1. Start the loop - Clear PROMPT.md, focused task
  2. Walk away - Make coffee, take a walk, context switch
  3. Return to review - Check results, not process
  4. Trust the system - Or fix the system (not the symptom)

4Prompt Evolution Pattern

Your PROMPT.md templates improve over time:

  1. Start simple - Basic task description
  2. Note failures - What did Claude miss?
  3. Add constraints - "Always check X before Y"
  4. Build templates - Reusable prompt structures
# PROMPT.md - Version 1 (naive) Add a user settings page. # PROMPT.md - Version 5 (evolved) Add a user settings page. ## Constraints - Match existing page layout in src/pages/ - Use existing form components from src/components/Form/ - Add route in src/routes.ts - Include loading states and error handling - Add tests in __tests__/ ## Verification Run `npm test -- settings` before completing.

5Session Boundary Management

Fresh context prevents hallucination drift:

  • Clean slate each iteration - Don't carry context between loops
  • LOG.md captures learning - Decisions persist externally
  • Fresh context = fresh thinking - Prevents compounding errors
The Philosophy

Boris Cherny's "fresh context > extended sessions" principle applied to automation. Each loop iteration is a new conversation with full context from files, not memory.

The Complete Workflow

Basic Ralph Loop

#!/bin/bash # ralph.sh - The basic loop while true; do # Read prompt and run Claude cat PROMPT.md | claude --print # Attempt commit (pre-commit hooks verify) git add -A git commit -m "Ralph iteration $(date +%s)" # If commit fails, Claude will see errors on next iteration # If commit succeeds, we have working code sleep 5 # Brief pause between iterations done

Enhanced Version

#!/bin/bash # ralph-enhanced.sh MAX_ITERATIONS=20 iteration=0 while [ $iteration -lt $MAX_ITERATIONS ]; do echo "=== Iteration $iteration ===" # Run Claude with current prompt cat PROMPT.md | claude --print # Check if task is complete (look for marker in LOG.md) if grep -q "TASK_COMPLETE" LOG.md; then echo "Task completed successfully!" exit 0 fi # Attempt commit if git add -A && git commit -m "Ralph iteration $iteration"; then echo "Iteration $iteration committed successfully" else echo "Iteration $iteration failed verification - will retry" fi ((iteration++)) sleep 2 done echo "Max iterations reached" exit 1

Setup Checklist

Implementation Checkpoint
  1. Create the 3 files: PROMPT.md, LOG.md, CONTEXT.md
  2. Configure pre-commit hooks: Tests, lint, typecheck
  3. Write your first PROMPT.md: Simple, focused task
  4. Start the loop: ./ralph.sh
  5. Walk away: Seriously, go make coffee
  6. Review results: Check LOG.md and commits

You should see: Multiple commits with passing tests, working code.

Implement Their Patterns

Quick Wins (< 10 minutes)

  1. Create PROMPT.md, LOG.md, CONTEXT.md in your project root
  2. Write a simple task in PROMPT.md
  3. Run cat PROMPT.md | claude --print once manually
  4. Review the output before automating

Medium Lift (1-2 hours)

  1. Set up pre-commit hooks (Husky, lint-staged, or similar)
  2. Create the ralph.sh script
  3. Run your first automated loop on a small task
  4. Iterate on PROMPT.md based on results

Deep Investment (days)

  1. Build a library of PROMPT.md templates for common tasks
  2. Create project-specific CONTEXT.md with patterns and conventions
  3. Set up notifications for loop completion
  4. Integrate with CI/CD for production-grade verification

Troubleshooting

Problem: "Loop running but not making progress"

Symptoms: Ralph loops endlessly, commits fail repeatedly, same errors keep appearing.

Causes:

  • PROMPT.md is too vague - Claude doesn't know what "done" means
  • Pre-commit hooks failing on unrelated issues
  • Claude can't find the right files
# Diagnose: Check last few iterations git log --oneline -10 # See what got committed git diff # See uncommitted changes # Fix: Add explicit completion criteria to PROMPT.md ## Success Criteria - [ ] File src/components/Button.tsx exists - [ ] All tests pass: npm test -- Button - [ ] No TypeScript errors: npm run typecheck - Write "TASK_COMPLETE" to LOG.md when done # Fix: Check if pre-commit hooks are the bottleneck git commit -m "test" --no-verify # Temporarily bypass # If this works, your hooks are too strict

Prevention: Always include explicit "done" criteria in PROMPT.md. Use the "TASK_COMPLETE" marker pattern.

Problem: "Pre-commit hooks failing repeatedly"

Symptoms: Every iteration fails at commit stage. Hook output shows same errors.

Causes:

  • Hooks too strict (failing on unrelated files)
  • Claude not seeing hook output
  • Hook timeout too short
# View hook output manually npm run lint # Run lint directly npm test # Run tests directly npm run typecheck # Run typecheck directly # Make hooks targeted (only check changed files) # In lint-staged.config.js: module.exports = { '*.{ts,tsx}': ['eslint --fix', 'tsc --noEmit'], '*.{ts,tsx,test.ts}': ['jest --findRelatedTests'] }; # Increase hook timeout (in .husky/pre-commit) timeout 120 npm test # 2 minute timeout

Recovery: If hooks are legitimately failing on Claude's code, add the error message to PROMPT.md so Claude can see what went wrong.

Problem: "Context getting stale - hallucinations increasing"

Symptoms: Later iterations reference code that doesn't exist. Claude invents APIs. Quality degrades over time.

Causes:

  • Not using fresh context between iterations
  • CONTEXT.md outdated
  • Loop running too long without reset
# Ralph should use fresh context each iteration # BAD: Running continuous Claude session while true; do echo "Continue from where you left off" | claude done # GOOD: Fresh Claude invocation each time while true; do cat PROMPT.md | claude --print # Fresh session git add -A && git commit -m "..." sleep 2 done # Update CONTEXT.md with current project state: # Project Structure - src/components/ - React components - src/utils/ - Utility functions - src/hooks/ - Custom hooks # Key APIs (actual, not imagined) - useAuth() from src/hooks/useAuth.ts - Button from src/components/Button.tsx

Prevention: Keep CONTEXT.md accurate. Update it whenever project structure changes. Fresh context every iteration is the key.

Problem: "Loop runs but changes are wrong"

Symptoms: Commits pass all checks, but the code doesn't actually solve the problem. Wrong approach taken.

Causes:

  • PROMPT.md lacks constraints
  • Tests don't actually verify requirements
  • Claude optimized for tests, not correctness
# Add architectural constraints to PROMPT.md ## Requirements Add user settings page with email preferences. ## Constraints - Use existing UserSettingsContext (src/contexts/User.tsx) - Follow the pattern in src/pages/Profile.tsx - Must integrate with existing EmailService - Do NOT create new services or contexts ## Verification 1. Settings should persist after page refresh 2. Email preference changes should trigger toast notification 3. Must work without JavaScript (form submission fallback)

Prevention: Your PROMPT.md should anticipate wrong approaches and explicitly forbid them. Add "Do NOT" constraints.

Problem: "PROMPT.md too complex - overwhelming Claude"

Symptoms: Claude attempts everything at once, makes partial progress on multiple fronts, finishes nothing.

Causes:

  • Too many requirements in single prompt
  • Dependencies between tasks not explicit
  • No incremental milestones
# BAD: Everything at once Add a complete user authentication system with login, logout, registration, password reset, email verification, OAuth, 2FA, and session management. # GOOD: Focused iterations # PROMPT.md (Iteration 1) Add user login form. - Create src/pages/Login.tsx - Add route in src/routes.ts - No backend integration yet - just the form - Write "PHASE_1_COMPLETE" to LOG.md when done # PROMPT.md (Iteration 2) Connect login form to auth API. - Use existing authService.login() - Handle success: redirect to /dashboard - Handle error: show error message - Write "PHASE_2_COMPLETE" to LOG.md when done # Key: One focused task per iteration

Prevention: Break complex features into 3-5 focused iterations. Each PROMPT.md should be completable in one loop cycle.

Emergency Recovery

# Loop stuck in infinite retry pkill -f "claude" # Stop the loop git status # See current state git diff # See uncommitted changes git stash # Save changes temporarily # Start fresh git checkout main # Clean state git stash pop # Apply saved changes if wanted # Loop made bad commits git log --oneline -10 # Find last good commit git reset --hard abc123 # Reset to that commit # WARNING: This loses all commits after abc123 # More careful: revert specific commits git revert HEAD~3..HEAD # Revert last 3 commits # This creates new commits that undo the bad ones # Loop created chaos, need to start over git log --all --oneline # See all branches git checkout main # Return to main git branch -D ralph-chaos # Delete failed branch # Pre-commit hooks broken beyond repair # Temporarily disable: rm .husky/pre-commit # Remove hook # Fix the underlying issue, then: git restore .husky/pre-commit # Restore hook
