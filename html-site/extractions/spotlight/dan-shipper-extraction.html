"Think like an engineering manager, not a coder."
PLAN → WORK → REVIEW → COMPOUND → (repeat)
You specify, review, and direct. Agents write all the code.

The mental model shift is profound: you're not a programmer anymore. You're a product manager for AI agents. You write specs, not code. You review output, not syntax. You direct parallel workers, not your own fingers.

3 Design Decisions

WHY 4+ PARALLEL INSTANCES?
One Claude instance = one engineer's output. Four instances = team-scale output. Dan runs 4+ Claude instances simultaneously, each in its own git worktree.
What this means for you
You need git worktrees, one branch per agent, and coordination on file ownership. "Two engineers at Every shipped six features, five bug fixes, and three infrastructure updates in one week."
WHY GIT WORKTREES (NOT DOCKER)?
Git worktrees provide file isolation without virtualization overhead. Each agent works in its own directory with its own branch. No container startup time. No resource duplication.
What this means for you
Learn the git worktree commands. Each worktree is a lightweight clone sharing the same .git directory. Agents can work in parallel without stepping on each other's files.
WHY THE CLAUDE.MD FLYWHEEL?
Every mistake becomes a rule. Every review feedback becomes a pattern. CLAUDE.md accumulates institutional knowledge that survives across sessions, features, and team members.
What this means for you
You MUST update CLAUDE.md after every feature. The discipline is non-negotiable. Week 1: basic instructions. Week 12: comprehensive team brain. Week 52: years of wisdom in a file.
WHY 12 REVIEW AGENTS (NOT 3 OR 1)?
Each agent sees through a specialized lens. A security agent thinks differently than a performance agent. Multiple perspectives catch more issues than one generalist.
What this means for you
The review agents run in parallel. Cost scales linearly but quality scales superlinearly. When agents disagree, you're the human tiebreaker for architectural decisions.
WHY SLASH COMMANDS?
Repeated workflows deserve named shortcuts. /issues pulls from Linear. /work starts a ticket. /review triggers the 12-agent review. Consistency beats improvisation.
What this means for you
Build custom slash commands for your patterns. Store them in .claude/commands/. Every team member uses the same workflows with the same quality.
WHY META-PROMPTS FOR ISSUES?
Dan uses prompts that generate other prompts. Voice memo becomes text. Text becomes detailed spec. Spec becomes GitHub issue. Issue becomes implementation.
What this means for you
Your ideas don't need to be polished. The meta-prompt chain refines them. "I have a prompt that writes prompts for GitHub issues."

#!/bin/bash
# setup-worktrees.sh - Initialize parallel workspaces

set -e

echo "Creating worktrees for parallel development..."

# Create worktree for Feature A
git worktree add ../feature-a feature-a-branch 2>/dev/null || \
  git worktree add ../feature-a -b feature-a-branch

# Create worktree for Feature B
git worktree add ../feature-b feature-b-branch 2>/dev/null || \
  git worktree add ../feature-b -b feature-b-branch

# Create worktree for Tests
git worktree add ../test-coverage test-branch 2>/dev/null || \
  git worktree add ../test-coverage -b test-branch

# Create worktree for Docs
git worktree add ../docs-update docs-branch 2>/dev/null || \
  git worktree add ../docs-update -b docs-branch

echo "Worktrees ready. Run Claude Code in each directory."

6 Gotchas

Real problems you'll hit, with concrete numbers and fixes:

#1 Worktree merge conflicts
Symptom: Same file modified in multiple worktrees simultaneously
Cause: Agents working on overlapping code without coordination
Fix: Coordinate file ownership per agent. One agent owns one area. Time to fix: 10-30 min
#2 CLAUDE.md grows too large
Symptom: Context window fills with instructions, less room for code
Threshold: >30KB starts degrading quality
Fix: Prune old/irrelevant rules quarterly. Archive to separate files. Time to fix: 30 min
#3 Review agents disagree
Symptom: Security agent says one thing, Performance agent says opposite
Cause: Genuine trade-offs with no objectively correct answer
Fix: You're the human tiebreaker for architectural decisions. Time to fix: 5 min
#4 Cost spike from parallel agents
Symptom: $200+ day unexpectedly, budget blown
Cause: 4+ agents running long sessions in parallel
Fix: Monitor token usage, set budget alerts. $400/month for 2 subscriptions is normal. Time to fix: Immediate
#5 Knowledge not compounding
Symptom: Same mistakes repeated across features
Cause: Skipping the CLAUDE.md update step after features
Fix: Audit CLAUDE.md updates after every feature. Non-negotiable discipline. Time to fix: 15 min
#6 Slash command not found
Symptom: /issues returns error, command not recognized
Cause: .claude/commands/ directory missing or path wrong
Fix: Check directory exists and command files have correct names. Time to fix: 2 min
#7 Voice memo transcription errors
Symptom: Spec contains misheard words, wrong feature described
Cause: Transcription passed directly to meta-prompt without review
Fix: Review transcription before meta-prompt runs. Time to fix: 5 min
#8 Linear integration fails
Symptom: /issues cannot pull tickets, auth error
Cause: API key expired or permissions wrong
Fix: Check Linear API key and permissions in .env. Time to fix: 10 min
#9 Git worktree branch conflicts
Symptom: "fatal: branch already exists" error when creating worktree
Cause: Reusing branch names from previous features
Fix: Use unique branch names per feature (date-based or ticket-based). Time to fix: 2 min
#10 Review agent timeout
Symptom: One of 12 review agents takes 10+ minutes, blocks merge
Cause: Complex PR or slow agent on detailed analysis
Fix: Set timeout, use Haiku for simple style checks, Opus for security/architecture. Time to fix: 5 min

7 What's Hard

These are fundamental tensions, not bugs to fix:

Cost vs. Throughput
The tension: More agents = more cost, always. There's no free throughput.
Symptoms: Budget blown mid-month, scrambling to reduce agent count, velocity drops.
Mitigation: Budget caps, monitor daily spend, use Haiku for simple tasks. Accept that 5x output costs 5x. The ROI math still works for most teams.
Knowledge Decay
The tension: CLAUDE.md can become stale or bloated. Old patterns conflict with new architecture.
Symptoms: Agents following outdated rules, conflicting instructions, context waste.
Mitigation: Quarterly audits, remove outdated rules, version CLAUDE.md sections by date. Accept some maintenance overhead is the price of compounding.
Coordination Overhead
The tension: 4+ agents need orchestration. Someone must decide who works on what.
Symptoms: Agents stepping on each other, merge conflicts, duplicated work.
Mitigation: Clear file ownership, branch naming conventions, one agent per feature area. The coordination cost is real but lower than the coordination cost of 4 human engineers.
Mental Model Shift
The tension: You're a PM now, not a programmer. The urge to code yourself is strong.
Symptoms: Taking over when agents struggle, not learning to prompt better, bottlenecking on your own typing.
Mitigation: Weeks of deliberate practice. Resist the coding urge. When you catch yourself typing code, stop and prompt instead. "Haven't typed a function in weeks."
Review Noise
The tension: 12 agents = potentially 12 opinions. Some will conflict. Some will be wrong.
Symptoms: Analysis paralysis, contradictory feedback, spending more time reviewing reviews than code.
Mitigation: Human tiebreaker protocol, prioritize security and test feedback over style, train yourself to filter signal from noise.
Debugging Distributed Work
The tension: When something breaks, it could be any of 4+ agents, any of 12 reviewers, any of N worktrees.
Symptoms: "Who broke this?" investigations, unclear commit history, hard to reproduce issues.
Mitigation: Good logging, explicit handoff notes in commits, clear agent assignment per feature. Accept that distributed debugging is harder than single-thread debugging.

8 When to Use / When Not

USE COMPOUND ENGINEERING WHEN
You're Stage 4+ (comfortable with Claude Code basics)
Daily budget of $30-100+ (multiple Claude instances required)
Solo or 2-3 engineer team (designed for team amplification)
Ongoing development (months+) where knowledge compounds
Multiple features in parallel (single features don't need this)
Quality matters (the 12-agent review adds real value)
DON'T USE COMPOUND ENGINEERING WHEN
Single feature iteration - Use Ralph Wiggum (simpler, cheaper)
Budget under $30/day - Parallel costs add up fast
Below Stage 4 - Build foundation first with simpler patterns
One-off scripts - No compounding benefit, overhead dominates
Team already has review process - Don't replace what works
Factory-scale (8+ roles) - Use Gas Town instead
