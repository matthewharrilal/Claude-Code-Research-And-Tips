2 Core Abstraction: Learn from Failure

"It might take 2 minutes or a day. But the loop continues to experiment and look at prior work to ultimately get you there."
/plugin install ralph-wiggum@claude-plugins-official
One command. The stop hook intercepts exits and re-injects the prompt with updated context.

The Ralph plugin installs a stop hook that intercepts when Claude finishes. Instead of ending the session, it:

  1. Checks if the completion promise was emitted
  2. If not, re-injects the original prompt with updated context
  3. The agent sees file changes + git history from prior iterations
  4. Loop continues until completion promise found OR max iterations reached
"Goal-conditioned loops with memory, evaluation & retry baked in. Once agents can persist, reflect, and verify outcomes autonomously, time becomes the main variable - not human attention."
- Arvid Kahl

This is fundamentally different from extended sessions. Each iteration is a fresh Claude instance. Git history and progress files provide memory. The agent learns from what the previous instance wrote, not from accumulated context.

3 Design Decisions

WHY PLUGIN (NOT CUSTOM SCRIPT)?
The official Ralph plugin is maintained, receives updates, and has a one-command install. Custom bash scripts work but require manual maintenance and miss bug fixes the community pushes.
What this means for you
Start with the plugin. Type /plugin, search "ralph", install. Community fixes bugs for you. Only build custom when you need behavior the plugin doesn't support.
WHY GOAL-CONDITIONED (NOT TIME-BASED)?
Time-based loops are arbitrary. "Run for 2 hours" has no connection to task completion. Goal-conditioned means the agent defines when it's done by emitting a completion promise. The loop ends when the goal is verifiably reached.
What this means for you
You must define clear, verifiable success criteria before starting. "Migrate Jest to Vitest" becomes "MIGRATION COMPLETE when all tests pass with Vitest". Vague goals = infinite loops.
WHY COMPLETION PROMISE PATTERN?
The agent must emit a specific, parseable string to signal completion. This is more reliable than inferring "done" from behavior. The stop hook does a simple grep for the exact promise text.
What this means for you
Include the exact completion promise text in your prompt: --completion-promise "MIGRATION COMPLETE". The agent must emit this exact string. No fuzzy matching.
WHY LOOP WITH FRESH CONTEXT?
Each iteration spawns a new Claude instance. No memory of previous runs except what's written to files. This avoids context rot (quality degradation after ~80K tokens) and ensures each iteration is independent.
What this means for you
Each iteration is a clean slate. Errors don't cascade. If iteration 5 produces bad code, iteration 6 can recover by reading the git diff and trying a different approach.
WHY GIT AS MEMORY?
Git commits are durable, diffable, and agent-readable. The stop hook shows the agent what files changed since last iteration. Git history becomes the agent's "memory" of what previous iterations accomplished.
What this means for you
Your project must be a git repository. Run git init before using Ralph. The agent reads git log and git diff to understand what previous iterations did.
WHY MAX ITERATIONS (NOT INFINITE)?
Unbounded loops are dangerous. Each iteration costs ~$2-3. 50 iterations = $50-100+. Max iterations is a safety backstop that prevents runaway costs when tasks can't converge.
What this means for you
Always set --max-iterations. Choose based on budget and expected complexity. Simple tasks: 10-20. Complex migrations: 30-50. Never run unbounded.

Running a Ralph Loop

# Basic usage
/ralph-loop "Migrate all Jest tests to Vitest" \
  --completion-promise "MIGRATION COMPLETE" \
  --max-iterations 50

# With PRD file
/ralph-loop --prd prd.json \
  --completion-promise "ALL STORIES COMPLETE" \
  --max-iterations 30

[T+0.5min] Iteration 1: Fresh context window loads. Plugin injects prompt + completion promise requirement. Agent analyzes codebase, finds 47 Jest test files.

[T+4min] Iteration 2 ends. Tests migrated: 1/47. Test run: 12 tests passed.
7
[Iterations 3-22] Each iteration: 30-90 seconds. Agent learns from git log what was done. Migrates roughly one file per iteration (conservative).
8
[T+18min] Iteration 23: Last file migrated: tests/utils/helpers.test.ts. Full test run: 312 tests passed.
9
Agent removes Jest dependencies, updates package.json. Outputs: "MIGRATION COMPLETE"
10
[T+18min] Loop terminates. Completion promise detected. Output: "Ralph finished! Iterations used: 23 of 50"

6 Gotchas

Critical warnings from the community - real problems with real fixes:

1. Plugin not found
Symptom: /plugin ralph shows no results
Fix: The plugin may have been renamed. Try searching "wiggum" or check the official plugins list. Plugin was broken ~10 hours ago and fixed.
Time to fix: 2 minutes
2. Windows compatibility issues
Symptom: Script fails on Windows
Cause: Bash scripts assume Unix environment
Fix: Use WSL2 or install jq manually. The stop hook relies on bash.
Time to fix: 15 minutes
3. Runaway costs
Symptom: $100+ bill unexpectedly
Cause: Running without max-iterations or task that never converges
Fix: ALWAYS set --max-iterations. Budget: $2-3 per iteration. 50 iterations = $50-100+.
Time to fix: Prevention only
4. No verification layer
Symptom: Agent claims "done" but code is broken
Cause: Completion criteria don't include tests
Fix: Add explicit tests in completion criteria: "MIGRATION COMPLETE when npm test passes"
Time to fix: 10 minutes setup
5. Completion promise not detected
Symptom: Loop runs to max despite agent saying "done"
Cause: Exact match required. Agent said "Migration complete!" not "MIGRATION COMPLETE"
Fix: Include the exact promise text in your prompt. The stop hook greps for exact match.
Time to fix: 5 minutes
6. Git not initialized
Symptom: Agent can't track history, each iteration repeats work
Cause: Ralph needs git for memory
Fix: Run git init first. Ralph reads git log/diff to understand prior iterations.
Time to fix: 1 minute
7. Poor planning upfront
Symptom: Agent wanders, doesn't converge
Cause: Vague goal definition
Fix: Define clear goal condition BEFORE starting. "Make it better" = infinite loop. "All tests pass" = convergence.
Time to fix: 15 minutes planning

7 What's Hard

These are fundamental tensions, not bugs to fix:

Cost vs Thoroughness
The tension: More iterations = better results = more cost. There's no free lunch.
Coping strategy: Set budget upfront, choose iteration limit accordingly. Accept that some tasks need 30+ iterations and that's a real cost.
Autonomy vs Verification
The tension: True AFK means trusting without checking. But agents make mistakes that compound across iterations.
Coping strategy: Build robust test suite BEFORE using Ralph. Your tests are your verification layer. Without them, you're flying blind.
Task Scope vs Context Limits
The tension: Big tasks need many iterations, but each iteration is constrained by context window. You can't plan across iterations from within an iteration.
Coping strategy: Break into small verifiable chunks. Use prd.json to track dependencies. Let the PRD be the cross-iteration memory.
Brute Force vs Planning
The tension: Ralph enables "just try it" but planning is still better. Brute force costs tokens; planning costs human time.
Coping strategy: Ralph doesn't replace upfront design. Invest 15 minutes planning to save $50+ in iterations. The loop is for execution, not discovery.
Goal Definition
The tension: Vague goals = vague results. But defining precise goals requires knowing the solution in advance.
Coping strategy: Invest time in clear, verifiable completion criteria. If you can't write a test for "done", the task isn't Ralph-suitable.
Learning Curve
The tension: Looks simple (it's just a loop!) but has many failure modes. You'll waste iterations learning what works.
Coping strategy: Start with small, low-stakes tasks. Learn the failure modes on cheap problems. Graduate to overnight runs after you've calibrated.

8 When to Use / When Not

Well-defined goals with verifiable completion
Mechanical, repetitive tasks
Migrations (Jest to Vitest, etc.)
Architecture implementation from spec
Budget: $20-100+ per task
Good existing test coverage
DON'T USE RALPH WHEN
Ambiguous/subjective tasks ("make it pretty")
Security-sensitive code
Tasks without clear success criteria
Budget-constrained situations
Need parallel execution (use CC Mirror)
