The Iron Law
YOU DO NOT WRITE CODE.
YOU DO NOT RUN COMMANDS.
YOU DO NOT EXPLORE CODEBASES.
You are the CONDUCTOR. Your agents play the instruments.

The orchestrator has ONE job: coordinate, delegate, synthesize. Workers execute using tools directly. This separation is load-bearing - violate it and the pattern collapses.

Tool Ownership Separation

Orchestrator Uses
  • Read (references, guides, 1-2 files max)
  • TaskCreate, TaskUpdate, TaskGet, TaskList
  • AskUserQuestion
  • Task (to spawn workers)
Workers Use
  • Read, Write, Edit, Bash
  • Glob, Grep, WebFetch, WebSearch, LSP
  • CAN'T: Spawn sub-agents
  • CAN'T: Manage task graph

3 Design Decisions

WHY THE IRON LAW (ORCHESTRATOR DOESN'T CODE)?
If the orchestrator writes code, it loses oversight. Its context fills with implementation details instead of coordination. Separation of concerns isn't just clean architecture - it's context protection.
What this means for you
Never let the orchestrator get pulled into implementation. If you catch yourself thinking "I'll just do this quick fix," STOP. Spawn a worker. The temptation to violate the Iron Law is strongest when tasks seem trivial.
WHY TOOL OWNERSHIP SEPARATION?
Workers CAN'T call TaskCreate/TaskUpdate. This prevents recursive spawning (workers spawning sub-workers spawning sub-sub-workers...) which would cause context explosion and uncontrollable agent proliferation.
What this means for you
The worker preamble must explicitly block Task tools. Without this, a worker might "helpfully" decompose its own task. This is never helpful - it corrupts the task graph and confuses the orchestrator.
WHY MODEL SELECTION (HAIKU/SONNET/OPUS)?
Models have different cost/capability tradeoffs. Haiku is fast and cheap but limited. Opus is powerful but expensive and slow. Matching model to task type optimizes for both quality AND cost.
What this means for you
Use Haiku liberally for reconnaissance (file lookups, grepping, pattern finding). Use Sonnet for the bulk of implementation work. Reserve Opus for architecture decisions and ambiguous tasks that require judgment.
WHY DOMAIN REFERENCES?
Different task types require different decomposition patterns. A feature request needs different subtasks than a code review. Domain references teach the orchestrator HOW to decompose before it decomposes.
What this means for you
Load the appropriate domain reference (software-development.md, code-review.md, testing.md) BEFORE decomposing a task. This front-loads domain expertise into the orchestrator's context.

5 The Path of a Task

Follow a feature request through the CC Mirror orchestration system:

1
[T+0s] Request arrives: "Build a user authentication system with JWT tokens"
2
[T+5s] Orchestrator reads domain reference: software-development.md
3
[T+10s] Task decomposition into graph:
  • #1 Explore auth patterns (reconnaissance)
  • #2 Create user model and migrations
  • #3 Implement password hashing
  • #4 Create JWT token service
  • #5 Build auth endpoints
  • #6 Write security tests
4
[T+15s] Model selection per task:
  • #1: Haiku (simple file exploration)
  • #2-#5: Sonnet (structured implementation)
  • #6: Opus (security edge cases)
5
[T+20s] Parallel worker spawn (3 concurrent):
  • [HAIKU] #1 Explore auth patterns - running
  • [SONNET] #2 Create user model - running
  • [SONNET] #3 Password hashing - running
  • [wait] #4 JWT service - blocked by #3
  • [wait] #5 Auth endpoints - blocked by #4
  • [wait] #6 Security tests - blocked by #5
6
[T+30s] Haiku #1 completes (10-30s typical). Returns file list. Orchestrator doesn't act on it - just notes it for context.
7
[T+60s] Sonnet #2 completes (30-90s typical). Created src/models/user.py. #3 already running (no dependency).
8
[T+90s] Sonnet #3 completes. Calls TaskUpdate to unblock #4. Orchestrator spawns Sonnet for #4.
9
[T+5min] All tasks complete. Orchestrator synthesizes results, reports to user.

Timing expectations: Skill loading 5-10s, Haiku tasks 10-30s, Sonnet tasks 30-90s, Opus tasks 60-180s, auto-compact 10-20s when triggered.

6 Gotchas

Real problems with specific symptoms, numbers, and fixes:

#1: Context collapse during orchestration
Symptom: "Unable to compact", agent loses track of task graph
Cause: Context filled with worker outputs, no room to think
Fix: Enable auto-compact beta feature (@nummanali: "Just make sure auto compact is on")
Time to fix: 5 minutes (settings change)
#2: Worker spawns sub-agents
Symptom: Recursive explosion, unexpected agent proliferation, context overflow
Cause: Worker preamble missing or doesn't explicitly block spawning
Fix: Use standard worker preamble with explicit "Do NOT spawn sub-agents"
Time to fix: 2 minutes (preamble update)
#3: Wrong model for task
Symptom: Haiku fails on complex task, Opus overkill for simple lookup
Cause: Ad-hoc model selection instead of following guide
Fix: Follow model selection guide strictly. When in doubt, use Sonnet.
Time to fix: 0 minutes (planning discipline)
#4: Worker calls TaskCreate
Symptom: Task graph corrupted, orchestrator confused about state
Cause: Worker preamble doesn't explicitly block Task tools
Fix: Worker preamble must include "Do NOT call TaskCreate or TaskUpdate"
Time to fix: 2 minutes (preamble update)
#5: High initial token cost
Symptom: 5299 tokens consumed just loading skill file
Reality check: This is the investment - it saves tokens via delegation
Fix: Accept upfront cost. Measure delegation ROI over session, not per-call.
Trade-off: "The additional instructions guide the orchestrator to manage sub agents, saving its own tokens significantly" - @nummanali
#6: Domain reference not loaded
Symptom: Orchestrator decomposes poorly, wrong task structure
Cause: Jumped straight to decomposition without reading domain reference
Fix: Read appropriate domain reference BEFORE decomposing any task
Time to fix: 1 minute (add read step)
#7: Orchestrator starts coding
Symptom: Iron Law violation, context bloats, loses oversight
Cause: "Quick fix" temptation, especially for trivial tasks
Fix: Re-read Iron Law. Spawn worker for ANY coding, no exceptions.
Time to fix: Immediate (discipline)
#8: Blocked tasks not unblocking
Symptom: Dependency cascade broken, tasks stuck waiting
Cause: TaskUpdate not called correctly after worker completion
Fix: Verify TaskUpdate call with correct task ID on worker completion
Time to fix: 5 minutes (debug task graph)
#9: Team feature causing issues
Symptom: Unexpected behavior in multi-agent coordination
Cause: Team feature conflict with orchestration skill
Fix: "You can disable the team feature under variant settings" - @nummanali
Time to fix: 2 minutes (settings)
#10: Workers not reporting absolute paths
Symptom: Can't find created files, relative paths don't resolve
Cause: Worker preamble doesn't require absolute paths in output
Fix: Worker preamble must include "Report your results with absolute file paths"
Time to fix: 2 minutes (preamble update)

7 What's Hard

These are fundamental tensions, not bugs. You manage them, not fix them.

Skill Size vs. Capability
The tension: 5299 tokens is expensive upfront, but enables sophisticated coordination. Smaller skill = less capability. Larger skill = more upfront cost.
Symptoms: Sticker shock at token count. Temptation to trim skill file.
Mitigation: Accept upfront cost. Measure delegation savings over full session. The 5299 tokens enable orchestrator to spawn 50K+ tokens of parallel work efficiently.
Orchestrator Discipline
The tension: It's always faster to "just do it" than spawn a worker. But this violates the Iron Law and corrupts the pattern.
Symptoms: Orchestrator context bloating. Lost oversight. "It seemed simpler to just..."
Mitigation: Strict Iron Law adherence. No exceptions. The discipline IS the pattern.
Model Selection Judgment
The tension: Not always clear which model for a task. Haiku vs Sonnet boundary is fuzzy. Opus seems like overkill until it's not.
Symptoms: Haiku failures on "simple" tasks that weren't. Opus waste on tasks Sonnet could handle.
Mitigation: Start with defaults (Haiku for lookups, Sonnet for implementation, Opus for architecture). Tune based on failure patterns. Accept that some waste is the cost of learning.
Parallel vs. Sequential Dependencies
The tension: Some tasks have hidden dependencies. Parallelizing them causes race conditions or wasted work.
Symptoms: Workers overwriting each other. Tasks completing but output incompatible.
Mitigation: Conservative dependency marking. When in doubt, sequence. Better to be slower than broken.
Worker Context Limits
The tension: Each worker has its own context limit. Scoping tasks too large causes worker context overflow.
Symptoms: Workers losing track mid-task. Incomplete implementations.
Mitigation: Scope tasks appropriately. If task description is >500 words, split it. Workers are for execution, not planning.
Debugging Distributed Work
The tension: When something fails, which worker failed? Why? Hard to trace across parallel agents.
Symptoms: "Something broke but I don't know what." Hours lost to investigation.
Mitigation: Good logging. Worker preamble output format. TaskUpdate with failure details. Accept that distributed debugging is hard.

8 When to Use / When Not

USE CC MIRROR ORCHESTRATION SKILL WHEN
3+ parallelizable subtasks (worth orchestration overhead)
Stage 5+ on complexity ladder (multi-agent familiarity)
Multi-domain project (frontend + backend + tests)
Budget for parallel workers ($10-50/day)
High coordination need (workers need orchestrated handoffs)
Willing to invest 5299 token upfront cost
DON'T USE CC MIRROR ORCHESTRATION SKILL WHEN
Single focused task - Use basic Ralph or single Claude session
Sequential work only - No parallelization benefit
Very simple project - Orchestration overkill
Below Stage 5 - Learn basic multi-agent first
Budget constrained - Use native Task API (lower overhead)
Need maximum control - Use Gas Town (explicit roles)
