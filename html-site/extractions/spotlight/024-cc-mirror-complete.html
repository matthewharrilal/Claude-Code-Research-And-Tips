2 The Core Abstraction

"Multi-agent shouldn't require new tools. It should unlock what's already there."
npx cc-mirror quick --provider mirror --name mclaude
3 commands to multi-agent orchestration. That's the discovery.

Numman Ali reverse-engineered Claude Code and discovered Anthropic had built a complete multi-agent orchestration system but disabled it in production. CC Mirror re-enables TaskCreate, TaskList, TaskGet, TaskUpdate, addBlocks, and addBlockedBy - primitives that let you decompose complex work into parallel task graphs.

Hub-and-Spoke Architecture

+------------------+ | ORCHESTRATOR | | (Conductor) | +--------+---------+ | | Task() with run_in_background=True | +---------------+---------------+ | | | v v v +---------+ +---------+ +---------+ | Worker1 | | Worker2 | | Worker3 | | (Coder) | | (Tester)| | (Docs) | +---------+ +---------+ +---------+ | | | +-------------+-------------+ | Shared Filesystem (handoff files)

The orchestrator manages the task graph. Workers execute in parallel, isolated contexts. Results flow back through the filesystem. Dependencies auto-resolve via BlockedBy relationships.

The Iron Law

The Iron Law of CC Mirror Orchestration
YOU DO NOT WRITE CODE. YOU DO NOT RUN COMMANDS. YOU DO NOT EXPLORE CODEBASES.

You are the CONDUCTOR. Your agents play the instruments.

Violating the Iron Law causes context pollution (orchestrator loses strategic view), context bloat (less room for coordination), verification bypass (same agent writes and approves), and uncontrolled recursion (exponential agent explosion).

3 Design Decisions

WHY SKILL-BASED (NOT EXTERNAL PROCESS)?
"Multi-agent shouldn't require new infrastructure." CC Mirror injects a ~5,299 token skill that enables orchestration. No Docker containers, no external processes, no servers to maintain.
What this means for you
Zero infrastructure to manage. One npx command and you have multi-agent. The skill IS the orchestration layer.
WHY HIDDEN ORCHESTRATION?
The user shouldn't need to think about coordination. Hidden orchestration reduces cognitive load - you describe what you want built, the system figures out how to parallelize it.
What this means for you
Multi-agent that FEELS like single-agent. You say "Build a REST API with tests and docs" and the decomposition happens automatically. Team mode available for debugging visibility.
WHY TASK PRIMITIVES (NOT AGENT PRIMITIVES)?
Tasks are the meaningful abstraction. Agents are implementation details. TaskCreate + BlockedBy creates a dependency DAG. The system manages which agent does what.
What this means for you
Focus on WHAT needs doing, not WHO does it. Define acceptance criteria, let the system assign and coordinate. This is declarative orchestration.
WHY NPX INSTALLATION?
Zero setup friction means more adoption. npx cc-mirror quick runs immediately without global installs or configuration files.
What this means for you
From zero to multi-agent in under 5 minutes. No package.json changes, no dotfiles, no CI updates. Works immediately.

Task API Reference

// TaskCreate - Define a task with dependencies
{
  "subject": "Implement authentication middleware",
  "description": "Create Express middleware for JWT verification",
  "acceptanceCriteria": [
    "Middleware validates JWT signature",

  "blockedBy": ["task-id-001"],
  "blocks": ["task-id-003"]

// TaskList - Filter and view tasks
TaskList (filter: status='open', not blocked)

// TaskUpdate - Claim, comment, resolve

  "status": "in_progress",

5 The Path of a Task

Follow "Add JWT authentication to the API" through the CC Mirror system:

1
[T+0s] Main Claude receives request. Orchestration skill activates, analyzes complexity.
2
[T+2s] Decomposition phase. TaskCreate: "task-001: JWT middleware", TaskCreate: "task-002: User model", TaskCreate: "task-003: Auth routes" (BlockedBy: [001, 002])
3
[T+5s] Parallel subagent spawn. Worker A claims task-001, Worker B claims task-002. Worker C waits (task-003 blocked).
4
[T+30s] Worker A completes. TaskComplete: task-001. Result merged to main context.
5
[T+45s] Worker B completes. TaskComplete: task-002. task-003 now unblocked.
6
[T+50s] Worker C spawned. Works on task-003 with context from task-001 and task-002.
7
[T+90s] Worker C completes. TaskComplete: task-003. All tasks complete.
8
[T+95s] Main Claude synthesizes. Presents final result to user: "JWT authentication added with middleware, user model, and routes."

Total time: ~95 seconds. Parallelism gain: Tasks 001 and 002 ran simultaneously, saving ~30s vs sequential.

6 Gotchas

Real problems you'll hit, with concrete fixes:

#1: Skill not loading
Symptom: "No orchestration capability" message
Cause: Skill file not injected or path wrong
Fix: Re-run npx cc-mirror quick, verify skill at ~/.claude/skills/orchestration/
Time to fix: 5 minutes
#2: Context collapse
Symptom: Subagents lose track mid-task, produce garbage
Cause: Auto-compact not enabled
Fix: Enable auto-compact beta feature in variant settings
Time to fix: 2 minutes
#3: Task stuck blocked
Symptom: Task never becomes claimable, stays "blocked"
Cause: Circular dependency in BlockedBy DAG (A blocks B, B blocks A)
Fix: Check DAG for cycles. Use npx cc-mirror tasks graph to visualize.
Time to fix: 10 minutes
#4: Merge conflicts from subagents
Symptom: Multiple workers edit same file, conflicts on merge
Cause: Task decomposition didn't isolate file ownership
Fix: Add file ownership to task descriptions. Use git worktrees for isolation.
Time to fix: 10-30 minutes
#5: Windows installation fails
Symptom: npx cc-mirror errors on Windows
Cause: Known Windows path compatibility issues
Fix: Use WSL2. Community PRs in progress for native Windows support.
Time to fix: 15 minutes (WSL setup)
#6: Token cost spike
Symptom: Higher than expected API costs
Cause: Team mode enabled when not needed
Fix: Disable team mode in variant settings if you don't need the visibility dashboard
Time to fix: 2 minutes

7 What's Hard

These are fundamental tensions, not bugs to fix:

Hidden Complexity = Harder Debugging
The tension: Hidden orchestration reduces cognitive load BUT makes failures opaque. When something breaks, you can't see inside.
Symptoms: "The task failed" with no visibility into which subagent did what.
Mitigation: Use team mode for debugging sessions. Trust logs. Accept that visibility has a cost (tokens).
Context Limits Still Apply
The tension: Each subagent has bounded context. Orchestration doesn't give you infinite context - it gives you parallel bounded contexts.
Symptoms: Large tasks fail even with parallelization because individual workers hit limits.
Mitigation: Design smaller tasks. Each task must be completable within one context window. If it's not, decompose further.
80% Not 100%
The tension: CC Mirror covers "80% of multi-agent use cases" but complex orchestration needs Gas Town. Know when to graduate.
Symptoms: Hitting walls at ~10+ parallel tasks. Need named persistent workers. Need inbox-based async.
Mitigation: CC Mirror is the on-ramp, not the final destination. When you consistently hit limits, move to Gas Town.
Skill Injection Fragility
The tension: CC Mirror patches minified JS to enable hidden APIs. Claude Code updates can break this.
Symptoms: Suddenly orchestration stops working after Claude Code update.
Mitigation: Pin versions in production. Follow CC Mirror GitHub for update compatibility. Run npx cc-mirror doctor after updates.
DAG Design is Non-Trivial
The tension: BlockedBy requires thinking upfront about dependencies. Bad DAG design = stuck tasks, wrong order, wasted work.
Symptoms: Tasks execute in wrong order. Circular dependencies. Excessive sequentialization kills parallelism.
Mitigation: Start simple (linear chains), evolve to DAGs. Use npx cc-mirror tasks graph to visualize before running.

8 When to Use / When Not

USE CC MIRROR WHEN
Task parallelism would speed up delivery (decomposable work)
Zero infrastructure tolerance (no Docker, no processes)
Setup time constraint < 5 minutes
Complexity beyond Ralph but below Gas Town
Stage 4+ developer experience (comfortable with Claude Code)
DON'T USE CC MIRROR WHEN
Simple sequential tasks -> Ralph Wiggum is simpler
Factory scale (20+ agents) -> Gas Town scales better
Distributed multi-machine -> Station is designed for this
Maximum control needed -> CC Mirror abstracts too much
Mission-critical production -> Beta/disabled feature risk
