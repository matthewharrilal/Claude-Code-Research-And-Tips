2 The Core Abstraction

"Gas Town is an Idea Compiler. You just make up features, design them, file the implementation plans, and sling work around to your polecats and crew."
go install github.com/steveyegge/gastown/cmd/gt@latest && gt init
You're not a programmer anymore. You're a product manager with an AI engineering team.

Gas Town inverts your role. Instead of writing code, you file feature requests. Instead of debugging, you review completed work in your inbox. The mental shift is profound: you become the head chef of a kitchen with AI robotic sous chefs. Work becomes fluid, an uncountable that you sling around freely.

Mayor Opus 4.5
Chief coordinator and concierge. Receives high-level directives from you (Overseer), routes work to appropriate rigs, manages cross-rig coordination.
Deacon Haiku
Health monitoring and agent coordination. Watches agent health across the town, handles handshakes, reports anomalies to Mayor.
Dogs Sonnet
Quality gates and watchdogs. Reviews work before acceptance, enforces quality standards, can reject/rollback work.
Refinery Opus 4.5
Task decomposition engine. Breaks down large tasks into agent-sized work, manages the merge queue with sequential rebasing.
Witness Haiku
Per-rig observer. Watches all work, records history and context, provides state to new workers, unsticks drifting agents.
Polecat Sonnet
Named, persistent agent workers (alice, bob, carol). Execute assigned tasks, maintain context across sessions, specialize in rig-specific knowledge.
Crew Haiku
Ephemeral burst workers. Short-lived, task-specific, cheap and disposable. Spawn, complete, disappear.

Beads: The Memory System

Agents suffer from "50 First Dates" syndrome - they wake up with no memory of yesterday. Beads (~225K lines of Go) solves this with git-backed universal context: 

# Beads provides queryable task state
bd ready           # List tasks with zero blockers (what can be done NOW)
bd create "Title"  # Create new task
bd dep add X Y     # X is blocked by Y
bd show <id>       # Full task state and history

Hash-based IDs (e.g., bd-a1b2) prevent merge conflicts. Memory decay through compaction summarizes old closed tasks. All state persisted in git - survives crashes, provides audit trail, enables rollback.

3 Design Decisions

WHY 8 ROLES (NOT 3 OR 20)?
The 8 roles evolved through iteration. They cover the full development lifecycle without excessive coordination overhead. Fewer roles means insufficient specialization; more means coordination explodes.
What this means for you
8 is enough specialization to parallelize work, few enough to coordinate. Don't try to add roles. If you need something different, it's probably a Polecat specialization, not a new role.
WHY GO FOR BEADS?
"Go is just... good. It's boring. You can always understand it!" Type safety for complex state management. Readable by both AI and humans. No TypeScript type wrestling (1/3 to 1/2 of diffs in TS are type fixes).
What this means for you
You need Go installed. If you don't know Go, you can still use Gas Town - you just can't extend Beads. The CLI abstracts the implementation.
WHY GIT AS DATABASE?
Durable, diffable, auditable storage with zero infrastructure. No Postgres to maintain. Everything survives crashes. Full history of all agent actions. Roll back to any point in time.
What this means for you
Limited query power (can't do complex joins). But you get reliability without ops burden. No heavyweight message bus required.
WHY TMUX (NOT DOCKER)?
Local development with visual observation. You can see what every agent is doing in real time. Docker would add latency and hide agent behavior.
What this means for you
tmux proficiency is mandatory. No alternative interface currently exists. This is both the limitation and the power - direct access to agent state.

Startup Script (gastown-start.sh)

#!/bin/bash
SESSION="gastown"

# Kill existing session if requested
if [ "$1" = "--fresh" ]; then
  tmux kill-session -t $SESSION 2>/dev/null
fi

# Create session
tmux new-session -d -s $SESSION -n 'overview'

# Infrastructure windows
tmux new-window -t $SESSION -n 'mayor'
tmux new-window -t $SESSION -n 'infra'
tmux split-window -h -t $SESSION:infra

# Per-rig windows
for RIG in project1 project2; do
  tmux new-window -t $SESSION -n "rig-$RIG"
  tmux split-window -h -t "rig-$RIG"
done

# Polecat windows
for POLECAT in alice bob carol; do
  tmux new-window -t $SESSION -n "$POLECAT"
done

# Start roles
tmux send-keys -t $SESSION:mayor 'gt mayor up' C-m
tmux send-keys -t $SESSION:infra.0 'gt deacon up' C-m
tmux send-keys -t $SESSION:infra.1 'gt dogs up' C-m
tmux send-keys -t $SESSION:rig-project1.0 'gt witness up project1' C-m
tmux send-keys -t $SESSION:rig-project1.1 'gt refinery up project1' C-m
tmux send-keys -t $SESSION:alice 'gt polecat up project1 alice' C-m
tmux send-keys -t $SESSION:bob 'gt polecat up project1 bob' C-m

# Attach
tmux select-window -t $SESSION:overview
tmux attach -t $SESSION

5 The Path of a Task

Follow "Build user authentication" through Gas Town to see how the 8 roles coordinate:

1
[T+0 min] OVERSEER (YOU) files request: gt inbox send my-project "Build user authentication with JWT"
2
[T+1 min] MAYOR receives and routes. Analyzes scope, selects rig, routes to Refinery for decomposition.
3
[T+3 min] REFINERY decomposes into subtasks in Beads: bd-a1b2: "Design JWT structure", bd-c3d4: "Implement token generation" (blocked by bd-a1b2), etc.
4
[T+5 min] DEACON checks agent health. Monitors alice, bob, carol - all healthy and available.
5
[T+6 min] POLECAT ALICE assigned first task: gt polecat assign alice bd-a1b2. Works in isolated worktree.
6
[T+20 min] ALICE completes, commits. WITNESS logs: "Alice completed bd-a1b2, JWT uses RS256". bd-c3d4 UNBLOCKS. BOB assigned next task.
7
[T+45 min] DOGS run quality check: gt dogs bark. Reviews alice's changes. Approves security, suggests follow-up task.
8
[T+90 min] REFINERY manages merge queue. Sequences rebases: alice -> bob -> carol.
9
[T+120 min] FEATURE COMPLETE. OVERSEER notified via inbox: "6 subtasks, 3 polecats, 2 hours."

Key insight: You filed ONE request. The factory decomposed it into 6 subtasks, parallelized across 3 workers, quality-checked everything, and merged in sequence. You reviewed decisions in your inbox. That's the PM-to-engineering-team mental model.

6 Gotchas

Real problems you'll hit, with concrete numbers and fixes:

#1: Context Collapse
Symptom: Agents stop making sense, circular reasoning
Cause: Beads state corruption or no auto-compact
Fix: gt beads sync --force (5 min recovery)
#2: Merge Wall
Symptom: Conflicting commits from multiple polecats
Cause: Same files edited by different workers
Fix: Use Refinery merge queue, avoid same files (10-30 min recovery)
#3: Runaway Crew
Symptom: Cost spike, dozens of Crew spawned unexpectedly
Cause: Refinery misconfiguration or task explosion
Fix: gt crew purge --all then set limits in config (2 min)
#4: Mayor Deadlock
Symptom: No work getting assigned, town seems stuck
Cause: Mayor waiting for human decision in inbox
Fix: gt inbox then respond to pending items (2 min)
#5: GUPP Not Triggering
Symptom: Agent starts but doesn't pick up work
Cause: Claude Code is "so miserably polite" - needs a nudge
Fix: gt nudge 30-60s after start (1 min)
#6: Cost Explosion
Symptom: $500+ day unexpectedly
Cause: Uncapped crew, Opus overuse, or stuck loops
Fix: gt pause --all immediately, review crew count and model assignments

7 What's Hard

These are fundamental tensions, not bugs to fix:

Cost vs. Throughput
The tension: More agents = more output = more cost. Always. No way around it.
Symptoms: Budget anxiety while watching polecats burn through tokens.
Mitigation: Budget caps, use Haiku where possible (Witness, Deacon, Crew). Accept that $50-200/day is table stakes.
Chaos vs. Control
The tension: Agents make mistakes. Some bugs get fixed 2-3 times. Some fixes get lost. Can't prevent all errors.
Symptoms: Frustration at rework, feeling loss of control.
Mitigation: Focus on THROUGHPUT, not perfection. Dogs provide quality gates. Accept some rework as cost of parallelism.
Mental Model Shift
The tension: You're a PM now, not a programmer. This feels wrong at first. Takes weeks to adjust.
Symptoms: Urge to "just do it myself," micromanaging polecats.
Mitigation: Trust the factory. Review inbox, not code. Let Dogs handle quality. Your job is direction, not implementation.
Prerequisite Cliff
The tension: Gas Town requires Stage 7+ experience. If you're not there yet, "you'll get your face ripped off by superintelligent chimpanzees."
Symptoms: Constant firefighting, can't recover from failures, town always broken.
Mitigation: Can't shortcut this. If struggling, go back to CC Mirror or Ralph loops. Build foundation first.

8 When to Use / When Not

USE GAS TOWN WHEN
Stage 7+ developer (100+ hours multi-agent experience)
Budget: $50-200/day is acceptable (multiple Claude accounts)
Solo developer wanting team-scale output
Large, ongoing project with multiple domains
High chaos tolerance (some bugs, some rework expected)
tmux proficiency (mandatory, no alternative)
DON'T USE GAS TOWN WHEN
Single feature (use Ralph Wiggum - simpler, cheaper)
Budget-constrained (use CC Mirror - lower cost multi-agent)
Need stability and predictability (use Manual + Ralph)
Below Stage 6 on dev evolution (build foundation first)
Team already exists (Gas Town replaces team, doesn't augment)
