Support
Essence (15 words)
Break complex tasks into agent-sized pieces with verifiable criteria. Decomposition quality determines orchestration success.

2 The Core Abstraction

"Each task must fit in one context window, have verifiable completion, and explicit dependencies."
DECOMPOSITION = f(Task Size, Dependencies, Verification, Context)
Master this before mastering any orchestration tool.

AI agents fail when tasks exceed context window (quality degrades), verification is impossible (vague "done" signals), dependencies are unclear (building on non-existent foundations), or scope is ambiguous (wandering fills context with tangents).

Decomposition Strategies

Strategy Description When to Use
Horizontal (Parallel) Independent subtasks, no dependencies Tasks can run simultaneously
Vertical (Sequential) Ordered subtasks, each depends on previous Order matters (schema -> backend -> UI)
Hybrid Mix of parallel and sequential Complex projects with both types
Recursive Subtasks may need further decomposition Large complex tasks (keep splitting)
Domain-Based Split by technical domain Feature development (FE/BE/DB)
Phase-Based Split by workflow phase Project workflows (design -> test)
Strategy Selection Decision Tree 
START: "I have a complex task"
|
+-- Can subtasks run independently?
|   +-- YES -> Horizontal (parallel) decomposition
|   +-- NO  -> Vertical (sequential) decomposition
|
+-- Is the task domain-specific or workflow-oriented?
|   +-- DOMAIN   -> Domain-based decomposition
|   +-- WORKFLOW -> Phase-based decomposition
|
+-- Does any subtask not fit in one session?
|   +-- YES -> Recursive decomposition needed
|   +-- NO  -> Stop decomposing
|
+-- Are all acceptance criteria verifiable?
|   +-- NO  -> Refine criteria
|   +-- YES -> Continue
|
+-- Are all dependencies explicit?
    +-- NO  -> Document dependencies
    +-- YES -> Decomposition complete

3 Design Decisions

WHY VERIFIABLE ACCEPTANCE CRITERIA?
Vague criteria like "works correctly" or "good UX" lead to false "done" signals. The agent marks tasks complete without proper verification. This compounds across iterations, producing garbage.
What this means for you
Every criterion must be checkable by the agent. "npm run typecheck passes" not "code is clean". "Button is green" not "looks nice". Include test commands as final criteria.
WHY AGENT-SIZED TASKS?
Context degrades with tokens ("context rot"). At ~80K tokens, quality drops dramatically. If a task can't complete in one context window, the agent loses track mid-task and produces broken code.
What this means for you
The 2-3 sentence test: if you can't describe the task in 2-3 sentences, split it. "Add investorType column with default 'cold'" is good. "Build the investor management system" is bad.
WHY EXPLICIT DEPENDENCIES?
Hidden dependencies cause race conditions and wasted work. Agent tries to build UI component before API exists. "Cannot find type X" errors everywhere. Previous work must be declared.
What this means for you
Every task should declare blockedBy and blocks. Standard order: schema -> backend -> frontend. Watch for "Module not found" errors - they signal missing dependencies.

4 Task Templates

Basic Task Format 
## Task: [Title]

### Description
[What to do in 2-3 sentences]

### Acceptance Criteria
- [ ] [Verifiable criterion 1]
- [ ] [Verifiable criterion 2]
- [ ] npm run typecheck passes
- [ ] npm run test passes

### Context
[Relevant information for this task]

### Dependencies
- Blocked by: [task-id or "none"]
- Blocks: [task-id or "none"]

Domain-Based Template

## Feature: [Feature Name]

### Database Tasks (run first)
- DB-001: [Task] | Criteria: [Verifiable]

### Backend Tasks (can parallelize after DB)
- BE-001: [Task] | Criteria: [Verifiable]
- BE-002: [Task] | Criteria: [Verifiable]

### Frontend Tasks (can parallelize after BE)
- FE-001: [Task] | Criteria: [Verifiable]
- FE-002: [Task] | Criteria: [Verifiable]

### Integration Tasks (run last)
- INT-001: [Task] | Depends: FE-*, BE-* | Criteria: [Verifiable]

5 Path of a Task: From Vague to Executable

Like PostgreSQL's "follow the query," trace a vague request through decomposition until it becomes agent-executable subtasks. See every decision point and transformation.

Scenario: "Build a notification system"

1
Vague Task Arrives
Stakeholder says: "We need a notification system." No criteria, no scope, no boundaries. Classic enterprise request.
Red Flag: If given to agent now -> would build something, but probably wrong thing. Context overflow as agent explores.
2
Clarification Questions
What types of notifications? (email, push, in-app) What triggers them? (user actions, system events) Who configures them? (admin, users, both)
Decision Branch: Answers clear? YES -> Continue. NO -> Get answers before proceeding.
3
Scope Emerges
After clarification: "Email notifications for user signups and password resets. Admin configures templates." Now we have boundaries.
4
Identify Natural Boundaries
Database: Notification templates table, notification logs table.
Backend: Template CRUD API, notification trigger service, email sending service.
Frontend: Admin template editor, notification history view.
Pattern Match: This is Domain-Based decomposition (FE/BE/DB).
5
Check Dependency Order
DB tables must exist before BE API can use them. BE API must exist before FE can call them. This is Vertical ordering on top of Domain boundaries.
Decision Branch: Can any tasks run parallel? YES -> Mark them. Template CRUD and notification service can parallel after DB.
6
Size Check: 2-3 Sentence Test
"Create notification_templates table with id, name, subject, body, created_at columns. Add migration." - Passes.
"Build complete admin interface for template management" - FAILS. Too vague, too big.
Action: Recursive decomposition needed on "admin interface" -> Split into: list view, edit form, delete confirmation.
7
Add Verifiable Criteria
Each task gets machine-checkable acceptance:
- "Migration runs without error" -> npm run migrate
- "API returns 200 on GET /templates" -> curl test
- "Tests pass" -> npm run test
Decision Branch: Criteria machine-verifiable? YES -> Continue. NO -> Rewrite until testable.
8
Add Context
Each task includes what agent needs to know: existing patterns in codebase, relevant files to reference, naming conventions, any constraints (e.g., "use Resend for email, not Sendgrid").
9
Document Dependencies
DB-001: notification_templates table (blocks: BE-001, BE-002)
DB-002: notification_logs table (blocks: BE-003)
BE-001: Template CRUD API (blockedBy: DB-001, blocks: FE-001)
BE-002: Trigger service (blockedBy: DB-001, blocks: FE-002)
...
10
Final Validation Pass
All tasks fit one session?
All criteria verifiable?
All dependencies explicit?
Parallelism maximized?
If any NO: iterate on that aspect until YES
11
Hand Off to Orchestrator
Ralph: Load into prd.json, start loop
CC Mirror: Create tasks via TaskCreate API
Gas Town: Feed to Refinery (which validates your decomposition)
12
Monitor Execution
Watch for: stuck loops (bad criterion), dependency errors (missing blockedBy), context overflow (task too big). Each signals decomposition quality issue.
Total transformation: ~25 minutes from vague request to 8+ executable subtasks
THE TRANSFORMATION
"Build a notification system" (5 words, unexecutable) became 8+ tasks, each with:
- 2-3 sentence description
- Machine-verifiable criteria
- Explicit dependencies
- Necessary context
Quality metric
Good decomposition = agent executes without asking clarifying questions. Each task self-contained. If agent gets stuck, decomposition failed.

6 Gotchas

15 real problems with concrete fixes:

Tasks Too Large (context overflow)
Symptom: Agent loses track mid-task, quality degrades, hallucinations appear
Threshold: >40K tokens for implementation + testing
Fix: Split until each fits one session. 15 min to fix.
Tasks Too Small (coordination overhead)
Symptom: More time describing tasks than implementing them
Threshold: Task description longer than expected code change
Fix: Combine related tasks. 10 min to fix.
Vague Acceptance Criteria
Symptom: Agent marks tasks done without verification, false completion
Cause: Criteria like "works correctly" or "looks good"
Fix: Make verifiable - "tests pass", "button is green". 5 min to fix.
Hidden Dependencies
Symptom: Race conditions, "Cannot find type X" errors
Cause: Implicit assumptions about what exists
Fix: Document all explicitly with blockedBy/blocks. 10 min to fix.
Circular Dependencies
Symptom: Tasks blocked forever, no progress possible
Cause: A blocks B, B blocks C, C blocks A
Fix: Review dependency graph, break cycles. 15 min to fix.
Missing Context
Symptom: Wrong implementation despite correct criteria
Cause: Agent doesn't have enough information
Fix: Include relevant context, reference files. 5 min to fix.
Wrong Decomposition Strategy
Symptom: Inefficient execution, tasks blocking unnecessarily
Cause: Chose sequential when parallel was possible
Fix: Re-evaluate with decision tree. 15 min to fix.
Over-Parallelization
Symptom: Coordination overhead dominates, thrashing
Threshold: >5 parallel tasks typically problematic
Fix: Limit parallel tasks, sequence some. 10 min to fix.

7 What's Hard

Fundamental tensions without perfect solutions:

Finding the Right Size
The tension: Too small = coordination overhead dominates. Too big = context overflow causes failure.
Why no formula: It depends on task complexity, agent capability, verification overhead.
Mitigation: "One session" heuristic. 2-3 sentence test. Iterate based on results.
Making Criteria Verifiable
The tension: Some things are genuinely hard to specify. "Correct behavior" vs "tests pass" may diverge.
Why it's fundamental: Verification is the foundation of autonomous execution.
Mitigation: Focus on observable outcomes. When impossible, use HOTL (Human On The Loop).
Capturing All Dependencies
The tension: Some dependencies are invisible until you hit them. "Unknown unknowns."
Why it's fundamental: Complex systems have emergent coupling.
Mitigation: Review with checklist. Accept some re-planning as cost of exploration.
Human vs Agent Decomposition
The tension: Human decomposition takes time but is more reliable. Agent decomposition (Refinery) scales but may miss nuance.
When to delegate: Simple, well-defined domains. Not for novel or safety-critical.
Mitigation: Complex = human, simple = agent. Review agent decomposition before execution.
Parallelism vs Coordination
The tension: More parallel = faster execution but more coordination overhead. Bottleneck shifts.
Inflection point: ~5 parallel tasks is typical limit before coordination dominates.
Mitigation: Measure actual time. Don't parallelize for parallelism's sake.

8 When to Use / When Not

DECOMPOSITION IS ESSENTIAL WHEN
Multi-agent orchestration (CC Mirror, Gas Town)
Tasks exceeding one context window
Parallel execution desired
Complex dependencies exist
Multiple people/agents involved
Task length >5 paragraphs
Acceptance criteria >5 items
DECOMPOSITION IS OVERKILL WHEN
Single simple task fitting one session
One-shot operations (quick fixes)
Exploratory work with no defined end
No dependencies between parts
Task description < 2-3 sentences
