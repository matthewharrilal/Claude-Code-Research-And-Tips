2 The 6-Component Unhobbled Stack

"Treat Claude Code not as a chat tool, but as a persistent autonomous developer agent that works continuously while you sleep, clean, drive, or exercise."
Each component unlocks a specific capability. Together, they transform a chat tool into an autonomous development partner.
1 LSP Integration
WHY: Claude navigates codebases like an IDE user
Real-time diagnostics, type information, go-to-definition, find references. Claude "sees" code structure, not just text. Semantic understanding beats grepping.
2 Hooks Architecture
WHY: Deterministic automation without AI reasoning
PreToolUse, PostToolUse, SessionStart, Stop. Auto-formatting, linting gates, notifications, security checks. Hooks are shell commands, NOT AI prompts.
3 Subagent Orchestration
WHY: Parallel processing + context isolation
Isolated Claude instances with own context windows. Expensive operations don't pollute main session. Specialization: code-simplifier, verify-app, adversarial-critic.
4 Adversarial Validation
WHY: Single-agent systems drift toward confirmation bias
Architect proposes, Critic challenges. Multiple perspectives catch edge cases. Productive conflict reveals hidden assumptions. Quality through opposition.
5 Ralph Wiggum Loops
WHY: Multi-hour autonomous sessions survive context limits
External state files (prd.json, progress.txt). Fresh context per iteration. Stop hook re-injects prompt. while :; do cat PROMPT.md | claude; done
6 2-Way Voice (STT/TTS)
WHY: Untethered development - cardio coding
Superwhisper/Wispr for speech-to-text. macOS say for text-to-speech. Work while walking, exercising. Natural language flow without typing friction.

3 Design Decisions

WHY HOOKS, NOT AI, FOR AUTOMATION?
"Hooks are shell commands, NOT AI prompts. They're for deterministic automation." AI reasoning for automation = unpredictable, expensive, slow. Hooks are cheap, fast, reliable.
What this means for you
Use hooks for formatting, notifications, security gates - never for decisions. If you need reasoning, spawn a subagent. Mixing AI into hooks destroys the reliability guarantee.
WHY SUBAGENTS FOR BROWSER WORK?
"Browser output is expensive. Protect main context from token bloat." A single Playwright session can consume 50K+ tokens. In main context, that's 25% of your window gone on one operation.
What this means for you
Isolate costly operations in subagents. Subagent returns summary only - "Login works, homepage renders, 3 console errors." Main context stays lean and focused.
WHY ADVERSARIAL VALIDATION?
"Single-agent systems drift toward confirmation bias." Claude will convince itself its code is correct. A DHH-persona critic asks "Why not use existing form library?" - and catches what self-review misses.
What this means for you
For architecture decisions, security code, public APIs - spawn a critic. For mechanical transformations and tests-passing code, skip it. Match validation intensity to risk.
WHY EXTERNAL STATE (PROGRESS.TXT)?
"Fresh starts + external memory beats growing conversations." Long conversations suffer context rot. Each Ralph iteration sees: static prompt + current file state + git history. State lives in files, not conversation.
What this means for you
Never rely on "Claude remembering" something. If it's not in a file, it doesn't exist. progress.txt must be append-only. The filesystem IS the memory.
WHY TIERED MODEL SELECTION?
"Haiku for errands, Opus for thinking." Not every task needs Opus. Eric's recommended split: Haiku 5%, Sonnet 75%, Opus 20%. Match model capability to task complexity.
What this means for you
Simple file operations, formatting, notifications: Haiku. Standard development work: Sonnet. Architecture decisions, complex debugging, adversarial review: Opus.

5 Path of an Overnight Session

Follow a feature through the unhobbled stack, from evening setup to morning review:

T-5min
User creates prd.json with stories: login form, validation, session management. All stories: passes: false
T-3min
User reviews PROMPT.md. Completion promise defined. Validators configured.
T-1min
User starts Ralph loop: ./scripts/ralph/ralph.sh 50 Component 5: Ralph
T+0min
ITERATION 1 BEGINS. Claude reads PROMPT.md + prd.json + progress.txt.
T+2min
Claude picks story: "Login form". Starts implementation.
T+5min
Edit tool triggers PostToolUse hook. smart-format.sh runs Prettier. Component 2: Hooks
T+7min
Claude needs type information. Uses LSP for hover documentation. Component 1: LSP
T+10min
Claude spawns verify-app subagent. Isolated context runs tests. Returns: PASS + summary. Component 3: Subagents
T+12min
Complex decision needed. Spawns adversarial critic (DHH persona). Critic: "Why not use existing form library?" Component 4: Adversarial
T+15min
Story complete. Updates prd.json: passes: true. Commits: feat: login form component. APPENDS to progress.txt.
T+16min
Claude attempts to exit. SessionEnd hook fires. Checks prd.json - stories incomplete. Re-injects PROMPT.md. Component 5: Ralph
T+3hrs
All stories complete. Claude outputs: <promise>COMPLETE</promise>. Ralph loop exits successfully.
MORNING
User reviews prd.json status, git log, runs npm run test. Merges successful work.

6 Gotchas (12 Practical Problems)

Real problems you'll hit, with concrete numbers and fixes:

#1: Runaway loop costs
Symptom: $500+ unexpected overnight bill
Threshold: >50 iterations without cost monitoring
Fix: Set MAX_ITERATIONS in ralph.sh. Use API cost limits. Time to fix: 2 min
#2: Hook not firing
Symptom: Formatting/notifications don't run
Cause: Matcher syntax wrong, path not found
Fix: Check matcher patterns, verify script paths are absolute. Time to fix: 10-15 min
#3: Subagent context pollution
Symptom: Main agent gets confused after subagent returns
Cause: Subagent returning too much context
Fix: Use worker preamble, enforce summary-only returns. Time to fix: 15-30 min
#4: Adversarial deadlock
Symptom: Critic never approves, loop stalls
Cause: Critic too strict, no approval threshold
Fix: Set max review rounds (3-5), time-box review. Time to fix: 5-10 min
#5: Ralph infinite loop
Symptom: Loop never exits, max iterations always hit
Cause: Completion promise not detected, task impossible
Fix: Check promise format exactly, add stuck detection (same HEAD 3x = exit). Time to fix: 5-10 min
#6: Voice transcription errors
Symptom: Wrong commands executed, code messed up
Cause: STT misinterprets technical terms
Fix: Use Superwhisper AI correction, review before execute. Continuous
#7: progress.txt bloat
Symptom: Late iterations slower than early ones
Threshold: >20KB progress.txt degrades quality
Fix: Archive to AGENTS.md periodically, keep recent learnings only. Time to fix: 15-30 min
#8: Validator false positives
Symptom: Good code rejected, iteration wasted
Cause: Validator too strict, doesn't match project conventions
Fix: Tune validator criteria, add exception handling. Time to fix: 20-40 min
#9: LSP server crash
Symptom: Lose semantic navigation mid-session
Cause: Memory limits, corrupt project state
Fix: Restart language server, check memory limits. Time to fix: 2-5 min
#10: Security gate too strict
Symptom: Legitimate commands blocked
Cause: Gate doesn't whitelist common patterns
Fix: Whitelist npm, git, common CLI tools. Time to fix: 10-20 min
#11: Model mismatch
Symptom: Simple tasks use Opus, complex tasks fail on Haiku
Cause: No tiered model selection implemented
Fix: Implement Haiku 5% / Sonnet 75% / Opus 20% selection. Time to fix: 15-30 min
#12: Context window overflow
Symptom: Claude stops making sense in late iterations
Cause: Costly operations in main context
Fix: Use subagents for browser/test output, prune aggressively. Time to fix: 10-20 min

7 What's Hard (Fundamental Tensions)

These are NOT bugs to fix. They're genuine dilemmas to manage:

Autonomy vs. Cost
The tension: More autonomy = more iterations = more tokens = more cost. The 10x productivity comes with 10x API bills.
Symptoms: Overnight runs cost $300. Team budget blown in a week.
Coping strategy: Budget caps, tiered models, clear completion criteria. Accept that some tasks aren't worth autonomous execution.
Validation vs. Speed
The tension: More validation = higher quality but slower throughput. Adversarial review on every task is overkill.
Symptoms: Simple tasks take 10x longer. Critic always finds something.
Coping strategy: Risk-based validation. Critical paths get full review. Routine changes skip it. Match validation to consequence.
Isolation vs. Context
The tension: Subagents are protected from pollution, but they can't see the full picture. Sometimes they need context they don't have.
Symptoms: Subagent makes locally-optimal choice that breaks global invariant.
Coping strategy: Careful context passing in task descriptions. Summarized project constraints. Accept some re-work.
Trust vs. Verification
The tension: Full trust is fast but risky. Full verification is slow but safe. No perfect middle ground exists.
Symptoms: Trusted code breaks production. Or: nothing ships because everything needs approval.
Coping strategy: Graduated trust based on task type and history. Build trust incrementally, not all at once.
Voice vs. Precision
The tension: Voice is fast but transcription is imperfect. Technical terms, code syntax, exact variable names - voice struggles.
Symptoms: "Create userSchema" becomes "create user scheme".
Coping strategy: Use voice for high-level direction, keyboard for precision. Review before execute. Don't voice-code into production.
Complexity vs. Reliability
The tension: More components = more capability, but also more failure modes. 6 components means 6 things that can break.
Symptoms: Hook fails silently. Subagent spawns infinitely. LSP crashes. Debugging takes longer than manual work.
Coping strategy: Start with Phase 1, add complexity incrementally. Validate each gate before proceeding. Accept that some domains stay simple.

8 When to Use / When Not

USE UNHOBBLED STACK WHEN
Stage 4+ developer (comfortable with Claude Code basics)
Daily budget $30-300 (components increase cost)
Ongoing project (weeks+) - setup investment needs payoff
Medium-high chaos tolerance (more components = more failure modes)
Terminal comfort (hooks, scripts, voice all terminal-based)
Desire for autonomy (if you prefer manual control, stay vanilla)
DON'T USE UNHOBBLED WHEN
New to Claude Code - learn basics first (Boris vanilla approach)
Simple one-off task - overkill for quick work
Budget-constrained - components add cost
Team environment - custom configs hard to share
Need predictability - autonomy is less predictable
