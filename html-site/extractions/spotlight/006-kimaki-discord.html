2 The Core Abstraction

"Each channel is a window into a specific project on your machine."
npx -y kimaki@latest
That's all it takes. Discord becomes your AI coding interface.

Kimaki is a Discord bot that bridges chat messages to AI coding agents running on your local machine. When you send a message in a linked Discord channel, an AI agent reads/edits files, executes terminal commands, and searches your codebase.

One Bot Per Machine Architecture

Discord Server
├── #project-a (Bot 1)
├── #project-b (Bot 1)
└── #project-c (Bot 2)
      │
      ▼
Machine 1 (Kimaki Bot 1)
├── OpenCode Server A → ~/projects/a
└── OpenCode Server B → ~/projects/b

Machine 2 (Kimaki Bot 2)
└── OpenCode Server C → ~/projects/c

Key Principle: Each Kimaki bot instance connects to exactly one machine. Multiple channels can map to different project directories on that machine. For multiple machines, run separate bot instances with different tokens.

Key Features

Text & File Interactions
Send messages in linked channels to trigger AI sessions. Attach files for inline context.
Voice Channel Support
Join voice channels and speak naturally. Real-time transcription via Gemini API.
Session Management
/session, /resume, /fork, /share - full session control via slash commands.
Message Queue
Chain follow-up tasks with /queue without waiting for completion.
Model Selection
Switch models per channel or session with /model command.
CI/CD Integration
Trigger sessions from GitHub Actions with npx kimaki send.

3 Design Decisions

WHY ONE BOT PER MACHINE?
Each bot connects to exactly one machine because the bot needs local filesystem access. You can't have a single bot securely accessing multiple machines without complex VPN/tunnel setups. The constraint simplifies security and ownership.
What this means for you
If you have a laptop, a server, and a workstation, you need three bot instances with three tokens. Use --data-dir for isolated configurations on the same machine.
WHY CHANNEL = PROJECT?
Discord channels are natural organizational units. Mapping each channel to a project directory creates a clear mental model: #frontend talks to ~/projects/frontend, #backend talks to ~/projects/backend. No confusion about which codebase you're modifying.
What this means for you
Create channels that match your project structure. Messages in #my-api will only affect the codebase linked to that channel. Thread = session, channel = project.
WHY GEMINI FOR VOICE?
Gemini has a good audio transcription API that handles real-time streaming audio. The voice processing pipeline requires: Opus decoding, downsampling, chunking, and transcription. Gemini handles this reliably with low latency.
What this means for you
Voice features require a Gemini API key in addition to your AI coding API key. Voice is optional - if you only need text, skip the Gemini setup.
WHY THREADS = SESSIONS?
Discord threads maintain context naturally - they're scoped conversations under a channel. This maps perfectly to coding sessions: start a task, create a thread, maintain context throughout, then archive when done.
What this means for you
Each new task creates a thread automatically. Use /resume to continue old sessions, /fork to branch from a point in history. Session state persists in SQLite at ~/.kimaki/discord-sessions.db.
WHY SQLITE FOR STATE?
No external database needed. SQLite is simple, local, and requires no setup. The database tracks 7 tables: thread_sessions, part_messages, bot_tokens, bot_api_keys, channel_directories, channel_models, and session_models.
What this means for you
All state is in ~/.kimaki/discord-sessions.db. Back it up if you care about session history. Delete it to start fresh.

5 The Path of a Task

Follow a text message through the entire system:

1
[T+0ms] User sends "Can you add error handling to the login function?" in #backend channel
2
[T+10ms] Discord.js event handler receives message → Permission check (admin/owner/Kimaki role)
3
[T+20ms] Channel-to-directory lookup via SQLite: channel_directories → /path/to/backend/
4
[T+30ms] Thread created with title: "Add error handling to login" → session_id generated
5
[T+50ms] OpenCode server spawned or retrieved for /path/to/backend/
6
[T+100ms] AI agent executes: reads src/auth/login.ts, generates error handling code
7
[T+2000ms] Response formatted: tool calls shown as src/auth/login.ts (+15-3)
8
[T+2100ms] Discord message updated in thread → Typing indicator stops
9
[T+2200ms] Session persisted to SQLite: thread_sessions, part_messages

Voice Message Flow

1
[T+0ms] User speaks "Can you refactor the user service?" in voice channel
2
[T+10ms] Discord voice stream received (48kHz Stereo Opus)
3
[T+50ms] Audio processing: Opus decode → Downsample (48kHz → 16kHz) → Mono → 100ms chunks
4
[T+100ms] Base64 encoding → Gemini transcription → "Can you refactor the user service?"
5
[T+200ms] Same flow as text: Channel lookup → OpenCode → AI execution
6
[T+N ms] Response via text in channel (or voice TTS if configured)

6 Gotchas

10 real problems you'll hit, with concrete fixes:

#1: Missing intents error
Symptom: "Missing required intents" on startup
Fix: Enable Server Members Intent + Message Content Intent in Discord developer portal → 5 min fix
#2: Port already in use
Symptom: Kimaki fails to start with port conflict
Fix: Kill existing process or use --data-dir ~/.kimaki-alt → 2 min fix
#3: OpenCode not found
Symptom: CLI not installed or not in PATH
Fix: npm install -g @opencode-ai/cli → 5 min fix
#4: Bot doesn't respond
Symptom: Messages ignored, no thread created
Fix: User needs admin, owner, or "Kimaki" role → 2 min fix
#5: Session times out
Symptom: OpenCode health check failing
Fix: Check project config, test opencode serve --port 9999 manually → 10 min fix
#6: Voice not transcribing
Symptom: Speaking produces no response
Fix: Run --restart, enter valid Gemini API key → 2 min fix
#7: Bot won't join voice
Symptom: No response when joining voice channel
Fix: Check Connect + Speak permissions in Discord → 2 min fix
#8: Channel not linked
Symptom: Messages in channel produce no response
Fix: Run --add-channels and select the project → 5 min fix
#9: Multi-instance conflict
Symptom: Two bots fighting for same data-dir
Fix: Use different --data-dir for each instance → 2 min fix
#10: Webhook issues for CI
Symptom: npx kimaki send not working from GitHub Actions
Fix: Check channel ID format, use --notify-only for simple messages → 5 min fix

7 What's Hard

These are fundamental tensions in Discord-as-interface, not bugs to fix:

One Bot Per Machine Constraint
The tension: You can't control multiple machines with one bot without complex networking. Each machine needs its own bot instance and Discord token.
Coping strategy: Accept the constraint. Create multiple bot applications in Discord. Use descriptive names: "Kimaki-Laptop", "Kimaki-Server", "Kimaki-Workstation".
Voice Processing Latency
The tension: Transcription adds ~200-500ms latency. Real-time voice coding feels slightly delayed compared to direct terminal input.
Coping strategy: Accept the latency as the cost of hands-free operation. Use voice for high-level guidance, text for precision commands.
Discord Rate Limits
The tension: Discord's API has request limits. Heavy usage can trigger throttling, especially with many concurrent sessions.
Coping strategy: Kimaki has built-in rate limiting. For high-volume use, consider staggering requests or using queue commands.
Permission Management Complexity
The tension: Discord's role system is powerful but complex. Who can use the bot? Who can see outputs? Permissions cascade in non-obvious ways.
Coping strategy: Create a single "Kimaki" role. Assign it to trusted users. Keep permissions simple rather than granular.
OpenCode (Not Claude Code) Dependency
The tension: Kimaki uses OpenCode, not Claude Code directly. Different backend, different patterns, different ecosystem.
Coping strategy: Learn OpenCode-specific patterns. The concepts transfer (channels, sessions, tools), but implementation details differ.
Gemini API Key for Voice
The tension: Voice requires another API key to manage. More credentials, more cost tracking, more potential failure points.
Coping strategy: Consider voice optional. Start with text-only. Add voice when the workflow demands hands-free operation.

8 When to Use / When Not

USE KIMAKI WHEN
Team collaboration - multiple developers, shared context visibility
Mobile-first - need to fix code from phone/tablet
Voice coding - hands-free development during commute or sketching
Async monitoring - start task, check Discord later
CI/CD integration - build notifications and fix requests via webhooks
DON'T USE KIMAKI WHEN
Single user, local only → Direct CLI is simpler
Mac-specific automation → SYSTEM has more features
No Discord in your workflow → Don't add dependency
Latency-critical work → Direct terminal is faster
Security-sensitive environments → Discord adds exposure
