Essence (15 words)
Multi-agent CLI with blind validators: workers implement, validators assess without context, preventing self-approval bias.

2 The Core Abstraction

"Validators can't lie about code they didn't write. Fail the check? Fix and retry until it actually works."
zeroshot run 123 # Implement GitHub issue #123 with blind validation
One command. Full multi-agent validation pipeline.

Single-agent systems have an inherent conflict of interest: the same agent that writes code reviews its own work. It knows what it meant to do, not just what it did. Zeroshot solves this with blind validation - validators see ONLY the resulting files, never the worker's reasoning or implementation history. 

        Traditional Ralph:
        Agent writes code -> Agent checks its own work -> Commits
        (Self-validation bias: agent knows intent, rationalizes results)

        Zeroshot Multi-Agent:
        WORKER writes code -> Worker context ENDS HERE
                          -> VALIDATORS check work BLIND
                          -> Validators see ONLY file state
                          -> Honest assessment without context bias

The key innovation: information barriers between agents. Workers implement. Validators judge. Neither sees the other's context. This is "zero-shot validation" - assessing from results alone, like ML's zero-shot learning classifies without prior examples.

Complexity-Based Routing

The Conductor classifies every task, routing to appropriate resources:

Level What It Means Agents Deployed
TRIVIAL Single file changes, no validation needed Worker only
SIMPLE One worker + one generic validator Worker + 1 Validator
STANDARD Multi-file, needs planning and review Planner + Worker + 2 Validators
CRITICAL Security-relevant, thorough validation Senior Planner + Worker + 5 Specialized Validators

3 Design Decisions

WHY BLIND VALIDATION?
When a validator sees the worker's reasoning, they rationalize the implementation. "They meant to do X, so this must be right." Blind validation forces pure assessment: does this code work, or doesn't it?
What this means for you
Validators give honest pass/fail, not rubber-stamp approval. If your acceptance criteria are vague ("make it work"), validators can't verify them. The discipline of blind validation forces discipline in task definition.
WHY SQLITE (NOT EXTERNAL DB)?
Single-file deployment. ACID transactions. No infrastructure to provision. If your session crashes, state persists. Resume from where you left off with zeroshot resume <cluster-id>.
What this means for you
No database setup. Crash recovery built-in. You can inspect state with standard SQLite tools: sqlite3 ~/.zeroshot/*.db "SELECT * FROM messages". The ledger IS your debug log.
WHY COMPLEXITY ROUTING?
Multi-agent validation costs more API calls. TRIVIAL tasks don't need 5 validators. CRITICAL tasks need thorough review. Routing optimizes: simple tasks cheap, complex tasks thorough.
What this means for you
Configure model levels per complexity. Use cheaper models (Haiku, GPT-4o-mini) for TRIVIAL/SIMPLE. Reserve expensive models (Opus, o1) for CRITICAL. Match resources to risk.
WHY ASYNC-FIRST EXECUTION?
Non-blocking, parallel validation possible. Start a task, continue working, check results later. Message bus architecture enables agents to work independently.
What this means for you
You can run multiple clusters in parallel. Use zeroshot list to see active runs. The pub/sub architecture means validators don't wait for each other - they assess in parallel when possible.
WHY PROVIDER-AGNOSTIC DESIGN?
Lock-in is expensive. New models emerge constantly. Provider-agnostic design lets you experiment: Claude for conductors, Codex for workers, Gemini for validators. Mix and match without code changes.
What this means for you
Switch providers with --provider codex. Set defaults with zeroshot providers set-default claude. Run A/B tests across models. Future-proof against any single provider.
WHY 4 SPECIALIZED VALIDATORS?
Different failure modes need different lenses. Requirements validator checks acceptance criteria. Code validator does senior-level review. Security validator looks for OWASP vulnerabilities. Test validator checks coverage. Multi-angle assessment catches what single review misses.
What this means for you
CRITICAL tasks deploy all 4+ validators. Each publishes ACCEPT or REJECT independently. Consensus required. One rejection means retry with findings. Comprehensive coverage at CRITICAL level.
WHY WORKTREE/DOCKER ISOLATION?
Safe experimentation. PR workflows need branch isolation. Risky changes need container isolation. Git worktrees are lightweight (~1 second setup). Docker provides complete sandboxing for untrusted code.
What this means for you
Use --worktree for PR workflows (creates zeroshot/{clusterId} branches). Use --docker for risky experiments (container isolation, auto-cleanup). Match isolation to risk.

5 Path of a Task Through Zeroshot

Follow GitHub Issue #123 ("Add rate limiting to API") through the system:

1
[T+0s] You run zeroshot run 123. CLI fetches issue from GitHub via gh.
2
[T+2s] CONDUCTOR classifies: Complexity STANDARD (multi-file, security-relevant). Task Type: TASK. Selects: Planner + Worker + 2 validators.
3
[T+5s] PLANNER creates strategy: 1) Identify endpoints, 2) Add middleware, 3) Configure limits. Publishes plan to message bus.
4
[T+10s] WORKER reads plan, implements. Modifies rate-limit.ts, api/routes.ts. Commits to worktree branch.
5
[T+45s] Worker publishes "work_complete" to bus. WORKER CONTEXT ENDS HERE. No more access to reasoning.
6
[T+50s] VALIDATORS receive notification. Each sees ONLY file state, not worker reasoning. Validation is BLIND.
7
[T+70s] Validator 1 (Requirements): Checks acceptance criteria. Validator 2 (Code): Reviews implementation quality. Each publishes ACCEPT or REJECT.
8
[T+90s] CONSENSUS CHECK: Both validators ACCEPT? Task complete. Any REJECT? Findings returned, retry loop initiated.
9
[T+95s] Task complete. If --pr flag was set, PR created. Output summary displayed. Cluster cleaned up.

6 Gotchas

Real problems you'll hit, with concrete fixes and time-to-resolve:

1. "No providers configured" error
Symptom: Zeroshot fails immediately on startup
Cause: Provider CLI not installed or not authenticated
Fix: npm install -g @anthropic-ai/claude-code && claude auth login
Time to fix: 5-10 minutes
2. Rate limited
Symptom: API errors, slow responses, 429 status codes
Cause: Too many parallel validators hitting API limits
Fix: Wait 60s, reduce model level, or use different providers for different agents
Time to fix: 1-2 minutes
3. Context overflow
Symptom: Task too complex, agent loses track mid-implementation
Cause: Task exceeds single context window capacity
Fix: Ensure task classified as STANDARD/CRITICAL, or split into smaller issues
Time to fix: 2-5 minutes (to reclassify or split)
4. Validator always rejects
Symptom: Endless retry loop, no progress
Cause: Acceptance criteria are vague or unverifiable
Fix: Make criteria specific: "API returns 429 after 100 requests/minute" not "add rate limiting"
Time to fix: 5-10 minutes (to rewrite criteria)
5. Docker fails
Symptom: "Docker daemon not running" or container errors
Cause: Docker Desktop not started
Fix: Start Docker Desktop, or use --worktree instead
Time to fix: 1 minute
6. Worktree conflict
Symptom: "Branch already exists" or worktree creation fails
Cause: Previous run left stale worktree
Fix: git worktree prune && git branch -D zeroshot/*
Time to fix: 2 minutes
7. Resume fails
Symptom: Cluster state corrupted, resume doesn't work
Cause: SQLite database corruption or invalid state
Fix: zeroshot kill <id> and start fresh
Time to fix: 1 minute
8. Agents receive no context (Issue #140)
Symptom: Agents seem to have no knowledge of the task, build wrong thing
Cause: Known issue in message bus context propagation
Fix: Update to latest version. Check zeroshot --version against GitHub releases.
Time to fix: 0-5 minutes (if update available)
9. Windows errors
Symptom: Various path or permission errors on Windows
Cause: Platform not fully supported natively
Fix: Use WSL2 for full compatibility
Time to fix: Variable (WSL2 setup)
10. Validator timeout
Symptom: Long validation exceeds time limit, cluster fails
Cause: Complex validation taking too long
Fix: Increase timeout in config, or reduce validation complexity
Time to fix: 1-2 minutes

7 What's Hard

These are fundamental tensions, not bugs to fix:

Cost vs. Validation Depth
The tension: More validators = better coverage = higher API costs. Each validator is an API call. CRITICAL tasks with 5 validators cost 5x a single-agent approach.
Coping strategy: Use complexity routing. TRIVIAL tasks skip validation entirely. Reserve CRITICAL (5 validators) for security-relevant changes. Match investment to risk.
Clear "Done" Criteria Required
The tension: Blind validators need verifiable criteria. "Make it work" isn't verifiable. "Returns 200 status on valid input" is. If you can't define "done," validators can't verify it.
Coping strategy: Invest time upfront in acceptance criteria. Write them as testable assertions. If you can't write a test for it, you can't validate it blind.
Multi-Agent Coordination Overhead
The tension: Message bus, SQLite persistence, agent spawning, consensus checking - all add complexity compared to single-agent loops. More moving parts = more to debug.
Coping strategy: Accept overhead for validation guarantees. Use zeroshot export to dump cluster history. Ledger IS your debug log. The complexity pays off in correctness.
Provider Rate Limits
The tension: Parallel validators can exhaust API rate limits. 3 validators at once = 3x the calls = faster limit hits.
Coping strategy: Stagger validation (built-in). Use different providers for different agents (Claude for conductor, Codex for workers). Distribute load across accounts.
Context Limits Per Agent
The tension: Each agent has bounded context. Complex codebases may exceed what a single agent can process. Task decomposition becomes critical.
Coping strategy: Break large tasks into smaller issues. Let the Conductor classify complexity. Trust CRITICAL routing to deploy more resources. Context limits are real - don't fight them.
Debugging Distributed System
The tension: Harder to debug than single-agent. "Why did validator 2 reject?" requires understanding message flow, agent state, consensus logic.
Coping strategy: Use zeroshot export <cluster-id> to get full conversation. Query SQLite directly: sqlite3 ~/.zeroshot/*.db. The ledger has everything.

8 When to Use / When Not

USE ZEROSHOT WHEN
You need validation guarantees (blind review catches issues)
Task complexity is STANDARD or above (worth multi-agent overhead)
You can define clear acceptance criteria (validators need targets)
You need crash recovery (SQLite persistence enables resume)
You want to experiment with multiple providers (provider-agnostic)
You need isolated experimentation (worktree/Docker modes)
DON'T USE ZEROSHOT WHEN
Simple, well-defined tasks - Original Ralph is cheaper
Exploratory work - Can't define "done" yet, use direct Claude
Cost-constrained - Multi-agent = more API calls
PRD-driven development - Production Ralph has better story-by-story fit
Learning orchestration - Basic Ralph has simpler mental model
Subjective quality - "Make it pretty" isn't verifiable
