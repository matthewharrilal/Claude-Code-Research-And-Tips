2 The Core Abstraction

"Claude Code is the world's biggest ant. Everyone is focused on making their ant run longer... when nature prefers colonies."
gt mayor up && gt polecat up myrig alice bob carol
Gas Town shifts from single agent to agent factory. You become the Product Manager.

Gas Town treats agents as disposable, specialized workers. Unlike Ralph's fresh-context-per-iteration model, Gas Town runs multiple named agents simultaneously (Polecats), coordinated by a hierarchy of management roles (Mayor, Deacon, Refinery).

Research finding: Polecats develop natural specialization over time. Community reports show "alice becomes the database expert" after handling similar tasks repeatedly. The research consensus: don't rotate polecats randomly - let expertise emerge.

Beads integration: All state persists through Beads (~225K lines of Go), a git-backed data plane. This solves the "50 First Dates" problem - each agent can reconstruct context from the shared repository rather than starting fresh.

3 Design Decisions (Research-Backed)

WHY LET POLECATS SPECIALIZE?
Research finding: Community practitioners observed that polecats naturally develop implicit "expertise" over time. Alice handles database tasks repeatedly and becomes faster at them. This isn't anthropomorphization - it's context accumulation in Beads.
What this means for you
Don't rotate polecats randomly across task types. Let alice keep doing database work, bob keep doing frontend. The Beads context for each polecat accumulates domain-specific learnings.
WHY BUDGET CAPS MATTER?
Research finding: Community reports include "$500+ day" cost explosion stories when crew limits weren't set. Steve's explicit warning: "If you care about costs, don't use this." But even Steve uses budget caps.
What this means for you
Set budget.daily_limit in config.yaml BEFORE your first run. Set crew.max_per_rig to prevent runaway spawning. $50-200/day is typical for serious usage. Plan for it.
WHY BEADS USES GIT?
Research finding: Steve's "boring technology" philosophy - Git is battle-tested, durable, and everyone understands it. Trade-off: worse query performance than a database, but survives crashes and provides full history.
What this means for you
Accept that Beads queries are slower than SQL. The durability and auditability are worth it. If you need to debug what happened, git log shows everything.
WHY STAGE 7+ REQUIRED?
Research consensus: Steve's warning is not gatekeeping - it's honest. "If you're not at Stage 7, you'll get your face ripped off by superintelligent chimpanzees." Multiple practitioners confirm: ~100 hours of multi-agent experience is the minimum.
What this means for you
If you haven't managed 10+ agents manually, start with CC Mirror or Ralph. Gas Town assumes you already understand context management, cost tracking, and multi-agent coordination. The learning curve is ~100 hours.

6 Gotchas (Research-Discovered)

Real problems discovered by the community with verified fixes:

#1: Stage 4 face-rip
Symptom: Everything breaks. Agents spin, cost explodes, output is garbage.
Cause: You're not Stage 7+. This is the "chimpanzee" scenario Steve warned about.
Fix: Stop immediately. Go back to Ralph or CC Mirror for 100 more hours. No shortcut exists.
#2: $500+ day
Symptom: Unexpected bill, API account suspended
Cause: Crew uncapped, no budget limits set
Fix: Set budget.daily_limit and crew.max_per_rig in config.yaml. Takes 2 minutes. Do it now.
#3: Polecat rotation mistake
Symptom: Quality drops over time, polecats seem "dumber"
Cause: Rotating polecats randomly destroys accumulated specialization
Fix: Prevention only. Let alice keep doing database work. Beads context for each polecat is domain-specific.
#4: GUPP not triggering
Symptom: Agent sits idle after startup, no work happening
Cause: Gas Town's event system didn't fire
Fix: gt nudge after 30-60 seconds of inactivity. Steve's official workaround.
#5: Merge queue backup
Symptom: All polecats stuck waiting, nothing completing
Cause: Refinery merge queue congested from conflicting changes
Fix: Check Refinery logs. Better task decomposition prevents conflicts. May need 5-30 minutes to clear.
#6: Context collapse
Symptom: Agents repeat themselves, lose track of state
Cause: Beads auto-compact not enabled, context windows full
Fix: Enable auto-compact in config.yaml. Takes 5 minutes to configure.

7 What's Hard (Fundamental Tensions)

These are research-documented tensions that can't be "fixed" - only managed:

Cost vs. Throughput
Research evidence: "$800 day" community reports exist. More agents = more cost = more throughput. Always linear.
Why it's hard: No way to get parallel throughput without parallel cost. The economics are physics.
Coping strategy: Set hard budget caps. Use Haiku for Crew wherever possible. Accept that cost visibility is a health metric like cloud spend.
Chaos vs. Control
Research evidence: Steve admits "some bugs fixed 2-3 times, some fixes get lost." This is design, not bug.
Why it's hard: Agents make mistakes. More agents = more mistakes. The trade is throughput for occasional rework.
Coping strategy: Focus on THROUGHPUT, not perfection. Dogs gates help but don't eliminate. Accept 1.2x rework for 3x throughput.
Prerequisite Cliff
Research evidence: "Stage 7+ or face ripped off by chimpanzees." Multiple practitioners confirm ~100 hours minimum.
Why it's hard: You can't shortcut multi-agent intuition. It's pattern recognition built over time.
Coping strategy: No shortcut. Start with simpler patterns. Build up. The learning curve is the prerequisite.
Mental Model Shift
Research evidence: You're a "PM not programmer" now. Years of habits must change.
Why it's hard: You instinctively want to debug code. Gas Town wants you to define features and review outcomes.
Coping strategy: Takes weeks to adjust. Start by consciously NOT looking at code. Review outputs only. Trust the vibe.
