CC Mirror enables a disabled beta feature in Claude Code: native Task management. Instead of building your own orchestration layer with days or weeks of effort, you get dependency-tracked, persistent, multi-agent coordination out of the box.

Task Management Tools

# The 4 Task Management Tools

TaskCreate  → Create with subject, description, acceptance criteria
TaskList    → Filter: status='open', no owner, not blocked
TaskGet     → Full context: description, comments, dependencies
TaskUpdate  → Claim (set owner), add comments, resolve, link references

Dependency Management

# Dependency Tools

addBlocks     → This task blocks another (must complete first)
addBlockedBy  → This task is blocked by another (wait for it)

The philosophy is simple: why spend weeks building custom orchestration when four native tools solve 80% of use cases? Save your engineering effort for the remaining 20% that genuinely requires custom solutions.

3 Design Decisions

WHY NATIVE (NOT CUSTOM)?
Custom orchestrators require days or weeks of development. Native Task tools are already built, tested, and integrated. Zero setup effort means you ship today, not next month.
What this means for you
Start with native. Only build custom when you hit the 20% edge cases. Most developers never need to graduate beyond native Task tools.
Aspect Native Task API Custom Build
Setup Zero effort Days/weeks
Dependencies Automatic Manual wiring
Persistence Built-in Build yourself
Visual tracking CLI included Build yourself
Recovery Native Build yourself
WHY DEPENDENCY TRACKING?
Tasks naturally have prerequisites. Building an API before defining the data model produces broken code. blockedBy automatically gates execution so tasks run in valid order.
What this means for you
Think in terms of task graphs, not task lists. Every task that depends on another should have explicit addBlockedBy. The system handles execution order automatically.
WHY PERSISTENT (NOT EPHEMERAL)?
Unlike TodoWrite, Tasks survive across sessions. If your machine crashes mid-orchestration, you pick up where you left off. No lost progress.
What this means for you
Use Tasks for anything that spans sessions. Use TodoWrite for single-session personal tracking. Different tools for different persistence needs.

Task Creation Script

# Example: Creating a dependency-tracked task graph
# Run these in sequence in Claude Code

# Step 1: Create root task (no dependencies)
TaskCreate:
  subject: "Design API architecture"
  description: "Analyze requirements, choose framework, document decisions"
  acceptanceCriteria:
    - "Framework selected and justified"
    - "Architecture diagram in docs/"
    - "ARCHITECTURE.md updated"

# Step 2: Create dependent task
TaskCreate:
  subject: "Implement data models"
  description: "Create SQLAlchemy models based on architecture"
  acceptanceCriteria:
    - "All models created in src/models/"
    - "Migrations generated"
    - "Tests pass"

TaskUpdate:
  taskId: "#2"
  addBlockedBy: ["#1"]

# Step 3: Create parallel tasks (both blocked by #2)
TaskCreate:
  subject: "Implement REST API endpoints"
  description: "Create FastAPI routes for all models"

TaskUpdate:
  taskId: "#3"
  addBlockedBy: ["#2"]

TaskCreate:
  subject: "Write integration tests"
  description: "pytest tests for all endpoints"

TaskUpdate:
  taskId: "#4"
  addBlockedBy: ["#3"]

Agent Spawning Script

# spawn-agents.sh - Launch parallel agents for ready tasks
#!/bin/bash

# Check for ready tasks (open, no owner, not blocked)
echo "Checking for available tasks..."

# Launch agents in separate terminal tabs/tmux panes
# Each agent will claim and work on one task

# Agent 1
claude-code --prompt "TaskList to find available work, then claim and complete one task" &

# Agent 2 (wait 5s to avoid race condition)
sleep 5
claude-code --prompt "TaskList to find available work, then claim and complete one task" &

# Agent 3
sleep 5
claude-code --prompt "TaskList to find available work, then claim and complete one task" &

echo "Agents launched. Monitor with /agents or Ctrl+Shift+A"

7 Gotchas

Real problems you'll hit with native Task orchestration, with specific fixes:

#1: Beta/disabled feature
Symptom: Behavior may change between CC Mirror versions
Cause: This is a disabled beta feature in Claude Code
Fix: Monitor CC Mirror releases. Keep CC Mirror updated.
#2: Relies on minified JS editing
Symptom: Task tools disappear after Claude Code update
Cause: CC Mirror patches Claude Code's minified JavaScript
Fix: Re-run CC Mirror setup after Claude Code updates.
#3: Generic limits for production
Symptom: Need state control, business logic, custom observability
Cause: Native is general purpose, not domain-specific
Fix: Graduate to Gas Town for the 20% that needs custom. (@nelsonpadil_)
#4: Agent autonomy risks
Symptom: Agents take uncontrolled actions
Cause: Less control than custom orchestrators
Fix: Watch agent monitoring view. Use clear acceptance criteria.
#5: Task order matters
Symptom: Tasks execute in wrong order
Cause: Missing or incorrect addBlockedBy
Fix: Double-check dependency graph before spawning agents.
#6: No automatic parallelization
Symptom: Only one agent working despite multiple ready tasks
Cause: Must manually spawn multiple agents
Fix: Launch agents in parallel explicitly.
#7: Forgotten tasks accumulate
Symptom: Old tasks from previous projects pollute TaskList
Cause: Tasks persist across sessions
Fix: Clean up with TaskUpdate: mark as done or cancelled.
#8: Circular dependencies
Symptom: Tasks block each other forever, nothing progresses
Cause: #A blocks #B, #B blocks #A
Fix: Check graph for cycles before spawning. No automated detection.
#9: Agent doesn't claim tasks
Symptom: Spawned agent sits idle
Cause: Task not open, already owned, or blocked
Fix: Check TaskList filter: status='open', no owner, not blocked.
#10: Results not visible
Symptom: Task marked done but can't see what happened
Cause: Didn't check TaskGet for full context
Fix: Use TaskGet for comments, references, resolution details.

8 What's Hard

These are fundamental tensions, not bugs to fix:

The 80% Solution
The tension: Native covers 80% of use cases, but some genuinely need full orchestration with custom roles, state machines, and observability.
Symptoms: Hitting limits feels like hitting a wall. No gradual degradation.
Mitigation: Know when you're in the 20%. Graduate to Gas Town for factory-scale orchestration. Don't try to force native beyond its design.
Generic vs. Specific
The tension: Native is general purpose. Domain-specific orchestration needs custom logic.
Symptoms: Wanting tasks to understand your business rules, not just dependency ordering.
Mitigation: Put domain logic in acceptance criteria and agent prompts. The Task layer handles coordination; you handle semantics.
Beta Feature Risk
The tension: This is a disabled feature enabled via patches. Could change or break.
Symptoms: Anxiety about depending on unofficial features.
Mitigation: Have a fallback. Track CC Mirror releases. Accept the trade-off: zero-effort orchestration vs. stability guarantees.
Agent Autonomy
The tension: Less control than custom orchestrators. Agents make decisions you might not approve.
Symptoms: Tasks marked complete but implementation differs from expectations.
Mitigation: Write explicit acceptance criteria. Monitor agent view closely. Use comments for mid-task guidance.
Production Limits
The tension: Production needs state control, business logic integration, custom observability, and evals.
Symptoms: "This works for demos but not for real systems."
Mitigation: Native is for development velocity, not production infrastructure. Build custom for production. (@nelsonpadil_)
Dependency Graph Complexity
The tension: Complex graphs are manual to construct. No visualization, no cycle detection.
Symptoms: Large projects become hard to reason about.
Mitigation: Keep graphs simple (< 10 tasks). For complex projects, use Gas Town or manual decomposition into smaller graphs.
