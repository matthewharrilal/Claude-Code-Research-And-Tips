2 The Core Abstraction

"Instead of long periods of intense focus, software development can now just fit into the gaps of your day."
mosh --ssh="ssh -p 47892" devuser@100.x.x.x
Phone + SSH + cloud VM = full development capability, anywhere.

Nader's paradigm inverts the traditional model. Instead of human-as-coder with AI-as-assistant, it's AI-as-coder with human-as-supervisor. You don't write code. You orchestrate agents that write code.

3 Design Decisions

WHY CLOUD VM OVER LOCAL MAC?
Your phone can't SSH to a sleeping Mac. A cloud VM is always on, always accessible. Pay ~$7/day but gain 24/7 development capability from any device.
What this means for you
Start/stop your VM with iOS Shortcuts to control costs. Stop when not coding (saves ~$210/month if you only code 8hrs/day vs always-on).
WHY TAILSCALE SPECIFICALLY?
Zero-config VPN. No firewall rules. No exposed SSH port. Survives network changes. Your VM gets a private IP (100.x.x.x) that only your devices can reach.
What this means for you
Security without complexity. No SSH brute force attacks. Phone can lock, wake up, and reconnect automatically.
WHY MOSH OVER SSH?
mosh survives WiFi-to-cellular transitions, device sleep, and packet loss. SSH dies on network changes. mosh is UDP-based with local echo - typing feels instant even on high latency.
What this means for you
Phone locks during commute? Connection persists. WiFi drops to cellular? No reconnection needed. This is essential for mobile-first development.
WHY GIT WORKTREES FOR PARALLEL WORK?
Each worktree is a separate directory, separate branch. 6 agents can edit code simultaneously without conflicts. No merge headaches between parallel features.
What this means for you
Agent 1 on auth, Agent 2 on payments, Agent 3 on dashboard - all running in parallel, each in isolated file systems. Merge when ready.
WHY POKE FOR NOTIFICATIONS?
iOS-native, webhook-based, free. When Claude hits AskUserQuestion, your phone buzzes with the actual question. No custom infrastructure needed.
What this means for you
Push notifications are the async interface. You're only pulled into development when Claude needs human input. Otherwise, agents work autonomously.

Push Notification Hook

~/.claude/settings.json - Hook configuration: 

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "AskUserQuestion",
        "hooks": [
          {
            "type": "command",
            "command": "~/.claude/hooks/notify-mobile.sh"
          }
        ]
      }
    ]
  }
}

~/.claude/hooks/notify-mobile.sh - The notification script: 

#!/bin/bash
# Extract the question from Claude's event data
QUESTION=$(echo "$CLAUDE_HOOK_EVENT_DATA" | jq -r '.tool_input.questions[0].question // .tool_input.question // "Input needed"')
PROJECT=$(basename "$PWD")

# Send to Poke (iOS notification service)
curl -s -X POST "$POKE_API_URL" \
  -H "Authorization: Bearer $POKE_TOKEN" \
  -H "Content-Type: application/json" \
  -d "{
    \"title\": \"Claude: $PROJECT\",
    \"body\": \"$QUESTION\"
  }"

VM Lifecycle Scripts

#!/bin/bash
# ~/scripts/vm-start
INSTANCE_ID="your-instance-id"
curl -X POST "https://api.vultr.com/v2/instances/$INSTANCE_ID/start" \
  -H "Authorization: Bearer $VULTR_API_KEY"

#!/bin/bash
# ~/scripts/vm-stop
INSTANCE_ID="your-instance-id"
curl -X POST "https://api.vultr.com/v2/instances/$INSTANCE_ID/halt" \
  -H "Authorization: Bearer $VULTR_API_KEY"
echo "VM stopping (saving ~$7/day)..."

6 Gotchas

Real problems you'll hit, with concrete numbers and fixes:

#1: VM costs if forgotten
Symptom: $210/month bill instead of expected $70-100
Cause: VM running 24/7 instead of only when coding
Fix: iOS Shortcut "Stop Dev" when done. Takes 1 minute. Saves $140+/month.
#2: Notification fatigue
Symptom: Phone buzzes constantly for every confirmation
Cause: Every AskUserQuestion triggers notification, including routine confirmations
Fix: Use smart-notify.sh with filtering. Skip "confirm" and "proceed" messages. 10 min setup.
#3: Phone keyboard too slow for complex commands
Symptom: Typing bash commands on phone is frustrating
Cause: Phone keyboards not designed for complex syntax
Fix: Pre-saved snippets in Termius. Store common commands. 5 min setup.
#4: Port conflicts with parallel branches
Symptom: Multiple dev servers clash, "port already in use"
Cause: All branches trying to use default ports (3000, 8000)
Fix: Hash-based port allocation: port = 8001 + (hash(branch) % 99)
#5: Tailscale connection drops
Symptom: "No route to host" error after phone sleep
Cause: Tailscale app needs to reconnect after network change
Fix: Open Tailscale app briefly to reconnect. 30 seconds.
#6: tmux session lost on VM reboot
Symptom: All agents gone after VM restart
Cause: tmux sessions don't persist across reboots
Fix: Re-run start-agents.sh. 2 minutes. Consider tmux-resurrect for persistence.
#7: Push notification not arriving
Symptom: Agent waiting for input, no phone notification
Cause: POKE_API_URL or POKE_TOKEN misconfigured
Fix: Test webhook manually: curl -X POST "$POKE_API_URL" .... 5 min debug.

7 What's Hard

These are fundamental tensions, not bugs to fix:

Small Screen vs. Code Complexity
The tension: Code requires visual context. Phone screens are small. You cannot read a full file, trace dependencies, or understand architecture on a 6-inch display.
Why it's fundamental: Phone form factor is fixed. Code complexity is fixed. These don't change.
Coping strategy: Accept that deep review needs a laptop. Phone is for direction, not deep work. Use CI as a verification proxy - green = trust, red = investigate on larger device.
Async vs. Urgent Issues
The tension: Push notifications don't convey urgency levels. "Missing env var" and "security vulnerability found" look the same.
Why it's fundamental: Async communication loses nuance. You've traded synchronous awareness for flexibility.
Coping strategy: Smart filtering script with priority levels. CRITICAL = immediate notification. Low priority = batch updates.
Cost vs. Availability
The tension: Always-on VM costs money ($210/month). Stopping loses session state. You're paying for availability you might not use.
Why it's fundamental: Cloud compute has a cost. State requires memory. These are physics.
Coping strategy: Accept the cost for the flexibility. Use tmux to persist across reconnects. Stop VM when truly idle (overnight if not running agents).
Trust vs. Verification
The tension: Mobile makes code review difficult. You must trust agents without seeing their full work.
Why it's fundamental: The whole point is to NOT be synchronously engaged. If you reviewed everything, you're just coding slowly on a phone.
Coping strategy: Green CI = trust. Red CI = investigate. Batch reviews in evening sessions. Accept that some issues will slip through.

8 When to Use / When Not

USE MOBILE WORKFLOW WHEN
You have 5-30 minute gaps throughout the day (commute, exercise, waiting)
Tasks are feature branches, tests, docs, bug fixes (well-defined, verifiable)
CI/tests can verify work (can't easily review code on phone)
Budget allows $7/day for VM ($210/month if always on)
You travel frequently (maximum value when laptop impractical)
DON'T USE MOBILE WORKFLOW WHEN
Deep architecture decisions -> Need full context, diagrams, laptop
Complex debugging -> Need to see logs, traces, stack frames
UI/UX work -> Can't see rendered output for visual verification
Security audits -> High stakes require careful review
Have 2+ hours focused time -> Use laptop, vanilla Claude Code
