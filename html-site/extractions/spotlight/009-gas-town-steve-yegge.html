2 The Core Abstraction Level 7

"Gas Town is a factory for agents, not a better single agent."

The insight: Stop optimizing one Claude instance. Instead, build an ant colony: multiple specialized agents coordinating through shared external memory (Beads), each doing what it does best.

Gas Town is an Idea Compiler. You're the PM, agents are the engineering team.

The 8 Stages of Developer Evolution

Steve Yegge's progression model. Gas Town is for Stage 7+. If you're not there, "you'll get your face ripped off by superintelligent chimpanzees."

Stage Description
1 Zero/near-zero AI usage
2 Agent in IDE, permissions on
3 Agent in IDE, YOLO mode
4 Agent fills screen, code is just diffs
5 CLI single agent, YOLO
6 CLI multi-agent (3-5 parallel)
7 10+ agents, hand-managed
8 Building your own orchestrator (Gas Town)

The 7 Worker Roles

Each role is specialized. Together they form a complete development factory.

Role Function
Mayor Town-level coordination, routes work to rigs
Deacon Monitoring, handshakes, health checks
Dogs Quality gates, watchdogs, code review
Refinery Task decomposition into subtasks (Beads)
Polecat Named persistent workers (alice, bob, carol)
Witness Per-rig observer, syncs shared context
Crew Ephemeral workers for overflow work
Overseer (YOU) Human boss with inbox for decisions

Beads: The Shared Memory

Gas Town runs on Beads (github.com/steveyegge/beads):

  • Git-backed data plane - Durable, diffable, auditable, zero infrastructure
  • Universal context management - All agents read/write shared state
  • ~225K lines of Go - Vibe-coded by Steve

3 Design Decisions

WHY 7 ROLES (NOT 3 OR 20)?
7 roles covers the full development lifecycle: coordination (Mayor), decomposition (Refinery), execution (Polecats), quality (Dogs), observation (Witness), scaling (Crew), and human oversight (Overseer). Fewer roles = bottlenecks. More roles = excessive coordination overhead.
What this means for you
7 is "enough specialization to parallelize, few enough to coordinate." You can name your Polecats (alice, bob, carol) and develop mental models of their work.
WHY GO FOR BEADS?
Steve's finding from ~1M lines vibe-coded: TypeScript = 1/3 to 1/2 of diffs are type wrestling. Python = "fine" but feels like scripts. Go = "boring" = evolutionary advantage. Diffs are understandable by AI and humans.
What this means for you
"Go is just... good. It's boring. You can always understand it!" The lack of magic makes AI edits predictable.
WHY TMUX (NOT DOCKER)?
tmux provides visual observability. You can watch all agents working in split panes. Docker isolates but hides. Gas Town is designed for direct observation during development.
What this means for you
You must learn tmux basics (30-60 minutes). But you can literally watch your agents work, debug in real-time, and understand the system behavior.
WHY "VIBE CODING" PHILOSOPHY?
"Work becomes fluid, an uncountable that you sling around freely, like slopping shiny fish into wooden barrels at the docks." Some bugs fixed 2-3 times. Some fixes get lost. Focus: THROUGHPUT, not perfection.
What this means for you
Accept rework as the cost of speed. Steve claims ~1M lines in 2025 without reading the code. If that sounds insane, Gas Town isn't for you yet.

4 The File Structure

The complete ~/.gt structure. This is your "Town" - the factory infrastructure. 

~/.gt/                           # Gas Town home (The Town)
├── config.yaml                  # Town-level configuration
├── rigs/                        # Project registrations
│   └── my-project/
│       ├── worktrees/           # Git worktrees per polecat
│       │   ├── alice/
│       │   ├── bob/
│       │   └── carol/
│       └── rig.yaml             # Rig-specific config
├── beads/                       # Beads data plane
│   └── .git/                    # Git-backed storage
├── logs/                        # Agent logs
│   ├── mayor.log
│   ├── deacon.log
│   └── polecats/
└── inbox/                       # Human decision queue (Overseer)

Startup Commands

# Install Gas Town
go install github.com/steveyegge/gastown/cmd/gt@latest

# Initialize your town
gt init

# Add a rig (project)
gt rig add /path/to/repo

# Launch roles
gt mayor up
gt refinery up beads
gt polecat up beads alice
gt polecat up beads bob
gt polecat up beads carol
gt witness up beads
gt dogs up

Environment Requirements

  • Go 1.21+ - For Beads and gt binary
  • tmux - Primary UI (no alternative)
  • Multiple Claude API accounts - Each role consumes tokens
  • Git - For Beads storage and worktrees

5 The Path of a Task

Follow a feature request through the Gas Town system. Example: "Build user authentication"

1
[T+0s] Human files request to Overseer inbox: gt inbox send my-project "Build user auth"
2
[T+30s] Mayor receives, routes to appropriate rig. Creates feature branch.
3
[T+1min] Deacon acknowledges with handshake protocol. Health checks running.
4
[T+2min] Refinery decomposes task into Beads: bd-a1b2 (DB schema), bd-c3d4 (API routes), bd-e5f6 (Frontend). Sets dependency: bd-c3d4 blocked by bd-a1b2.
5
[T+3min] Polecat alice assigned bd-a1b2. Works in her worktree: ~/.gt/rigs/my-project/worktrees/alice/
6
[T+15min] Alice completes schema. bd-a1b2 status: complete. bd-c3d4 unblocks, goes to bob.
7
[T+20min] Witness observes progress, syncs context to shared Beads state.
8
[T+30min] Dogs run quality check: gt dogs bark. Reviews alice's changes. Reports issues or approves.
9
[T+45min] All subtasks complete. Refinery merges via queue. Feature branch ready.
10
[T+50min] Mayor notifies Overseer. Human reviews in inbox. Approves or requests changes.
11
[T+55min] Feature complete. Merged to main. Crew cleaned up.

6 Gotchas

10 real problems with specific fixes. From Steve's warnings and community reports.

#1: Code is 3 weeks old
Steve's words: "smuggled 400 miles upriver in my ass"
Reality: Expect unexpected behavior, bugs, rough edges
Fix: Check GitHub issues, expect instability. Accept.
#2: 100% vibe-coded
Reality: Steve has never read the code
Implication: Don't expect polish. Report issues.
Fix: Accept. This is the vibe coding philosophy made manifest.
#3: Expensive as hell
Reality: Multiple Claude accounts burn budget. Each role = tokens.
Budget: $50-200+/day is realistic for active development
Fix: Set spending limits. Use Haiku for simple work. Accept cost.
#4: Requires tmux
Reality: No alternative UI. tmux is the interface.
Time to learn: 30-60 minutes for basics
Fix: Learn tmux. Or don't use Gas Town.
#5: Can wreck your shit
Steve's warning: "Chimps rip faces off"
Reality: Agents make mistakes, break things, delete code
Fix: Sandbox, backups, staged rollout. Prevention.
#6: Stage 7+ required
Prerequisite: 100+ hours of multi-agent experience
Below this: "Face ripped off by superintelligent chimpanzees"
Fix: Build up through stages 1-6 first. Don't skip.
#7: Beads desync
Symptom: Agents have stale context, work conflicts
Cause: Git-backed storage can lag
Fix: gt beads sync --force (5 min)
#8: Polecat naming confusion
Symptom: Lost track of which polecat did what
Cause: Poor naming conventions
Fix: Use consistent naming, check logs. Planning.
#9: Merge conflicts
Symptom: Multiple polecats edit same file
Cause: Poor task decomposition
Fix: Use Refinery queue, worktree isolation. Architecture.
#10: Cost explosion overnight
Symptom: Woke up to 5x expected bill
Cause: Uncapped Crew spawning
Fix: Set crew limits in config.yaml (5 min config)

7 What's Hard

These are fundamental tensions, not bugs to fix. Managing, not solving.

Cost vs Throughput
The tension: More agents = more throughput = more cost. Always.
Why unsolvable: Tokens cost money. No free lunch.
Coping strategy: Budget caps, Haiku for simple work, accept the trade-off.
Chaos vs Control
The tension: Agents make mistakes, and you can't prevent all of them.
Why unsolvable: LLMs are probabilistic. Perfect reliability is impossible.
Coping strategy: Accept rework. Focus on throughput, not zero bugs. Dogs role for quality gates.
Vibe Coding Trade-off
The tension: Some bugs fixed 2-3 times. Some fixes get lost.
Why unsolvable: Speed and perfection are opposites. You chose speed.
Coping strategy: Trust the process. Ship anyway. The goal is output, not elegance.
Stage 7+ Cliff
The tension: You can't shortcut experience. Gas Town requires prior multi-agent fluency.
Why unsolvable: Understanding emerges from hours of practice, not reading docs.
Coping strategy: Build up through stages 1-6 first. There are no shortcuts.
Debugging a Distributed System
The tension: 8 roles, hard to trace failures. Where did it go wrong?
Why unsolvable: Distributed systems have emergent failure modes.
Coping strategy: Good logging, Witness role, tmux observability.
Mental Model Shift
The tension: You're a Product Manager now, not a programmer.
Why unsolvable: Identity change is slow. Years of programming habits to unlearn.
Coping strategy: Takes weeks to adjust. Be patient with yourself.

8 When to Use / When Not

USE GAS TOWN WHEN
You're at Stage 7+ (100+ hours multi-agent experience)
Daily budget is $50-200+ (multiple Claude accounts)
Solo developer wanting team-scale output
Large, ongoing project with multiple domains
High chaos tolerance (some bugs, rework expected)
tmux proficiency (mandatory, no alternative)
DON'T USE GAS TOWN WHEN
Not comfortable with chaos - use Ralph or CC Mirror
Care about costs - use CC Mirror (lower multi-agent cost)
Below Stage 6 - build foundation with Level 0-5 patterns first
Want polish - use Manual + Ralph for more control
Team already exists - Gas Town replaces, doesn't augment
