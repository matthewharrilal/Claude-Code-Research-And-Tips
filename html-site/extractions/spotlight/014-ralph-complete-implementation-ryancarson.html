Ryan Carson's implementation is the most complete, reproducible Ralph guide available. It includes a GitHub repository with all files, a live demo site, and real production results: 13 user stories completed in ~15 iterations over ~1 hour.

2 The Core Abstraction

"Ralph is an autonomous AI coding loop that ships features while you sleep."
while :; do cat prompt.md | amp --dangerously-allow-all ; done
Created by @GeoffreyHuntley, this runs your agent repeatedly until all tasks complete. Memory persists via git + text files.

6 Critical Success Factors

1 Small Stories
Must fit in one context window. "Build auth system" is too big. "Add login form", "Add email validation" are right-sized.
2 Feedback Loops
Ralph needs fast feedback: npm run typecheck, npm test. Without these, broken code compounds.
3 Explicit Criteria
Not "works correctly" but specific: "Email/password fields", "Validates email format", "typecheck passes".
4 Learnings Compound
By story 10, Ralph knows patterns from stories 1-9. progress.txt = session memory, AGENTS.md = permanent docs.
5 AGENTS.md Updates
"When modifying X, also update Y." Permanent knowledge for future agents. Don't add story-specific details.
6 Browser Testing
For UI changes, use dev-browser skill. "Not complete until verified with screenshot." Tests pass != looks right.

3 Design Decisions

WHY scripts/ralph/ DIRECTORY?
Organized structure that's portable across projects. All Ralph files in one place, easy to copy to any new project.
What this means for you
Copy the entire scripts/ralph/ folder to any project. Customize prompt.md and prd.json for your needs. The structure is project-agnostic.
WHY PRIORITY FIELD IN STORIES?
Deterministic ordering. Agent always picks the same story (highest priority, passes=false). No ambiguity about what to work on next.
What this means for you
Set priority 1 for foundational work (schema), then 2 for dependent features, etc. Agent won't try to build UI before the data layer exists.
WHY progress.txt FOR LEARNINGS?
Session memory that survives iterations. Each fresh Claude instance reads progress.txt first, learns what worked in previous iterations.
What this means for you
Use "APPEND" (caps) in prompt.md. Models interpret "update" as "replace". Add reusable patterns to the TOP under "Codebase Patterns" section.
WHY branchName IN prd.json?
Feature isolation. Agent checks it's on the correct branch before working. All feature work stays isolated until you merge.
What this means for you
Create branch before running Ralph. Set branchName in prd.json. Review entire feature as one PR when complete.
WHY "FAIL FAST, COMMIT ANYWAY"?
Recoverability. Even failed iterations leave a git commit. You can bisect to find where things went wrong. Better than silent corruption.
What this means for you
Check git log --oneline after runs. You'll see exactly what happened at each iteration. Rollback is easy with git reset.

4 The File Structure 

scripts/ralph/
├── ralph.sh      # The loop script
├── prompt.md     # Instructions for each iteration
├── prd.json      # Task list with user stories
└── progress.txt  # Learnings + patterns

ralph.sh - The Loop Engine

#!/bin/bash
set -e

MAX_ITERATIONS=${1:-10}
SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"

echo "Starting Ralph"

for i in $(seq 1 $MAX_ITERATIONS); do
  echo "=== Iteration $i ==="

  OUTPUT=$(cat "$SCRIPT_DIR/prompt.md" \
    | amp --dangerously-allow-all 2>&1 \
    | tee /dev/stderr) || true

  if echo "$OUTPUT" | grep -q "<promise>COMPLETE</promise>"; then
    echo "Done!"
    exit 0
  fi

  sleep 2
done

echo "Max iterations reached"
exit 1

Make executable: chmod +x scripts/ralph/ralph.sh. For Claude Code: replace amp --dangerously-allow-all with claude --dangerously-skip-permissions.

prompt.md - Agent Instructions

# Ralph Agent Instructions

## Your Task

1. Read `scripts/ralph/prd.json`
2. Read `scripts/ralph/progress.txt` (check Codebase Patterns first)
3. Check you're on the correct branch
4. Pick highest priority story where `passes: false`
5. Implement that ONE story
6. Run typecheck and tests
7. Update AGENTS.md files with learnings
8. Commit: `feat: [ID] - [Title]`
9. Update prd.json: `passes: true`
10. APPEND learnings to progress.txt

## Progress Format

APPEND to progress.txt:

---
## [Date] - [Story ID]
- What was implemented
- Files changed
- **Learnings:**
  - Patterns discovered
  - Gotchas encountered

## Codebase Patterns

Add reusable patterns to the TOP of progress.txt:

## Codebase Patterns
- Migrations: Use IF NOT EXISTS
- React: useRef<Timeout | null>(null)

## Stop Condition

If ALL stories pass, reply:
<promise>COMPLETE</promise>

Otherwise end normally.

prd.json - Task Board

{
  "branchName": "ralph/feature",
  "userStories": [
    {
      "id": "US-001",
      "title": "Add login form",
      "acceptanceCriteria": [
        "Email/password fields",
        "Validates email format",
        "typecheck passes"
      ],
      "priority": 1,
      "passes": false,
      "notes": ""
    }
  ]
}
Key fields: branchName = branch to use | priority = lower = first | passes = set true when done | notes = hints for stuck agent

6 Gotchas

Real problems from production use, with concrete fixes:

Idempotent Migrations
Symptom: "Column already exists" errors
Fix: Use ADD COLUMN IF NOT EXISTS email TEXT;
Time to fix: 5 minutes
Interactive Prompts
Symptom: Script hangs waiting for input
Fix: echo -e "\n\n\n" | npm run db:generate
Time to fix: 2 minutes
Schema Changes Cascade
Symptom: Type errors in unrelated files after schema change
Fix: Check server actions, UI components, API routes - fix them (not scope creep)
Time to fix: 10 minutes
Story Too Big
Symptom: Context overflow, incomplete implementations
Fix: Split into smaller stories. If you can't describe in 2-3 sentences, too big.
Time to fix: Planning phase
Vague Acceptance Criteria
Symptom: Agent interprets differently than you expected
Fix: Be specific: "dropdown has 3 options" not "dropdown works correctly"
Time to fix: Planning phase
Notes Field Unused
Symptom: Agent stuck on same issue repeatedly
Fix: Add hints to the notes field in prd.json: "Use the existing auth helper"
Time to fix: 2 minutes
progress.txt Overwritten
Symptom: Lost learnings from previous iterations
Fix: Use "APPEND" in CAPS in prompt.md - the capitalization matters
Time to fix: Prevention

7 What's Hard

These are fundamental tensions, not bugs to fix:

Story Sizing Art
The tension: Too big = context overflow. Too small = overhead dominates.
Coping strategy: "One context window" rule. If description > code change, you've decomposed too far.
Explicit Criteria
The tension: Easy to be vague because humans understand implicit requirements.
Coping strategy: Write like test assertions. "Renders login button" not "login works".
Learning vs Speed
The tension: Documenting learnings takes time each iteration.
Coping strategy: Pays off by iteration 10. Early investment = compound returns.
Review Burden
The tension: 13+ commits to check in the morning.
Coping strategy: Trust CI, check progress.txt for anomalies, review via git diff.
UI Verification
The tension: Tests pass but UI is broken.
Coping strategy: dev-browser skill is mandatory for UI stories. Add screenshot verification to acceptance criteria.

8 When to Use / When Not

USE THIS RALPH WHEN
Feature with clear user stories (verifiable completion)
Overnight development is acceptable
Well-factored codebase with clean patterns
Good test coverage (feedback loops work)
Solo developer wanting to scale capacity
DON'T USE WHEN
Exploratory work with no defined end
Major refactors without clear criteria
Security-critical code (needs human review)
Anything requiring subjective judgment
Legacy code without tests (add tests first)
