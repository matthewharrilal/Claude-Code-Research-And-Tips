3 Design Decisions

WHY ASYNC OVER SYNC?
Phone input is inherently slow and interruptible. You're in line, on a train, between meetings. Synchronous coding (type-run-type-run) doesn't fit. Async means you give intent ("add dark mode"), agent executes while you do other things, you review results later.
What this means for you
Don't try to "code on your phone" like you code at a desk. Give high-level direction, let agents run, review when convenient. Your phone is a remote control, not a keyboard.
WHY 5 LAYERS (NOT 3 OR 7)?
Every mobile setup needs: compute (where Claude runs), connectivity (how you reach it), notifications (how it reaches you), session management (persistence), and human interface (your device). Fewer layers means missing functionality. More layers adds unnecessary complexity.
What this means for you
When troubleshooting, check each layer. Can't connect? Layer 2 (connectivity). Not getting notifications? Layer 3. Session died? Layer 4. The 5-layer model is your diagnostic framework.
WHY 4 ARCHITECTURES (NOT ONE "BEST")?
No single approach wins on all dimensions. Cloud VM maximizes control but requires technical skill. Replit minimizes setup but limits customization. Superconductor offers best UX but costs more and requires Apple. Home Mac is free but requires hardware you may not have.
What this means for you
Choose based on YOUR priorities: technical comfort, budget, platform, desired control level. There's no universally "best" - only best for your situation.
WHY TMUX FOR SESSION MANAGEMENT?
Mobile connections are unreliable. You'll lose connectivity in tunnels, elevators, spotty wifi. Without persistent sessions, your Claude work dies when connection drops. tmux (or equivalent) keeps sessions alive server-side so you can reattach anytime.
What this means for you
For VM/Home Mac setups, tmux is non-negotiable. Add auto-attach to your shell config. For Replit/Superconductor, session persistence is built-in - that's part of their value proposition.
WHY PUSH NOTIFICATIONS?
Async work requires knowing when to check back. Without notifications, you either poll constantly (defeating async benefits) or miss completion/questions for hours. Push notifications close the loop - agent needs input, you get pinged immediately.
What this means for you
Set up hooks (ntfy, Pushover, Poke) for VM setups. Native apps handle this automatically. The notification layer is what makes async actually work in practice.
WHY TAILSCALE OVER RAW SSH?
SSH to a public IP exposes your server to the internet. Tailscale creates a private overlay network - your devices can reach each other but no one else can. Zero firewall config, automatic key rotation, works across NAT.
What this means for you
For VM/Home Mac: Install Tailscale on both ends. Connect via Tailscale IP (100.x.x.x), never public IP. Free tier covers personal use.

scripts/idle-shutdown.sh - Cost Control

#!/bin/bash
# scripts/idle-shutdown.sh
# Auto-shutdown VM after 2 hours idle (prevents $210/month surprise)
# Add to crontab: */15 * * * * ~/scripts/idle-shutdown.sh

IDLE_THRESHOLD=7200  # 2 hours in seconds
LAST_ACTIVITY=$(stat -c %Y /tmp/claude-activity 2>/dev/null || echo 0)
NOW=$(date +%s)
INACTIVE=$((NOW - LAST_ACTIVITY))

if [ "$INACTIVE" -gt "$IDLE_THRESHOLD" ]; then
  # Send warning notification
  curl -d "VM shutting down due to inactivity" ntfy.sh/your-topic

  # Wait 5 minutes for user to respond
  sleep 300

  # Re-check activity
  LAST_ACTIVITY=$(stat -c %Y /tmp/claude-activity 2>/dev/null || echo 0)
  NOW=$(date +%s)
  INACTIVE=$((NOW - LAST_ACTIVITY))

  if [ "$INACTIVE" -gt "$IDLE_THRESHOLD" ]; then
    sudo shutdown now
  fi
fi

# Touch activity file when Claude is active
# Add to .claude/hooks: touch /tmp/claude-activity

6 Gotchas by Approach

Common to ALL Approaches

No offline mode
Symptom: Can't code without internet connection
Reality: ALL mobile approaches require network. Period.
Mitigation: Accept it. Or use local models (Ollama) for degraded offline mode.
API costs add up invisibly
Symptom: Higher bills than expected
Threshold: Parallel agents = multiplied cost (4 agents = 4x API)
Fix: Monitor usage religiously. Set billing alerts at $50, $100, $200.

Cloud VM Specific

VM costs if not stopped
Symptom: $210/month surprise bill (Vultr vhf-8c-32gb always-on)
Cause: Forgot to stop VM after session
Fix: Use vm-stop religiously. Or use Hetzner (~$13/month always-on).
mosh doesn't forward SSH agent
Symptom: git push fails with permission denied
Cause: SSH agent forwarding doesn't work over mosh
Fix: Run ssh-add in tmux session. Or use SSH instead of mosh for git operations.
Tailscale disconnects silently
Symptom: Can't reach VM, connection times out
Fix: Re-auth in Tailscale app. Check "Connected" status before SSH.

Replit Specific

Replit Agent uses credits (not Claude Code CLI)
Symptom: Unexpected Replit credit charges
Cause: Using Replit's built-in "Agent" feature instead of Claude Code CLI
Fix: Install Claude Code CLI separately. Use terminal, not Replit Agent.
Session timeout loses work
Symptom: Return to find session ended, unsaved work gone
Fix: Commit frequently. Replit doesn't have tmux persistence.

Superconductor Specific

iOS only - no Android
Symptom: Can't find Superconductor in Play Store
Reality: Apple-only app
Alternative: Use Cloud VM approach for Android
GitHub-centric workflow
Symptom: Want to use GitLab/Bitbucket
Reality: Superconductor is GitHub-native only
Alternative: Cloud VM for other VCS platforms

7 What's Hard: Fundamental Tensions

These apply to ALL mobile approaches. They're physics, not bugs.

Small screen vs. code context
The tension: Phones can't show full files. You can't see the whole system at once.
Symptoms: Missing context, making wrong decisions, needing to scroll constantly
Mitigation: Accept it. Mobile is for OVERSIGHT, not deep work. Use phone for direction, laptop for immersion.
Notification fatigue vs. responsiveness
The tension: Too many notifications = burnout. Too few = delayed agents.
Symptoms: Either ignoring all notifications OR being interrupted constantly
Mitigation: Smart filtering. Priority levels. Quiet hours. Batch notifications.
Input limitations
The tension: Phone typing is slow. Voice has errors. Neither is as fast as keyboard.
Symptoms: Frustration, typos, voice misinterpretation
Mitigation: External Bluetooth keyboard for serious sessions. Voice dictation for quick responses. Snippets for common commands.
Async paradigm requires mental shift
The tension: Mobile development feels "wrong" if you expect laptop-style focus sessions.
Symptoms: Frustration that you "can't really code" on phone
Mitigation: Reframe: Mobile is a DIFFERENT mode, not a worse laptop. You're directing agents, not typing code.
