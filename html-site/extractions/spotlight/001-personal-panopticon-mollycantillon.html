2 The Core Abstraction

"Institutions see you but you cannot see back. Claude Code lets you build your own watchtower."
~/nox ~/metrics ~/email ~/growth ~/trades ~/health ~/writing ~/personal
8 isolated directories. 8 Claude instances. One self-legible life.

Pre-modern states were blind to their citizens. They built "apparatus of sight": censuses, surnames, maps. Now corporations have legibility over YOU (behavioral data) but you don't have it over yourself. Personal Panopticon reverses this asymmetry.

Each Claude instance operates independently in its own directory. No cross-contamination. Each spawns short-lived subagents for specific tasks. They exchange context through explicit file-based handoffs. The filesystem IS the persistent memory.

3 Design Decisions

WHY 8 DOMAINS (NOT 3 OR 20)?
8 domains cover the major areas of life without excessive coordination overhead. Fewer means gaps in coverage. More means coordination complexity explodes and maintenance becomes a full-time job.
What this means for you
Start with 3-4 high-impact domains. Add more only when existing domains are stable. The sweet spot is between coverage and manageability.
WHY ISOLATED DIRECTORIES?
Cross-contamination is the death of clarity. If your trades Claude instance starts reading health data, its recommendations become muddied. Each instance is "contained" to its domain through directory boundaries.
What this means for you
Add explicit boundaries to each domain's CLAUDE.md: "DO NOT access other domains." Use file permissions if needed. Cross-domain communication happens ONLY through the handoff queue.
WHY FILESYSTEM OVER DATABASE?
Claude natively reads and writes files. Zero infrastructure required. No database setup, no schema migrations, no connection strings. The filesystem is the persistence layer Claude already speaks.
What this means for you
Start immediately. Create directories and CLAUDE.md files. No infrastructure sprint needed. If you later need more structure, migrate then. Premature optimization is the root of complexity.
WHY CRON OVER REAL-TIME?
Overnight/scheduled runs enable batch processing without babysitting. Most life automation doesn't need sub-second response. Morning briefs, inbox processing, workout generation - all can wait for their scheduled slot.
What this means for you
Schedule heavy processing overnight. Use cron for regular cadences. For time-sensitive triggers (WHOOP sleep detection), use file watchers or webhooks that spawn Claude on demand.
WHY HUMAN-IN-THE-LOOP VIA TEXT?
Checkpoint approvals prevent runaway automation. Text notifications reach you anywhere. Reply to approve, adjust, or abort. You remain in control at decision points while automation handles the grunt work.
What this means for you
Set up Twilio or similar for text notifications. Define checkpoint moments for each domain. Don't make everything autonomous - that's how $2000 in subscriptions renews while you sleep.
WHY GUI AUTOMATION FALLBACK?
Not everything has an API. Some services only offer web interfaces. GUI automation (mouse/keystroke injection) lets Claude control desktop apps directly when APIs fail.
What this means for you
Use APIs when available - they're more reliable. Fall back to GUI automation only when necessary. Log all GUI actions. Be prepared for breakage when apps update their interfaces.

Sample Domain CLAUDE.md (~/trades/CLAUDE.md)

# Trades Domain - Claude Instructions

## Domain Purpose
This instance manages personal finance: brokerage data,
congressional disclosures, market sentiment, morning briefs.

## Data Sources
- Brokerage API (overnight scraping)
- Congressional disclosure RSS
- Polymarket odds API
- X/Twitter sentiment (via scraping)

## Output Expectations
- Daily: ~/trades/morning-brief/YYYY-MM-DD.md
- Signals: ~/trades/signals/active.json
- Alerts: Text notification for actionable items

## Boundaries
- DO NOT access other domains
- DO NOT make trades (advisory only)
- DO NOT store credentials in files

## Handoff Protocol
- Incoming: Check ~/trades/inbox/ for tasks
- Outgoing: Write to ~/.panopticon/handoffs/ with domain prefix

Automation Layer

# Keep system awake during long runs
caffeinate -i

# Simple alias for natural language CLI
alias c='claude'
# Usage: c "Show me everyone who hasn't paid in 90 days"

# Cron jobs for scheduled runs (crontab -e)
# Finance brief at 5am
0 5 * * * cd ~/trades && claude "Generate morning brief" >> ~/logs/trades.log 2>&1

# Email triage at 8am
0 8 * * * cd ~/email && claude "Process inbox" >> ~/logs/email.log 2>&1

# Process handoffs every hour
0 * * * * ~/scripts/handoff-process.sh >> ~/logs/handoffs.log 2>&1

5 The Path of a Task

Path 1: Morning Finance Brief

1
[T+0min] Cron triggers at 5:00 AM: cd ~/trades && claude "Generate morning brief"
2
[T+1min] Claude loads domain context: reads CLAUDE.md, positions/current.json, yesterday's brief
3
[T+2min] Data collection: scrapes brokerage, fetches congressional disclosures (RSS), Polymarket odds, X sentiment
4
[T+10min] Synthesis: correlates position changes vs news, identifies unusual activity, generates recommendations
5
[T+15min] Output: writes morning-brief/2026-01-25.md, updates signals/active.json, sends text if urgent
6
[T+16min] Session ends. Claude terminates. Brief ready for human review. Fresh context tomorrow.

Path 2: Email Triage with Handoff

1
[T+0min] Cron triggers at 8:00 AM: cd ~/email && claude "Process inbox"
2
[T+1min] Claude loads email domain: reads CLAUDE.md, sorted/ patterns, templates/
3
[T+5min] Inbox processing: categorizes (support, sales, personal, spam), drafts routine replies, flags items needing human decision
4
[T+15min] Handoff detected: Email about subscription renewal ($2000 found!). Creates ~/.panopticon/handoffs/personal-subscription-audit.md
5
[T+20min] Checkpoint notification: texts "Email processed: 47 sorted, 3 need review". Human replies to approve/adjust.
6
[T+25min] Session ends with handoff queued. Personal domain will pick up subscription audit on next run.

Path 3: WHOOP Sleep Trigger

1
[T+0min] External trigger: WHOOP API detects 6+ hours sleep, webhook writes to ~/health/triggers/sleep-complete.json
2
[T+1min] File watcher activates: cd ~/health && claude "Sleep complete, process trigger"
3
[T+2min] Claude reads trigger context: sleep-complete.json (sleep quality), yesterday's workout, travel schedule
4
[T+5min] Workout generation: adjusts today's workout based on recovery score, considers travel constraints
5
[T+8min] Projector activation: Recovery score > 80% triggers GUI automation to start morning routine projector
6
[T+10min] Session ends. Workout written to workouts/2026-01-25.md. Projector running. Human wakes to personalized morning.

6 Gotchas

Real problems you'll hit, with concrete numbers and fixes:

1. Domain Cross-Contamination
Symptom: Agent reads files outside its domain, recommendations get muddied
Fix: Add explicit boundary in CLAUDE.md: "DO NOT access other domains". Use directory permissions (chmod) if needed.
Time to fix: 5 minutes
2. Handoff Queue Backup
Symptom: ~/.panopticon/handoffs/ fills up, handoffs never processed
Fix: Add cron job to process handoffs hourly. Alert on queue size > 10 items.
Time to fix: 10 minutes
3. Cron Silent Failure
Symptom: Scheduled job doesn't run, no error visible
Fix: Add >> ~/logs/domain.log 2>&1 to every cron entry. Monitor logs daily.
Time to fix: 15 minutes
4. API Rate Limits
Symptom: Brokerage or service blocks overnight scraping
Fix: Add rate limiting between requests. Spread requests over 30+ minutes instead of bursting.
Time to fix: 30 minutes
5. Text Notification Failure
Symptom: Twilio down, no checkpoint received, agent stuck waiting
Fix: Add fallback notification (email). Add timeout + default action after 30 minutes.
Time to fix: 20 minutes
6. GUI Automation Breaks
Symptom: Desktop app updated, mouse coordinates wrong
Fix: Use accessibility APIs where possible (more stable). Log all clicks. Accept occasional manual intervention.
Time to fix: 30 minutes per app
7. Context Too Large
Symptom: Domain accumulates too much state, Claude slows down
Fix: Add cleanup routine. Archive briefs older than 30 days. Keep only recent data in active directories.
Time to fix: 10 minutes
8. Human Never Responds
Symptom: Checkpoint text sent, hours pass, agent stuck waiting
Fix: Add timeout + default action. Escalation path: text -> email -> Slack. Consider safe defaults.
Time to fix: 15 minutes
9. Credential Exposure
Symptom: API key accidentally written to brief or log
Fix: Use environment variables exclusively. Add credential scrubbing to output. Never persist secrets in files.
Time to fix: 5 minutes (but critical)
10. Domain Scope Creep
Symptom: Trades domain starts handling health concerns, boundaries blur
Fix: Review CLAUDE.md boundaries quarterly. Refactor when domains merge. Consider if domain split is still correct.
Time to fix: 20 minutes
Thread critique addressed: @flutterwhat: "Swarm of eight instances in parallel is a cute way to say I made eight folders and accidentally reinvented ~/bin plus crontab -e."
The technical implementation IS simple (directories + cron). The value is in the ORGANIZATION and domain isolation mental model, not novel infrastructure.

7 What's Hard

These are fundamental tensions, not bugs to fix:

Goodhart's Law
The tension: "Optimize for a metric and you game your way to hollow victory." When WHOOP says recovered but you feel like death - notice the gap.
Coping strategy: Keep a "meta-level outside the system" that can argue with the brief. Be ready to delete ~/health tomorrow if it stops serving. "We must continue to live outside it."
Anthropic Visibility
The tension: Every query goes to Anthropic. They see your finances, health, communications. This is a privacy trade-off with no technical workaround (short of local LLMs).
Coping strategy: "Sometimes the tower has a landlord. The value exchange is explicit: their visibility into your thinking for access to a thousand-clone attention span. In this case, chosen beats imposed."
Maintenance Burden
The tension: 8 domains = 8x the configuration, 8x the monitoring, 8x the potential failure points. This is operational complexity, not technical complexity.
Coping strategy: Invest in automation, templates, and health checks. Start with 3-4 domains. Add more slowly. Accept that maintenance is the price of capability.
Diminishing Returns
The tension: Early domains (email, trades) have massive ROI. Later domains have unclear value. Domain #8 might cost more attention than it saves.
Coping strategy: Track time-to-value for each domain. Cut domains that don't pay for themselves in attention savings. Quality over quantity.
Human Bottleneck
The tension: The system can surface insights, but humans still need to ACT. 8 domains producing 8 briefs creates 8 decision backlogs.
Coping strategy: Design for actionability, not just information. Reduce decision load - "here's what I recommend" not "here are 10 options." Batch non-urgent decisions.
Integration Complexity
The tension: Each new data source needs custom integration. Brokerage APIs differ from WHOOP APIs differ from email APIs. No universal connector exists.
Coping strategy: Use MCP servers where available (they standardize the interface). Accept some manual data entry. Don't let perfect integration block good-enough automation.

8 When to Use / When Not

USE PANOPTICON WHEN
Multiple distinct life domains need automation (not just one project)
You're comfortable with cron, shell scripts, basic sysadmin
Willing to invest 20+ hours in initial setup
You accept Anthropic data visibility (explicit trade-off)
Have API access to relevant services (not GUI-only)
Planning to use this for 1+ years (long-term investment)
DON'T USE PANOPTICON WHEN
Single domain to automate - use simple Claude + cron instead
No time for setup investment - use manual Claude sessions
Privacy-critical data (medical, legal) - consider local LLMs or no AI
Rapidly changing needs - architecture locks in structure
Team context - use Gas Town or CC Mirror instead
Real-time requirements - Panopticon is batch/cron oriented
