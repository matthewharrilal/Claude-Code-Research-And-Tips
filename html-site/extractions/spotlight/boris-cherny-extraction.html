2 Core Abstraction: The Vanilla Workflow

"Do the simple thing first. CLAUDE.md is another example of this idea - we had all these crazy ideas about memory architectures, but in the end, the thing we did is ship the simplest thing: a file that has some stuff that's auto-read into context."
Simplicity won over sophistication. Every time.
Philosophy

"Fresh context > extended sessions"

After ~80K tokens, LLM quality degrades. Instructions get lost. The model contradicts itself. This is "context rot" - a fundamental limitation of transformer attention.

Implementation

The /clear command

Use /clear liberally. Start new sessions when context feels bloated. Don't try to extend sessions - restart instead.

# When context feels bloated or quality degrades:
/clear
# Or exit and start fresh:
exit && claude
Philosophy

"Verification loops prevent drift"

LLMs hallucinate. Code proves reality. If Claude has a feedback loop, it will 2-3x the quality of the final result.

Implementation

Test after every change

Run npm test and npm run typecheck after every modification. Claude tests every single change using the Chrome extension for UI verification.

# After every code change:
npm run typecheck && npm test

# For UI changes, add visual verification:
# Chrome extension opens browser, tests UI
Philosophy

"CLAUDE.md is permanent memory"

Files persist, conversation fades. Every mistake becomes a rule. The filesystem IS the memory.

Implementation

Shared CLAUDE.md in Git root

Anytime the team sees Claude do something incorrectly, add it to CLAUDE.md. The longer the team works together, the smarter the agent becomes.

The Compounding System

"Think of Claude Code as a compounding system - not a one-shot tool, but an evolving teammate. Every correction becomes permanent context. The cost of a mistake pays dividends forever."
- Boris Cherny

Each session that updates CLAUDE.md increases effective capability for all future sessions. This is the mathematics of compounding: 

# Linear usage:
Output = Capability x Time

# Compounding usage (Boris's approach):
Output = Capability x Time x (1 + Learning Rate)^Sessions

3 Design Decisions

WHY VANILLA (NO HEAVY CONFIGURATION)?
Complexity creates debt. Claude is already optimized. Boris explicitly describes his own setup as "surprisingly vanilla" despite being the creator. This is deliberate: if the creator doesn't need heavy customization, the tool works well out of the box.
What this means for you
Start simple, add only when needed. Having a long list of complex, custom slash commands is an anti-pattern. The entire point of an agent like Claude is that you can type almost whatever you want.
WHY VERIFICATION LOOPS?
Probably the most important thing to get great results out of Claude Code: give Claude a way to verify its work. If Claude has that feedback loop, it will 2-3x the quality of the final result. Verification is not optional - it's the difference between 1x quality and 2-3x quality.
What this means for you
Always run tests after changes. Use typecheck commands. For UI, use Chrome extension or Playwright. Generic bash tests are insufficient for UI/mobile - domain-specific verification is required.
WHY FRESH CONTEXT OVER LONG SESSIONS?
Context rot degrades quality. At 80K tokens, Claude's quality degrades. Instructions get lost. The model contradicts itself. This is a fundamental limitation of transformer attention, not a bug.
What this means for you
Use /clear liberally. Start new sessions when stuck. If iteration 20 produces worse code than iteration 5, your context is rotting. The fix is to restart, not debug.
WHY PLAN MODE FIRST?
This prevents the classic failure: when the agent thinks it's helpful and makes 40 changes you didn't want. Don't let a system act before you've agreed on intent and constraints. Plan Mode can 2-3x success rates for complex tasks.
What this means for you
Start in Plan Mode (Shift+Tab twice). Iterate back-and-forth until plan is solid. THEN switch to auto-accept mode. The ones struggling often skip planning to save time, then spend more time fixing mistakes.
WHY PARALLEL SESSIONS (5-10)?
Boris doesn't see AI as a tool you use, but as a capacity you schedule. He runs 5 terminal instances with notifications, 5-10 web sessions, and mobile sessions started in morning and checked later. Each session is a separate worker with its own focus.
What this means for you
Run multiple Claude instances. Assign tasks, check progress, harvest results. Traditional thinking: "I ask Claude a question, it answers." Boris thinking: "I have 10 workers. I assign tasks, check progress, harvest results."
WHY ALWAYS OPUS 4.5 + EXTENDED THINKING?
The bottleneck isn't generation speed, it's correction time. Opus requires less steering, makes fewer tool use errors, produces higher-quality first drafts, and reduces iteration loops. Total Time = Generation + Correction + Re-prompting. Opus has higher generation but dramatically lower correction time.
What this means for you
Use think (4K tokens), think hard (10K tokens), or ultrathink (31,999 tokens) keywords. Net result: faster despite slower generation because you're not fixing mistakes.

5 Path of a Task: The Agentic Dev Loop

Boris's complete cycle for adding a feature like "user authentication":

1
[T+0min]
Start fresh Claude session. Command: claude (or /clear if in existing session). Claude reads CLAUDE.md automatically.
2
[T+1min]
Describe task clearly: "Add basic email/password authentication using Passport.js". Claude proposes approach in Plan Mode.
3
[T+2min]
Human reviews approach. Approve or redirect: "Yes, but use bcrypt for password hashing".
4
[T+3min]
Switch to auto-accept mode. Claude implements - edits files directly, creates auth routes, user model, middleware.
5
[T+8min]
Claude runs verification: npm run typecheck then npm test.
6
[T+10min]
Tests fail? Claude reads error output, corrects issues, re-runs verification.
7
[T+12min]
Tests pass, Claude creates commit message. Human reviews diff.
8
[T+15min]
Context check: Task complete? Done. Context feeling bloated? /clear and continue. CLAUDE.md updated if needed.

6 Gotchas: Anti-Patterns Boris Warns Against

#1: Skipping Plan Mode
Symptom: Claude makes 40 changes you didn't want
Boris says: "The ones struggling often skip planning to save time, then spend more time fixing mistakes. Plan Mode isn't training wheels - it's measuring before you cut."
Fix: Always start complex tasks in Plan Mode (Shift+Tab twice). Agree on intent before execution.
#2: Context rot in long sessions
Symptom: Quality degrades, Claude forgets earlier work, contradicts itself
Threshold: Quality noticeably drops after ~80K tokens
Fix: Use /clear, start new session. Don't debug context rot - restart.
Time to fix: 1 minute
#3: Missing CLAUDE.md
Symptom: Claude asks obvious questions, makes wrong assumptions
Impact: Same mistakes repeat indefinitely without institutional memory
Fix: Create CLAUDE.md with project context. Add every mistake as a rule.
Time to fix: 10-15 minutes initial setup
#4: No verification infrastructure
Symptom: Features "complete" that don't actually work
Impact: 2-3x worse output quality
Fix: Add test commands to every task. Use Chrome extension for UI. Set up verification hooks.
#5: Over-customization
Symptom: Hooks break, settings conflict, complex magic command systems
Boris says: "Having a long list of complex, custom slash commands is an anti-pattern. The entire point of an agent like Claude is that you can type almost whatever you want."
Fix: Start vanilla, add only when clearly needed.
#6: Sequential instead of parallel sessions
Wrong: Run one Claude, wait for it, start another
Right: Run 5 Claudes, context-switch between them as they work
Why: Each session is a separate worker. Parallelization beats optimization.
#7: Optimizing for token speed over quality
Symptom: Using Sonnet to "save time" but spending more time on corrections
Reality: Total Time = Generation + Correction + Re-prompting. Opus has higher generation but dramatically lower correction time.
Fix: Default to Opus 4.5 with extended thinking. Measure total completion time, not generation speed.
#8: Not reading Claude's plan
Symptom: Accept bad approach, then spend time undoing it
Fix: Review proposed approach before saying "proceed"
Time to fix: 2 minutes of reading
#9: Vague task descriptions
Symptom: Claude guesses wrong, implements the wrong thing
Bad: "Add auth"
Good: "Add basic email/password authentication using Passport.js with bcrypt for password hashing"
Fix: Be specific about libraries, patterns, and constraints
#10: Using /permissions skip (YOLO mode)
Anti-pattern: Dangerous blanket permission grants
Pattern: Use /permissions add Bash:npm* instead
Why: Permissions are team assets - shared, reviewable, and versioned. Pre-allowing safe actions makes the right thing the default.

7 What's Hard: Fundamental Tensions

These are genuine dilemmas, not bugs to fix:

Context limits vs task complexity
The tension: Some tasks genuinely need more context than fits in a window. But extending sessions leads to context rot.
Symptoms: Complex features can't complete in one session. Starting fresh loses important context.
Mitigation: Break into smaller tasks. Use filesystem as memory. CLAUDE.md carries forward essentials. Accept that /compact preserves essence but loses detail.
Verification vs speed
The tension: Testing takes time. Skipping feels faster. But unverified code compounds errors.
Symptoms: Pressure to ship without tests. "It works on my machine" syndrome.
Mitigation: Invest in fast test suites. Verification pays off 2-3x. The time you "save" skipping tests costs more in debugging.
Trust vs oversight
The tension: You can't review every line Claude writes. But blind trust leads to subtle bugs.
Symptoms: Reviewing everything is slow. Not reviewing leads to problems.
Mitigation: Focus on tests and behavior, not line-by-line review. Git provides safety net. Tests catch problems. Trust the verification loop, not the code.
Vanilla vs customization
The tension: Boris says vanilla is enough. But sometimes you genuinely need configuration.
Symptoms: Feeling like you're "doing it wrong" when adding configuration.
Mitigation: Start vanilla, customize only when clearly needed. Don't add complexity preemptively. If you need hooks, add ONE hook. Test if it helps.
Direct edit vs safety
The tension: Letting AI edit files directly feels risky. But manual copying is slow and error-prone.
Symptoms: Fear of letting Claude write directly. Wasted time on manual copy-paste.
Mitigation: Git provides safety net (git reset --hard origin/main). Tests catch problems. The fear is worse than the reality. Verify with tests, not manual review.
Building for today vs building for the future model
The tension: Current models have limitations. Workarounds help now but become technical debt.
Boris's approach: "Don't build for the model of today, build for the model six months from now." The team deletes code with each model release - removing half the system prompt when upgrading.
Mitigation: Don't over-engineer workarounds. Accept that some limitations will go away. Extreme scaffolding frameworks will hurt more than help.

8 When to Use / When Not

USE VANILLA WORKFLOW WHEN
Starting with Claude Code (universal starting point)
Daily development work - features, fixes, refactoring
Solo or small team (minimal coordination overhead)
Any project size - scales from script to app
You have tests and typecheck as verification layer
Tasks can complete in one focused session
DON'T USE VANILLA WHEN
Need parallel development on same feature - use Ralph + worktrees
Complex multi-step features - use Ralph Wiggum loop
Overnight autonomous work - use Ralph with external state
Team of AI agents - use CC Mirror, Gas Town
Specialized repeated workflows - use Skills/Hooks
Production-critical deployments - add additional guardrails
