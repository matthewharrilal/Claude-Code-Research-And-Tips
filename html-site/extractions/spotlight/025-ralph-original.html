2 The Core Abstraction

"Each time Ralph does something bad, Ralph gets tuned - like a guitar." - Geoffrey Huntley
while :; do cat PROMPT.md | claude-code ; done
That single line IS the pattern. Everything else is scaffolding.

The fundamental insight: Each iteration spawns a NEW Claude instance with NO memory of previous work. The agent is stateless. Memory is architecture, not magic.

This is counterintuitive. Surely accumulated context is valuable? No. After ~80K tokens, LLM quality degrades. "Context rot" means the model forgets early instructions, contradicts itself, loses track of goals. Fresh context beats accumulated context.

"It's a 'we learn from failure'-centric approach. You define your goal condition and let the agent loop over and over until it has verifiably reached that promised goal."
- @arvidkahl describing Huntley's concept

Origin Story

3 Design Decisions

WHY FRESH CONTEXT?
Long AI coding sessions hit fundamental problems: context windows fill up, quality drops as tokens accumulate, the model loses track of original goals, and errors compound without fresh perspective. Internal summarization is lossy - detail gets lost.
What this means for you
Kill and restart. Don't try to extend sessions. If iteration 20 produces worse code than iteration 5, your context is rotting. The fix is to restart, not debug.
WHY EXTERNAL STATE?
If each iteration is a fresh Claude with no memory, how does progress persist? Through files. Git commits, prd.json (task tracking), progress.txt (learnings). The filesystem IS the memory.
External State Architecture
Artifact Purpose Lifespan
prd.json Task tracking, completion status Per-feature
progress.txt Session learnings, patterns Per-feature
CLAUDE.md Permanent project knowledge Project-wide
AGENTS.md Agent-specific documentation Project-wide
Git history Code changes, implicit context Forever
WHY "RALPH WIGGUM"?
The name comes from The Simpsons character Ralph Wiggum - the simple-minded child of Police Chief Wiggum. The metaphor is precise: Ralph Wiggum is persistent despite setbacks, simple in approach but effective through repetition, unaware of "failure" as a stopping condition, and eventually accomplishes goals through sheer persistence.
Philosophy mapping
Iteration over perfection - Don't aim for perfect on first try
Failures are data - "Deterministically bad" failures are predictable
Operator skill > model capability - Your prompts matter more than the model version
Persistence wins - Keep trying until success

Fresh Context vs Extended Sessions

Long Sessions Fresh Context (Ralph)
Context fills up Always starts at 0%
Quality degrades over time Peak quality each iteration
Errors compound Each iteration can self-correct
Hard to recover from mistakes Easy to resume from any point
Lossy internal compaction Full external history preserved
One chance to get it right Unlimited iterations to succeed

ralph.sh - 6 Implementation Variations

1. Basic Rate-Limited

#!/bin/bash
set -e
MAX_ITERATIONS=${1:-10}
RATE_LIMIT_SLEEP=5

for (( i=1; i<=$MAX_ITERATIONS; i++ )); do
  echo "=== Iteration $i / $MAX_ITERATIONS ==="
  cat PROMPT.md | claude --dangerously-skip-permissions 2>&1 | tee output.txt
  if grep -q "<promise>COMPLETE</promise>" output.txt; then
    echo "Done!"
    exit 0
  fi
  sleep $RATE_LIMIT_SLEEP
done
echo "Max iterations reached"
exit 1

2. With Error Handling

#!/bin/bash
MAX_ITERATIONS=${1:-10}
CURRENT_ITERATION=1

while [ $CURRENT_ITERATION -le $MAX_ITERATIONS ]; do
  echo "=== Iteration $CURRENT_ITERATION / $MAX_ITERATIONS ==="
  OUTPUT=$(cat PROMPT.md | claude 2>&1) || true
  echo "$OUTPUT" | tee output.txt

  # Check for rate limiting
  if echo "$OUTPUT" | grep -q "rate_limit"; then
    echo "Rate limited, waiting 60s..."
    sleep 60
    continue  # Don't increment, retry
  fi

  if grep -q "<promise>COMPLETE</promise>" output.txt; then
    echo "All stories complete!"
    exit 0
  fi
  CURRENT_ITERATION=$((CURRENT_ITERATION + 1))
  sleep 5
done

3. With Stuck Detection

#!/bin/bash
MAX_ITERATIONS=${1:-10}
STUCK_THRESHOLD=3
LAST_COMMIT=""
STUCK_COUNT=0
CURRENT_ITERATION=1

while [ $CURRENT_ITERATION -le $MAX_ITERATIONS ]; do
  cat PROMPT.md | claude 2>&1 | tee output.txt

  CURRENT_COMMIT=$(git rev-parse HEAD 2>/dev/null)
  if [ "$CURRENT_COMMIT" = "$LAST_COMMIT" ]; then
    STUCK_COUNT=$((STUCK_COUNT + 1))
    if [ $STUCK_COUNT -ge $STUCK_THRESHOLD ]; then
      echo "ERROR: Stuck for $STUCK_COUNT iterations"
      exit 1
    fi
  else
    STUCK_COUNT=0
    LAST_COMMIT=$CURRENT_COMMIT
  fi

  if grep -q "<promise>COMPLETE</promise>" output.txt; then
    exit 0
  fi
done

4. jq-Based Completion (Maurice Kleine Style)

#!/bin/bash
# No max iterations - runs until PRD is complete

while jq '.tasks[] | select(.passes == false)' prd.json | grep -q .; do
  echo "Incomplete tasks remaining..."
  cat PROMPT.md | claude
  sleep 5
done

echo "All tasks complete!"

prompt.md - Agent Instructions

# Ralph Agent Instructions

## Your Task

1. Read `scripts/ralph/prd.json`
2. Read `scripts/ralph/progress.txt` (check Codebase Patterns first)
3. Check you're on the correct branch
4. Pick highest priority story where `passes: false`
5. Implement that ONE story
6. Run typecheck and tests
7. Commit: `feat: [ID] - [Title]`
8. Update prd.json: `passes: true`
9. APPEND learnings to progress.txt

## Progress Format

APPEND to progress.txt:

---
## [Date] - [Story ID]
- What was implemented
- Files changed
- **Learnings:**
  - Patterns discovered
  - Gotchas encountered

## Codebase Patterns

Add reusable patterns to the TOP of progress.txt:

## Codebase Patterns
- Migrations: Use IF NOT EXISTS
- React: useRef<Timeout | null>(null)

## Stop Condition

If ALL stories pass, reply:
<promise>COMPLETE</promise>

Otherwise end normally.

5 The Path of a Task

Follow a task through the complete mechanism to see how the pieces connect:

1
Start with fresh context - New Claude instance spawns (zero memory of previous iterations)
2
Read task state from files - prd.json, progress.txt, git history
3
Pick highest priority incomplete task - First story where passes: false
4
Implement task - Write code, modify files
5
Run verification - npm run typecheck && npm test
6
If pass: Commit, update prd.json with passes: true
7
If fail: Log learnings, try again (or iteration ends)
8
APPEND to progress.txt - What was learned, patterns discovered
9
Check completion condition - All done? Output <promise>COMPLETE</promise>
10
Exit - Claude terminates, bash loop restarts with fresh instance

Key insight: Continuity comes entirely from Git history (code changes), prd.json (task status), progress.txt (learnings), and the file system itself. The agent has NO internal memory of previous iterations.

6 Gotchas: 10 Common Mistakes

Real problems you'll hit, with concrete solutions:

1. Tasks too large
Why it fails: Context overflow, broken code
Fix: 2-3 sentence rule for each task
2. No verification
Why it fails: Errors compound across iterations
Fix: Require typecheck + tests in acceptance criteria
3. Vague acceptance criteria
Why it fails: Agent interprets differently each iteration
Fix: Be explicit: "dropdown has 3 options"
4. Editing progress.txt (instead of appending)
Why it fails: Learnings lost
Fix: Use "APPEND" in CAPS in prompt.md
5. No max iterations
Why it fails: Runaway costs, infinite loops
Fix: Always set MAX_ITERATIONS
6. Dependencies in wrong order
Why it fails: Agent builds on non-existent schema
Fix: Order: Schema -> Backend -> Frontend
7. Skipping tests for UI
Why it fails: Visual bugs despite "passes"
Fix: Add browser verification / Playwright
8. Non-idempotent migrations
Why it fails: Database inconsistency
Fix: Use IF NOT EXISTS, explicit defaults
9. Interactive prompts in script
Why it fails: Script hangs
Fix: Pipe empty input or use --yes flags
10. No stuck detection
Why it fails: Silent failures, wasted iterations
Fix: Check git diff between iterations
"If you don't do this [verification], you're hamstringing future agent runs with bad code, and they'll need to bisect to find bugs. Super nasty."
- Matt Pocock

7 What's Hard: Fundamental Tensions

These are tensions, not bugs to fix:

Context has no memory
The tension: Agent forgets EVERYTHING between iterations
Coping strategy: External state architecture (prd.json, progress.txt, git)
Task sizing is art, not science
The tension: Too big = overflow, too small = overhead
Coping strategy: 2-3 sentence rule, iterate on sizing
Verification discipline is non-negotiable
The tension: Easy to skip, but compound failures are catastrophic
Coping strategy: Non-negotiable CI in acceptance criteria
Operator skill matters more than model capability
The tension: Your prompts determine success, not the model version
Coping strategy: "Tune like a guitar" - iterate on prompts
Failures are data, not waste
The tension: Feels inefficient to run failing iterations
Coping strategy: Embrace eventual consistency - each failure refines
Review burden scales with iterations
The tension: Many commits to check in the morning
Coping strategy: Trust CI, spot check critical paths

8 When to Use / When Not

USE RALPH WHEN
Feature with clear specs (verifiable completion criteria)
Migrations (Jest to Vitest) - mechanical, repetitive
Overnight development (can run unattended)
Greenfield projects (no legacy constraints)
Test suite generation (clear pass/fail)
Database schema changes (verifiable via migrations)
DON'T USE RALPH WHEN
"Make it pretty" (no verifiable completion)
Security-critical code (needs human review)
Ambiguous requirements (wastes tokens on exploration)
One-shot operations (loop overhead not justified)
Design decisions (Ralph implements, humans decide)
Legacy without tests (no verification possible)

  |         NO -> Add tests/verification first, then Ralph
  |
