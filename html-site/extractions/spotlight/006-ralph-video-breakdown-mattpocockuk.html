2 The Core Abstraction: Kanban for AI

Matt's key insight: Stop writing multi-phase plans. Use a Kanban board. The old way of AI development was sequential phases that were hard to modify. The Ralph PRD approach treats tasks as independent cards on a board. 

OLD WAY (Multi-Phase Plans):
Phase 1 -> Phase 2 -> Phase 3 -> Phase 4
Problems: Hard to add tasks mid-plan, human must figure out exact placement

RALPH WAY (Kanban Loop):
+--------------------------------------+
|  Pick task from board                |
|         |                            |
|         v                            |
|  Complete task                       |
|         |                            |
|         v                            |
|  Back to board                       |
|         |                            |
|         v                            |
|  All done? -> Exit                   |
|  Not done? -> Loop                   |
+--------------------------------------+
"Focus on WHAT it should do at the end, not HOW to get there."
prd.json: { task: { description, verification, passes: false } }
Each task is a user story with clear acceptance criteria. Agent picks tasks, completes them, marks passes: true.

The binding between philosophy and implementation: prd.json IS the Kanban board. Each task entry is a card. passes: false means "in backlog". passes: true means "done". The agent reads this file, picks a task, completes it, updates the file. Fresh context each iteration.

3 Design Decisions

WHY KANBAN (NOT MULTI-PHASE)?
Multi-phase plans are rigid. Adding a new task means figuring out where it fits in the sequence. With Kanban, you just add a card to the board. The agent picks the next available task each iteration.
What this means for you
You can add tasks mid-run. Discovery during implementation? Add a new entry to prd.json. The agent will pick it up next iteration. No restructuring required.
WHY JSON (NOT MARKDOWN)?
Per Matt: "Just copied Anthropic's approach." JSON is structured, making it easier for LLMs to parse and update. Markdown is ambiguous - where does one task end and another begin?
What this means for you
The agent can reliably update passes: true in JSON. In Markdown, the agent might misplace the completion marker, corrupt the file format, or miss the field entirely.
WHY APPEND-ONLY progress.txt?
If you tell the agent to "update" progress.txt, it will overwrite the entire file. Learnings from previous iterations are lost. "APPEND" is explicit: add to the end, don't replace.
What this means for you
Use "APPEND" in all caps in your prompt. The agent respects emphasis. progress.txt becomes an audit trail: iteration 1 learned X, iteration 5 learned Y. Knowledge compounds.
WHY ONE FEATURE PER ITERATION?
Context rot: LLMs get stupider with more tokens. If you try to do multiple features in one iteration, you'll fill the context window before completing any. Quality degrades.
What this means for you
Each iteration is a complete unit: read PRD, pick ONE task, complete it, update PRD, commit, exit. Fresh context for the next task. Never multi-task within an iteration.
WHY GIT COMMIT EACH LOOP?
Creates checkpoints. If iteration 15 breaks something, you can bisect back to find when. Each commit is a unit of working code (tests pass).
What this means for you
Morning inspection: git log --oneline shows what was done. If something is broken, git bisect finds the culprit. Safe recovery path.
WHY passes: BOOLEAN (NOT PERCENTAGE)?
"80% complete" is ambiguous. Does the agent keep working? Start a new task? With boolean, it's clear: done or not done. No "mostly done" state.
What this means for you
Write acceptance criteria that are verifiable. "Tests pass" not "tests mostly pass". "TypeScript compiles" not "fewer errors than before".
WHY REQUIRE VERIFICATION (TESTS)?
Per Anthropic best practices: "Claude's tendency to mark features complete without proper testing... did much better at verifying end-to-end once explicitly prompted to use browser automation tools."
What this means for you
Without verification, Ralph produces garbage. Invest in tests BEFORE using Ralph. Typecheck + unit tests + (optionally) Playwright for UI verification.

4 The File Structure

│

prd.json - The Task Board

{
  "project_name": "Video Editor Features",
  "description": "New features for AI Hero video app",
  "tasks": {
    "delete_video_shows_confirmation": {
      "description": "Delete video shows confirmation dialog before deleting",
      "verification": "pnpm test && pnpm typecheck",
      "passes": true
    },
    "beats_display": {
      "description": "Beats display as three orange ellipses dots below clip",
      "details": "Add beat to clip, verify 3 orange dots appear",
      "verification": "pnpm test && pnpm typecheck",
      "passes": false
    }
  }
}

Each task has: description (what to build), details (optional additional context), verification (how to check), and passes (boolean completion).

ralph.sh - The Loop Engine

#!/bin/bash
set -e

MAX_ITERATIONS=$1

for (( i=1; i<=$MAX_ITERATIONS; i++ )); do
  echo "Iteration $i / $MAX_ITERATIONS"

  claude "Review plans/prd.json and plans/progress.txt.
    Mark the right PRD items as passes: true.
    Append your progress to progress.txt.
    Make a git commit of that feature.
    Only work on a single feature.
    If PRD is complete, output PROMISE COMPLETE HERE." | tee output.txt

  if grep -q "PROMISE COMPLETE HERE" output.txt; then
    break
  fi
done

Key details: set -e makes errors throw. tee output.txt captures output for the grep check. PROMISE COMPLETE HERE is the stop signal - when all tasks pass, the loop exits.

progress.txt - Append-Only Memory

# Progress Log

Use this to leave a note for the next person working on your codebase.

---
Iteration 1 - delete_video_shows_confirmation
Learned: Use Dialog component from /ui/dialog

---

5 The Path of a Task: Beats Display Demo

Matt's live demo: implementing "beats display" - three orange dots below video clips. Follow the task through the system:

1
[T+0min] Iteration begins. ralph.sh calls claude with the prompt. Fresh context, zero memory of previous iterations.
2
[T+0.5min] Agent reads prd.json. Sees beats_display: passes = false. This is the task to pick.
3
[T+0.5min] Agent reads progress.txt. Previous iteration recorded: "For beats, use orange (#FF6B35)". Knowledge from prior work!
4
[T+1min] Agent creates BeatIndicator.tsx component. Uses #FF6B35 (learned from progress.txt). Positions below clip thumbnail.
5
[T+2min] Agent writes BeatIndicator.test.tsx. Tests: renders three dots, dots are orange color.
6
[T+2.5min] Verification: pnpm typecheck passes. pnpm test passes. Both green.
7
[T+3min] Agent updates prd.json: beats_display: passes = true. The task is complete.
8
[T+3min] Agent commits: git commit -m "feat: Add beat indicator with orange dots"
9
[T+3min] Agent APPENDs to progress.txt: learnings from this iteration.
10
[T+3.5min] Output captured. grep checks for "PROMISE COMPLETE HERE". Not found (other tasks remain). Loop continues.
11
[T+3.5min] Next iteration begins. Fresh Claude instance reads updated files. Checks prd.json: if all tasks pass, outputs "PROMISE COMPLETE HERE" and loop exits.

Result: Three orange dots appeared below clips. Feature complete. One iteration, ~3 minutes, verified by tests.

6 Gotchas

10 concrete problems you will hit, with symptoms and fixes:

#1: Context Rot
Symptom: Quality degrades over iterations within a single session
Cause: LLMs get stupider with more tokens (~80K threshold)
Fix: Keep tasks small enough to complete in 1-5 iterations. Fresh context beats extended sessions. Time to fix: Design time
#2: Tasks Too Large
Symptom: Agent fills context window before completing, produces broken code
Cause: Task doesn't fit in one context window
Fix: Break into smaller tasks, each <1 context window. Time to fix: 15 min refactor
#3: MCP Tools Expensive
Symptom: High token usage per iteration, costs adding up
Cause: MCP calls consume context budget
Fix: Minimize MCP calls, batch operations, keep tasks small. Time to fix: Ongoing optimization
#4: Parallel Agents Conflict
Symptom: Merge conflicts when running multiple Ralph loops
Cause: Two agents editing the same files
Fix: Run serial, not parallel. For parallel work, use CC Mirror with worktrees. Time to fix: Architecture decision
#5: No Verification Layer
Symptom: Agent marks done but feature is broken
Cause: No tests/typecheck in verification criteria
Fix: Add tests + typecheck to verification BEFORE using Ralph. Time to fix: 30 min setup
#6: Audio Quality (Video Source)
Symptom: Can't hear demo clearly in Matt's video
Cause: OBS settings issue
Fix: Read transcript or text guides instead. Time to fix: N/A
#7: LLM Style Quirks
Symptom: Uses w-/h- instead of size- in Tailwind, minor style differences
Cause: Model has preferences that differ from your codebase conventions
Fix: Accept minor differences, fix post-run, or add style guide to CLAUDE.md. Time to fix: 5 min
#8: progress.txt Overwritten
Symptom: History lost, learnings from previous iterations gone
Cause: Prompt says "update" instead of "APPEND"
Fix: Use "APPEND" in CAPS in your prompt. Models respect emphasis. Time to fix: Prompt fix
#9: Commit Without Tests Passing
Symptom: Broken code in git history
Cause: Agent committed before verification step
Fix: Add explicit test check before commit in prompt. Time to fix: Prompt fix
#10: Vague Acceptance Criteria
Symptom: Agent doesn't know when done, marks complete incorrectly
Cause: Acceptance criteria like "make it look good"
Fix: Make criteria explicit and testable: "pnpm test passes" not "tests work". Time to fix: 10 min per task

7 What's Hard

These are fundamental tensions, not bugs to fix. You manage them, you don't solve them:

Task Sizing: Too Big vs Too Small
The tension: Too big = context rot, agent loses track mid-task. Too small = overhead dominates, more iterations than necessary.
Why it's hard: There's no formula. "The right size" depends on codebase complexity, task type, and model capability. Experience develops intuition.
Coping strategy: Start conservative (small). Review progress.txt after runs to see what worked. Adjust based on evidence.
Verification Investment
The tension: Ralph requires tests before it works well. But you want to use Ralph to avoid writing code. Chicken and egg.
Why it's hard: Without tests, Ralph produces garbage. But writing tests takes time upfront.
Coping strategy: Invest upfront. Ralph amplifies existing quality. If your codebase has good tests, Ralph is a multiplier. If not, Ralph multiplies garbage.
Context Window Limits
The tension: LLMs have hard limits. You can't "fix" this. 200K tokens is the ceiling, quality degrades at 80K.
Why it's hard: This is transformer physics. No prompt engineering workaround. No "just add more context" solution.
Coping strategy: Design around the constraint, not against it. Ralph's fresh-context-per-iteration IS the design for this constraint.
Serial vs Parallel
The tension: Serial is slow but safe. Parallel is fast but causes conflicts.
Why it's hard: Ralph is inherently single-threaded. Multiple Ralph loops editing the same codebase = merge conflicts.
Coping strategy: Accept serial for Ralph. If you need parallel, graduate to CC Mirror with git worktrees for isolation.
AFK Trust
The tension: Walking away overnight requires confidence. But confidence comes from experience. How do you get experience without trying?
Why it's hard: First AFK run is a leap of faith. You don't know if your task sizing, verification, and prompt are right until you try.
Coping strategy: Build confidence with HOTL (Human On The Loop) first. Run iterations manually, review between each. Graduate to AFK once patterns are proven.
Prompt Engineering
The tension: Small changes in prompts have big effects. "APPEND" vs "append" matters. But you can't predict which changes matter.
Why it's hard: Model behavior is not deterministic. What works today may not work tomorrow.
Coping strategy: Iterate on prompt, treat as code. Version control it. When something breaks, diff against last working version.

8 When to Use / When Not

Two Modes: AFK vs HOTL

AFK Ralph (Away From Keyboard)
Full loop, runs overnight. Wake up to completed work. Requires bulletproof feedback loops and proven task sizing.
./plans/ralph.sh 50
HOTL Ralph (Human On The Loop)
Interactive, one iteration at a time. Human reviews between loops. Great for learning, steering, and uncertain tasks.
./plans/ralph-once.sh (review) repeat
USE RALPH PRD WHEN
Tasks can be expressed as user stories with passes: true/false
Each task completable in 1-5 iterations (fits context window)
Good existing tests OR plan to add them first
Hours available for AFK OR interactive time for HOTL
Can rollback via git if something goes wrong
Agent can navigate your codebase (familiar structure)
DON'T USE RALPH PRD WHEN
Need parallel execution - use CC Mirror instead
Factory-scale output - use Gas Town for agent coordination
Subjective tasks ("make it pretty") - use manual Claude
Security-critical code - needs human review gates
No test coverage - build tests first, Ralph later
Ambiguous requirements - clarify before automating

Graduation Path
