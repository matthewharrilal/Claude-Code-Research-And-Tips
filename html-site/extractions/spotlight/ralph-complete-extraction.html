2 The Core Abstraction

"The agent is stateless. Memory is architecture, not magic."
while :; do cat PROMPT.md | claude ; done
That's the whole pattern. Everything else is scaffolding.

The Fresh Context Principle

Each iteration spawns a NEW Claude instance with NO memory of previous work. This is the key insight. Why does it matter?

  • Context rot is real — LLMs degrade with accumulated tokens (after ~80K)
  • Quality drops — The more context, the worse the output
  • Lossy compaction — Internal summarization loses critical detail
  • Fresh perspective — Each iteration can see the problem anew

External State Architecture

Continuity comes entirely from files, not memory:

Artifact Purpose Lifespan
prd.jsonTask tracking, completion statusPer-feature
progress.txtSession learnings, patternsPer-feature
CLAUDE.mdPermanent project knowledgeProject-wide
Git historyCode changes, implicit contextForever

3 Design Decisions

WHY FRESH CONTEXT EACH ITERATION?
Context rot is real. At 80K tokens, Claude's quality degrades. Instructions get lost. The model contradicts itself. This is "context rot" — a fundamental limitation of transformer attention, not a bug.
What this means for you
Peak quality every iteration, no degradation. Kill and restart rather than extend. If iteration 20 produces worse code than iteration 5, your context is rotting — restart, don't debug.
WHY EXTERNAL FILES (NOT AGENT MEMORY)?
If each iteration is a fresh Claude with no memory, how does progress persist? Through files. Agent is stateless, sessions die, files persist. prd.json tracks tasks, progress.txt accumulates learnings, git stores code.
What this means for you
You control the memory. It survives restarts. Never rely on "Claude remembering" something — if it's not in a file, it doesn't exist.
WHY SMALL TASKS (2-3 SENTENCES)?
Each task must complete in ONE iteration (one context window). Context limits mean large tasks overflow. Verification difficulty increases with task size. Small tasks = verifiable units = reliable progress.
What this means for you
Tasks succeed, quality stays high. If you can't describe a task in 2-3 sentences, split it. "Build auth system" becomes: "Add users table", "Add auth middleware", "Add login form", "Add session handling".
WHY TYPECHECK + TESTS MANDATORY?
Errors compound, bad code generates more bad code. Without verification, false completion wastes iterations. Each iteration inherits broken code from previous, spiraling into disaster.
What this means for you
Clean slate each iteration. Quote from Matt Pocock: "If you don't do this, you're hamstringing future agent runs with bad code, and they'll need to bisect to find bugs. Super nasty."
WHY ARCHIVE COMPLETED PRDs? (Ryan Carson's innovation)
Learnings transfer to future projects. Each project makes you smarter. Patterns accumulate across runs. What worked once can work again.
What this means for you
"What's amazing is I can ask it to look at previous runs to learn what worked or didn't work. So every time I run Ralph, it gets better and better." — Ryan Carson

ralph.sh — The Loop Engine

#!/bin/bash
# Ralph Wiggum Loop - Ship while you sleep
# Usage: ./ralph.sh [max_iterations]

MAX_ITERATIONS=${1:-10}
ITERATION=0

echo "Starting Ralph loop (max $MAX_ITERATIONS iterations)"
echo "Ctrl+C to stop"

while [ $ITERATION -lt $MAX_ITERATIONS ]; do
  ITERATION=$((ITERATION + 1))
  echo ""
  echo "=== Iteration $ITERATION of $MAX_ITERATIONS ==="
  echo ""

  # Run Claude with the prompt
  if claude -p "$(cat PROMPT.md)" --output-format stream-json 2>&1 | grep -q "COMPLETE"; then
    echo ""
    echo "=== Ralph detected COMPLETE signal ==="
    echo "Finished in $ITERATION iterations"
    exit 0
  fi

  echo "Iteration $ITERATION complete, continuing..."
  sleep 2  # Brief pause between iterations
done

echo ""
echo "=== Max iterations ($MAX_ITERATIONS) reached ==="
echo "Review progress and restart if needed"

PROMPT.md — Agent Instructions

# Ralph Wiggum - Development Loop

## Context
Read CLAUDE.md for project context. Read progress.txt for learnings from prior iterations.

## Your Task
1. Read prd.json and find the next user story where passes: false
2. Implement ONLY that user story
3. Run ALL acceptance criteria (including typecheck/tests)
4. If ALL criteria pass: set passes: true in prd.json
5. APPEND learnings to progress.txt (patterns discovered, issues solved)
6. If any criteria fail: fix the issue, re-verify

## Completion Signal
When ALL user stories have passes: true, output exactly:
<promise>COMPLETE</promise>

## Rules
- NEVER modify stories that already have passes: true
- NEVER skip acceptance criteria
- ALWAYS verify your changes work before marking done
- ALWAYS append learnings (don't overwrite progress.txt)

5 The Path of a Task

Follow a feature ("Build user authentication") through the Ralph system:

1
[T+0min] Human: Create PRD. Decompose feature into 5-7 small user stories. Each story is 2-3 sentences max. Write clear acceptance criteria including npm run typecheck passes and npm run test passes.
2
[T+25min] Human: Initialize progress.txt with header. Create empty file with project name and date.
3
[T+30min] Human: Start Ralph loop. Command: ./ralph.sh 10 (max 10 iterations as safety limit). Can walk away now — "ship while you sleep".
4
[T+31min] Iteration 1: Agent spawns with fresh context. Reads PROMPT.md, prd.json (finds US-001), progress.txt (empty). Implements US-001: database migration. Runs typecheck → passes. Runs tests → passes. Updates prd.json: US-001.passes = true. Appends learnings. Session ends.
5
[T+38min] Iteration 2: NEW agent spawns (fresh context, no memory). Reads prd.json (US-001 done, finds US-002). Reads progress.txt (knows patterns from Iteration 1!). Implements US-002. Updates state. Session ends.
6
[T+45min] Iteration 3: Fresh context, reads US-003 (API routes). Tests FAIL: forgot await on bcrypt. Agent detects failure, adds await to verifyPassword(), re-runs tests → passes. Marks done. Appends learning: "bcrypt is async!" Session ends.
7
[T+55-90min] Iterations 4-7: Each iteration: fresh agent, reads state, implements story. Learnings compound in progress.txt. Each agent is smarter than the last (via progress.txt).
8
[T+92min] Iteration 8: Agent reads prd.json: all stories have passes: true. Agent outputs: <promise>COMPLETE</promise>. ralph.sh detects COMPLETE, exits loop.
9
[T+93min] Human: Review completed work. Git commit, push, PR. Archive prd.json and progress.txt to archives/run-2026-01-15-auth/. Extract reusable learnings. NEXT PROJECT: Start from accumulated knowledge.

6 Gotchas

14 concrete problems you'll hit, with specific fixes and time estimates:

1. Stories too large
Symptom: Context overflow, task incomplete, agent loses track
Fix: 2-3 sentence rule. Split stories. If you can't describe it briefly, it's too big.
Time to fix: Rewrite PRD (15 min)
2. Weak acceptance criteria
Symptom: Agent claims done but broken
Fix: Be specific: "npm run test passes", not "works correctly"
Time to fix: 5 min per story
3. Context rot mid-task
Symptom: Quality degrades during long story
Fix: Task too big. Split it further.
Time to fix: 10 min
4. PRD drift
Symptom: Agent interprets criteria differently than intended
Fix: Explicit examples in acceptance criteria
Time to fix: 10 min
5. False completion
Symptom: passes: true but code broken
Fix: Require programmatic verification (typecheck/tests)
Time to fix: Add tests (30+ min)
6. Stop condition failure
Symptom: Loop never terminates
Fix: Clear COMPLETE signal + max iterations
Time to fix: 5 min
7. Progress overwriting
Symptom: Learnings lost between iterations
Fix: ALWAYS use APPEND in CAPS, not "update" or "edit"
Time to fix: 2 min
8. Non-idempotent code
Symptom: Database errors on re-run
Fix: Use IF NOT EXISTS, idempotent patterns
Time to fix: 5 min
9. No browser verification
Symptom: UI claimed done but visually broken
Fix: Add Playwright to acceptance criteria
Time to fix: 30 min setup
10. No max iterations
Symptom: Runaway loop, huge costs
Fix: ALWAYS set MAX_ITERATIONS in script
Time to fix: 1 min
11. Agent modifies done stories
Symptom: Regression in completed work
Fix: Explicit rule: NEVER modify passes: true
Time to fix: 2 min
12. No git commits between iterations
Symptom: Lost work on crash
Fix: Add "git commit" to PROMPT.md completion steps
Time to fix: 2 min
13. progress.txt too long
Symptom: Context overflow from learnings
Fix: Periodically summarize, archive old entries (20KB threshold)
Time to fix: 10 min
14. Flaky tests
Symptom: Intermittent failures waste iterations
Fix: Fix flaky tests first, add retry logic
Time to fix: Varies

7 What's Hard

These are fundamental tensions, not bugs to fix. You manage them; you don't solve them.

Task sizing
Why it's hard: Hard to know correct size until you try. Too small = overhead dominates. Too large = context overflow.
Coping strategy: Start small, split if iteration takes too long. Review progress.txt after runs to calibrate.
Verification investment
Why it's hard: Setting up Playwright takes hours. Tests take time. But without verification, Ralph produces garbage.
Coping strategy: Start with typecheck/test, add browser later. Worth it for 3+ task features.
Trust the loop
Why it's hard: Feels wrong to walk away while code ships. "What if it breaks while I sleep?"
Coping strategy: Monitor first 2-3 iterations, then trust. Good tests. Review in morning. Accept some risk.
PRD quality
Why it's hard: Bad PRD = wasted iterations. But writing good PRDs takes time and skill.
Coping strategy: Spend 25-30 min planning. It pays off. Quality here = quality everywhere.
Compounding discipline
Why it's hard: Easy to skip archiving when rushing. But skipping means losing learnings.
Coping strategy: Script the archive step into workflow. Make it automatic.
Context limits
Why it's hard: Long progress.txt fills context. But you want to preserve all learnings.
Coping strategy: Periodically summarize, archive old learnings. Move permanent patterns to AGENTS.md.
External dependencies
Why it's hard: Network, APIs, databases can fail. Not Claude's fault, but breaks the loop.
Coping strategy: Mock what you can. Retry logic for transient failures. Stuck detection (same HEAD for 3 iterations = exit).

8 When to Use / When Not

USE RALPH WHEN
Feature is well-defined and decomposable into small stories
Automated tests exist or can be created
You're at Stage 3+ developer (basic Claude proficiency)
1+ hours available for setup (loop overhead justified)
You want to "ship while you sleep"
DON'T USE RALPH WHEN
Exploratory work with no clear completion criteria
Very simple task (single prompt is enough)
Need parallelism (use CC Mirror or Gas Town)
No tests/typecheck (set up first)
Debugging (need conversation, not automation)
Learning new codebase (explore before defining)
