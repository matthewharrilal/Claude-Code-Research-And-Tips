3 Design Decisions

WHY 8 ROLES (NOT 3 OR 20)?
Evolved through iteration. 8 covers the full lifecycle: coordination (Mayor), monitoring (Deacon), quality (Dogs), merge management (Refinery), supervision (Witness), implementation (Polecat), and personal work (Crew).
What this means for you
Enough specialization to parallelize effectively, few enough to coordinate without chaos. You can start with just Mayor (tutorial mode) and add roles as you scale.
WHY GIT AS DATABASE (BEADS)?
"Git as database provides durable, diffable, auditable storage with zero infrastructure." No Postgres to maintain. Everything survives crashes. Full audit trail of all agent actions.
What this means for you
Slower queries than a real database, but you'll never lose state. Roll back to any point in time. Multiple agents share the same store safely.
WHY TMUX (NOT DOCKER)?
tmux provides local development, visual observation, and scriptability. You can watch agents work in real-time, switch between windows, and intervene immediately.
What this means for you
Must learn tmux (Ctrl-b navigation). Higher learning curve, but direct access to agents. No container overhead. GTGUI provides an "Age of Empires" style GUI alternative.
WHY NAMED POLECATS (ALICE, BOB, CAROL)?
Human names create a mental model. You develop a "relationship" with each polecat. Alice becomes "the database expert" through repeated assignment, not by design but by emergent specialization.
What this means for you
Easier to remember which agent is doing what. "Alice is stuck" is clearer than "polecat-3 is stuck". The naming is psychological, not technical.
WHY GO FOR BEADS (~225K LINES)?
"Go is just... good. It's boring. You can always understand it!" Type safety, readable by AI and humans alike. Contrast with TypeScript where "1/3 to 1/2 of diffs are type wrestling."
What this means for you
You need Go 1.23+ installed. The codebase is large but readable. Steve vibe-coded it himself, never reading the code—proof that Go works well with AI.

Beads Data Plane

Beads solves the "50 First Dates" problem—agents wake up with no memory. Every session is fresh, so all state must be external. 

# Key Beads Commands
bd ready                         # List tasks with zero blockers (actionable NOW)
bd create "Title"                # Create new task
bd dep add X Y                   # X is blocked by Y
bd show <id>                     # Full task state and history
bd cook <formula>                # Cook formula into protomolecule
bd mol pour <formula>            # Instantiate molecule (workflow)

# Step 3: Verify installation

# Step 5: Add your first rig (project)
gt rig add myproject https://github.com/you/myproject

# Step 6: Create crew workspace
gt crew add yourname --rig myproject

[T+1min] Mayor receives, analyzes scope, routes to appropriate rig

4
[T+3min] Polecat alice assigned bd-a1b2 via gt polecat assign alice bd-a1b2. Alice starts in her worktree.

6
[T+20min] Witness monitors progress, detects bob stuck on an import error, nudges bob with context.
7
[T+30min] Dogs run quality check via gt dogs bark. Reviews alice's schema changes, approves.
8
[T+45min] All subtasks complete. Refinery merges via sequential queue, resolves minor conflicts.
9
[T+50min] Feature complete. Overseer notified via inbox: "User authentication shipped."

GUPP: The Universal Propulsion Principle

"If there is work on your hook, YOU MUST RUN IT."

Work "slung" to agent hooks (persistent git worktrees) survives crashes and restarts. Agent checks hook on startup. If work exists, execution is immediate.

The Gupp Nudge: Claude is "so miserably polite" that GUPP alone doesn't always work. Agents get a startup poke with gt nudge 30-60 seconds after starting.

6 Gotchas

Real problems you'll hit, with concrete numbers and fixes:

#1: Context Collapse
Symptom: Agents stop making sense, instructions ignored
Cause: Beads state corruption or stale context
Fix: gt beads sync --force — Time: 5 min
#2: Merge Wall
Symptom: Conflicting commits, agents overwriting each other
Cause: Multiple agents on same files without coordination
Fix: Use Refinery merge queue. Each agent uses own worktree. — Time: 10-30 min
#3: Runaway Crew
Symptom: Cost spike, dozens of agents spawned unexpectedly
Cause: Refinery misconfiguration or uncapped crew limits
Fix: gt crew purge --all + set crew limits in config — Time: 2 min
#4: Mayor Deadlock
Symptom: No work being assigned, system appears frozen
Cause: Mayor waiting for human decision in inbox
Fix: Check inbox with gt inbox, respond to pending items — Time: 2 min
#5: Murderous Deacon (v0.3.x)
Symptom: Workers dying randomly during patrol
Cause: Known v0.3.x bug: "Deacon spree-killing workers while on patrol"
Fix: gt deacon down until v0.4.0+ — Time: 1 min
#6: GUPP Not Triggering
Symptom: Agent starts but doesn't begin work
Cause: Claude is "miserably polite" — waits for explicit permission
Fix: gt nudge 30-60 seconds after agent start — Time: 1 min
#7: Cost Explosion
Symptom: $500+ day unexpectedly
Cause: Too many agents running, Opus where Haiku would suffice
Fix: gt pause --all, review crew count, use Haiku for Witness — Time: Immediate

7 What's Hard

These are fundamental tensions, not bugs to fix:

Cost vs. Throughput
The tension: More agents = more output = more cost. Always. No free lunch.
Symptoms: Guilt about spending $150/day. Temptation to cut corners on quality roles.
Mitigation: Budget caps via config. Use Haiku where possible (Witness). Accept that this is what "team-scale output from one person" costs. "If you care about costs, don't use this."
Chaos vs. Control
The tension: Agents make mistakes. You can't prevent all errors. Some bugs get fixed 2-3 times. Some fixes get lost.
Symptoms: Frustration when agents produce bad code. Desire to micromanage.
Mitigation: Focus on THROUGHPUT, not perfection. Dogs provide quality gates. Accept some rework as the cost of velocity. "You're not a programmer anymore. You're the head chef of a kitchen with AI robotic sous chefs."
Mental Model Shift
The tension: You're the PM now, not the programmer. This is a fundamental identity shift.
Symptoms: Instinct to "just fix it yourself" when agent struggles. Difficulty delegating.
Mitigation: Takes weeks to adjust. Trust the system. Your job is to define features, review inbox, and provide judgment—not to write code. Let go.
Prerequisite Cliff
The tension: Gas Town requires Stage 7+ experience. You can't shortcut this. Below Stage 7, you will fail.
Symptoms: Everything breaks and you don't know why. Agents seem "stupid." Costs spiral without results.
Mitigation: No mitigation—prerequisite is real. Use Ralph first. Build up to CC Mirror. Only attempt Gas Town after 100+ hours multi-agent experience.
Debugging Distributed Systems
The tension: 8 roles, multiple rigs, async messaging. When something breaks, tracing the cause is non-trivial.
Symptoms: "Where did this bug come from?" "Why did alice do that?"
Mitigation: Good logging (logs/ directory). Witness role provides observation. Beads provides audit trail. But you're debugging a distributed system now—accept the complexity.
