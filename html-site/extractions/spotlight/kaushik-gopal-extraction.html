2 The Core Abstraction

"Single source of truth for AI instructions."
See AGENTS.md for instructions.
That's all your CLAUDE.md needs. Point, don't duplicate.

AGENTS.md is a master instructions file in your repository root. Every AI tool's config file (CLAUDE.md, .cursorrules, .gemini/settings.json) simply points to it. Edit AGENTS.md once, and Claude Code, Cursor, Gemini CLI, and Firebender all follow the same rules.

This solves the multi-tool coordination problem. Without a single source, you're maintaining identical rules in 5 different files. When you update one, you forget another. Behavior diverges. The AI in Cursor does something different from Claude Code. Chaos.

The mechanism: Claude scans each Skill's YAML frontmatter at startup, building a lightweight index. Full content loads only when relevant. This preserves context window for actual work instead of wasting tokens on rarely-needed instructions.

3 Design Decisions

WHY A SINGLE AGENTS.MD?
"Updating the same rules for each quickly becomes untenable." When you use 5 AI tools, you have 5 config files. Each update requires 5 edits. You forget one. Tools diverge. Consistency breaks.
What this means for you
Create AGENTS.md in your repo root. Every tool-specific config file contains one line: "See AGENTS.md for instructions." One edit point. Zero divergence.
WHY POINTERS, NOT COPIES?
Duplication is the enemy of maintenance. When CLAUDE.md, .cursorrules, and .gemini/settings.json each have their own copy of rules, you're not just doing 5x work-you're creating drift risk. Copy-paste introduces subtle differences. Over time, tools behave differently.
What this means for you
Tool configs should contain REFERENCES, not content. "Follow AGENTS.md" rather than duplicated rules. Modern tools support this: they read the referenced file.
WHY NESTED AGENTS.MD FILES?
Parent AGENTS.md files get bloated with context-specific rules. Backend conventions don't belong in frontend context. Testing rules shouldn't pollute production code guidance.
What this means for you
Use nested AGENTS.md for subdirectories: backend/AGENTS.md, frontend/AGENTS.md, tests/AGENTS.md. Tools recognize folder-specific files. Right scope for right context.
WHY CONTINUOUS PRUNING?
Instructions go stale. The framework changed. The convention evolved. The AI already knows this now. Stale rules confuse the model-it follows outdated guidance or gets contradictory signals.
What this means for you
"You need to be pruning and tweaking these instructions to get the best results." Regular maintenance required. Remove outdated rules, delete redundancies, resolve conflicts. This isn't one-time setup-it's ongoing hygiene.

6 Gotchas

Real problems you'll hit, with concrete fixes:

Stale instructions confuse the model
Symptom: AI does outdated things, follows conventions you've abandoned
Cause: Instructions that were added months ago, never removed
Fix: Weekly pruning ritual. Spend 10 minutes reviewing AGENTS.md. Remove what's stale.
Redundant rules cause bloat
Symptom: AGENTS.md is 500 lines, mostly obvious stuff
Cause: "Use TypeScript" when the project is already TypeScript. AI already knows.
Fix: 15-minute audit. Remove obvious conventions. Keep only project-SPECIFIC rules.
Tool doesn't read the reference
Symptom: Tool ignores AGENTS.md, behaves generically
Cause: Older tool version, or tool doesn't support file references
Fix: Update tool OR inline the content for that specific tool. Test each tool's behavior.
Nested AGENTS.md not found
Symptom: Backend rules not applied in backend/ folder
Cause: Wrong path in reference, or tool doesn't support nested files
Fix: Use correct relative paths. Test nested file recognition with each tool.
No /init run
Symptom: AGENTS.md is generic, doesn't reflect your actual project
Cause: Wrote AGENTS.md from scratch without AI analysis
Fix: Run /init in Claude Code. Let AI analyze your codebase and generate project-specific instructions.

7 What's Hard

These are fundamental tensions, not bugs to fix:

Maintenance Burden
The tension: Instructions go stale, but pruning takes time you don't have.
Symptoms: AGENTS.md grows forever, AI gets confused by contradictory rules.
Mitigation: Schedule weekly 10-minute pruning. Make it a habit, like code review. Accept this is ongoing work, not one-time setup.
Tool Compatibility Varies
The tension: Not all tools read file references the same way. Some need inline content.
Symptoms: Pattern works for Claude Code but not Gemini CLI. One tool behind on features.
Mitigation: Test each tool's behavior explicitly. Fallback to inline for tools that don't support references. Accept some divergence may be necessary.
Discovering What to Document
The tension: You don't know your conventions until AI breaks them. Documentation follows mistakes.
Symptoms: AGENTS.md is empty because you don't know what to write. Or it's wrong because you guessed.
Mitigation: Run /init for AI-generated starting point. Add rules ONLY when AI makes mistakes. Don't pre-document-document reactively.
Balancing Detail vs. Bloat
The tension: Too little = generic output. Too much = context bloat and confusion.
Symptoms: Either AI ignores your project specifics, or it's drowning in instructions and slow.
Mitigation: Start minimal. Add ONLY on mistake. Use nested AGENTS.md for context-specific rules. If in doubt, leave it out.

8 When to Use / When Not

USE AGENTS.MD PATTERN WHEN
Using 2+ AI coding tools (Claude Code, Cursor, Gemini CLI, etc.)
Project duration is weeks+, setup investment pays off
You have medium+ convention complexity worth documenting
Willing to do weekly maintenance (pruning, updating)
Team needs shared AI conventions across members
DON'T USE AGENTS.MD WHEN
Single tool only - just use CLAUDE.md directly
Quick experiment - setup overhead exceeds benefit
No conventions yet - discover first, document later
Team won't maintain - unmaintained AGENTS.md is worse than none
Very simple project - overhead exceeds value
