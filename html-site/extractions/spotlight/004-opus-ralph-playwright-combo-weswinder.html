2 The Self-Correcting UI Loop

"Claude can write code but can't SEE what renders. Playwright gives Claude eyes."
Opus 4.5 + Ralph Loop + Playwright
Reasoning + Persistence + Visual Verification = Self-Correcting UI Development
The combination creates what feels like AGI for UI work: it doesn't give up, it sees what's wrong, it fixes itself.

Every time you build UI with Claude, you experience the gap: Claude writes code, says "done", you open the browser... and see a broken layout, missing elements, or completely wrong output. Claude marks features "complete" based on successful file writes and no TypeScript errors. But none of these verify visual correctness. 

# The transformation:

# BEFORE (Code-Only Loop)
Claude writes CSS → Claude says "done" → You check browser → Broken!

# AFTER (Browser-Verified Loop)
Claude writes CSS → Playwright screenshots → Claude analyzes screenshot
→ Claude sees it's broken → Iterates automatically → Repeat until correct

Opus 4.5 handles complex visual understanding and makes fewer mistakes per iteration. Ralph provides persistence through external state. Playwright provides actual visual feedback, not just code analysis.

3 Design Decisions

WHY OPUS 4.5 (NOT SONNET)?
Opus 4.5 "handles complex visual understanding, makes fewer mistakes" according to @weswinder. For UI work, you're asking Claude to reason about screenshots - layout, spacing, colors. Opus 4.5's multimodal reasoning is stronger, reducing iterations needed.
What this means for you
Fewer iterations needed per story, but higher per-iteration cost. For UI-heavy work, the reduced iteration count usually offsets the higher model cost.
WHY PLAYWRIGHT (NOT PUPPETEER)?
Playwright has official Anthropic MCP server integration. Cross-browser support. Modern async API. The MCP integration means Claude can call browser automation natively without shell hacks.
What this means for you
Use the MCP server for cleanest integration: @anthropic/mcp-server-playwright. If MCP isn't available, Playwright's CLI works too.
WHY BROWSER VERIFICATION IN SUBAGENT?
@TendiesOfWisdom's insight: "Browser output is expensive." Page HTML can be 50K+ tokens. Screenshot analysis adds 1K-3K tokens. If these bloat your main agent's context, you'll hit context overflow mid-feature. Subagent isolation protects the main context window.
What this means for you
Spawn a Task for browser verification. The subagent returns only PASS/FAIL + brief summary. Your main Ralph loop stays lean.
WHY VISUAL CRITERIA SEPARATE FROM ACCEPTANCE CRITERIA?
Tests pass != looks right. npm test verifies functional behavior. But layout issues, CSS bugs, wrong colors - tests don't catch these. Visual criteria are verified via screenshot analysis, not test runners.
What this means for you
In prd.json, have both acceptanceCriteria (tests) and visualCriteria (screenshot analysis) for UI stories.

ralph.sh - Loop Engine with Screenshot Support

#!/bin/bash
# scripts/ralph/ralph.sh - Opus 4.5 + Ralph + Playwright Loop
set -e

MAX_ITERATIONS=${1:-10}
SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
SCREENSHOTS_DIR="$SCRIPT_DIR/screenshots"

# Create screenshots directory
mkdir -p "$SCREENSHOTS_DIR"

echo "=========================================="
echo " OPUS 4.5 + RALPH + PLAYWRIGHT"
echo " Max iterations: $MAX_ITERATIONS"
echo "=========================================="

for i in $(seq 1 $MAX_ITERATIONS); do
  echo "=== Iteration $i of $MAX_ITERATIONS ==="

  # Clear previous screenshot
  rm -f "$SCREENSHOTS_DIR/current.png"

  OUTPUT=$(cat "$SCRIPT_DIR/prompt.md" \
    | claude --dangerously-skip-permissions 2>&1 \
    | tee /dev/stderr) || true

  if echo "$OUTPUT" | grep -q "<promise>COMPLETE</promise>"; then
    echo "ALL STORIES COMPLETE!"
    exit 0
  fi

  if echo "$OUTPUT" | grep -q "<promise>BLOCKED</promise>"; then
    echo "BLOCKED - Human input needed"
    exit 2
  fi

  sleep 2
done

echo "MAX ITERATIONS REACHED"
exit 1

prompt.md - With Browser Verification Protocol

# Opus 4.5 + Ralph + Playwright Agent

## Your Core Loop

1. Read `scripts/ralph/prd.json` for story list
2. Read `scripts/ralph/progress.txt` for learnings
3. Pick ONE story where `passes: false`
4. Implement the story
5. Verify with Playwright (spawn subagent)
6. If visual verification passes: mark `passes: true`, commit
7. If visual verification fails: note what's wrong, iterate
8. APPEND learnings to progress.txt

## Browser Verification via Subagent

For UI stories, spawn a verification subagent:

Task: Verify UI at [URL] against these criteria:
- [Visual criterion 1]
- [Visual criterion 2]

Return only:
1. PASS or FAIL
2. 2-3 sentence summary
3. If FAIL, the specific criterion that failed

Save screenshot to: scripts/ralph/screenshots/[story-id].png

## Stop Conditions

- All stories pass → `<promise>COMPLETE</promise>`
- Blocked → `<promise>BLOCKED</promise>`
- Otherwise: end normally, loop restarts

5 Path of a Task: Login Form

Follow a UI feature through the system to see how browser verification changes the flow:

1
ralph.sh starts iteration 1. Reads prompt.md (~500 tokens)
2
Agent reads prd.json: finds UI-001 with passes: false
3
Agent reads progress.txt: no prior learnings yet
4
Agent implements: creates LoginForm.tsx, adds route handler
5
Runs npm run typecheck - PASSES
6
Browser verification via subagent: Spawns Task to verify at localhost:3000/login
7
Subagent: Playwright navigates, waits for networkidle, takes screenshot
8
Screenshot analysis: Email input? YES. Password input? YES. Form centered? NO (left-aligned!)
9
FAIL - Agent iterates within same session, updates CSS for centering
10
New screenshot - Form centered? YES. All visual criteria pass.
11
PASS - Commits: feat: UI-001 - Add login form
12
Updates prd.json: UI-001 passes: true. APPENDs to progress.txt
13
Iteration ends. More stories remain? Loop continues to iteration 2.

Time estimate: ~1-2 minutes per iteration. For a 5-story feature: 15-30 iterations, 30-60 minutes total.

6 Gotchas

Real problems you'll hit with browser verification, with specific fixes:

#1: Browser not found
Symptom: "Executable doesn't exist at /path/to/chromium"
Fix: npx playwright install chromium
Time to fix: 2 min
#2: Context overflow with browser output
Symptom: Claude responses get cut off, quality degrades mid-task
Cause: Browser HTML in main context (can be 50K+ tokens)
Fix: Use subagents for browser work - they isolate context
Time to fix: 5 min refactor
#3: Dev server not running
Symptom: "net::ERR_CONNECTION_REFUSED"
Fix: Start npm run dev in separate terminal before running Ralph
Time to fix: 1 min
#4: Screenshot shows blank page
Symptom: Screenshot is white but page works manually
Cause: Screenshot taken before page fully loads
Fix: Add waitForLoadState('networkidle') or wait for specific selector
Time to fix: 5 min
#5: Tests pass but visual is wrong
Symptom: npm test passes, browser shows broken UI
Cause: Tests don't cover visual rendering
Fix: Add visualCriteria separate from acceptanceCriteria
Time to fix: 5 min
#6: Screenshot too large (token explosion)
Symptom: Token count explodes, slow analysis
Fix: Use fullPage: false or crop to specific region
Time to fix: 2 min
#7: MCP server not connected
Symptom: "no browser tool available"
Fix: Restart Claude Code after config changes to ~/.claude/settings.json
Time to fix: 2 min
#8: Ralph stuck on same story
Symptom: Same story attempted 5+ iterations
Fix: Split story into smaller pieces, or add hints in notes field
Time to fix: 10 min
#9: Race condition on screenshot
Symptom: Screenshot taken mid-render, partially loaded
Fix: Wait for specific selector visibility before screenshot
Time to fix: 3 min
#10: Port mismatch
Symptom: Screenshots show wrong page or 404
Fix: Verify devServerUrl in prd.json matches your actual port
Time to fix: 1 min

7 What's Hard

These are fundamental tensions, not bugs to fix:

Visual Verification is Expensive
The tension: Screenshots + multimodal analysis = high token count. Each visual verification costs 1000-3000 tokens in the subagent.
Mitigation: Subagent isolation (don't pollute main context). Use fullPage: false to minimize screenshot size. Focus on presence/absence, not pixel-perfect.
"Looks Right" is Subjective
The tension: Claude's visual judgment != your visual judgment. What Claude thinks is "centered" might not match your expectation.
Mitigation: Write explicit visual criteria. Not "looks good" but "form width between 300-500px, centered horizontally". The more specific, the better.
Browser State Complexity
The tension: Auth, cookies, JS state, animations - all affect what the screenshot shows. The page at T+0 != T+2 seconds.
Mitigation: Start from consistent state. Fresh browser context. Wait for networkidle. If auth needed, include login steps in test setup.
Screenshot Analysis Has Limits
The tension: Claude can't see subtle CSS issues, animations, or hover states from a static screenshot. Micro-animations, transitions, responsive behavior at different breakpoints.
Mitigation: Focus on what screenshots CAN verify: element presence, basic layout, text content. Accept that some verification requires human eyes.
Cost Scaling Non-Linear
The tension: More visual stories = more browser overhead. Not just linear cost increase - each screenshot adds context pressure.
Estimates: 5 iterations: $2-5. 10 iterations: $4-10. 50 iterations (overnight): $20-50.
Mitigation: Set budget caps. Prioritize which stories need visual verification. Not every story needs screenshots.

8 When to Use / When Not

USE THIS PATTERN WHEN
3+ UI stories with visual components in backlog
Daily budget of $10-50 (browser adds ~50% cost)
Stage 4+ developer (Ralph familiarity assumed)
Frontend-heavy project with visual requirements
Low visual test coverage (Playwright fills the gap)
Medium to high visual complexity
DON'T USE THIS PATTERN WHEN
Backend-only APIs - basic Ralph is sufficient
Single UI change - setup overhead not justified
Highly interactive UI (animations, complex state)
Below Stage 4 - learn basic Ralph first
Tight budget - browser verification is expensive
CI/CD environment - needs different architecture
