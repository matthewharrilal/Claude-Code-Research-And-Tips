3 Design Decisions

WHY prd.json (NOT MARKDOWN)?
Markdown is human-readable but machine-opaque. JSON is parseable, queryable, and versionable. You can jq for status, programmatically track completion, and integrate with other tools.
What this means for you
Every story has passes: true/false. Check completion with: cat prd.json | jq '.userStories[] | select(.passes == false) | .title'
WHY ONE STORY = ONE ITERATION?
Context windows are finite. If a story is too big, the LLM runs out of context before finishing and produces poor code. The constraint isn't arbitrary - it's physics.
What this means for you
The 2-3 Sentence Rule: If you can't describe a story in 2-3 sentences, split it. "Build auth" becomes: Add users table, Add auth middleware, Add login form, Add session handling.
WHY TYPECHECK IN EVERY STORY?
Errors compound across iterations. One broken commit at iteration 5 means iterations 6-20 build on a broken foundation. By the time you check, 15 commits are garbage.
What this means for you
Non-negotiable: Every story's acceptance criteria MUST include "typecheck passes". UI stories also need "Verify in browser using dev-browser skill".
WHY APPEND-ONLY progress.txt?
If each Claude instance is fresh (no memory), how do learnings persist? Through files. And if you overwrite, you lose history. Append-only preserves the full learning trajectory.
What this means for you
Use "APPEND" in CAPS in prompt.md. Models interpret "update" as "replace" by default. The capitalization matters.
WHY COMPLETION DETECTION VIA GREP?
Deterministic exit condition. When all stories pass, Claude outputs <promise>COMPLETE</promise>. Bash greps for it and exits cleanly.
What this means for you
The exact string matters. grep -q "<promise>COMPLETE</promise>" - if Claude paraphrases, the loop runs forever.

prompt.md Template

# Ralph Agent Instructions

## Your Mission
You are Ralph, an autonomous coding agent. Implement user stories
from prd.json one at a time until all are complete.

## Workflow (Follow Exactly)
1. Read `prd.json` - find highest priority story where `passes: false`
2. Read `progress.txt` - check "Codebase Patterns" section first
3. Verify correct git branch (create if needed)
4. Implement that ONE story
5. Run quality checks: typecheck, tests
6. For UI: Verify in browser using dev-browser skill
7. Commit: `feat: [US-XXX] - [Story title]`
8. Update prd.json: `passes: true`
9. APPEND learnings to progress.txt (never overwrite)
10. Update relevant AGENTS.md with reusable patterns

## Stop Condition
If ALL stories have `passes: true`, respond:
<promise>COMPLETE</promise>

Otherwise end normally.

## Critical Rules
- ONE story per iteration only
- NEVER skip quality checks
- ALWAYS verify UI in browser
- APPEND to progress.txt, never overwrite

5 Path of a Task

Follow a feature through the Production Ralph system - from PRD to shipped:

1
[T+0min] User creates tasks/prd-task-filtering.md with feature description
2
[T+5min] PRD skill converts to structured requirements
3
[T+10min] Ralph skill converts PRD to prd.json with 6 user stories
4
[T+12min] Human reviews and adjusts story sizes (2-3 sentence rule)
5
[T+15min] Ralph loop starts: ./scripts/ralph/ralph.sh 25
6
[Iteration 1] US-001 - Fresh Claude reads prd.json, implements schema, runs typecheck, commits, marks passes:true, appends to progress.txt
7
[Iteration 2-14] Each iteration reads accumulated learnings, implements faster using discovered patterns
8
[Iteration 14] All stories pass:true, outputs <promise>COMPLETE</promise>, loop exits
9
[T+3hrs] Human reviews: git log shows clean commits, manual testing finds 2-3 edge cases
10
[T+3.5hrs] Quick Amp fixes for edge cases, feature shipped, archive created
"By story 10, Ralph knew our patterns. Learnings compound."
- Ryan Carson on the 14-iteration feature run

6 Gotchas

Concrete problems you'll hit, with numbers and fixes:

#1: Story too large - fails 3+ times
Symptom: Same story attempted multiple iterations without progress
Root cause: Story exceeds one context window
Fix: Split into smaller stories (2-3 sentence rule). Time: 10 min
#2: progress.txt overwritten
Symptom: Learnings from iteration 5 lost at iteration 6
Root cause: prompt.md says "update" not "APPEND"
Fix: Use "APPEND (never overwrite)" in CAPS. Time: 2 min
#3: UI claimed complete but visually broken
Symptom: Tests pass, but the actual UI is wrong
Root cause: No browser verification in acceptance criteria
Fix: Add "Verify in browser using dev-browser skill" to UI stories. Time: 5 min
#4: Completion not detected - runs forever
Symptom: All stories show passes:true but loop continues
Root cause: Claude paraphrased the completion signal
Fix: Check grep matches exact: <promise>COMPLETE</promise>. Time: 5 min
#5: Missing dependency - story fails with "not found"
Symptom: Story US-003 fails because US-002's output doesn't exist
Root cause: Priority ordering wrong - dependency runs after dependent
Fix: Reorder priorities: schema first, backend second, UI third. Time: 5 min
#6: jq not found - script fails immediately
Symptom: ralph.sh errors on first line reading prd.json
Root cause: jq not installed
Fix: brew install jq (macOS). Time: 2 min

7 What's Hard

These are fundamental tensions, not bugs to fix:

Story sizing is art, not science
The tension: No formula exists for "right size". It depends on codebase complexity, existing patterns, and domain.
Why it's hard: Too small = overhead dominates. Too large = context overflow. Only experience calibrates.
Coping strategy: Start with the 2-3 sentence rule. Review progress.txt after runs to see what worked. Expect to get better over time.
Spec quality determines success
The tension: "Garbage in, garbage out". Vague specs yield vague code. But good specs take time.
Why it's hard: Writing good specs IS the work. You can't skip it. Production Ralph just makes this visible.
Coping strategy: Use PRD skill for structure. Every criterion must be verifiable. "Works correctly" is useless - "Filter dropdown shows All, Cold, Friend" is testable.
No human review during AFK runs
The tension: The whole point is overnight execution. But errors can compound before you notice.
Why it's hard: Verification catches most issues, but not all. Some require human judgment.
Coping strategy: Start with smaller max_iterations. Review frequently at first. Increase confidence over time. Accept that some manual testing is still required post-run.
Acceptance criteria must be verifiable
The tension: Not everything is automatically verifiable. "Good UX" can't be grep'd.
Why it's hard: Production Ralph requires objective completion detection. Subjective goals don't fit the model.
Coping strategy: Translate subjective to objective where possible ("Lighthouse score 90" not "fast"). Use HOTL Ralph for tasks requiring judgment.

8 When to Use / When Not

USE PRODUCTION RALPH WHEN
5-20 user stories (too few = overkill, too many = Marathon Ralph)
Test/typecheck infrastructure exists (Production Ralph assumes verification works)
Feature is multi-file (single file changes = Basic Ralph)
Want learnings to compound across runs (archive system enables this)
Overnight/AFK execution is the goal ("ship while you sleep")
DON'T USE PRODUCTION RALPH WHEN
Quick one-off task - Use Basic Ralph (less overhead)
No test infrastructure - Build tests first or use Marathon Ralph (auto-generates)
Need Linear integration - Use Marathon Ralph (designed for Linear)
Exploratory work - Use HOTL Ralph (need human judgment each iteration)
>50 stories - Graduate to Marathon Ralph or Gas Town orchestration
