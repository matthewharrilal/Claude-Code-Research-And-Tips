3 Design Decisions

WHY STRICT ROLE SEPARATION?
If workers could spawn sub-agents, you'd get infinite recursion and context explosion. The conductor/worker boundary is a circuit breaker. Workers do tasks; conductors coordinate. No exceptions.
What this means for you
Every worker preamble MUST include "Do NOT spawn sub-agents or call TaskCreate/TaskUpdate." Test this. If a worker spawns, your orchestration is broken.
WHY A 5,299 TOKEN SKILL?
The orchestration skill seems expensive, but it pays for itself. Comprehensive guidance means the conductor delegates efficiently, saving tokens downstream. The alternative: re-explain orchestration every message, or build your own.
What this means for you
Accept the upfront cost. If you're budget-sensitive, disable team mode under variant settings. But don't try to shrink the skill—it will reduce quality.
WHY MODEL SELECTION GUIDANCE?
Different tasks need different models. Using Opus for file fetching is expensive overkill. Using Haiku for architecture decisions produces garbage. The guide prevents misallocation.
What this means for you
Haiku: Errands, file fetching, grep, simple lookups. Spawn 5-10 in parallel.
Sonnet: Capable worker—implementation, patterns, tests.
Opus: Complex reasoning, architecture decisions, ambiguous tasks.
WHY DOMAIN REFERENCES?
Decomposition quality determines outcome quality. Loading a domain reference (software-development.md, testing.md) before decomposing gives the conductor specialized patterns for that domain.
What this means for you
Always load the relevant domain reference BEFORE asking the conductor to decompose. "Load software-development.md" then "Build a REST API with auth."

5 Path of a Task

Follow a complex task through the orchestration system:

1
T-0: You give the conductor a complex task: "Build a REST API with authentication"
2
Conductor loads orchestration skill (~5,299 tokens)
3
T+1min: Conductor loads relevant domain reference (software-development.md)
4
Conductor analyzes and decomposes into subtasks with dependencies:
- Task 1: Design database schema (Opus)
- Task 2: Create project structure (Haiku)
- Task 3: Implement user model (Sonnet, blocked by Task 1)
- Task 4: Create auth endpoints (Sonnet, blocked by Task 3)
- Task 5: Write integration tests (Sonnet, blocked by Task 4)
5
T+3min: Conductor issues TaskCreate for each subtask
6
Parallel workers start: Tasks 1 and 2 have no blockers
7
Workers complete and report via TaskUpdate
8
Blocked tasks unblock automatically as dependencies resolve
9
If worker fails: conductor retries or reassigns to different model
10
T+Done: All tasks resolved. Conductor integrates and verifies.

6 Gotchas

14 real problems with specific fixes:

#1: Windows Compatibility
Symptom: Install fails on Windows
Fix: Use WSL2. Community PR in progress for native Windows.
Time to fix: 15 minutes (WSL2 setup)
#2: Context Collapse During Orchestration
Symptom: Conductor loses track of task graph after many workers
Fix: Enable auto-compact beta feature. @nummanali: "Just make sure auto compact is on."
Time to fix: 2 minutes
#3: High Token Cost (5,299 tokens)
Symptom: First bill higher than expected
Fix: Accept upfront cost—it saves via delegation. Or disable team feature under variant settings.
Trade-off: Lower cost = less orchestration guidance
#4: Worker Spawns Sub-Agent (Iron Law Violation)
Symptom: Infinite recursion, context explosion, bill shock
Fix: Every worker preamble MUST say "Do NOT spawn sub-agents"
Time to fix: Immediate (fix preamble template)
#5: Wrong Model for Task
Symptom: Opus running file fetches (expensive) or Haiku doing architecture (garbage)
Fix: Follow model selection guide. Haiku for errands, Sonnet for code, Opus for reasoning.
#6: Poor Decomposition Quality
Symptom: Tasks too vague, workers fail
Fix: Load domain reference BEFORE decomposing. Not after.
Time to fix: 2 minutes
#7: GLM Team Mode Skip
Symptom: Quick install skips team mode for GLM provider
Fix: Use variant options or update configuration after installation
#8-14: More Gotchas
#8: Worker doesn't complete (check task complexity, split if needed)
#9: Dependency cycle (review task graph manually)
#10: Lost worker output (workers MUST report absolute paths)
#11: Too many parallel workers (limit to 5-7 concurrent)
#12: Stale skill version (reinstall skill file)
#13: Conductor writes code (re-read Iron Law)
#14: Node.js version (requires Node 18+)

7 What's Hard

Fundamental tensions in CC Mirror, not bugs to fix:

Token Cost vs. Capability
The tension: The 5,299 token skill is expensive but comprehensive. Shrinking it reduces orchestration quality. Expanding it consumes more context.
Coping strategy: Accept the cost. Measure savings from delegation. If cost is prohibitive, use Ralph loops instead (simpler, cheaper).
Simplicity vs. Control
The tension: CC Mirror's 3-step setup means less customization. Gas Town gives full control but takes hours+ to configure.
Coping strategy: Start with CC Mirror for 80% of use cases. Graduate to Gas Town only when you hit its limits. Don't pre-optimize.
Conductor Bottleneck
The tension: Single conductor orchestrating many workers. If conductor overloads, entire orchestration stalls.
Coping strategy: Limit to 5-7 concurrent workers. Use dependency graph to serialize where needed. Don't spawn 20 workers at once.
Hidden Feature Risk
The tension: CC Mirror enables a disabled feature. Anthropic could remove or change it.
Coping strategy: Follow @nummanali for updates. Have a fallback plan (Gas Town, Ralph loops).
Decomposition Quality
The tension: Bad decomposition = bad outcomes. The conductor's decomposition skill varies. No automated quality check.
Coping strategy: ALWAYS load domain reference before decomposing. Review first task graph before execution. Learn to recognize good decomposition.
Worker Isolation
The tension: Workers can't coordinate with each other. All coordination flows through conductor. This creates bottleneck but prevents chaos.
Coping strategy: Accept isolation as a feature, not a bug. It prevents runaway agents. Design task boundaries assuming no worker-to-worker communication.

Graduation Criteria to Gas Town

Move from CC Mirror to Gas Town when you need:

  • 8+ parallel workers (CC Mirror caps around 5-7)
  • Custom role definitions beyond worker/conductor
  • Beads git-backed versioning for worker outputs
  • Cross-project coordination (multiple repositories)
  • Production-grade observability and logging
  • Custom evaluation framework for output quality
"Simplicity covers 80% with zero effort—mass-market ease." — @nummanali
CC Mirror is for the 80%. Gas Town is for the 20% who need factory-scale orchestration. Don't over-engineer.
