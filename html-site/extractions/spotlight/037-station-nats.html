2 The Core Abstraction

"Traditional agent setups isolate AI agents by team. Synthesizing results across tools requires manual coordination. NATS changes that."
stn init --provider anthropic && stn up
Two commands to production-grade agent orchestration.

Station is a mesh network for AI agents. Think of it as Kubernetes service discovery, but for Claude instances. Each Station hosts specialized agents. The Lattice mesh connects them with NATS pub/sub messaging. The result: agents on Station A can invoke agents on Station B in sub-milliseconds.

NATS isn't just "another message queue." It's purpose-built for sub-millisecond latency at scale. Station leverages the full NATS stack:

3 Design Decisions

WHY NATS (NOT KAFKA, RABBITMQ)?
Sub-millisecond latency matters for agent coordination. Kafka is optimized for throughput (millions of messages/sec) but adds latency. RabbitMQ requires separate broker deployment. NATS provides embedded server option for simple deployments AND can scale to distributed clusters.
What this means for you
Start with stn serve --orchestration (embedded NATS). Scale to external NATS cluster when needed. No broker management for simple deployments.
WHY JETSTREAM FOR PERSISTENCE?
Every agent invocation, work assignment, and status change is an event. JetStream provides exactly-once semantics and append-only storage. DenyDelete: true and DenyPurge: true create immutable audit trails—critical for compliance and debugging.
What this means for you
Full audit trail without additional infrastructure. Replay any time window for debugging. Query events for compliance reporting. Default 7-day retention, 1GB storage.
WHY THREE OPERATING MODES?
Different scales need different setups. Standalone: development, no mesh. Orchestrator: runs embedded NATS, central hub. Member: joins existing mesh. This enables gradual adoption—start simple, scale when needed.
What this means for you
Start with standalone. When you need multi-machine coordination, add orchestrator. Scale to multi-member mesh for production. Each mode is a progression, not a replacement.
WHY PRESENCE HEARTBEATS?
In distributed systems, stations fail. Presence heartbeats (default: 10 seconds) + TTL (default: 30 seconds) mean dead stations are detected automatically. Work reroutes to healthy stations without manual intervention.
What this means for you
No manual health checks. No orchestrator monitoring. Stations self-report liveness. Missed heartbeats trigger cleanup. Network partitions auto-heal on reconnection.
WHY GO (NOT PYTHON, RUST)?
NATS has first-class Go support (nats.go). Go's concurrency model matches the pub/sub paradigm. Performance is excellent. Readable codebase means easy contributions. Single-binary deployment simplifies operations.
What this means for you
One binary, no runtime dependencies. Easy to understand and extend. Cross-platform builds. No Python environment management for the Station itself.

5 The Path of a Task

Follow a security scan request through the Station mesh:

1
Orchestrator receives request: call_agent("security-scanner", "Scan /app for vulnerabilities")
T+0ms
2
AgentRouter queries Registry via NATS KeyValue: GET from lattice-agents
T+1ms - Finds security-scanner on Station-B
3
Invoker sends to Station-B: lattice.station.{stationB-id}.agent.invoke
T+2ms - Timeout: 5 minutes
4
Station-B receives, locates local agent, begins execution
T+5ms
5
Agent executes with configured tools (filesystem, web)
T+10ms to T+45s - Depends on task complexity
6
Agent completes, returns result via NATS reply
T+45s - Includes DurationMs, ToolCallCount
7
Event published to JetStream: lattice.events.AgentInvoked
T+45s - Stored in append-only stream for audit
8
Result returned to orchestrator, trace exported to Jaeger
T+45s - Full distributed trace available

6 Gotchas

Real problems you'll hit, with specific fixes:

1. Agent name collision across stations
Symptom: "Agent already exists" error, routing confusion
Cause: Same agent name registered on multiple stations
Fix: Use unique names per station, or prefix with station name: security-station-vuln-scanner
Time to fix: 2 minutes
2. Presence timeout - Station shows offline
Symptom: Station appears offline in mesh despite running
Cause: Heartbeat missed, TTL expired (default 30s)
Fix: Check network, increase presence_ttl_sec if latency is high
Time to fix: 5 minutes
3. JetStream not enabled
Symptom: KeyValue operations fail, "stream not found"
Cause: JetStream not configured on NATS server
Fix: Use stn serve --orchestration (auto-enables JetStream) or add JetStream config to external NATS
Time to fix: 10 minutes
4. Port conflict on startup
Symptom: "Address already in use" on 8585, 4222, etc.
Cause: Previous Station instance or conflicting service
Fix: lsof -i :8585 to find, then kill or change ports in config
Time to fix: 2 minutes
5. Authentication mismatch
Symptom: "Authorization failed" when member joins mesh
Cause: Token/credentials don't match between orchestrator and member
Fix: Verify auth config matches exactly. Check credential file permissions: chmod 600
Time to fix: 5 minutes
6. TLS certificate issues
Symptom: Connection refused despite correct URL
Cause: Cert path wrong, CA chain incomplete, or cert expired
Fix: Verify paths, check openssl x509 -in cert.pem -noout -dates
Time to fix: 15-30 minutes
7. Agent invocation timeout
Symptom: Agent call hangs, times out after 5 minutes
Cause: Agent task takes longer than default timeout
Fix: Increase timeout_sec in config, or break task into smaller pieces
Time to fix: 2 minutes
8. Event stream full
Symptom: Events not publishing, disk usage high
Cause: MaxBytes (default 1GB) or MaxAge (default 7 days) reached
Fix: Increase limits or enable cleanup policy in stream config
Time to fix: 5 minutes
9. Missing API key
Symptom: "Provider not configured" on agent execution
Cause: ANTHROPIC_API_KEY (or equivalent) not set
Fix: export ANTHROPIC_API_KEY="sk-ant-..." before stn up
Time to fix: 1 minute
10. Jaeger traces missing
Symptom: No traces in Jaeger UI at :16686
Cause: Jaeger not running, or OTEL endpoint not set
Fix: Run stn jaeger up BEFORE stn up, ensure OTEL_EXPORTER_OTLP_ENDPOINT set
Time to fix: 2 minutes

7 What's Hard

These are fundamental tensions, not bugs to fix:

Distributed Complexity
The tension: Multiple stations = multiple failure modes. Network partitions, clock skew, split brain scenarios. Single-station is simpler.
Why it's hard: Distributed systems are fundamentally harder than single-node. No framework eliminates this complexity.
Coping strategy: Start with standalone mode. Add mesh incrementally. Use Jaeger traces extensively. Design for eventual consistency.
NATS Learning Curve
The tension: NATS is a different paradigm from HTTP/REST. Pub/sub, subjects, streams, consumers—new mental models required.
Why it's hard: Most developers are HTTP-native. NATS concepts don't map directly to REST patterns.
Coping strategy: Study NATS basics before Station. Use nats CLI for debugging. Understand subjects as hierarchical namespaces.
Debugging Across Stations
The tension: When a request fails, which station is the problem? Traces span multiple processes, logs are distributed.
Why it's hard: Correlation across boundaries requires infrastructure. Human intuition fails at scale.
Coping strategy: Enable Jaeger from day one. Use trace IDs in all logging. Check NATS monitoring at :8222 for message flow issues.
Agent Manifest Consistency
The tension: When you update an agent definition, the change must propagate across the mesh. Registry updates aren't instant.
Why it's hard: Distributed state is eventually consistent. No global lock exists.
Coping strategy: Restart stations after agent changes. Use presence announcements to force refresh. Accept brief inconsistency windows.
Cost at Scale
The tension: Each station runs full Claude instances. More stations = more API costs. Mesh overhead isn't free.
Why it's hard: LLM costs scale linearly with usage. No caching reduces unique queries.
Coping strategy: Budget per-station. Monitor token usage. Use cheaper models for simple agents. Consider agent consolidation.

DON'T USE STATION WHEN

No audit requirements - JetStream overhead unnecessary

Invariant Connection
Shared with Ralph: External State
Station stores state in JetStream KeyValue just like Ralph uses progress.txt. Same "external state > internal memory" principle, same crash resilience.
