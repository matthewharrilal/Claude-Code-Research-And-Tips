Essence (15 words)
Simple-minded persistence wins: loop through failures until verifiable completion, fresh context every time.
"It's a 'we learn from failure'-centric approach. You define your goal condition and let the agent loop over and over until it has verifiably reached that promised goal."
- @arvidkahl

Named after Ralph Wiggum from The Simpsons - the simple-minded child who persists in tasks regardless of obstacles. The pattern embodies:

  • Persistence over perfection - Keep trying until it works
  • Failures are data, not stops - Each failure informs the next attempt
  • Eventually consistent completion - It might take 2 minutes or a day, but it gets there

2 Core Abstraction + Implementation

"Context degrades with tokens. Files persist. Fresh context beats extended sessions."
while :; do cat PROMPT.md | claude ; done
That's the whole pattern. Everything else is scaffolding.

The Mental Model

+---------------------------------------------+
|  FRESH CONTEXT (no rot)                     |
|            |                                |
|  EXTERNAL MEMORY (git, prd.json, progress)  |
|            |                                |
|  PICK NEXT TASK (highest priority undone)   |
|            |                                |
|  IMPLEMENT + VERIFY (tests must pass)       |
|            |                                |
|  COMMIT + LOG (git commit, progress.txt)    |
|            |                                |
|  CHECK COMPLETION (all done? -> EXIT)       |
|            |                                |
|  LOOP (repeat with fresh context)           |
+---------------------------------------------+

Each iteration spawns a NEW Claude instance with NO memory of previous work. Continuity comes entirely from:

  • Git history - Code changes persist
  • prd.json - Task status (passes: true/false)
  • progress.txt - Learnings, patterns, gotchas
  • File system - The codebase itself

#!/bin/bash
set -e

MAX_ITERATIONS=$1

for (( i=1; i<=$MAX_ITERATIONS; i++ )); do

  claude "Review plans/prd.json and plans/progress.txt.

    Append your progress to progress.txt.
    Make a git commit of that feature.

  fi
done

3 Design Decisions

WHY FRESH CONTEXT?
At 80K tokens, Claude's quality degrades. Instructions get lost. The model contradicts itself. This is "context rot" - a fundamental limitation of transformer attention. Matt Pocock: "LLMs get stupider with more tokens."
What this means for you
Kill and restart. Don't try to extend sessions. If iteration 20 produces worse code than iteration 5, your context is rotting. The fix is to restart, not debug.
WHY EXTERNAL STATE?
If each iteration is a fresh Claude with no memory, how does progress persist? Through files. Git commits, prd.json (task tracking), progress.txt (learnings). The filesystem IS the memory.
What this means for you
Never rely on "Claude remembering" something. If it's not in a file, it doesn't exist. progress.txt must be append-only. prd.json must be updated after each task. Every learning must be persisted.
WHY SMALL TASKS?
Each task must complete in ONE iteration (one context window). If a task is too big, Claude loses track mid-task and produces broken code. Ryan Carson's rule: "If you can't describe it in 2-3 sentences, it's too big."
What this means for you
"Build auth system" is wrong. Split it: "Add users table", "Add auth middleware", "Add login form", "Add session handling". Each is one iteration.

Decision Tree

START: "I want to automate development"
|
+-- Can I define verifiable completion criteria?
|   +-- NO -> Don't use Ralph (subjective tasks)
|   +-- YES -> Continue
|
+-- Is sequential (one task at a time) acceptable?

|   +-- YES -> Continue

ralph.sh - The Loop Engine

#!/bin/bash
set -e

MAX_ITERATIONS=${1:-10}
SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"

echo "Starting Ralph"

for i in $(seq 1 $MAX_ITERATIONS); do
  echo "=== Iteration $i ==="

  OUTPUT=$(cat "$SCRIPT_DIR/prompt.md" \
    | claude --dangerously-skip-permissions 2>&1 \
    | tee /dev/stderr) || true

  if echo "$OUTPUT" | grep -q "<promise>COMPLETE</promise>"; then
    echo "Done!"
    exit 0
  fi

  sleep 2
done

echo "Max iterations reached"
exit 1

5 Path of a Task

Follow a task through the system to see how the pieces connect:

1
PREPARATION: Write PRD with small, verifiable tasks (2-3 sentences per task)
2
Create progress.txt (empty or with header)
3
Create git branch for feature: ralph/feature-name
4
Verify tests/typecheck work before starting
5
START: Run ./ralph.sh 25 (or /ralph-loop if plugin)
6
Walk away (overnight feature mode)
7
EACH ITERATION: Fresh Claude reads prd.json, finds first passes: false
8
Claude reads progress.txt for previous learnings
9
Claude implements, runs npm run typecheck && npm test
10
If pass: marks passes: true, commits, appends to progress.txt
11
If fail: logs learnings, iteration ends (next iteration retries)
12
COMPLETION: All tasks pass, outputs <promise>COMPLETE</promise>
13
REVIEW: Check git log, review all changes, run full test suite, merge or request review

6 Gotchas

18 real problems you'll hit, with concrete symptoms and fixes:

Stories Too Large (Context Overflow)
Symptom: Agent loses track, produces broken code, loops on same task
Fix: Apply 2-3 sentence rule. Each task must fit one context window. (15 min to fix)
Vague Completion Criteria
Symptom: Agent marks done when not done. "Works correctly" is not verifiable.
Fix: Require explicit criteria: "npm run typecheck passes", specific UI states. (5 min to fix)
No Browser Verification for UI
Symptom: UI changes claimed complete but visually broken
Fix: Add Playwright MCP. "Verify at localhost:3000 using dev-browser skill." (30 min to add)
Wrong Task Ordering (Dependencies)
Symptom: Task fails because prerequisite doesn't exist yet
Fix: Order: Schema -> Backend -> Frontend. Dependencies first. (15 min to reorder)
Non-Idempotent Migrations
Symptom: Database in inconsistent state after failures
Fix: Test migrations, add rollback capability, manual SQL verification. (30 min)
Infinite Loop (No EXIT)
Symptom: $200+ bill, loop never terminates
Fix: Add completion signal. Set --max-iterations. Always. (2 min to add)
progress.txt Grows Unbounded
Symptom: After ~50 iterations, Claude spends more tokens reading than thinking
Threshold: >20KB starts degrading quality
Fix: Truncate to last 10 entries. Archive old learnings. (10 min)
Stuck on Same Task (No Progress)
Symptom: Same git HEAD for 3+ iterations
Fix: Add stuck detection. Split task or add hints to prd.json notes. (15 min)

7 What's Hard

These are fundamental tensions, not bugs to fix:

Task Sizing is a Skill, Not a Formula
The tension: Too small = overhead, too big = rot. No formula exists.
Why it's hard: Each codebase has different complexity. What fits in one context varies.
Mitigation: Practice the 2-3 sentence rule. Review progress.txt after runs to see what worked. Expect to get better over time.
Verification Design is Hard
The tension: Ralph only works if acceptance criteria are verifiable. But not everything can be verified automatically.
Why it's hard: "Correct" is often subjective. UI design, UX quality, security implications.
Mitigation: Write tests first (TDD). Use HOTL Ralph for judgment-requiring tasks. Accept that some domains aren't Ralph-suitable.
Trusting Automation
The tension: Fear of unchecked changes vs. desire for overnight productivity.
Why it's hard: Loops without review work until they build something broken in ways you don't notice.
Mitigation: Use HOTL Ralph for critical code. Strong test coverage. Review git diff every morning.
Cost Unpredictability
The tension: Token usage varies by complexity. Hard to budget.
Why it's hard: Same task might take 3 iterations or 15 depending on edge cases.
Mitigation: Set --max-iterations. Track costs. Budget $10-100 per overnight run.
Debugging Without Context
The tension: Fresh context means you can't ask "why did you do that?" - that Claude is gone.
Why it's hard: progress.txt captures learnings but not reasoning traces.
Mitigation: Better progress.txt logging. Include decision rationale. Use git commit messages as breadcrumbs.

8 When to Use / When Not

GOOD CANDIDATES
Overnight features with clear specs
Migration tasks (repetitive, well-defined)
Greenfield with specs (you can walk away)
Mechanical refactoring with tests

Tasks with automatic verification (tests, linters)
POOR CANDIDATES
Subjective tasks ("make it pretty")
Security-critical code (needs human judgment)
Ambiguous requirements
One-shot operations (no need for loops)
Exploratory work without defined end
Production debugging (needs intuition)
