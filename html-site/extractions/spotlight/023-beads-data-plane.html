2 The Core Abstraction

"Agents forget between sessions. Files persist. Git merges."
bd create "Implement login" -p 0 && bd sync
That's the core pattern: structured issues in git, synced between agents.

Beads is the memory layer for AI agents. When Agent A creates an issue and syncs, Agent B can pick up that work seamlessly. No shared database server, no infrastructure - just git.

The Three-Layer Model

Layer 1: SQLite Database
Fast queries, complex joins, dependency graphs. File: .beads/beads.db (gitignored). Latency: milliseconds. This is the performance layer.
Layer 2: JSONL Files
Git-tracked source of truth. File: .beads/issues.jsonl. One JSON object per line. Human-readable diffs, merge-friendly format. This is the durability layer.
Layer 3: Git Remote
Multi-agent/multi-machine collaboration. Standard git operations: push, pull, merge. This is the collaboration layer.

Why three layers? JSONL is git-mergeable but slow to query. SQLite is fast but not mergeable. The three-layer model gives you both: SQLite for speed, JSONL for git, remote for collaboration.

Three-Way Merge Strategy

When multiple agents edit the same issues, Beads uses three-way merge: 

# Three-way merge components:
BASE:   sync_base.jsonl   # Last sync point
LOCAL:  Current SQLite    # Your changes
REMOTE: Pulled JSONL      # Their changes

# Field-specific merge strategies:
Scalars (title, status):  Last-Write-Wins
Collections (labels):     Union merge
Comments:                 Append + dedupe

3 Design Decisions

WHY GIT-BACKED VS POSTGRESQL?
Traditional databases require server infrastructure, network connectivity, backup complexity. Git provides natural branching (feature branches have their own issues), merge capabilities (git's core competency), offline-first operation, and complete history built-in.
What this means for you
No database server to maintain. Works anywhere git works. Backup = git push. Rollback = git revert. Your issues live in the same repo as your code.
WHY JSONL FORMAT (NOT JSON)?
A single JSON file creates large diffs on any change, merge conflicts on every edit, and requires full file read/write. JSONL (one JSON per line) means small diffs (one line per change), independent line merges, and incremental read/write.
What this means for you
Git merge works naturally. When Agent A and Agent B create different issues, git can merge the JSONL files automatically. No merge conflicts except for editing the same issue.
WHY SQLITE CACHE LAYER?
Direct JSONL queries are O(n) for any search, with no indexing and no joins possible. SQLite provides O(log n) indexed queries, complex joins for dependency graphs, and full-text search. The 25x speedup (752ms to 29ms for 10K issues) is from the blocked issues cache alone.
What this means for you
bd ready returns instantly even with thousands of issues. Dependency tree traversal is fast. The cache rebuilds automatically from JSONL in under 50ms for 10K issues.
WHY DAEMON ARCHITECTURE?
Without a daemon, each CLI command opens a database connection, executes, and closes. The daemon provides persistent connections, batch operations (5-second debounce), background sync, and connection pooling. Result: 60% CPU reduction.
What this means for you
The daemon runs in the background, automatically syncing changes. You can disable it with BEADS_NO_DAEMON=1 or bd --no-daemon for worktree operations.

4 The File Structure 

.beads/
├── issues.jsonl         # Git-tracked source of truth
│                        # One JSON object per line
│
├── deletions.jsonl      # Deletion manifest (prevents resurrection)
│                        # Records deleted issue IDs + timestamps
│
├── beads.db             # SQLite cache (gitignored)
│                        # Fast queries, complex joins
│
├── sync_base.jsonl      # Last-synced snapshot (gitignored)
│                        # Used for three-way merge
│
├── bd.sock              # Daemon Unix socket (gitignored)
│
├── config.yaml          # Local configuration
│
└── metadata.json        # Database metadata

JSONL Issue Schema

# Each line is a complete issue:
{"id":"bd-a3f8","title":"Implement login","status":"open","priority":0,"type":"task","dependencies":[]}
{"id":"bd-b2c4","title":"Add validation","status":"open","priority":1,"dependencies":[{"type":"blocks","id":"bd-a3f8"}]}

Essential CLI Commands

# INITIALIZATION
bd init                              # Initialize beads in repo
bd init --stealth                    # Initialize without git tracking

# ISSUE MANAGEMENT
bd create "Title" -p 0               # Create high-priority issue
bd create "Title" --type bug         # Create bug
bd create "Title" --deps blocks:bd-xxxx  # Create with dependency
bd show bd-xxxx                      # Show issue details
bd show bd-xxxx --json               # Show as JSON (for scripting)
bd update bd-xxxx --status in_progress   # Update status
bd close bd-xxxx --reason "Done"     # Close issue

# QUERIES
bd list                              # List all open issues
bd ready                             # Show work with no blockers
bd stale --days 7                    # Find stale issues
bd dep tree bd-xxxx                  # Show dependency tree

# SYNC
bd sync                              # Full sync (pull -> merge -> push)
bd sync --force                      # Force reimport from JSONL

# DAEMON
bd daemons list                      # List running daemons
bd daemons stop                      # Stop current daemon

# MAINTENANCE
bd doctor                            # Check health
bd doctor --fix                      # Auto-fix issues
bd compact --older-than 30d          # Compact old issues

config.yaml Template

# .beads/config.yaml
prefix: bd                    # Issue ID prefix (max 8 chars)
remote-sync-interval: "30s"   # Sync frequency (min: 5s)
daemon-mode: "events"         # "poll" or "events"
auto-start: true              # Auto-start daemon on CLI use
auto-compact: true            # Auto-compact old issues
compact-after-days: 30        # Compaction threshold

5 The Path of a Task

Follow an issue through the complete lifecycle, from creation by Agent A to pickup by Agent B:

1
Agent A creates issue: bd create "Implement login" -p 0
2
Write to SQLite (instant, <1ms). Database marked as "dirty". Returns: bd-a3f8
3
5-second debounce window accumulates changes. Agent updates: bd update bd-a3f8 --status in_progress
4
Debounce expires. Daemon exports changed issues to issues.jsonl
5
Agent A completes work: bd close bd-a3f8 --reason "Implemented"
6
Agent A syncs before ending: bd sync (pull -> merge -> export -> commit -> push)
7
Agent B starts on different machine: bd sync (or daemon auto-sync)
8
JSONL newer than local SQLite detected. Parse line by line, compare content hashes.
9
Rebuild blocked cache (<50ms for 10K issues)
10
Agent B queries: bd ready - Returns issues with no open blockers (sub-ms)
11
Agent B sees bd-a3f8 is closed, picks next available task
12
"The plane has landed" - Full context transferred, no data loss, audit trail in git
"The plane has NOT landed until git push completes successfully."

6 Gotchas

Real problems you will hit, with concrete fixes:

#1: bd: command not found
Symptom: CLI not in PATH after installation
Fix: Add ~/.local/bin to PATH. Check with which bd. Time: 5 min
#2: No beads repository found
Symptom: Commands fail with "not initialized"
Fix: Run bd init in git repo root. Time: 1 min
#3: Database locked
Symptom: Multiple processes trying to access SQLite
Fix: Stop daemon: bd daemons stop, retry. Time: 1 min
#4: Zombie resurrection
Symptom: Deleted issue keeps reappearing after sync
Cause: Missing deletions.jsonl or another clone restored it
Fix: Check deletions.jsonl exists, re-delete issue. Time: 5 min
#5: Worktree daemon conflict
Symptom: Daemon issues in git worktrees (shared .git directory)
Fix: Disable daemon: BEADS_NO_DAEMON=1 or bd --no-daemon. Time: 1 min
#6: Slow queries
Symptom: List commands taking seconds instead of milliseconds
Fix: Rebuild cache: bd admin rebuild-cache. Time: 2 min
#7: Agent can't use bd edit
Symptom: bd edit launches interactive editor, agent hangs
Fix: Use bd update --field "value" instead. Add --json for scriptable output. Time: 0 min

7 What's Hard

These are fundamental tensions, not bugs to fix:

Git vs Real Database
The tension: Git is not designed for structured queries. SQLite cache provides most query power, but you cannot do arbitrary SQL joins across issues and external data.
Coping strategy: Accept limited query expressiveness. For complex analytics, export to real database. Beads is for agent memory, not business intelligence.
Merge Conflicts Are Possible
The tension: Git three-way merge has limits. When two agents edit the same issue field, conflict is possible. Field-level strategies (last-write-wins for scalars) help but don't eliminate conflicts.
Coping strategy: Use bd conflicts resolve bd-xxxx --ours/--theirs. Design workflows to minimize same-issue edits. Accept that some manual resolution is the cost of distributed operation.
Scale Limits
The tension: Beads is designed for 10K+ issues, not millions. Enterprise scale requires PostgreSQL or similar. Large JSONL files slow git operations.
Coping strategy: Use compaction: bd compact --older-than 60d. Fine for agent workflows. If you need enterprise scale, Beads is not your tool.
No Semantic Search
The tension: Beads is structured, not vector-based. Full-text search exists, but semantic similarity search does not. Use Claude-mem if semantic search is your primary need.
Coping strategy: Use Beads for structured queries (dependencies, status). Use Claude-mem for semantic memory. They complement, not compete.
