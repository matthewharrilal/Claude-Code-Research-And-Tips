2 The Core Abstraction

"View AI orchestrators as living, self-sustaining entities that monitor and upgrade themselves recursively, using containerization as a 'body' for isolation."
Goal: Improve itself
The goal is a command. The orchestrator figures out the rest.

Unlike Ralph (which loops fresh contexts) or Gas Town (which manages a factory of workers), Infinite Orchestra is fundamentally about self-improvement. When idle, the orchestrator doesn't wait. It analyzes its own limitations and implements fixes.

This is the key insight: idle time is improvement time. The agent monitors its own activity. When nothing is happening, it triggers a self-improvement workflow using its own worker profiles.

3 Design Decisions

WHY DOCKER CONTAINERIZATION?
Self-improvement is inherently dangerous. An agent modifying its own code could break its host machine. Docker provides an isolation boundary: the "body" that contains the orchestrator's actions.
What this means for you
Always run Orchestra in Docker. Never on bare metal. If the agent makes a catastrophic change, you can blow away the container and start fresh. Docker is not optional here; it is load-bearing.
WHY IDLE-STATE SELF-IMPROVEMENT?
Most agents idle between tasks. Wasted compute. Orchestra inverts this: idle time triggers recursive enhancement. The agent that improved itself to add Docker safety? That happened during idle state.
What this means for you
Set clear goals but expect the agent to evolve. Watch for unexpected improvements. Review git commits. The agent is designed to change itself, so you must verify those changes are beneficial.
WHY HUB-AND-SPOKE WORKERS?
Central coordination simplifies control. The orchestrator is the single point of decision-making. Workers are stateless specialists. This is simpler than peer-to-peer, easier to debug, easier to understand.
What this means for you
All work flows through the orchestrator. You have one place to look for state, one place to look for decisions. Workers don't talk to each other, they report to the center.
WHY 6 SPECIFIC PROFILES?
These cover the common development workflow: see (vision), learn (docs, research), build (coder), remember (memory), design (architect). Each profile has a focused purpose. Specialization enables better prompts.
What this means for you
Assign tasks to the appropriate specialist. Don't ask the coder to do research. Don't ask the architect to implement. Let each profile do what it's designed for.
WHY OPENCODE DEPENDENCY?
Orchestra is built as a plugin for OpenCode, leveraging existing agent infrastructure. This is not standalone; it extends an existing tool.
What this means for you
You need OpenCode installed first. Then install the Open Orchestra plugin. This is a prerequisite, not optional.
WHY NEO4J (OPTIONAL)?
Graph databases excel at complex memory relationships. For simple use cases, flat file memory works. For sophisticated context persistence across many sessions, Neo4j provides richer querying.
What this means for you
Start without Neo4j. Add it later if you need complex memory relationships. It's optional complexity.

  },
  "docker": {

    "resource_limits": {
      "memory": "4g",

    }
  }
}

Docker Setup

# docker-compose.yml
version: '3.8'
services:
  orchestra:
    build: .
    container_name: orchestra
    environment:
      - ANTHROPIC_API_KEY=${ANTHROPIC_API_KEY}
    volumes:
      - ./goals:/app/goals
      - ./memory:/app/memory
    deploy:
      resources:
        limits:
          memory: 4g
          cpus: '2'
    restart: unless-stopped
# Start Orchestra
docker-compose up -d

# Check logs
docker-compose logs -f orchestra

# Stop (preserves state)
docker-compose stop

# Destroy (fresh start)
docker-compose down -v

5 The Path of a Task

Two distinct flows: normal task execution and the self-improvement cycle.

Path A: Normal Task (e.g., "Build an ecommerce store")

1
goals/current.md contains "Build an ecommerce store"
2
Central Orchestrator reads goal, creates task graph
3
Tasks decomposed: design data models (architect), research payment APIs (research), implement backend (coder), write docs (docs)
4
Sequential execution - architect runs first, returns schema
5
Research worker investigates Stripe API integration
6
Coder worker implements backend (takes longest)
7
Docs worker creates README and API documentation
8
Task complete. Orchestrator enters idle monitoring.

Path B: Self-Improvement Cycle

1
Orchestrator monitors activity, detects idle state
2
Self-improvement trigger fires, creates improvement task
3
Research worker analyzes current limitations: "Can destroy host machine"
4
Architect worker designs solution: "Add Docker containerization"
5
Coder worker modifies orchestrator config, adds docker-compose.yml
6
Improvement complete. Orchestrator now runs in container.
7
Returns to idle monitoring, ready for next trigger.

Note: This is the actual demo @0xSero ran. The agent genuinely added Docker safety to itself.

6 Gotchas

Real problems you will hit, with concrete fixes:

#1: Sequential Only
Symptom: Tasks run one at a time, even when they could parallelize
Cause: Current architecture limitation (@Zay878458749421 discovered this)
Fix: Wait for git worktrees feature (@0xSero roadmap). For now, accept sequential.
#2: Subagent Explosion
Symptom: 5+ agents with 5 subagents each = conflicts and problems
Cause: Combinatorial explosion of coordination
Fix: Limit subagent spawning in configuration. Keep orchestration shallow.
#3: No Checkpointing
Symptom: Crashes lose all progress from current improvement cycle
Cause: No state persistence mid-task (@SahilDhankhad suggested this)
Fix: Future feature. For now, use git for recovery. Commit frequently.
#4: Auto-Updates Unreliable in Docker
Symptom: Container doesn't self-update cleanly after self-improvement
Cause: Container immutability conflicts with self-modification
Fix: Manual rebuild after changes. docker-compose build && docker-compose up -d
#5: OpenCode Dependency
Symptom: Orchestra won't start without OpenCode installed
Cause: Built as a plugin, not standalone
Fix: Install Open Orchestra plugin for OpenCode first. 10 min setup.
#6: Resource Limits Matter
Symptom: Docker container exhausts host resources despite isolation
Cause: No limits set, container takes everything
Fix: Set memory and CPU limits in config. See docker-compose.yml above.
#7: Self-Improvement is Experimental
Symptom: Unexpected changes to orchestrator behavior
Cause: Agent modifying itself in ways you didn't anticipate
Fix: Review all git commits. Use git bisect if something breaks. Accept higher risk.
#8: Goal Clarity is Critical
Symptom: Agent takes wrong approach, wastes compute
Cause: Vague goals = confused interpretation
Fix: Write specific, measurable goals. "Build ecommerce store" not "make something cool".
#9: Limited to Anthropic API
Symptom: Can't use OpenAI, Gemini, or local models
Cause: Built specifically for Claude
Fix: Have Anthropic API key ready. Factor API cost into budget.
#10: Neo4j Optional but Recommended
Symptom: Flat file memory is limited for complex projects
Cause: File-based memory doesn't capture relationships
Fix: Install Neo4j for better persistence. 15 min setup. Worth it for long-running orchestrators.

7 What's Hard

These are fundamental tensions, not bugs to fix:

Self-Improvement is Dangerous
The tension: An agent modifying itself could break things in subtle or catastrophic ways. Docker helps but isn't a complete solution.
Symptoms: Unexpected behavior after idle cycles. Breaking changes that work in testing but fail in production. Cascading failures from one bad improvement.
Mitigation: Git for recovery. Review commits. Accept that self-improvement = higher risk. This is the price of autonomy.
Sequential vs. Parallel
The tension: Current architecture is sequential only. Parallelization would speed things up but introduces coordination complexity.
Symptoms: Long wait times for multi-step tasks. Workers idle while one worker runs.
Mitigation: Wait for git worktrees feature. Accept sequential for now. Plan tasks to minimize blocking.
Crash Recovery
The tension: No checkpointing means crashes lose work. Adding checkpointing adds complexity.
Symptoms: Multi-hour improvement cycles lost to container restart.
Mitigation: Manual backups. Git commits frequently. Future checkpointing feature.
Trust vs. Autonomy
The tension: Full autonomy enables 24/7 operation but means the agent might do wrong things while you sleep.
Symptoms: Morning reveals unexpected changes. "What did it do last night?"
Mitigation: Set clear goals. Review changes daily. Use git diff to understand overnight work.
Experimental Maturity
The tension: Less battle-tested than Gas Town or CC Mirror. Fewer users means fewer discovered edge cases.
Symptoms: Hitting problems no one else has hit. Limited community support.
Mitigation: Accept higher risk. Contribute feedback. Help mature the pattern.
Resource Management
The tension: 24/7 operation = 24/7 cost. Anthropic API bills add up.
Symptoms: Unexpected API charges. Runaway compute in improvement loops.
Mitigation: Set resource limits. Monitor API usage. Budget for continuous operation.
