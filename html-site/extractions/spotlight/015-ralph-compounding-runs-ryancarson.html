2 Core Abstraction: Compounding Returns

"Each run makes the next better through systematic archiving."
Run N: Reference archives 1..N-1 → Better output → Archive Run N
This is the compounding loop. Knowledge accumulates exponentially.

Traditional Ralph: Each run starts fresh. Memory via progress.txt, git commits, current prd.json. Works great for single features.

Compounding Ralph Enhancement

Same as above PLUS: Archived PRDs from previous runs, archived user story JSONs, and explicit learnings.md files. Agent references past successes and failures before starting each task.

The insight: Fresh context is still essential (context rot is real), but SELECTIVE knowledge injection from archives gives each run a head start. You're not extending sessions—you're accumulating institutional memory in files.

The Mental Model

Think of AI coding as an extremely persistent but context-limited junior developer who works 24/7. You become the product manager/architect. The archives are your handoff documentation—each shift starts by reading what the previous shift learned.

3 Design Decisions

WHY ARCHIVE AFTER EVERY RUN?
Knowledge compounds only when systematically captured. If you archive sporadically, you lose the compounding effect. Even "failed" runs contain valuable learnings about what NOT to do.
What this means for you
Make archiving a ritual. After every Ralph run (success or failure), copy prd.json and progress.txt to a dated archive folder, then write 2-3 key learnings to learnings.md.
WHY JSON FORMAT FOR PRDs?
JSON is structured, parseable by agents, and diffable in git. Agents reason better over structured data than prose. You can also write tooling to analyze patterns across archived PRDs.
What this means for you
Keep PRDs in JSON. Include userStories as an array with id, title, acceptanceCriteria, priority, and passes fields. This structure is your task tracking system.
WHY INCLUDE learnings.md?
Explicit capture of what worked and what didn't. Forces reflection at the end of each run. Without it, learnings stay implicit and fade from memory (yours AND the agent's).
What this means for you
Write learnings.md even when tired. Keep each learning to 2-3 lines max. Include failures with cause analysis—they're often more valuable than successes.
WHY PARALLEL WORKTREES?
Git worktrees let you run 3 Ralph instances simultaneously on different features. Each worktree is an isolated copy of your repo at a different branch. 3x throughput without interference.
What this means for you
Before overnight runs, create worktrees for each feature. Each Ralph instance gets its own directory. In the morning, you have commits across 3 branches to review and merge.
WHY 25-30 MINUTES PLANNING?
PRD quality determines overnight success. Garbage in, garbage out. Spending 25-30 minutes on detailed user stories with clear acceptance criteria means Claude works on the RIGHT things while you sleep.
What this means for you
Treat planning as your primary job. Your role is PM/architect, Claude's role is implementation army. Heavy upfront planning pays massive dividends in overnight productivity.

Prompt Strategy for Archive Reference

In your prompt.md, instruct the agent to reference archives: 

## Learning from Previous Runs

Before starting, review archives in `archives/` folder:
- What patterns worked well?
- What caused failures?
- What shortcuts were discovered?

Apply these learnings to current implementation.

## Your Task

1. Read `archives/` for prior learnings
2. Read `scripts/ralph/prd.json` for current tasks
3. Read `scripts/ralph/progress.txt` for session learnings
4. Pick highest priority story where `passes: false`
5. Implement using patterns from archives
6. Run typecheck and tests
7. Commit: `feat: [ID] - [Title]`
8. Update prd.json: `passes: true`
9. APPEND learnings to progress.txt

5 Path of a Task: Compounding Flow

Follow a feature through multiple compounding runs to see knowledge accumulation:

1
Evening: Spend 25-30 minutes writing prd.json with detailed user stories and acceptance criteria
2
Run 1: First attempt. Claude reads current prd.json, implements stories, commits. Some succeed, some fail.
3
Archive Run 1: Copy prd.json + progress.txt to archives/run-2026-01-05/, write learnings.md
4
Run 2: New Claude instance reads archives FIRST, then current task. Avoids Run 1's mistakes.
5
Archive Run 2: Knowledge accumulates. learnings.md grows with proven patterns.
6
Run 3+: Each run inherits all prior learnings. Velocity increases. Errors decrease.
7
Morning: Wake up to 18+ commits across 3 parallel worktrees. Review, merge, ship.

6 Gotchas: 10 Problems with Fixes

#1: PRD quality determines success
Symptom: All runs fail the same way
Cause: Unclear acceptance criteria, ambiguous stories
Fix: Spend full 25-30 min on PRD. Every story needs testable acceptance criteria.
#2: Archive pollution
Symptom: Too many irrelevant learnings confuse the agent
Threshold: >50 learnings without curation
Fix: Curate learnings.md to essentials. Keep top 5 per category. Time: 10 min
#3: Stale learnings
Symptom: Agent applies outdated patterns that no longer work
Cause: Codebase evolved, learnings didn't
Fix: Add date/context to each learning. Review monthly. Time: 5 min/learning
#4: Cost accumulation
Symptom: Amp credits depleted faster than expected
Math: 3 parallel runs = 3x cost. Each run ~$3-5.
Fix: Budget and monitor overnight. Set cost alerts. Prioritize which features get parallel treatment.
#5: Catastrophic failures undetected
Symptom: Wake up to broken repo, all runs failed at iteration 2
Cause: No monitoring, slept through alerts
Fix: Set up morning alerts (Slack/Discord webhook on stuck detection). Time: 15 min setup
#6: Worktree conflicts
Symptom: Parallel runs touch same files, merge conflicts galore
Cause: Features not properly isolated
Fix: Enforce feature isolation in PRDs. One worktree = one domain. Time: 10 min planning
#7: Archive not referenced
Symptom: Agent ignores archives entirely, no compounding
Cause: Missing instruction in prompt.md
Fix: Explicit "Read archives/ FIRST" instruction in prompt.md. Time: 2 min
#8: Learnings too verbose
Symptom: Agent overwhelmed by context, performance degrades
Threshold: >20KB total learnings
Fix: Keep each learning to 2-3 lines. Be ruthless. Quality over quantity.
#9: No failure learnings
Symptom: Only successes archived, same failures repeated
Cause: Natural bias toward documenting wins
Fix: Mandate failure analysis in learnings.md. "What went wrong? Why? How to avoid?" Time: 5 min
#10: Missing timestamps
Symptom: Can't trace learning evolution, don't know which are recent
Cause: Archive folders not dated
Fix: Use run-YYYY-MM-DD format for archive folders. Include date in learnings. Time: 2 min

7 What's Hard: Fundamental Tensions

These aren't bugs to fix. They're tensions to manage:

Quality vs Quantity of Learnings
The tension: Too many learnings = noise. Too few = missed insights. There's no formula.
Why it's hard: Every learning feels valuable when you write it. Time reveals which are actually useful.
Coping strategy: Curate ruthlessly. Keep top 5 per category. Monthly review to prune outdated patterns.
Trusting Overnight Execution
The tension: You can't monitor, but you also can't NOT monitor. Fear of waking to broken repo.
Why it's hard: Autonomous execution requires trust. Trust requires good PRDs. Good PRDs require skill you build over time.
Coping strategy: Start with HOTL Ralph (human review between iterations). Graduate to overnight only after 10+ successful supervised runs.
Parallel Cost Scaling
The tension: 3x features = 3x cost. Budget isn't infinite. Which features deserve parallel treatment?
Why it's hard: ROI varies by feature. Some compound well, others don't.
Coping strategy: Use parallel for high-value, isolated features. Sequential for experimental or interdependent work.
Archive Maintenance Burden
The tension: Archives grow over time, become unwieldy. Maintenance is overhead that compounds.
Why it's hard: Short-term: more learnings help. Long-term: too many learnings hurt. Inflection is invisible.
Coping strategy: Monthly consolidation ritual. Merge similar learnings. Archive archives (meta-archive old runs).
Learning Transfer Fidelity
The tension: Agent may misapply learnings to wrong context. A pattern for auth might break payments.
Why it's hard: Context is implicit. Learnings are explicit. The mapping is lossy.
Coping strategy: Include specific context in each learning: "When building [X], do [Y]" not just "Do [Y]".

8 When to Use / When Not

USE COMPOUNDING RALPH WHEN
Project duration 1+ week — short projects don't benefit from compounding
5+ features to build — archive overhead justified by learning accumulation
Daily Ralph runs — frequent runs maximize compounding effect
Solo developer — archives are personal knowledge base
Similar feature patterns — learnings transfer between related work
DON'T USE COMPOUNDING RALPH WHEN
Single feature — use basic Ralph, no compounding benefit
Team project — archives are personal, not team. Use shared docs.
Budget constrained — parallel runs cost 3x. Sequential may suffice.
Highly novel projects — no patterns to compound, each run unique
Need coordination — use CC Mirror for inter-agent coordination
