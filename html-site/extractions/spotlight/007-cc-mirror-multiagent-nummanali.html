2 The Core Abstraction

"Think of Claude Code orchestration as a Kraken - a powerful, unleashed multi-agent system where you direct tentacles (agents) via simple speech in an isolated terminal."
npx cc-mirror quick --provider mirror --name mclaude
That's the unlock. Multi-agent was always there. CC Mirror enables it.

Claude Code has built-in multi-agent orchestration primitives - TaskCreate, TaskList, TaskUpdate, TaskGet - but they're disabled in the standard release. CC Mirror creates an isolated variant that enables these features without touching your main installation.

The pattern is pure Numman Ali philosophy: "You speak, it does anything you want." No special syntax. No orchestration DSL. Just natural language direction to a conductor that spawns parallel agents.

3 Design Decisions

WHY ISOLATED VARIANT (NOT PATCH)?
CC Mirror could have patched the main Claude Code installation directly. Instead, it creates a separate variant at ~/.mclaude/. This is intentional - it means your main Claude Code stays untouched and stable.
What this means for you
You can use both vanilla Claude Code AND CC Mirror simultaneously. If CC Mirror breaks after an update, your main installation is unaffected. Safe experimentation.
WHY PRE-PACKAGED SKILL?
The orchestration skill (5299 tokens) defines how the conductor breaks down tasks, tracks dependencies, and spawns workers. It's pre-packaged so you don't need to write prompt engineering for multi-agent coordination.
What this means for you
Works immediately after install. Just say "Load the orchestration skill and initiate" - no configuration needed. The skill handles task DAG, worker spawning, and dependency resolution.
WHY PROVIDER FLEXIBILITY?
Different models have different cost/quality tradeoffs. The "mirror" provider reflects Claude's internal primitives exactly. GLM-4.7 handles multi-agent coordination well at lower cost. OpenRouter and LiteLLM provide access to other models.
What this means for you
Start with mirror for reliability. Switch to GLM-4.7 for cost optimization. Use OpenRouter for model variety. The architecture is provider-agnostic.

6 Gotchas

Real problems you'll hit, with concrete numbers and fixes:

#1: npx cc-mirror: command not found
Symptom: Command fails, npm throws errors
Fix (2 min): npm install -g npm@latest then retry. If npx fails, use npm install -g cc-mirror
#2: EACCES permission denied
Symptom: npm global install fails with permission error
Fix (5 min): mkdir ~/.npm-global && npm config set prefix '~/.npm-global' then add to PATH
#3: Windows compatibility issues
Symptom: Installation fails or mclaude won't launch on Windows
Fix (30 min): Use WSL2: wsl --install then run cc-mirror inside WSL
Status: "@YashasGunderia: I'm fixing the windows issue right now" - Active PRs in progress
#4: Provider not found / API key issues
Symptom: Authentication errors, connection failures
Fix (2 min): echo $ANTHROPIC_API_KEY to verify. Test with curl https://api.anthropic.com/v1/models -H "x-api-key: $ANTHROPIC_API_KEY"
#5: Quick install skips team mode for GLM
Symptom: GLM-4.7 setup missing expected features
Fix (5 min): Use full variant: npx cc-mirror variant --provider glm --name mclaude --team-mode
#6: Orchestration skill fails to load
Symptom: "Load orchestration skill" doesn't work
Fix (2 min): Exit and restart mclaude. If persists: rm -rf ~/.mclaude and reinstall
#7: Context window exhausted mid-orchestration
Symptom: Agent stops, forgets task assignments
Fix (2 min): Enable auto-compact beta feature. Keep individual tasks smaller.
#8: Multiple agents stepping on each other
Symptom: File conflicts, race conditions
Fix (15 min): Use git worktrees: git worktree add ../agent-1 task-1-branch for file isolation. Or assign non-overlapping file ownership in task descriptions.
#9: Dependencies not resolving correctly
Symptom: Blocked tasks stay blocked
Fix (5 min): Check TaskUpdate syntax: TaskUpdate task-id, addBlockedBy=["blocking-id"]
#10: Orchestration stuck or looping
Symptom: Same task attempted repeatedly
Fix (2 min): Exit and restart. If tasks corrupted: "Clear all tasks and restart orchestration"
Nuclear (5 min): rm -rf ~/.mclaude && npx cc-mirror quick --provider mirror --name mclaude

7 What's Hard

These are fundamental tensions, not bugs to fix:

Disabled Feature Risk
The tension: CC Mirror relies on editing minified JS to enable disabled features. Claude Code updates may break this at any time.
Symptoms: After Claude Code update, CC Mirror stops working or behaves unexpectedly.
Mitigation: Watch for Claude Code updates before critical work. Keep a backup mclaude version. Accept that this is beta-level stability.
Beta Stability
The tension: CC Mirror is a community project enabling an officially disabled feature. It's not production-hardened.
Symptoms: Unexpected breaking changes, no guaranteed SLA, limited observability.
Mitigation: Use for development and experimentation, not production deployment. Accept the trade-off of features vs stability.
Context Consumption
The tension: Multi-agent orchestration uses more context per session than single-agent. Each agent reads the task state, project context, and skill definition.
Symptoms: Context exhaustion mid-orchestration, agents forgetting context, quality degradation in complex orchestrations.
Mitigation: Enable auto-compact. Keep individual tasks focused and small. Accept that very complex orchestrations may need to be split across sessions.
Observability Limits
The tension: CC Mirror provides less visibility than custom-built orchestration solutions. The trade-off is speed of setup vs depth of insight.
Symptoms: Can't easily debug why an agent made a decision. Limited logging. Hard to audit in regulated environments.
Mitigation: Accept the trade-off for development use. For production, consider Gas Town or custom solutions with full observability.
Windows Friction
The tension: Windows has known compatibility issues. The community is actively fixing them, but it's not yet seamless.
Symptoms: Installation failures, runtime issues specific to Windows.
Mitigation: Use WSL2 on Windows. Check GitHub for latest Windows-specific PRs. Or wait for fixes.

8 When to Use / When Not

USE CC MIRROR WHEN
You need parallel agent execution (CC Mirror's core strength)
Task complexity is medium-high and decomposable into DAG
Budget is moderate ($10-50/day for multiple agents)
You're in development/experimentation mode (not production)
Setup time under 10 minutes is acceptable
DON'T USE CC MIRROR WHEN
Single simple task - Orchestration overhead not worth it
Production deployment - Beta feature may change
Windows without WSL - Known compatibility issues
Need guaranteed stability - Relies on editing minified JS
Strict security requirements - Less visibility than audited tools
Simple sequential tasks - Use Ralph loop instead
