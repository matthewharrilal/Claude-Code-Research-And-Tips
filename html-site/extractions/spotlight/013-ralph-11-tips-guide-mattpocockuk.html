2 The Core Abstraction

The Mental Model
Think of the AI agent as an eager but somewhat dim junior developer (Ralph) who never sleeps, happily works on one tiny task at a time, commits often, and tells you when it's truly finished — but needs extremely tight scoping, strong feedback loops, and hard limits to avoid disaster.
"The AI is an infinite junior team. You're the requirements gatherer."

You define WHAT. Ralph figures out HOW.

Ralph is positioned as simpler than swarm/multi-agent systems while still enabling truly unattended development. The pattern trades complexity for reliability.

The 7 Key Principles

1
Finite Iterations
Never truly infinite loop. Always set max iterations. Prevents runaway costs.
2
Tiny, Clearly Scoped Tasks
One user story/feature at a time. Single focus per iteration.
3
Clean Slate Per Iteration
New context window each run. Avoids "context rot".
4
Hard Stop Condition
<promise>COMPLETE</promise> or all PRD items done.
5
Ruthless CI Green Enforcement
Tests + types must pass every commit. No broken code inheritance.
6
Progress Tracking via Files
progress.txt, Git history, PRD.json. External state is memory.
7
Human Defines Quality/Exit
Agent executes blindly. Quality bar set in advance by you.

3 Design Decisions

WHY FINITE ITERATIONS (NOT INFINITE)?
Unbounded loops lead to cost explosion. Without a cap, a stuck loop can burn through your budget overnight. Matt emphasizes: the iteration limit is your circuit breaker.
What this means for you
Always set MAX_ITERATIONS. Accept incomplete runs over unbounded runs. 50 iterations overnight is a reasonable starting point. Monitor costs and adjust.
WHY TINY TASKS (NOT LARGE)?
Context windows are finite. When a task is too large, Claude loses track mid-task, produces incomplete code, and marks it "done" anyway. Small tasks = verifiable units = reliable progress.
What this means for you
The 2-3 sentence rule: if you can't describe a task in 2-3 sentences, split it. "Build auth system" becomes 5+ separate tasks. Scope creep is the most common failure mode.
WHY RUTHLESS CI (NOT OPTIONAL)?
Without feedback loops, bad code compounds. If iteration 5 introduces a bug that passes silently, iterations 6-20 build on broken foundation. CI is the verification layer that makes autonomy safe.
What this means for you
Tests + typecheck are non-negotiable. No CI = no Ralph. If your repo doesn't have tests, add them first. This is a prerequisite, not optional.
WHY EXTERNAL FILES (NOT INTERNAL MEMORY)?
Each iteration spawns a fresh Claude instance with zero memory. The only way to persist learnings across iterations is through files. progress.txt, prd.json, and git commits ARE the memory system.
What this means for you
Never rely on "Claude remembering." If it's not in a file, it doesn't exist for the next iteration. progress.txt must be append-only. Every learning must be externalized.

4 File Structure 

project/
├── scripts/ralph/           # The Ralph system
│   ├── ralph.sh             # Loop with MAX_ITERATIONS
│   ├── PRD.json             # Stories with passes: false/true
│   └── progress.txt         # Learnings (append-only)
└── CLAUDE.md                # Project context

PRD.json Task Sizing Examples

// BAD: Too big - will overflow context
{
  "build_entire_auth_system": { "passes": false }
}

// GOOD: Properly decomposed
{
  "userStories": [
    { "id": "US-001", "title": "Create user model", "passes": false },
    { "id": "US-002", "title": "Add password hashing", "passes": false },
    { "id": "US-003", "title": "Create login endpoint", "passes": false },
    { "id": "US-004", "title": "Create logout endpoint", "passes": false },
    { "id": "US-005", "title": "Add session management", "passes": false }
  ]
}

Each task should be completable in ONE iteration (one context window). If a task takes multiple iterations to complete, it's too big. Split it further.

6 Gotchas: The 3 "Don't Do This"

Don't let Ralph run infinitely
Why it fails: No cost control → budget explosion
What happens: A stuck loop burns through your API budget overnight
Fix: Always set MAX_ITERATIONS. Accept incomplete over unbounded.
Don't give huge tasks
Why it fails: Context overflow → incomplete, broken code
What happens: Claude loses track mid-task, marks it "done" anyway
Fix: 2-3 sentence rule. Split ruthlessly. Scope creep is the #1 failure mode.
Don't skip tests
Why it fails: No feedback loop → bad code compounds
What happens: Iteration 5 breaks something, iterations 6-20 build on broken foundation
Fix: CI is non-negotiable. No tests = no Ralph.

Additional Caveats

Needs capable models
Issue: Smaller models (Haiku) can't handle complex tasks reliably
Fix: Use Claude Opus-class models for Ralph loops
Not for every repo
Issue: Works better for well-factored repos with good test coverage
Fix: Improve your codebase first. Add tests before adding Ralph.
Review burden
Issue: Many commits/PRs to check in the morning
Fix: Trust CI, spot check key commits. Don't review everything line-by-line.
Windows issues
Issue: Some implementations have compatibility problems on Windows
Fix: Use WSL or macOS for reliable Ralph loops

7 What's Hard: Fundamental Tensions

These are real tensions with no perfect solutions, only trade-offs:

Scope Discipline
The tension: Most common failure is scope creep. It's tempting to combine related tasks. Resist.
Symptom: Tasks that take multiple iterations to "complete" but still fail verification
Mitigation: Ruthless tiny tasks. When in doubt, split. The 2-3 sentence rule is a forcing function.
Review Burden
The tension: Many commits overnight = many things to check in the morning
Symptom: Spending 2 hours reviewing 23 commits defeats the purpose
Mitigation: Trust CI. Spot check key files. If CI passes, trust the output unless something seems wrong.
Model Capability Requirements
The tension: Not all models can handle Ralph-style autonomous loops
Symptom: Haiku gets stuck, produces lower quality, fails verification
Mitigation: Use Opus-class models. The cost difference is worth the reliability.
Well-Factored Requirement
The tension: Ralph works best on codebases with clear structure and good tests. Legacy code struggles.
Symptom: Ralph makes changes that break unrelated things because the codebase is tangled
Mitigation: Improve your codebase structure first. Add tests. Ralph is a reward for good architecture, not a fix for bad architecture.
Mental Model Shift
The tension: You're no longer the coder. You're the PM, the requirements gatherer, the quality definer.
Symptom: Frustration when Ralph doesn't do things "your way"
Mitigation: Accept the role shift. Define what you want, not how to do it. Trust the process.

8 When to Use / When Not

WHAT RALPH ENABLES
Truly unattended overnight/multi-day development
Scaling test coverage massively
Working on large backlogs while sleeping
"Ship while you sleep" workflows
Features with verifiable acceptance criteria
WHEN NOT TO USE
Exploratory work with no clear stop condition
Major refactors without verification criteria
Security-critical code needing human judgment
Anything requiring real-time human review
Legacy codebases without test coverage

Combines Well With

Safety isolation for overnight runs
Skills/Custom Tools
Steering behavior with specialized tools
Automatic Linting/Formatting
Quality baseline without human intervention
GitHub/Linear Integration
Notification when Ralph completes
Heavy TDD
Strong feedback loops for verification
