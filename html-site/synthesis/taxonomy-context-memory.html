Essence (15 words)
LLMs forget everything. Files persist. The filesystem IS the memory for Claude Code.

2 The Memory Problem

"No persistent memory across sessions. Every conversation starts from zero."
Session A → [Work] → Lost | Session B → [Re-explain everything]
That's the fundamental limitation. Everything else is workarounds.

LLMs face a fundamental limitation: no persistent memory across sessions. Every new conversation starts from zero. This creates multiple challenges:

Challenge Impact
Context window limits Cannot process entire codebases at once
Session boundaries Must re-explain project context each time
Context rot Performance degrades as context fills up
Token waste Repetitive information consumes expensive tokens
Lossy compaction Auto-summary loses important details

The patterns in this taxonomy represent the community's solutions to these challenges, ranging from simple file-based approaches to sophisticated multi-agent memory architectures.

3 Design Decisions

WHY EXTERNAL STATE?
If Claude has no memory across sessions, progress must persist somewhere. Files survive session boundaries. Git survives machine restarts. The filesystem becomes the brain's long-term memory.
What this means for you
Never rely on "Claude remembering" something. If it's not in a file, it doesn't exist. Every learning, every decision, every state change must be persisted externally. The files ARE the memory.
WHY FRESH CONTEXT PER ITERATION?
At ~80K tokens, Claude's quality degrades. Instructions get lost. The model contradicts itself. This is "context rot"—a fundamental limitation of transformer attention. Fresh context each iteration avoids this.
What this means for you
Kill and restart beats extending sessions. If iteration 20 produces worse code than iteration 5, your context is rotting. The fix is to restart with fresh context, not to debug.
WHY LAYERED FILE HIERARCHY?
Different scopes need different persistence. Enterprise policies apply everywhere. Project knowledge applies per-codebase. Personal preferences apply per-user. A single CLAUDE.md cannot serve all these purposes.
What this means for you
Use the right layer for each type of knowledge. Security policies → Enterprise level. Architecture decisions → Project level. Personal shortcuts → User level. Local secrets → CLAUDE.local.md (gitignored).

4 Memory Persistence Patterns

4.1 File-Based State

The simplest and most widely-used persistence pattern. Information survives sessions through filesystem reads. 

project/
├── CLAUDE.md              # Project-level knowledge (git-checked)
├── CLAUDE.local.md        # Personal/secret config (gitignored)
└── .claude/
    └── rules/*.md         # Path-specific rules with frontmatter

Pattern: CLAUDE.md Knowledge Base

  • Claude Code scans for CLAUDE.md files at startup
  • Content injected into system prompt automatically
  • Path-targeted rules activate based on file context
  • Updates persist via git (CLAUDE.md) or locally (CLAUDE.local.md)

Pattern: progress.txt Append-Only Log

# progress.txt - APPEND ONLY - NEVER EDIT

---
Iteration 1 (2026-01-09 10:00):
- Completed: US-001 - Tool skeleton created
- Learned: Use orange (#FF6B35) for visibility
- Next: Consider beat playback timing

---
Iteration 2 (2026-01-09 10:15):
- Completed: US-002 - Category mapping
- Issue encountered: Zod schema needed adjustment
- Pattern discovered: Always use .optional() explicitly

Critical rule: NEVER UPDATE — only append new entries.

Pattern: prd.json Structured Task State

{
  "project": "Untangle",
  "branchName": "ralph/feature-name",
  "userStories": [
    {
      "id": "US-001",
      "title": "Create upsertIncome tool skeleton",
      "acceptanceCriteria": [
        "Zod schema matches PRD",
        "npm run typecheck passes"
      ],
      "priority": 1,
      "passes": false
    }
  ]
}

4.2 Git-Based Memory

Git provides distributed, versioned memory that Claude can read through its standard tooling. 

# Agent can reconstruct project evolution
git log --oneline -20
git diff HEAD~5
git show <commit-hash>

# Create worktree for parallel agent
git worktree add ../wt-feature feature-branch

# Check worktree list
git worktree list

Git Worktrees enable parallel Ralph loops. Multiple working directories share the same git history, providing true file isolation between agents while maintaining shared context through version control.

4.3 External Memory Systems

Beyond files and git, specialized tools provide sophisticated memory capabilities.

Claude-Mem Plugin - Automatic session capture with semantic compression:

  • Captures lifecycle events (SessionStart, PostToolUse, prompts)
  • Compresses to ~50-100 tokens (95% savings)
  • Stores in SQLite + Chroma vector DB
  • 3-layer retrieval: search() → timeline() → get_observations()

MCP Server State - External tools via Model Context Protocol:

Claude Code ←→ MCP Server ←→ External System
                              (Slack, BigQuery, Sentry, etc.)

4.4 Memory Hierarchy

Layer 1: Enterprise Policy
/Library/Application Support/ClaudeCode/CLAUDE.md
Organization-wide security policies, compliance requirements
Layer 2: Project Memory
./CLAUDE.md or ./.claude/CLAUDE.md
Architecture decisions, coding standards, team workflows
Layer 3: Modular Rules
./.claude/rules/*.md
Path-specific rules with frontmatter targeting
Layer 4: Personal + Local
~/.claude/CLAUDE.md + ./CLAUDE.local.md
Agent-specific context, sandbox URLs, test credentials

Loading order: Enterprise → Project → Modular Rules → Personal → Local

5 The Path of Memory

Follow how memory flows through a Ralph-style iteration:

1
Session starts. Claude loads CLAUDE.md automatically into system prompt.
2
Claude reads prd.json to understand task state (what's done, what's pending).
3
Claude reads progress.txt for learnings from previous iterations.
4
Claude implements the task, using knowledge from all three memory sources.
5
Task completes. Claude updates prd.json with passes: true.
6
Claude APPENDS learnings to progress.txt (never overwrites).
7
Claude commits code changes with meaningful message referencing task ID.
8
Session ends. All context is lost—but files persist.
9
New session starts. Fresh Claude reads files. Memory reconstructed from files.
10
Repeat. Each iteration builds on prior iterations via external state.

6 Gotchas

Real problems you'll hit with memory patterns, with concrete numbers and fixes:

Context window fills up too quickly
Symptom: Quality degrades mid-task, agent forgets earlier context
Threshold: ~80K tokens is when rot typically starts
Fix: Move expensive tools to subagents. Use fresh context per iteration. Verify CLAUDE.md isn't bloated (should be <500 lines).
progress.txt grows unbounded
Symptom: Claude spends more time reading than thinking
Threshold: >20KB starts degrading quality
Fix: Archive old entries. Keep last 10-50 entries active. Move permanent learnings to CLAUDE.md.
Claude-Mem not finding relevant context
Symptom: Semantic search returns irrelevant results
Cause: Query too broad, or observations not properly tagged
Fix: Use specific type filters: search(query="auth bug", type="bugfix"). Always batch IDs in get_observations.
Multi-agent memory conflicts
Symptom: Agents overwrite each other's state, miss handoffs
Cause: Unclear ownership or timing issues
Fix: Use explicit file ownership (each agent owns its directory). Use handoff protocol with timestamps. Consider file locks for critical sections.
Git history not providing enough context
Symptom: Future iterations don't understand prior work
Cause: Poor commit messages or missing context
Fix: Use meaningful commit messages with story IDs. Include "why" not just "what". Commit after each story, not in bulk.

7 What's Hard

These are fundamental tensions in memory management, not bugs to fix:

Context Rot vs. Memory Loss
The tension: Extended sessions provide rich context but quality degrades at 80K tokens. Fresh context provides peak quality but loses accumulated learnings.
Symptoms: Quality drops in late iterations (rot), OR early iterations repeat mistakes from archived runs (memory loss).
Mitigation: Accept that fresh context beats rot. Invest in better file hygiene (progress.txt under 20KB, meaningful CLAUDE.md) rather than longer sessions.
Compression vs. Fidelity
The tension: Semantic compression saves 95% tokens but loses nuance. Full transcripts preserve everything but consume expensive context.
Symptoms: Compressed summaries miss edge cases. Full context causes rot.
Mitigation: No perfect answer exists. Use 3-layer retrieval (Claude-Mem style): search returns IDs, detailed fetch only for selected items. Accept that some information will be lost.
Isolation vs. Coordination
The tension: Multi-agent patterns require isolation (each agent owns its domain) but also coordination (agents must communicate). Too much isolation → duplication. Too much sharing → conflicts.
Symptoms: Agents overwrite each other's state. Or agents don't know about each other's work.
Mitigation: Use explicit handoff protocols. File-based handoffs for loose coupling. Task graphs (CC Mirror) for tight coordination. Filesystem ownership boundaries.
Optimal Compaction Timing
The tension: progress.txt must be compacted eventually but compacting too early loses useful learnings, too late causes context bloat.
Symptoms: Late iterations worse than early ones (too late). Repeated mistakes (too early).
Mitigation: No automation exists yet. Manual compaction when quality degrades. General heuristic: ~20KB or ~50 iterations, then archive.
