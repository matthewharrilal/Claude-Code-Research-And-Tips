Essence (15 words)
Async coordination enables parallel agents to share work without stepping on each other's files.

2 The Core Abstraction

"Agents that can't talk end up fighting over the same files."
register_agent() → reserve_files() → do_work() → release()
That's the whole pattern. Everything else is preventing collisions.

When multiple agents work on the same codebase asynchronously, they can overwrite each other's work. Agent A edits routes.py, Agent B edits routes.py, Agent B's commit overwrites Agent A's work. Hours of debugging follow. 

+===========================================================================+
|                    THE FILE CONFLICT PROBLEM                              |
+===========================================================================+
|                                                                           |
|  Time T0: Agent A starts editing src/api/routes.py                       |
|  Time T1: Agent B starts editing src/api/routes.py (unaware of A)        |
|  Time T2: Agent A commits their changes                                   |
|  Time T3: Agent B commits their changes                                   |
|           --> Agent B's commit OVERWRITES Agent A's work                  |
|           --> Both agents confused by unexpected diffs                    |
|           --> Human spends hours reconciling changes                      |
|                                                                           |
+===========================================================================+

The solution: coordination infrastructure. Agents register themselves, reserve files before editing, and release when done. Two paradigms exist: email-like (Agent Mail) for single-machine coordination, and message-queue (NATS/Station) for distributed systems.

3 Design Decisions

WHY ADVISORY LOCKS?
Traditional locks are enforced by the filesystem - if locked, you cannot proceed. Advisory locks are signals that agents CHOOSE to respect. This prevents deadlocks and allows graceful degradation.
What this means for you
File reservations are "Do Not Disturb" signs, not locked doors. Well-behaved agents check and respect them. If you need hard enforcement, add pre-commit guards.
WHY TTL LEASES?
Without TTL (Time-To-Live), a crashed agent holds reservations forever. TTL-based leases automatically expire after the configured duration, preventing permanent deadlocks.
What this means for you
Always set appropriate TTL. Too short = losing reservations mid-work. Too long = blocking others unnecessarily. 1 hour is typical for feature work.
WHY DUAL PERSISTENCE?
Agent Mail stores messages in both SQLite (for fast queries) and Git (for human-readable audit trails). This serves two audiences: agents need speed, humans need readability.
What this means for you
You can search messages programmatically via SQLite FTS5, or browse them as markdown files in the git repo. Both views are always in sync.

4 Agent Mail File Reservations

Agent Mail implements file reservations as TTL-based advisory leases. Reserve before editing, release when done. 

# Step 1: Register identity
register_agent(
    project_key="/my/project",
    agent_name="GreenCastle",       # Memorable adjective+noun
    program="Claude Code",
    model="Opus 4.5"
)

# Step 2: Reserve work surface
result = file_reservation_paths(
    project_key="/my/project",
    agent_name="GreenCastle",
    paths=["src/api/**/*.py"],      # Glob patterns work
    ttl_seconds=3600,              # 1 hour lease
    exclusive=True,                # No shared access
    reason="br-123"                # Link to task/issue
)

if result.conflicts:
    # Another agent holds conflicting reservation
    print(f"Conflict with: {result.conflicts}")
    # Options: Wait, work on different files, or request shared access

# Step 3: Do the work...

# Step 4: Release reservation
release_file_reservations(
    project_key="/my/project",
    agent_name="GreenCastle"
)

Message Threading

Agents communicate through threaded messages. All messages about a feature share a thread_id, creating traceable conversation trails. 

# All messages about feature br-123 share thread_id
send_message(
    project_key="/my/project",
    from_agent="GreenCastle",
    to_agents=["RedCat", "BlueLake"],
    thread_id="br-123",                    # Links to task/issue
    subject="[br-123] Start: Auth API",
    body="Implementation plan and status..."
)

# Later response in same thread
send_message(
    thread_id="br-123",                    # Same thread
    from_agent="RedCat",
    to_agents=["GreenCastle"],
    subject="RE: [br-123] Start: Auth API",
    body="Frontend ready for integration..."
)
Property Behavior
TTL-Based Automatically expires after ttl_seconds
Advisory Not enforced by filesystem, agents choose to respect
Glob Support Reserve src/api/**/*.py not just single files
Exclusive/Shared Exclusive blocks all others; shared allows readers
Traceable reason field links to task IDs for audit

6 Gotchas

Real problems you'll hit, with concrete numbers and fixes.

from_agent not registered
Symptom: Message send fails with registration error
Cause: Sending message before calling register_agent()
Fix: Always register_agent() first. It's a one-time call per session.
FILE_RESERVATION_CONFLICT
Symptom: Cannot reserve files, conflict returned
Cause: Another agent holds exclusive reservation
Fix: Wait for TTL expiry (check expires_in), use exclusive=False for reading, or work on different files.
Agent name rejected
Symptom: Registration fails with invalid name error
Cause: Name doesn't match adjective+noun pattern
Fix: Use names like GreenCastle, RedCat, SwiftFalcon. Not steve, claude, or agent123.
NATS connection refused
Symptom: Station member can't connect to lattice
Cause: Orchestrator not running or wrong port
Fix: Verify orchestrator: stn status. Default NATS port is 4222.

7 What's Hard

These are fundamental tensions, not bugs to fix.

Advisory vs Enforced
The tension: Advisory locks are flexible but rely on agent cooperation. Enforced locks prevent conflicts but can cause deadlocks.
Symptoms: Rogue agents ignoring reservations, or legitimate agents blocked by crashed holders.
Mitigation: Use pre-commit guards for enforcement layer. TTL handles crashed agents. Accept some risk for flexibility.
Polling vs Push
The tension: Agent Mail uses polling (higher latency, simpler). NATS uses push (real-time, more complex). No single best choice.
Symptoms: Delayed message handling with Agent Mail, or deployment complexity with NATS.
Mitigation: Use Agent Mail for same-machine coordination. Use NATS when sub-millisecond latency or cross-machine coordination is required.
Audit Trail Granularity
The tension: Fine-grained audit trails (every action) consume storage and slow queries. Coarse trails miss important details.
Symptoms: Can't debug coordination failures, or audit data overwhelming.
Mitigation: Thread-based logging captures conversation context. JetStream has configurable retention (default 7 days).

8 When to Use / When Not

USE ASYNC COORDINATION WHEN
Multiple agents editing same codebase (prevents file conflicts)
Need human-readable audit trails (Agent Mail's git storage)
Cross-machine agent teams (Station/NATS lattice)
Integration with Gas Town, Ralph, or CC Mirror (natural fit)
Long-running tasks requiring handoffs between agents
DON'T USE WHEN
Single agent workflow (no coordination needed)
Agents work on completely separate files (use git worktrees)
Need real-time synchronous coordination (use CC Mirror tasks)
Simple sequential Ralph loops (progress.txt is sufficient)
Infrastructure overhead not justified (single-machine, few agents)
