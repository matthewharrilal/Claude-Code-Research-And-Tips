2 The Extension Hierarchy

"Skills don't add new tools - they inject procedural knowledge about HOW to use existing tools effectively."
CLAUDE.md (always) → Skills (on-demand) → Commands (triggered) → MCP (persistent)
Four layers with different loading behaviors and use cases.
Layer 1: CLAUDE.md + Memory
Always loaded - Token overhead: Constant
Use for: Project context, universal instructions
Layer 2: Skills (SKILL.md)
On-demand loading - Token overhead: Per invocation
Use for: Domain expertise, procedural knowledge
Layer 3: Commands (/command)
User-triggered - Token overhead: Zero until invoked
Use for: Repeatable workflows, multi-step operations
Layer 4: MCP Servers
Persistent connections - Token overhead: Connection metadata
Use for: External capabilities, APIs, databases

The decision framework is simple: Do you need it in EVERY session? Put it in CLAUDE.md. Is it domain expertise loaded on-demand? Create a Skill. Is it a repeatable multi-step workflow? Create a Command. Is it an external capability? Use MCP.

Layer Comparison Matrix

Aspect CLAUDE.md Skills Commands MCP
Loading Always On-demand User-triggered Persistent

Best For Universal context Domain expertise Repeatable workflows External tools
Portability Per-project Cross-platform Per-installation Platform-specific

3 Design Decisions

WHY PROGRESSIVE LOADING?
Full skill loading costs 5,000-10,000 tokens even when you only need part of it. Progressive 3-tier loading (discovery → core → domain) typically saves 58% of tokens by loading only what's needed.
What this means for you
Tier 1 (discovery ~100 tokens) stays in AGENTS.md always. Tier 2 (core ~1500 tokens) loads when skill invoked. Tier 3 (domain ~500 each) loads only when that specific domain is needed.
WHY SKILLS VS CLAUDE.md?
CLAUDE.md is always loaded, consuming tokens even when irrelevant. Skills load only when triggered, preserving context budget for the actual task. Put permanent project context in CLAUDE.md; put specialized domain expertise in Skills.
What this means for you
If you find yourself adding 1000+ tokens of specialized instructions to CLAUDE.md that only apply sometimes, extract them into a Skill. The rule: CLAUDE.md = "always relevant", Skills = "sometimes relevant".
WHY COMMANDS?
Commands capture expert workflows as reusable packages. When you invoke /tools:security-scan, you're not just running a prompt - you're activating accumulated security expertise packaged into a repeatable process with OWASP coverage, multiple scanners, and structured output.
What this means for you
If you do something more than 3 times, make it a command. The wshobson/commands collection has 57 production-ready commands you can use immediately. Clone to ~/.claude/commands/ and they're available in every session.

6 Gotchas

Real problems you'll hit when using skills and commands:

Skill overload (too many loaded at once)
Symptom: Context budget exhausted, Claude forgetting instructions
Threshold: >3 skills loaded simultaneously = context bloat
Fix: Load skills on-demand, unload when done. Never load orchestration + gastown + dev-browser + zai-cli all at once.
Plugin validation errors
Symptom: "Schema validation failed", plugin won't load
Cause: plugin.json has $schema key, or SKILL.md YAML is malformed
Fix: Remove $schema from plugin.json. Validate SKILL.md frontmatter. Run npx @agentskills/skills-ref validate ./skill-path
Commands not recognized
Symptom: /workflows:* returns "unknown command"
Cause: Commands not cloned to correct location
Fix: Clone to ~/.claude/commands/ (exact path matters). Restart Claude Code session.
cc-mirror tasks not persisting
Symptom: Tasks created but lost between iterations
Cause: cc-mirror not initialized in project, or permissions issue
Fix: npm i -g cc-mirror && npx cc-mirror init in project root. Check write permissions on task storage.
Dev-browser connection refused
Symptom: Browser automation scripts fail with connection errors
Cause: Server not started, or not waited for "Ready" message
Fix: Start server: ./skills/dev-browser/server.sh &. Wait for "Ready" before running scripts. Try --headless flag.

7 What's Hard

These are fundamental tensions in the extension ecosystem:

Token Budget vs. Capability
The tension: More skills = more capability = less context budget for actual work. Every skill costs tokens even when sitting idle.
Symptoms: Claude forgets task details mid-execution. Quality degrades as conversation progresses. "Too many cooks" feeling.
Mitigation: Load ONE skill at a time. Unload before loading another. Use progressive disclosure (Tier 1 → 2 → 3). Monitor context usage with /context.
Skill Discovery Problem
The tension: For Claude to suggest the right skill, it needs to know what skills exist. But listing all skills wastes tokens.
Symptoms: Claude uses native capabilities when a skill would be better. You have to explicitly request skills you know about.
Mitigation: Keep AGENTS.md Tier 1 entries minimal but descriptive. Use trigger keywords that match natural language. Accept that you'll need to prompt explicitly sometimes.
Cross-Platform Compatibility
The tension: Skills work across Claude Code, Cursor, Copilot, etc. - but each platform has quirks. What works in Claude Code may break in Cursor.
Symptoms: Skills that work locally fail on other platforms. Tool permissions differ. MCP server behaviors vary.
Mitigation: Test on target platform before relying on it. Use agentskills.io spec for portability. Keep platform-specific workarounds documented.
Command Maintenance Burden
The tension: 57 commands sounds great - until you need to update them. Claude Code updates may break commands. Dependencies rot.
Symptoms: Commands that worked last month fail today. Output formats change. Agent coordination breaks.
Mitigation: Pin to specific command repository versions. Test critical workflows after Claude Code updates. Maintain your own fork if you customize heavily.

8 When to Use / When Not

USE SKILLS WHEN
Task requires domain expertise beyond general coding
Multi-agent orchestration is needed (orchestration skill)
Browser automation required (dev-browser skill)
Web research with vision needed (zai-cli skill)
Factory-scale agent coordination (gastown skill)
DON'T USE SKILLS WHEN
Native Claude Code capability suffices (don't over-engineer)
Context budget is already tight (add skill = lose context)
Single-file edit or simple task (skill overhead not worth it)
Exploratory work with no clear structure (skills want structure)
USE COMMANDS WHEN
Repeatable workflow you run 3+ times
Multi-step process with defined phases
Need consistent output format (security reports, etc.)
TDD workflow enforcement (/workflows:tdd-cycle)
DON'T USE COMMANDS WHEN
One-off task (command overhead not worth setup)
Need flexibility (commands enforce structure)
Simple query or explanation (just ask directly)
