Essence (15 words)
The complete lexicon defining every term in the Claude Code agent pattern language.

You Are Here: This is the dictionary for the Claude Code pattern language. When you encounter unfamiliar terms in other synthesis documents (like "fresh-context", "orchestrator", or "polecat"), look them up here. This document defines every primitive building block that appears across the research corpus.

Start here when terminology is confusing, or use it as a reference while reading more advanced documents like grammar-syntax.md or mastery-ralph-complete.md.

2 The Core Abstraction

"An agent that ORCHESTRATES must NEVER directly EXECUTE. An agent that EXECUTES must NEVER directly ORCHESTRATE."
This is THE IRON LAW. Every term in this vocabulary exists in service of this principle.
The Iron Law
An agent that ORCHESTRATES must NEVER directly EXECUTE.
An agent that EXECUTES must NEVER directly ORCHESTRATE.

Mixing these roles leads to context pollution, scope creep, verification bypass, and uncontrolled recursion. The vocabulary is organized around this separation: orchestrators coordinate, workers execute, and state containers bridge them.

3 Loop Constructs

These primitives define how autonomous agents iterate. The choice between them determines safety, cost, and termination behavior.

while-loop
Definition: Bash construct for indefinite iteration until an external condition is satisfied.
while :; do <command>; done
Use: Autonomous execution until completion signal detected.
Example: Original Ralph pattern - while :; do cat PROMPT.md | claude ; done
No inherent termination Requires external signal
for-loop
Definition: Bounded iteration with explicit maximum iterations.
for (( i=1; i<=$MAX_ITERATIONS; i++ )); do <command>; done
Use: Safe autonomous execution with guaranteed termination.
Example: Ryan Carson's Ralph - for (( i=1; i<=25; i++ )); do claude ... ; done
Predictable cost Safe for production
iteration
Definition: A single pass through the Ralph loop, from fresh context to commit/exit.
Structure:
  1. Read state files (prd.json, progress.txt)
  2. Pick next unpassed story
  3. Implement feature
  4. Run quality gates
  5. Commit if passing
  6. Update state files
  7. Check for COMPLETE or loop
Key Property: NEW agent instance each time
fresh-context
Definition: The principle that each iteration starts with zero accumulated context, relying on external state for continuity.
Rationale: Prevents context rot, lossy compaction, and quality degradation over long runs.
Contrast: Long-running sessions accumulate context until quality degrades at 85%+ usage.
Core Ralph principle
stop-condition
Definition: The signal that terminates the loop, indicating all work is complete.
<promise>COMPLETE</promise>
Detection:
if grep -q "<promise>COMPLETE</promise>" output.txt; then break; fi
Variants: PROMISE COMPLETE HERE, ALL TASKS DONE

4 State Containers

These files persist state across iterations. Each serves exactly one purpose. The agent remembers nothing; the filesystem remembers everything.

progress.txt
Definition: Append-only session memory file that accumulates learnings across iterations. 
# Progress Log - Append Only

## Codebase Patterns
- [Discovered pattern 1]
- [Discovered pattern 2]

---

## [Date] - [Story ID]
- What was implemented
- Files changed
- **Learnings:**
  - Patterns discovered
  - Gotchas encountered
Rules: APPEND ONLY - never edit previous entries. Patterns section at TOP for cross-session learning.
Session memory Ephemeral (per session)
prd.json
Definition: JSON task tracking file with machine-readable task states. 
{
  "project": "ProjectName",
  "branchName": "ralph/feature-name",
  "userStories": [
    {
      "id": "US-001",
      "title": "Task description",
      "acceptanceCriteria": ["Criterion 1", "typecheck passes"],
      "priority": 1,
      "passes": false
    }
  ]
}
Key Fields: passes: boolean (completion status), priority: number (lower = first)
Task board Deterministic selection
AGENTS.md
Definition: Permanent learnings document for humans and future agents.
Distinction from progress.txt:
  • progress.txt: Session-specific, ephemeral notes
  • AGENTS.md: Permanent, curated, project-level documentation
Rule: If a human will benefit from this in 6 months, it goes in AGENTS.md.
Permanent learnings Curated, not raw
CLAUDE.md
Definition: Project context file that onboards stateless agents to the codebase.
Best Practices:
  • Under 500 tokens
  • Essential information only
  • Verifiable facts over opinions
Agent onboarding <500 tokens

5 Agent Types

These define the roles agents can play. The Iron Law determines which tools each type may use.

orchestrator
Definition: Coordinating agent that distributes tasks but NEVER executes.
Characteristics:
  • NEVER writes code
  • NEVER runs bash commands
  • NEVER explores codebases directly
  • Uses: Read (1-2 files), Task*, AskUserQuestion
Metaphor: "The conductor. Your agents play the instruments."
Model: Opus Strategy, reasoning
worker
Definition: Executing agent that implements tasks but NEVER coordinates.
Characteristics:
  • ALWAYS writes code
  • Uses: Read, Write, Edit, Bash, Glob, Grep, WebFetch
  • NEVER spawns sub-agents
  • NEVER modifies task definitions
# Worker Preamble
CONTEXT: You are a WORKER agent, not an orchestrator.
RULES:
- Complete ONLY the task described below
- Use tools directly
- Do NOT spawn sub-agents
- Do NOT call TaskCreate/TaskUpdate
Model: Sonnet Implementation
subagent
Definition: Short-lived agent spawned by parent for specific isolated task.
Characteristics:
  • Own isolated context window
  • Returns summary to parent
  • Context doesn't pollute parent
  • Ephemeral - terminates after task
Task(description=..., run_in_background=True|False)
Isolated context Ephemeral

6 Common Confusions

Real terminology problems you'll hit when reading the research corpus:

Term Confusion: worker vs subagent vs polecat
Symptom: Using terms interchangeably when they have different meanings
Cause: Multiple source patterns (Gas Town, Ralph, Panopticon) use different terminology
Fix:
  • subagent = short-lived spawned agent (structural)
  • worker = role constraint: no coordination (behavioral)
  • polecat = named persistent worker (Gas Town-specific)
State File Purpose Confusion
Symptom: Putting session-specific notes in AGENTS.md or permanent patterns in progress.txt
Cause: Both files store learnings, but have different scopes and lifecycles
Fix:
  • progress.txt: Session-specific, ephemeral (delete after session)
  • AGENTS.md: Permanent, curated (persists across sessions)
  • Rule: If a human benefits in 6 months, use AGENTS.md
Acceptance Criteria Too Vague
Symptom: Agent marks tasks "complete" that don't actually work
Cause: Criteria like "works correctly" or "good UX" can't be verified
Fix: Every criterion must be automatable: 
# Bad
- "Handles edge cases"
- "Works correctly"

# Good
- "npm run typecheck passes"
- "Returns 401 on invalid credentials"
- "Column 'email' has unique constraint"
Iron Law Violation
Symptom: Orchestrator tries to write code, or worker spawns sub-agents
Cause: Role boundaries not enforced in prompts
Fix: Include explicit role constraints in agent preambles

7 Vocabulary Tensions

Fundamental tensions in the vocabulary itself, not bugs to fix:

Multiple Source Vocabularies
The tension: Ralph, Gas Town, Panopticon, and CC Mirror all contributed terminology. Sometimes they use different words for similar concepts.
Symptoms: "Worker" vs "polecat" vs "crew" confusion. "Progress file" vs "memory file" naming.
Mitigation: This document provides canonical definitions. When in doubt, check the source pattern context.
Evolving Terminology
The tension: The field is young. Terms shift as patterns mature. What's called "prd.json" in Ralph might become "task-board.json" in a future variant.
Symptoms: Different sources use different file names for same concept.
Mitigation: Focus on the ROLE (task tracking file), not the NAME (prd.json vs tasks.json).
Abstraction Level Mismatch
The tension: Some terms describe structure (subagent = spawned child), others describe behavior (worker = no coordination), others describe identity (polecat = named expert).
Symptoms: "Is a polecat a worker?" - Yes, behaviorally. But polecat emphasizes identity; worker emphasizes constraints.
Mitigation: Check the definition-tag metadata. Structure/behavior/identity are orthogonal dimensions.

Trade-Off Tension
while-loop vs for-loop
THE DILEMMA:
* while-loop: Runs until done, but no cost bound
* for-loop: Cost-bounded, but may not finish

WHY NO PERFECT ANSWER: You can't know in advance how many iterations a feature needs.

Heuristic: Use for-loop (25 iterations) for production. Use while-loop only when monitoring actively.
