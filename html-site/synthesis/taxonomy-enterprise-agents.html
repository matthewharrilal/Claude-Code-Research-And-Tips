3 Design Decisions

WHY DOMAIN VS METHODOLOGY?
Domain agents (99-Agents, Claude-007) answer "How do I build X?" They have deep technical expertise but no project structure. Methodology agents (BMAD) answer "What should I build next?" They provide SDLC structure but invoke domain expertise when needed.
What this means for you
Know what to build? Use domain catalog. Starting from scratch? Use BMAD methodology. Then combine: BMAD for phases 1-3, domain agents for phase 4 implementation.
WHY SPECIALIZE AGENTS?
Specialized agents produce higher quality output for their domain. A React expert knows hooks deeply; a generalist knows them superficially. Specialization also improves context efficiency: smaller, focused prompts leave more room for your code.
What this means for you
For quality-critical tasks (security, architecture), use specialists. For exploration or one-off tasks, generalists are fine. The rule: specialize when the task repeats OR when quality matters.
WHY MODEL TIERS?
99-Agents assigns agents to Opus/Sonnet/Haiku based on task complexity. Opus for critical architecture (42 agents), Sonnet for complex reasoning (39 agents), Haiku for fast execution (18 agents), Inherit for user preference (42 agents).
What this means for you
Don't use Opus for grep. Don't use Haiku for architecture. The 75% cost reduction claim comes from routing: simple tasks to WASM/Haiku, complex to Sonnet/Opus.

6 Gotchas

Real problems you'll hit when integrating agent ecosystems:

Multiple agents responding simultaneously
Symptom: Invoke one agent, get responses from multiple ecosystems
Cause: Multiple ecosystems active without clear orchestration rules
Fix: Establish orchestration in CLAUDE.md: "Only one agent active at a time unless explicitly multi-agent"
Agent persona wrong or generic
Symptom: Winston doesn't mention "boring technology," Mary doesn't act like a treasure hunter
Cause: Context pollution from previous agent, or agent file not loaded
Fix: Start fresh conversation, explicitly load: "Load the BMAD analyst agent Mary"
Context window exceeded with multiple agents
Symptom: Claude gets confused, loses track of conversation
Cause: Too many plugins loaded, all skills at Tier 3
Fix: Reduce installed plugins to essentials. Configure skills for on-demand loading.
Agents duplicating work
Symptom: Two agents implement the same feature differently
Cause: No shared state between ecosystems
Fix: Use BMAD workflow-status.yaml as shared state. Add anti-duplication protocol to CLAUDE.md.

7 What's Hard

Fundamental tensions in enterprise agent orchestration:

Specialization Depth vs. Context Efficiency
The tension: Deep specialists produce better output but consume more context. 108 agents loaded = no room for your code.
Symptoms: Claude forgets instructions, context fills with agent definitions instead of your codebase.
Mitigation: Load only needed agents. Use progressive disclosure (trigger phrases activate skills). Haiku for support tasks.
Ecosystem Lock-in vs. Best-of-Breed
The tension: Committing to one ecosystem is simpler but limits options. Mixing ecosystems creates orchestration complexity.
Symptoms: Need a capability that your chosen ecosystem lacks. Or: juggling three different agent invocation syntaxes.
Mitigation: Start with one ecosystem, add others for specific gaps. Document integration rules in CLAUDE.md so the complexity is managed.
Fixed Roster (BMAD) vs. Unlimited Scale (Swarms)
The tension: BMAD's 9 personas are clear but limiting. Swarms scales to 1000+ but requires manual topology selection.
Symptoms: BMAD: "I need a specialized agent that doesn't exist." Swarms: "Which of 14 topologies do I use?"
Mitigation: BMAD for SDLC structure, 99-Agents for domain depth. Swarms only when scale > 100 agents needed.
