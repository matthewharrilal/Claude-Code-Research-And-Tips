Essence (15 words)
Eight levels from interactive prompts to autonomous agent factories. Master each before advancing.

"Complexity is not arbitrary. Each level unlocks new capabilities that require the previous level's foundations."

The Complexity Ladder is a progression framework for Claude Code usage. It maps the journey from basic interactive sessions to full team-scale autonomous development. Each level represents a distinct capability plateau with specific prerequisites, patterns, and cost implications.

The ladder exists because Claude Code capabilities compound. Level 4 (PRD-driven loops) doesn't work if you haven't mastered Level 1 (CLAUDE.md). Level 7 (Gas Town) requires understanding all previous levels. Skipping levels leads to brittle implementations and wasted API credits.

3 Design Decisions

WHY 8 LEVELS?
The ladder has 8 levels because that's where natural capability boundaries fall. Each level introduces fundamentally new capabilities, not just incremental improvements. L0-L2 are single-agent foundations. L3-L4 introduce autonomous loops. L5-L7 introduce multi-agent coordination.
What this means for you
Self-assess honestly. Most developers think they're at Level 4 when they're actually at Level 1. The checkpoints at each level exist to prevent self-deception. If you can't pass the checkpoint, you haven't mastered the level.
WHY SEQUENTIAL?
Each level's patterns assume mastery of previous levels. Level 4 (PRD loops) assumes you understand external state (Level 2), which assumes you understand CLAUDE.md (Level 1). Skip a level and you'll build on a broken foundation.
What this means for you
Never skip levels. If Level 3 isn't working, the fix is usually at Level 1 or 2, not Level 3. Spend more time on foundations. The time investment table exists for a reason.
WHY COST MATTERS?
Cost scales non-linearly with level. Level 7 costs 10-40x more than Level 1 per month. If you're burning $800/month without Level 6 mastery, you're wasting money. Cost is a forcing function for discipline.
What this means for you
Match your budget to your level. Don't attempt Level 5+ until you can afford the API costs AND have the foundations to use them effectively. Level 3-4 is the sweet spot for most individual developers.

4 The 8 Levels

0
Single Interactive Session
Basic AI coding assistance
Basic Claude Code usage in a single terminal. No persistence between sessions. Manual approval for all operations. You're the driver, Claude is the copilot. Run claude, ask questions, approve each operation manually, session ends, context is lost.
Unlock: Basic AI coding assistance. Every session starts from scratch.
1
CLAUDE.md + Slash Commands
Consistent behavior, workflow shortcuts
Project context via CLAUDE.md files. Reusable workflows via slash commands in .claude/commands/. Claude "remembers" your project's architecture, coding standards, and preferences across sessions. 
# CLAUDE.md in project root
## Tech Stack
- TypeScript + React
- PostgreSQL with Drizzle ORM

## Commands
npm run dev       # Start dev server
npm run typecheck # TypeScript check
Unlock: Consistent Claude behavior. Commands take seconds instead of paragraphs.
2
Subagents + Hooks
Task delegation, automated workflows
Delegate to specialized agents with isolated contexts in .claude/agents/. Automate responses to lifecycle events with hooks in ~/.claude/settings.json. Claude becomes a small team rather than a single assistant. 
# .claude/agents/security-reviewer.yaml
name: Security Reviewer
tools: [Read, Grep, Glob]
prompt: |
  Analyze code for hardcoded secrets,
  SQL injection, XSS vulnerabilities.
Unlock: Specialized agents for specific tasks. Automatic formatting on file save.
3
Simple Loops (Basic Ralph)
Autonomous execution
Claude Code runs in a loop until a task is complete. Each iteration gets fresh context. External memory via git and files. Named after Geoffrey Huntley's "Ralph Wiggum" pattern. 
#!/bin/bash
while :; do
  claude "Check progress.txt, complete next task, update progress" | tee output.txt
  if grep -q "ALL TASKS COMPLETE" output.txt; then
    break
  fi
  sleep 5
done
Unlock: Autonomous execution. Come back to find tasks completed.
4
PRD-Driven Loops (Full Ralph)
Structured autonomous development
JSON-based task tracking with acceptance criteria, priority ordering, and quality gates. Full product requirement documents drive autonomous development. The prd.json structure with userStories, acceptanceCriteria, and passes fields. 
{
  "userStories": [{
    "id": "US-001",
    "title": "Add login form",
    "acceptanceCriteria": ["Email field renders", "npm run typecheck passes"],
    "priority": 1,
    "passes": false
  }]
}
Unlock: Real features shipped overnight. Learnings compound across iterations.
5
Multi-Agent Orchestration
Parallel specialized work
Central orchestrator delegates to specialized workers. Hub-and-spoke architecture. Agents coordinate via task management and file-based handoffs. Powered by CC Mirror or similar tools. Orchestrator spawns: Architect (Opus), Coders (Sonnet), Tester (Sonnet), Docs (Haiku). 
# Worker preamble template
CONTEXT: You are a WORKER agent, not an orchestrator.
RULES:
- Complete ONLY the task below
- Use tools directly (Read, Write, Edit, Bash)
- Do NOT spawn sub-agents
- Report results with absolute file paths
Unlock: Multiple agents working simultaneously on coordinated tasks.
6
Parallel Swarms + Worktrees
Massive parallelization
Git worktrees provide complete isolation for parallel agents. Multiple Ralph loops running simultaneously. Merge orchestration handles conflicts. True parallel development with git worktree add for each feature. 
# Create isolated worktrees for parallel agents
git worktree add ../wt-auth feature-auth
git worktree add ../wt-payments feature-payments
git worktree add ../wt-notifications feature-notifications

# Run Ralph in each worktree simultaneously
for feature in auth payments notifications; do
  (cd ../wt-$feature && ./ralph.sh 20) &
done
wait
Unlock: 10x productivity through true parallel development.
7
Full Agent Factory (Gas Town)
Team-scale productivity solo
Factory architecture for agent production. 7+ specialized roles working continuously. Self-improving systems that add capabilities. Human becomes Product Manager, agents become the engineering team. The 7 Roles: Mayor, Deacon, Dogs, Refinery, Polecat, Witness, Crew. 
# Gas Town installation
go install github.com/steveyegge/gastown/cmd/gt@latest
gt init
gt rig add ~/my-project

# Daily workflow
gt status            # Morning standup
gt feature add "Build invoice system"  # File new feature
gt inbox review      # Review completed work
Unlock: One person with Gas Town = small engineering team output.

5 Path of Progression

Moving through the ladder is not about time spent but about mastery demonstrated. Here's the flow:

1
Start at Level 0. Run claude, complete simple tasks, understand the approval flow.
2
When you find yourself repeating instructions, advance to Level 1. Create CLAUDE.md, make 3 slash commands.
3
When you need specialized analysis, advance to Level 2. Add a security-review subagent, add format-on-save hook.
4
When you want overnight execution, advance to Level 3. Write a simple bash loop, let it run unattended.
5
When you need structured acceptance criteria, advance to Level 4. Create prd.json, run Ralph on a real feature.
6
When you need parallelism, advance to Level 5. Set up CC Mirror, orchestrate 3 agents on one task.
7
When you need file isolation, advance to Level 6. Use worktrees, run 3 parallel Ralph loops.
8
When you want a persistent agent team, advance to Level 7. Install Gas Town, create your first rig, name your first polecat.
9
At each transition: Use the checkpoint checklist. If you can't pass, stay and practice.
Example: Level 4 Checkpoint
[ ] prd.json structure with userStories, acceptanceCriteria, passes fields
[ ] How to write atomic, verifiable tasks (2-3 sentence rule)
[ ] Quality gate enforcement (typecheck + tests before commit)
[ ] The <promise>COMPLETE</promise> termination signal

6 Gotchas

Common mistakes when applying this framework:

Skipping levels to reach the "cool stuff"
Symptom: Level 5 orchestration keeps failing in mysterious ways
Why: Without Level 1-2 foundations, your CLAUDE.md is missing critical context
Numbers: 90% of Level 5+ failures trace back to Level 1-2 gaps
Fix: Go back. Pass the checkpoints honestly. The foundations matter.
Confusing "have tried" with "mastered"
Symptom: "I've done Level 4 before" but can't write a prd.json from scratch
Why: Trying once is not mastery. Mastery is repeatable success without reference docs.
Fix: Use the checkpoints as tests. Can you do each item from memory? If not, you haven't mastered it.
Budget mismatch with level
Symptom: Running out of API credits mid-feature at Level 5+
Why: Higher levels burn tokens exponentially. Level 7 can hit $50/day easily.
Numbers: Level 7 is ~40x the cost of Level 1
Fix: Match budget to level. If you can't afford Level 5+ costs, stay at Level 4 until you can.
Overcomplicating early levels
Symptom: 500-line CLAUDE.md at Level 1, 10 subagents at Level 2
Why: More is not better. Each level has optimal complexity.
Numbers: CLAUDE.md should be <500 tokens. 1-3 subagents is plenty at Level 2.
Fix: Simplify. The best implementations at each level are surprisingly minimal.

7 What's Hard

Fundamental tensions in applying this framework:

Honest Self-Assessment
The tension: Everyone overestimates their level. The checkpoints feel tedious. Skipping feels faster. But skipping leads to failure.
Symptoms: Frustration at higher levels. "It should work but doesn't." Blaming the tools instead of the foundations.
Mitigation: Treat checkpoints as non-negotiable. Have a peer verify. Build something non-trivial at your claimed level.
Knowing When to Stay vs. Advance
The tension: Staying too long wastes potential. Advancing too fast wastes effort. There's no formula for "ready."
Symptoms: Repeated failures at new level (advanced too fast). Boredom but not advancing (stayed too long).
Mitigation: Three successful projects at current level before advancing. Not three attempts, three successes.
Level 7 is a Different Game
The tension: Level 7 (Gas Town) requires a mindset shift from "developer" to "product manager." Most developers resist this shift.
Symptoms: Trying to micromanage agents. Fighting the inbox pattern. Missing the philosophy.
Mitigation: Read Steve Yegge's posts. Understand: "You're a PM now, not an engineer." Accept that you won't understand the code your agents write.

8 When to Use / When Not

USE THIS FRAMEWORK WHEN
You want structured progression (not random exploration)
You're investing in Claude Code long-term (not one-off projects)
You want to avoid wasting API credits on premature complexity
You're building team capability, not just personal workflows
You want checkpoints to verify actual skill acquisition
DON'T USE THIS FRAMEWORK WHEN
One-off project, no repeat usage planned - just use Level 0-1
You already have working patterns - don't fix what works
Exploration mode, unclear goals - frameworks need direction
Budget-constrained and already effective at Level 2-3
Team with existing Claude Code culture - adapt, don't replace
