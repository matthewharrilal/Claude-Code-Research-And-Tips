Essence (15 words)
Workflow patterns are task management + verification + stop conditions orchestrating eventual consistency over perfect first attempts.

2 The Core Abstraction

"View AI agents as persistent, file-memory-based loops where AI acts as an autonomous worker, breaking projects into small, verifiable stories that compound over iterations."
Task → Verify → Persist → Loop
Every workflow pattern is a variation on this fundamental cycle.

The workflow taxonomy covers how to structure AI agent work from single-shot prompts to factory-scale swarms. Each level trades oversight for throughput, and the right choice depends on task verifiability, risk tolerance, and available infrastructure.

3 Task Management Approaches

Six approaches to structuring work for AI agents. Each trades structure for flexibility.

PRD-Driven Development
The dominant pattern for structured, multi-task autonomous work. 
{
  "project": "MyProject",
  "branchName": "feature/my-feature",
  "userStories": [
    {
      "id": "US-001",
      "title": "Task title",
      "acceptanceCriteria": [
        "Criterion 1",
        "npm run typecheck passes"
      ],
      "priority": 1,
      "passes": false
    }
  ]
}
Why JSON Over Markdown?
JSON parsing is deterministic. Markdown parsing is ambiguous. LLMs reliably update passes: true/false in JSON but often mangle markdown checkboxes.
Kanban-Style Mental Model
Treat tasks as cards. Agent picks from "todo", moves to "doing", then "done". 
# Matt Pocock's workflow description:
"Pick task from board -> Complete -> Back to board -> All done? Exit : Loop"

# Agent workflow:
1. Read prd.json
2. Find story with passes: false and highest priority
3. Skip if blocked by unfinished dependencies
4. Implement story
5. Run verification
6. If pass: set passes: true, commit
7. If fail: add notes, try again
8. Repeat until all passes: true
Goal-Conditioned Loops
Define the end state, let agent loop until it reaches it. 
#!/bin/bash
# Ralph Wiggum Implementation
while :; do
  output=$(claude @prompt.md)
  if echo "$output" | grep -q "<promise>COMPLETE</promise>"; then
    echo "Goal reached!"
    break
  fi
done

# jq-based alternative (Maurice Kleine)
while jq '.tasks[] | select(.passes == false)' prd.json | grep -q .; do
  claude @prompt.md
done
Phase-Based Development
Break large features into sequential phases with gates. 
{
  "epic": "User Authentication System",
  "phases": [
    {
      "phase": 1,
      "name": "Database Foundation",
      "prd_file": "plans/auth-phase1.json",
      "gate": ["Schema exists", "Migrations run"]
    },
    {
      "phase": 2,
      "name": "Backend Logic",
      "blockedBy": "phase1",
      "gate": ["All API endpoints return expected status codes"]
    },
    {
      "phase": 3,
      "name": "Frontend Integration",
      "blockedBy": "phase2",
      "gate": ["E2E login flow passes"]
    }
  ],
  "total_budget": "$50"
}

Verification Types

Negative Verification
"Does not modify files outside {directory}"
"Does not add new dependencies without approval"
"Does not increase bundle size > 5KB"
"No console errors"
Property-Based
"Output length equals input length"
"All items from input appear in output"
"Adjacent items satisfy: a.date <= b.date"
"Idempotent: sort(sort(x)) === sort(x)"
Approval Testing
Generate output → Compare to approved baseline → Present diff if different → Human approves → Update baseline
Screenshot Validation
Navigate → Wait for load → playwright.screenshot() → Compare to expected state → Log to progress.txt

5 External Memory Systems

"Each iteration spawns a NEW Claude instance with NO memory of previous work."
Continuity comes entirely from external sources.
State Type Storage Update Pattern
Code changes Git history Commit each iteration
Task status prd.json Update passes: true/false
Learnings progress.txt APPEND only
Context CLAUDE.md Rarely updated
Project structure File system Modified as needed
progress.txt Pattern
# Use this to leave a note for the next person working on your codebase.

---

Iteration 1 (2026-01-09 10:00):
- Completed: US-001 - Tool skeleton created
- Learned: Use orange (#FF6B35) for visibility on dark backgrounds
- Next: Consider beat playback timing

---

Iteration 2 (2026-01-09 10:15):
- Completed: US-002 - Category mapping
- Issue encountered: Zod schema needed adjustment for optional fields
- Pattern discovered: Always use .optional() explicitly
Critical Rule: NEVER UPDATE progress.txt
Why: Updating would overwrite history. Always APPEND new entries.
What to log: What was completed (reference story ID), what was learned (patterns, gotchas), what blocked or failed, what to do next.

CLAUDE.md for Ralph Projects

# Project Name

## Ralph Loop Configuration
This project uses the Ralph Wiggum pattern for autonomous development.

### PRD Location
- Task tracking: `plans/prd.json`
- Progress log: `plans/progress.txt`

### Verification Requirements
For any story to be marked passes: true:
1. `npm run typecheck` must pass
2. `npm run test` must pass
3. `npm run lint` must pass

### Workflow
1. Read prd.json to find next task with passes: false
2. Implement the feature following existing patterns
3. Run verification commands
4. If all pass, mark passes: true and commit
5. Append learnings to progress.txt
6. If ALL tasks complete, output: <promise>COMPLETE</promise>

6 Multi-Agent Coordination

Five patterns for coordinating multiple Claude instances.

Hub-and-Spoke (Gas Town)
Central orchestrator decomposes and distributes work.
Hub: Decomposes tasks
Spokes: Execute independently
Hub: Merges results
Peer-to-Peer (CC Mirror)
Agents coordinate via TaskCreate/TaskUpdate.
No central orchestrator. Blocking relationships coordinated through shared task system.
Assembly Line
Sequential handoff between specialized agents.
Design → Develop → Test → Deploy
Each agent specializes. Output becomes next input.
Parallel Ralph (Worktrees)
True isolation with git worktrees.
git worktree add ../wt-$feature $branch
No file conflicts. Shared git history. Merge at end.

Parallel Ralph Script

#!/bin/bash
# Run multiple Ralph loops in parallel

for feature in auth payments notifications; do
  git worktree add ../wt-$feature $feature-branch
  (cd ../wt-$feature && ./ralph.sh 20) &
done
wait

# Merge results
for feature in auth payments notifications; do
  git merge wt-$feature
done

Stop Condition Patterns

# 1. Promise Tag
if grep -q "<promise>COMPLETE</promise>" output.txt; then break; fi

# 2. JSON State Check
while jq '.tasks[] | select(.passes == false)' prd.json | grep -q .; do
  # continue
done

# 3. Max Iterations
MAX_ITERATIONS=25
while [ $CURRENT_ITERATION -le $MAX_ITERATIONS ]; do ... done

# 4. Stuck Detection (no git commits for 3 iterations)
if [ "$CURRENT_COMMIT" = "$LAST_COMMIT" ]; then
  STUCK_COUNT=$((STUCK_COUNT + 1))
  if [ $STUCK_COUNT -ge 3 ]; then exit 1; fi
fi

# 5. Time-Based
MAX_SECONDS=3600  # 1 hour
if [ $ELAPSED -gt $MAX_SECONDS ]; then break; fi

7 Gotchas and What's Hard

Real problems you'll hit, with concrete numbers and fixes.

Agent Marks Task Complete But It's Not Done
Symptom: Story marked passes: true but feature doesn't work
Cause: Missing verification steps, or acceptance criteria too vague
Fix: Always include npm run typecheck passes. Add specific, testable criteria. Use browser verification for UI tasks. Add negative criteria.
Loop Gets Stuck on Same Story
Symptom: Same story attempted repeatedly across iterations
Cause: Tests failing, or criteria impossible to meet
Fix: Check progress.txt for failure pattern. Verify tests actually work. Split story if too large. Add stuck handling: "After 3 failed attempts, add BLOCKED note and move to next"
Cost Spiraling Out of Control
Symptom: Ralph loop consuming much more than expected
Cause: Too many iterations, wrong model, or tasks too large
Fix: Set max iterations (./ralph.sh 25). Use model stepping (start Haiku, escalate if needed). Right-size tasks. Use subagents for expensive operations.
Fresh Context Loses Important Information
Symptom: New iteration doesn't know critical context from previous
Cause: Not capturing learnings in progress.txt or CLAUDE.md
Fix: End each iteration with progress.txt update: what was completed, what was learned, what to do next. Update CLAUDE.md with discovered patterns.

What's Hard (Fundamental Tensions)

Context Rot vs. Memory Loss
The tension: Fresh context avoids rot but loses learnings. progress.txt helps but grows unbounded. No perfect solution exists.
Symptoms: Quality drops in late iterations (rot), OR early iterations repeat mistakes (memory loss).
Mitigation: Keep progress.txt under 20KB. Start fresh runs from AGENTS.md (permanent), not progress.txt (session).
Verification Dependency
The tension: Ralph only works if acceptance criteria are verifiable. But not everything can be verified automatically.
Symptoms: Tasks marked "complete" that aren't actually complete. Garbage compounding.
Mitigation: Don't use Ralph for subjective tasks. Use HOTL variant for tasks requiring judgment.
Task Decomposition Skill
The tension: Ralph's success depends on your ability to decompose features into iteration-sized tasks. This is a skill, not a formula.
Symptoms: Tasks too big = context overflow. Tasks too small = overhead dominates. Wrong order = broken dependencies.
Mitigation: Start with HOTL Ralph to learn task sizing. Review progress.txt after runs to see what worked.

Task Sizing Heuristics

Size Lines Files Iterations Example
XS 1-20 1 1 Add column, fix typo
S 20-100 1-2 1 New component, endpoint
M 100-300 2-5 1-2 Feature piece, integration
L 300-1000 5-10 3-5 Small feature, refactor
XL 1000+ 10+ 5+ MUST SPLIT
The 2-3 Sentence Rule
"If you can't describe the task in 2-3 sentences, it's too big." Split triggers: >40K tokens estimated, >10 files modified, >5 test runs expected, >3 MCP tool sequences.

8 Decision Guide

Match workflow to situation. The right pattern depends on verifiability, risk, and available infrastructure.

Scenario Recommended Workflow Key Factors
Simple bug fix Single-shot Clear solution, low risk
Explore codebase Interactive Discovery, learning
Well-defined feature Ralph loop Clear criteria, testable
Multi-component feature PRD phases Dependencies, milestones
Team collaboration CC Mirror + TaskCreate Coordination, visibility
Enterprise migration Gas Town Scale, resources
Security-sensitive HOTL Ralph Human review required
Time-sensitive Parallel Ralph Speed over efficiency
Budget-constrained Model stepping Cost optimization
USE AUTONOMOUS WORKFLOWS WHEN
Tasks have verifiable acceptance criteria
Overnight execution is acceptable
Feature can be split into <30min subtasks
You have tests/typecheck as verification layer
You want to "ship while you sleep"
DON'T USE AUTONOMOUS WHEN
Subjective quality goals ("make it pretty")
Security-critical code without HOTL gates
Exploratory work with no defined end
Legacy code without tests
Design decisions (agent implements, humans decide)
