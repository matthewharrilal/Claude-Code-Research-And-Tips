Core Concepts

Context Window

The limited amount of text (tokens) an LLM can process at once. Claude's nominal context is ~200K tokens, but effective quality work happens in ~100K tokens.

Why it matters: Context is the primary constraint in agent engineering. As context fills, quality degrades ("context rot").

fundamental constraint
Context Rot

The degradation of LLM performance as the context window fills up. Symptoms include: repetition, forgetting earlier work, ignoring instructions, and hallucination.

"Context rot: LLMs get stupider with more tokens." - Matt Pocock

Source: Community observation, Matt Pocock
warning fundamental
CLAUDE.md

A markdown file automatically loaded into Claude Code's context at session start. Contains project standards, architecture decisions, and coding conventions.

Locations (in loading order):

  • ~/.claude/CLAUDE.md - Global user preferences
  • ./CLAUDE.md - Project-level (git-checked)
  • ./.claude/rules/*.md - Path-targeted rules
  • ./CLAUDE.local.md - Personal/secret (gitignored)
Source: Boris Cherny (Claude Code creator), Official docs
essential memory
MCP (Model Context Protocol)

An open-source standard created by Anthropic for connecting AI applications to external systems. Think of it as "USB-C for AI".

MCP provides:

  • Tools - Functions LLMs can actively call
  • Resources - Passive read-only data sources
  • Prompts - Pre-built workflow templates
Source: Anthropic official, modelcontextprotocol.io
protocol integration
Extended Thinking
Also: think, think hard, ultrathink

Trigger words that activate deeper reasoning in Claude. When included in prompts, Claude spends more tokens on reasoning before responding.

  • think - Basic extended reasoning
  • think hard - More deliberate reasoning
  • ultrathink - Maximum reasoning depth
Source: Community discovery, Anthropic internal
technique reasoning
Subagent

A spawned Claude instance that operates in its own isolated context window. Subagents protect the main agent's context from expensive operations.

Key benefit: Expensive operations (browser automation, large file searches) happen in isolated contexts, only returning summaries to main context.

Source: Boris Cherny, @TendiesOfWisdom

Patterns

Ralph Wiggum Loop
Also: Ralph Loop, Ralph Pattern, AFK Ralph

A persistence pattern for AI coding agents that achieves reliability through repetition and external memory.

while :; do cat PROMPT.md | claude ; done

Philosophy:

  • Fresh Context > Accumulated Context
  • Failure is Data, not Defeat
  • Small + Verifiable = Reliable

Named after The Simpsons character - the simple-minded child who persists at tasks regardless of obstacles.

Source: Geoffrey Huntley (creator), Matt Pocock, Ryan Carson

Gas Town

A factory for coding agents - Steve Yegge's multi-agent orchestration system with 8 specialized roles.

"Claude Code is the world's biggest fuckin' ant. Everyone is focused on making their ant run longer... when nature prefers colonies." - Steve Yegge

Key concept: Human as Product Manager, AI as Implementation Army.

Warning: Stage 7+ only, expensive, requires chaos tolerance.

Source: Steve Yegge (@steve_yegge)

CC Mirror

A reverse-engineered Claude Code distribution that enables hidden multi-agent orchestration primitives that Anthropic built but disabled.

Enabled tools:

  • TaskCreate - Create tasks with dependencies
  • TaskList - Filter open tasks
  • TaskUpdate - Claim, comment, resolve
  • addBlocks/addBlockedBy - Dependency management

Hub-and-Spoke

A multi-agent architecture where a central orchestrator delegates to specialized workers.

+------------------+ | ORCHESTRATOR | +--------+---------+ | +---------------+---------------+ | | | v v v +------+ +------+ +------+ |Worker| |Worker| |Worker| +------+ +------+ +------+

Key rule: Orchestrator NEVER writes code - it delegates.

Source: CC Mirror, multi-agent patterns

Compounding

The pattern of accumulating learnings across iterations so each run becomes better than the last.

Mechanism: Archive previous runs, reference past successes/failures, apply learnings to current implementation.

"What's amazing is I can ask it to look at previous runs to learn what worked or didn't work. So every time I run Ralph, it gets better and better." - Ryan Carson

Source: Ryan Carson (@ryancarson)

Techniques

/clear Habit

The practice of regularly clearing context to prevent context rot. Rather than extended sessions, prefer frequent fresh starts.

When to clear:

  • After completing a task
  • When context > 70%
  • When quality degrades
  • Between different tasks
Source: Boris Cherny, community best practice
workflow essential
thoughts/ Directory

A pattern for externalizing Claude's reasoning to files. By writing thoughts to files, the agent can preserve reasoning across context clears.

thoughts/ ├── 001-initial-analysis.md ├── 002-architecture-decision.md ├── 003-implementation-plan.md └── 004-review-findings.md

Benefit: Converts ephemeral context into persistent memory.

Source: Community pattern
memory workflow
Subagent Spawning

Delegating expensive operations to isolated subagents to protect main context window.

Use for:

  • Browser automation (Playwright)
  • Large file searches
  • Web research
  • Database queries

"Put costly tools like browser control in subagents to protect your main context window tokens." - @TendiesOfWisdom

Source: @TendiesOfWisdom, Boris Cherny
architecture optimization
Progress.txt Pattern

An append-only log file for cross-iteration learning in Ralph loops.

# progress.txt - APPEND ONLY --- Iteration 1 (2026-01-09 10:00): - Completed: US-001 - Tool skeleton - Learned: Use .optional() for Zod - Files: src/tools/upsert.ts

Critical rule: NEVER update - only append. The word "APPEND" in instructions is essential.

Source: Ralph pattern, Ryan Carson
memory ralph
prd.json

A JSON file tracking task completion state in Ralph loops.

{ "userStories": [ { "id": "US-001", "title": "Add login form", "acceptanceCriteria": ["npm run typecheck passes"], "passes": false } ] }

Why JSON: Machine-parseable, unambiguous state, smaller context footprint.

Source: Ralph pattern, Anthropic practices

Gas Town Roles (8 Specialized Agents)

Overseer (YOU)

Type: Human operator | Scope: Town-wide

The human authority in Gas Town. You are the Product Manager, decision-maker, and ultimate arbiter of quality.

Responsibilities: Define features, approve decisions, manage budgets, handle escalations.

"You're not a programmer anymore. You're the head chef of a kitchen with AI robotic sous chefs."

Mayor

Type: AI Agent | Scope: Town-level | Model: Opus 4.5

Your concierge and chief-of-staff. Receives directives, decomposes work, spawns agents, summarizes results.

Minimum viable Gas Town: You only need the Mayor to start.

gt mayor up
Deacon

Type: AI Agent (Patrol) | Scope: Town-level | Model: Sonnet

The "daemon beacon" - Gas Town's central heartbeat. Runs patrol workflows, manages worker recycling, broadcasts "Do Your Job" signals.

Warning (v0.3.x): Known bug causes Deacon to "spree-kill" workers. Run without until v0.4.0.

Dogs

Type: AI Agent (Workers) | Scope: Town-level | Model: Sonnet

The Deacon's personal crew - town-level grunts handling maintenance. Branch cleanup, handyman tasks, plugin execution.

Includes Boot: A special Dog that monitors Deacon health every 5 minutes.

Refinery

Type: AI Agent (Patrol) | Scope: Rig-level | Model: Opus 4.5

The intelligent merge queue manager. Merges polecat changes sequentially to main, prevents conflicts, escalates unresolvable issues.

Essential when: Running 3+ polecats generating PRs.

Witness

Type: AI Agent (Patrol) | Scope: Rig-level | Model: Haiku

The supervisor and troubleshooter for polecat swarms. Monitors health, unsticks blocked workers, runs rig-level plugins.

"Once you spin up enough polecats, you need an agent just to watch over them and help them get un-stuck."

Polecat

Type: AI Agent (Ephemeral) | Scope: Rig-level | Model: Sonnet | Count: 3-30

Ephemeral workers generating pull requests. Complete assigned implementation, produce code, submit to Refinery, then decommission.

Naming: Use human names (alice, bob, carol) for mental clarity.

gt polecat up <rig> alice
Crew

Type: AI Agent (Persistent) | Scope: Personal | Model: Sonnet/Opus

Long-lived personal coding agents. Unlike polecats, they have persistent identities, work directly for Overseer, support back-and-forth collaboration.

Use for: Sustained design work, architecture, exploratory coding.

Tools & Plugins

Claude-Mem

A plugin providing automatic session capture, semantic compression, and retrieval injection for persistent memory.

Stats: Up to 95% fewer tokens per session, ~20x more tool calls.

Components: SQLite DB, Chroma Vector DB, Claude Agent SDK compression.

Source: @thedotmack
plugin memory
Beads

Steve Yegge's git-backed universal data plane - the foundation of Gas Town. ~225K lines of Go.

Layers: Beads (atomic units) - Epics (parent-child) - Molecules (chains) - Protomolecules (templates) - Formulas (TOML definitions).

Source: github.com/steveyegge/beads
gas-town data-plane
Hooks

Claude Code's lifecycle hooks for automation. Trigger actions at key points: session start, tool use, completion.

Types:

  • PreToolUse - Before a tool runs
  • PostToolUse - After a tool completes
  • SessionStart - When session begins
  • Stop - When agent stops
Source: Claude Code official

Core Principles (The 8 Laws)

1. Context Is The Primary Constraint
All patterns optimize around finite context windows. Every decision must answer: "How does this affect context consumption?"
2. External State > Internal Memory
Files persist, context evaporates. Write state to files immediately; never trust context for continuity.
3. Fresh Context > Extended Sessions
New instances beat long conversations. Extended sessions guarantee degradation; fresh context guarantees consistent quality.
4. Separation of Concerns
Orchestrators coordinate, workers execute. Mixing causes context pollution and role confusion.
5. Atomic, Verifiable Tasks
Complete in one iteration, verify before commit. If you can't describe it in 2-3 sentences, it's too big.
6. Quality Gates Before Commit
Tests must pass, no broken code inheritance. Broken code propagates; working code accumulates.
7. Isolation Prevents Contamination
Own your space, don't conflict. Shared state causes conflicts; isolation enables parallelization.
8. Explicit Communication
File-based handoffs, not implicit coordination. LLMs don't share memory - write state to files.
