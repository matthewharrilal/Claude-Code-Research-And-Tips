Planning vs Execution Separation

A core architectural pattern for long-running agent workflows: use a high-reasoning model to plan and decompose goals, then switch to an execution-focused model that grinds through queued tasks. The filesystem becomes your stateful queue for resilience and oversight.

Platform: X (via Codex CLI user)
Date: ~January 2026
Context: Codex CLI workflow, principles transfer directly to Claude Code

Core Insight

View the AI as a specialized dev team. Use a high-level planner model to map codebases and decompose goals into persistent, auditable files (scope, rules, tasks). Then switch to an execution-focused model as a disciplined "worker" that grinds through queued tasks autonomously.

The filesystem becomes a stateful queue for resilience and oversight.

Two-Phase Architecture

The separation creates a clear handoff point between strategic thinking and tactical execution.

Phase 1

Planning Phase

High-reasoning model (Opus in plan mode)

  • Map codebase holistically - understand architecture
  • Decompose goals into atomic tasks
  • Generate artifacts - scope.md, rules.md, task queue
  • Avoid questions - just decompose, don't clarify
Filesystem Queue
scope.md, rules.md, tasks/
Phase 2

Execution Phase

Execution-focused model (Sonnet/Haiku workers)

  • Strict tool-using implementation
  • Follow rules.md guardrails without deviation
  • Process queued tasks atomically (15-60 min each)
  • Commit, test, push after each task
Checkpoint: Understanding the Separation

You should understand:

  • Planning phase outputs FILES, not actions
  • Execution phase reads FILES, not history
  • The filesystem is the handoff mechanism
  • Each phase uses a different model (or model mode)

Key Principles

1. Separate Planning from Execution

  • Planning: Holistic, question-avoiding decomposition
  • Execution: Strict, tool-using implementation
  • Don't mix them - leads to drift or excessive clarifications
Checkpoint: Why Separation Matters

When you mix planning and execution:

  • The planner gets distracted by implementation details
  • The executor asks too many clarifying questions
  • Context fills with noise instead of progress
  • Sessions drift from the original goal

2. Enforce Guardrails via rules.md

Example rules for the executor:

  • Always branch before changes
  • Test fully before committing
  • Reduce bloat in legacy projects
  • Commit and push after each task

3. Atomic Tasks (15-60 minutes)

  • Tasks should be completable in 15-60 minutes
  • Minimizes drift
  • Enables resumability
  • Prevents scope creep

4. Filesystem as State

  • Queue prompts in files for multi-hour runs
  • Survives interruptions
  • Enables auditing
  • Manual re-queue on failures

5. Reliability > Speed

  • Prioritize cleaner, functional code over speed
  • Let the agent grind through tasks
  • CI-grade output is the goal

Your CLAUDE.md should have:

  • Project understanding section (what the planner produced)
  • Decomposed goals as a task list
  • Clear rules section for the executor
  • Current task focus (what to work on now)

What Can Go Wrong

Poorly defined goals lead to suboptimal task breakdowns

Symptom: Executor produces work that doesn't align with your intent.

Cause: Planning phase wasn't thorough enough in decomposition.

Fix: Spend more time in planning. Have the planner write out acceptance criteria for each task. Review the plan before starting execution.

Long runs (24+ hours) require stable environments

Symptom: Tasks fail partway through due to environment issues.

Cause: Network interruptions, API rate limits, system restarts.

Fix: Design for resumability. Each task should checkpoint progress. Use filesystem state so you can restart from where you left off.

Large codebases strain context limits during planning

Symptom: Planner can't hold full codebase understanding.

Cause: >100k LOC exceeds practical context limits.

Fix: Have planner focus on relevant subsystems. Build incremental understanding. Use existing documentation as summaries.

Interruptions handled via resumability, but need manual re-queue

Symptom: Agent stops mid-task and doesn't resume.

Cause: Context loss, session timeout, user interrupt.

Fix: Check progress.txt or git log to see what was completed. Update task queue to mark completed work. Restart executor on next task.

Non-atomic tasks cause scope creep and failures

Symptom: Single task takes hours and drifts from goal.

Cause: Task was too broad, couldn't fit in one focused session.

Fix: Break tasks into 15-60 minute chunks. If you can't describe it in one paragraph, it's too big. Decompose further.

Don't Do This

Using execution model for planning

"It sucks at it and asks too many questions." The execution model wants to ACT, not THINK. Don't ask it to strategize.

Skip tests or delete them to pass

Defeats the purpose of CI-grade output. If tests fail, fix the code, don't disable verification.

Making tasks too broad

Causes scope creep and failures. A task should be completable without asking questions.

Neglecting to review generated files

Planning phase output needs human review before execution begins. The plan becomes the contract.

Agentic engineering in 2025-2026: shifting from prompt engineering to structured workflows with filesystem-based state management for resilience. The pattern works because:

  • Files survive context resets
  • Git provides audit trail
  • Separation matches model strengths
  • Atomic tasks enable parallelization
