Superwhisper Code Mode

Superwhisper has a "code mode" that:

  • Preserves technical terminology
  • Handles camelCase/snake_case
  • Recognizes programming keywords
# Example dictation:
"create a function named get user by id that takes an integer
id parameter and returns a user object"

# Superwhisper outputs:
"Create a function named getUserById that takes an integer
id parameter and returns a User object"

iOS Shortcuts Voice Integration

Shortcut: "Hey Claude"

  1. Trigger: Voice (Siri)
  2. Get text input: (dictation)
  3. Run shell script over SSH: ssh devuser@claude-vm "echo '$input' | claude --print"
  4. Speak result

Limitations: Siri dictation quality varies, SSH timeout may occur, long responses truncated.

Bidirectional Voice Script

#!/bin/bash
# voice-claude.sh - Full voice loop

# Capture voice input (requires Superwhisper or similar)
VOICE_INPUT=$(superwhisper --capture --timeout 30)

# Send to Claude
RESPONSE=$(echo "$VOICE_INPUT" | claude --print)

# Speak response
say "$RESPONSE"  # macOS
# espeak "$RESPONSE"  # Linux

Offline/Intermittent Connectivity Strategies

Mobile connectivity is unreliable: airplane mode, spotty cellular, roaming data limits. Here's how to work around it.

Strategy 1: Queue-Based Development

Prepare tasks offline, execute when connected.

# task-queue.md - Prepare while offline

## Pending Tasks
1. [ ] Add input validation to form component
2. [ ] Write tests for auth module
3. [ ] Refactor database queries

## Ready to Execute
When connected, run:
```bash
claude "Complete the tasks in task-queue.md"
```

Strategy 2: Pre-Caching Project Context

#!/bin/bash
# cache-for-offline.sh - Run before going offline

CACHE_DIR="$HOME/.claude-offline/$(basename $PWD)"
mkdir -p "$CACHE_DIR"

# Cache all source files
find . -name "*.ts" -o -name "*.tsx" -o -name "*.py" | \
  xargs -I{} cp --parents {} "$CACHE_DIR/"

# Cache project structure
tree -L 4 > "$CACHE_DIR/structure.txt"

# Cache recent git history
git log --oneline -100 > "$CACHE_DIR/git-log.txt"
git diff HEAD~10 > "$CACHE_DIR/recent-changes.diff"

# Cache CLAUDE.md and context files
cp CLAUDE.md "$CACHE_DIR/" 2>/dev/null || true

echo "Cached $(find $CACHE_DIR -type f | wc -l) files"

Strategy 3: Local Models for Degraded Mode

# Install Ollama (do this while online!)
curl -fsSL https://ollama.com/install.sh | sh

# Pull capable models
ollama pull codellama:34b
ollama pull deepseek-coder:33b
ollama pull qwen2.5-coder:32b

# Offline usage
ollama run qwen2.5-coder:32b "Refactor this function: $(cat function.ts)"
Local Model Limitations
  • Lower reasoning capability than Claude
  • No tool use or file editing
  • Best for: code review, explanations, simple edits

Strategy 4: mosh for Poor Connections

mosh handles poor connections better than SSH:

  • Local echo (type before network confirms)
  • UDP-based (survives connection changes)
  • Automatic reconnection
# Connect via mosh instead of SSH
mosh --ssh="ssh -p 47892" devuser@100.x.x.x

# Low-bandwidth tmux settings (~/.tmux.conf)
set -g status-interval 30  # Less frequent updates
set -g visual-activity off
set -g visual-bell off

Airplane Mode Development Pattern

The Pattern

  1. Before flight: Cache project, prepare task list
  2. During flight: Use local model for thinking/drafting
  3. On landing: Execute cached tasks with Claude

Checkpoint: Offline Strategies

  • cache-for-offline.sh script created and tested
  • At least one local model (Ollama) installed
  • mosh configured as primary connection method
  • task-queue.md template saved for offline use

Security Hardening for Mobile Access

Mobile access increases attack surface. Here's how to lock it down properly.

Advanced SSH Configuration

# /etc/ssh/sshd_config - Hardened for mobile access

# Only listen on Tailscale interface
ListenAddress 100.x.x.x

# Strong algorithms only
KexAlgorithms curve25519-sha256@libssh.org
Ciphers chacha20-poly1305@openssh.com,aes256-gcm@openssh.com
MACs hmac-sha2-512-etm@openssh.com

# Prevent brute force
MaxAuthTries 3
MaxSessions 5
LoginGraceTime 30

# Logging for audit
LogLevel VERBOSE

# Restrict to specific user
AllowUsers devuser

# Require key authentication
PasswordAuthentication no
PermitRootLogin no

2FA with SSH (Google Authenticator)

# Install PAM module
apt install libpam-google-authenticator

# Configure per user
google-authenticator

# Add to /etc/pam.d/sshd:
auth required pam_google_authenticator.so

# Update /etc/ssh/sshd_config:
AuthenticationMethods publickey,keyboard-interactive

API Key Security Pattern

Never store API keys on mobile device

Your phone can be lost or stolen. Keep keys on the VM only.

# On VM: Store key securely (macOS)
security add-generic-password -a "$USER" -s "anthropic-api" -w "your-key"

# In .zshrc (VM only)
export ANTHROPIC_API_KEY=$(security find-generic-password \
  -a "$USER" -s "anthropic-api" -w)

# Phone never sees the actual key

Tailscale ACLs for Access Control

// Tailscale ACL policy
{
  "acls": [
    {
      "action": "accept",
      "src": ["tag:phone", "tag:laptop"],
      "dst": ["tag:claude-vm:22"]
    },
    {
      "action": "accept",
      "src": ["tag:claude-vm"],
      "dst": ["*:443"]
    }
  ],
  "tagOwners": {
    "tag:phone": ["autogroup:admin"],
    "tag:laptop": ["autogroup:admin"],
    "tag:claude-vm": ["autogroup:admin"]
  }
}

Audit Logging for Remote Sessions

#!/bin/bash
# /etc/profile.d/session-audit.sh

if [[ -n "$SSH_CONNECTION" ]]; then
  LOGFILE="/var/log/ssh-sessions.log"
  echo "[$(date)] Session start: $USER from $SSH_CLIENT" >> "$LOGFILE"

  # Log all commands
  export PROMPT_COMMAND='echo "$(date) $(whoami) $(history 1)" >> /var/log/user-commands.log'
fi

Checkpoint: Security Hardening

  • sshd_config hardened with strong algorithms
  • 2FA enabled (optional but recommended)
  • API keys stored on VM, not mobile device
  • Tailscale ACLs configured to restrict access
  • Audit logging enabled for session tracking

Advanced tmux for Mobile

tmux is the backbone of mobile development. Here are advanced configurations for mobile-optimized workflows.

Mobile-Optimized Status Bar

# ~/.tmux.conf - Mobile context status bar

# Show git branch and Claude status
set -g status-right-length 100
set -g status-right '\
#[fg=cyan]#{pane_current_path} \
#[fg=yellow]#(cd #{pane_current_path} && git branch --show-current 2>/dev/null || echo "no-git") \
#[fg=green]#(pgrep -f "claude" >/dev/null && echo "CLAUDE" || echo "idle") \
#[fg=white]| %H:%M'

# Highlight window with active Claude session
setw -g window-status-current-format '#[fg=green,bold]#I:#W*'

# Bell when Claude needs input
set -g visual-bell on
set -g bell-action any

Session Naming Convention

Session: work +-- Window 1: main (Primary Claude session) +-- Window 2: tests (Test runner) +-- Window 3: agent-1 (Feature branch 1) +-- Window 4: agent-2 (Feature branch 2) +-- Window 5: monitor (rpai or logs)
# Quick navigation aliases (~/.zshrc)
alias t1='tmux select-window -t 1'
alias t2='tmux select-window -t 2'
alias ta='tmux select-window -t agent-1'
alias tm='tmux select-window -t monitor'

Session Persistence Plugins

# ~/.tmux.conf - Session persistence

set -g @plugin 'tmux-plugins/tmux-resurrect'
set -g @plugin 'tmux-plugins/tmux-continuum'

# Auto-save every 15 minutes
set -g @continuum-save-interval '15'

# Auto-restore on tmux start
set -g @continuum-restore 'on'

# Restore Claude processes
set -g @resurrect-processes 'claude node python'

# Initialize TMUX plugin manager
run '~/.tmux/plugins/tpm/tpm'
Session Survival

With tmux-resurrect + continuum, you can reboot the VM and tmux auto-restores all sessions including Claude state.

Checkpoint: Advanced tmux

  • Status bar showing git branch and Claude status
  • Window naming convention established
  • tmux-resurrect and continuum installed
  • Quick navigation aliases in .zshrc

Advanced Notification Patterns

Beyond basic notifications: priority routing, batching, and Do Not Disturb integration.

Priority-Based Routing

#!/bin/bash
# ~/.claude/hooks/multi-channel-notify.sh

PRIORITY=$(echo "$CLAUDE_HOOK_EVENT_DATA" | jq -r '.priority // "normal"')
MESSAGE=$(echo "$CLAUDE_HOOK_EVENT_DATA" | jq -r '.tool_input.questions[0].question')
PROJECT=$(basename "$PWD")

case $PRIORITY in
  "critical")
    # Phone call via Twilio for truly critical
    curl -X POST "https://api.twilio.com/..." \
      -d "Body=CRITICAL: Claude needs immediate input on $PROJECT"
    ;;
  "high")
    # Push notification + sound
    curl -H "Priority: urgent" \
         -d "$PROJECT: $MESSAGE" \
         ntfy.sh/claude-urgent
    ;;
  "normal")
    # Standard push
    curl -d "$PROJECT: $MESSAGE" ntfy.sh/claude-work
    ;;
  "low")
    # Silent/batch for later
    echo "$MESSAGE" >> ~/.claude/pending-notifications.txt
    ;;
esac

Notification Batching

#!/bin/bash
# Aggregate notifications, send summary every 30 minutes

BATCH_FILE="/tmp/claude-notification-batch.txt"
echo "[$(date)] $PROJECT: $QUESTION" >> "$BATCH_FILE"

# Check if it's time to send batch
LAST_SEND=$(stat -f %m ~/.claude/last-batch-send 2>/dev/null || echo 0)
NOW=$(date +%s)
INTERVAL=$((30 * 60))

if (( NOW - LAST_SEND > INTERVAL )); then
  COUNT=$(wc -l < "$BATCH_FILE")
  curl -d "$(cat $BATCH_FILE)" \
       -H "Title: $COUNT Claude notifications" \
       ntfy.sh/claude-batch

  rm "$BATCH_FILE"
  touch ~/.claude/last-batch-send
fi

Do Not Disturb Integration

#!/bin/bash
# Respect DND hours (10pm-7am)

HOUR=$(date +%H)
if (( HOUR >= 22 || HOUR < 7 )); then
  # Store for morning digest
  echo "$MESSAGE" >> ~/.claude/morning-digest.txt
  exit 0
fi

# Otherwise send normally
curl -d "$MESSAGE" ntfy.sh/your-topic

ntfy Interactive Actions

# Send notification with tap-to-respond buttons
curl \
  -H "Actions: http, Approve, https://your-api/respond, method=POST, body={\"response\": \"yes\"}" \
  -H "Actions: http, Reject, https://your-api/respond, method=POST, body={\"response\": \"no\"}" \
  -d "Deploy to production?" \
  ntfy.sh/your-topic

Checkpoint: Notification Patterns

  • Priority-based routing script implemented
  • DND hours configured (or batching enabled)
  • Tested high-priority vs low-priority notifications

Multi-Device Synchronization

Seamlessly move between phone, tablet, and laptop without losing context.

Native Claude Code Teleportation

# In terminal Claude session
&
# or
--teleport

# Outputs URL like: https://claude.ai/code/session/abc123
# Open URL on any device to continue

tmux Session Teleportation

tmux sessions persist on the server - detach anywhere, attach from anywhere.

On laptop: tmux attach -t work (work for a while) (close laptop lid - connection drops) | v On phone (via Termius): ssh devuser@vm (auto-attaches to existing tmux session) (continue exactly where you left off)

Session State File Pattern

# ~/project/.claude-state.json - Maintain state in files
{
  "current_task": "Implementing user authentication",
  "branch": "feature/auth",
  "last_action": "Created login endpoint",
  "pending_questions": [],
  "context_files": [
    "src/auth/login.ts",
    "src/auth/session.ts"
  ],
  "timestamp": "2026-01-11T10:30:00Z"
}

# Claude can read this at session start to restore context

Device Handoff Checklist

When Switching Devices

  1. Save any local changes (Working Copy on iPad)
  2. Push to branch
  3. Detach tmux (or let it auto-detach)
  4. Connect from new device
  5. Verify tmux session state
  6. Continue working

Checkpoint: Multi-Device

  • Successfully teleported a session between devices
  • .claude-state.json template created
  • Tested disconnect/reconnect from different devices

Advanced Troubleshooting

iPad Stage Manager not showing all windows

Problem: Stage Manager limits visible windows.

Solution: Enable "More Space" in Display settings. Use external monitor for more Stage Manager stages. Consider Split View instead for 2-app workflows.

Voice dictation quality is poor

Problem: Technical terms not recognized correctly.

Solution: Use Superwhisper's code mode. Add custom vocabulary for project-specific terms. Consider Whisper.cpp with custom fine-tuning for better accuracy.

mosh reconnection taking too long

Problem: mosh doesn't reconnect after network change.

Solution: Check firewall allows UDP 60000-61000. Try mosh --predict=experimental for faster reconnection. Consider SSH keepalive as fallback.

Spot instance interrupted mid-work

Problem: AWS terminated spot instance, lost work.

Solution: Enable tmux-resurrect for session persistence. Store project on persistent EBS volume, not instance storage. Set up spot instance termination notice handler to auto-save.

Notifications overwhelming during busy day

Problem: Too many Claude notifications, notification fatigue.

Solution: Implement priority-based routing (section 07). Enable batching during focus hours. Use ntfy topics to separate urgent vs routine. Set DND hours.

2FA making mobile access cumbersome

Problem: Entering 2FA codes on phone is tedious.

Solution: Use Tailscale SSH (keyless auth via Tailscale identity). Or set up SSH certificate auth. Consider security tradeoffs - Tailscale ACLs may be sufficient.
