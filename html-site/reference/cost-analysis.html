Model Selection Strategies

Tiered Model Selection (CC Mirror Pattern) Savings: 50-70%

Match model capability to task complexity. Use the cheapest model that can reliably complete each task.

ModelUse CaseToken Cost Ratio
HaikuErrand runner: fetch files, grep, simple lookups. Spawn 5-10 in parallel1x (baseline)
SonnetCapable worker: structured implementation, following patterns, test gen12-20x
OpusComplex reasoning, architecture decisions, ambiguous tasks60-100x
Worker Preamble Template
CONTEXT: You are a WORKER agent, not an orchestrator.

RULES:
- Complete ONLY the task described below
- Use tools directly (Read, Write, Edit, Bash, etc.)
- Do NOT spawn sub-agents
- Do NOT call TaskCreate or TaskUpdate
- Report your results with absolute file paths

TASK:
[Your specific task here]
The "Haiku Army" Pattern Savings: 10-20x

For embarrassingly parallel tasks, spawn many Haiku instances instead of one Opus instance.

Cost Comparison for 10 Operations
  • 1 Opus session: ~$0.50-1.00 (sequential, context accumulation)
  • 10 Haiku workers: ~$0.05-0.10 (parallel, isolated context)

Best for: File searches, bulk refactoring, documentation generation, test file creation.

Fallback Cascade Pattern Savings: 60-80%

Start with cheapest model, escalate only on failure or low confidence.

MODELS_BY_COST = [
    "claude-haiku-3-5-20241022",
    "claude-sonnet-4-20250514",
    "claude-opus-4-5-20251101"
]

for model in MODELS_BY_COST:
    response = get_response(model, prompt)
    if confidence >= quality_threshold:
        return response  # Use cheapest that works
    # Otherwise try next model

Result: Most queries resolve at Haiku/Sonnet level. Opus only when truly needed.

Caching Strategies

Cache-Friendly Prompt Architecture Savings: 85-95% on repeat requests

Structure all prompts to maximize cache hit rates.

[CACHEABLE - 90% of prompt]
├── System prompt (instructions)
├── Reference documentation
├── Code context (if static)
├── Examples/few-shot
└── cache_control: ephemeral

[VARIABLE - 10% of prompt]
└── User's actual question
Cache Hit Maximization
  • Keep cached portion identical across requests
  • Don't include timestamps or dynamic data in cached section
  • Use ephemeral type (5-minute TTL) for session work
  • Batch similar requests together within TTL window
Multi-Layer Cache Architecture Savings: 50-100% depending on hit rate

Implement caching at multiple levels:

L1: Application deduplicationExact match hash of prompts100% on duplicates

Context Management for Cost Reduction

Ralph Wiggum: File-Based State Savings: 60-80%

Keep state in files instead of growing conversation context. Each iteration reads fresh from files.

Why This Saves Tokens
  • No lossy compaction (traditional chats hit limits, trigger summarization)
  • Lean API calls (only PROMPT.md + current file state)
  • Sustainable for multi-hour/day projects
plans/
├── ralph.sh        # The loop script
├── prd.json        # Task list with passes: true/false
└── progress.txt    # Append-only memory log
Subagent Token Isolation Savings: 50-70%

Put costly operations (like browser control) in subagents to protect main context window.

"The feedback loop is everything, but put costly tools like browser control in subagents to protect your main context window tokens"

Key insight: Subagents have their own context window. When they complete, only their summary returns to main agent, not the full conversation.

Context Window Monitoring (Claude HUD) Savings: 20-30% (prevents waste)

Monitor context usage in real-time to compact or reset before hitting limits.

/plugin marketplace add jarrodwatts/claude-hud
/plugin install claude-hud
/claude-hud:setup

Sample output:

[Opus 4.5] ████████░░ 35% | 1 MCPs | 1m
✓ Read x2 | ✓ WebFetch x1 | ✓ Skill x1 | ✓ Edit x1

Action thresholds:

  • 60-70%: Consider compacting proactively
  • 80%: Compact or prepare fresh session
  • 90%+: Stop and save state immediately

Real-World Case Studies

Code Review Pipeline

Scenario: Automated code review on every PR using Claude.

Unoptimized
  • Full codebase context: 100K tokens input
  • Opus for everything
  • $1.50 per review
  • 50 PRs/week = $300/month
Optimized
  • Git diff only: 5K tokens average
  • Tiered: Haiku for lint, Sonnet for logic
  • Prompt caching for review instructions
  • 50 PRs/week = $20/month
93% reduction - $280/month saved

Customer Support Bot

Scenario: 24/7 customer support chatbot handling 10K conversations/month.

Unoptimized
  • Sonnet for all queries
  • 2K tokens in/out per conversation
  • 10K conversations x 4K tokens
  • $720/month
Optimized
  • Haiku for FAQ (70%): $42
  • Sonnet for complex (25%): $180
  • Opus for escalations (5%): $180
  • + Prompt caching + Semantic cache
  • $220/month
69% reduction - $500/month saved

Document Processing

Optimized
