Role Breakdown

1. Mayor

Chief coordinator - your concierge and chief-of-staff. Receives directives, routes work, manages cross-rig coordination.

Model: Opus 4.5 (complex coordination)
2. Deacon

Health monitoring and agent coordination. Handles handshakes, reports anomalies, manages worker recycling.

Model: Haiku (simple health checks)
3. Dogs

Quality assurance and safety gates. Reviews work, enforces standards, can reject/rollback. Town-level grunts.

Model: Sonnet (quality review)
4. Refinery

Task decomposition. Breaks large tasks into agent-sized work, assigns to Polecats/Crew, manages merge queue.

Model: Opus 4.5 (architectural work)
5. Witness

Per-rig observer. Watches all work, records history, provides state to new workers, unsticks drifting agents.

Model: Haiku (observation/logging)
6. Polecat

Named persistent workers. Execute tasks, maintain context, specialize in rig-specific knowledge. Human names (alice, bob).

Model: Sonnet (implementation)
7. Crew

Ephemeral burst workers. Single tasks, no persistent state, cheap and disposable. Spawn, complete, disappear.

Model: Haiku (simple tasks)
8. Overseer (YOU)

Product Manager and decision maker. Define features, review inbox, approve major changes. Has identity in system.

Beads: The Memory System

"Agents suffer from '50 First Dates' syndrome - they wake up with no memory of yesterday."

Beads (github.com/steveyegge/beads) is a ~225K line Go project solving this problem.

What Beads Provides

  • Git-backed storage: All state persisted in git
  • Universal context: Shared context across all agents
  • Hash-based IDs: (e.g., bd-a1b2) prevent merge conflicts
  • Dependency tracking: Explicit links between tasks
  • Memory decay: Compaction summarizes old closed tasks

The Memory Model

+-------------------------------------------------------------+
|                    BEADS MEMORY MODEL                        |
+-------------------------------------------------------------+
|                                                              |
|  LAYER 1: Hot Context (RAM-cached)                          |
|  +-- Current task state                                     |
|  +-- Last N interactions                                    |
|  +-- Active file references                                 |
|                                                              |
|  LAYER 2: Warm Context (SQLite + Git index)                 |
|  +-- Recent commit history                                  |
|  +-- Cross-agent message log                                |
|  +-- Rig state snapshots                                    |
|                                                              |
|  LAYER 3: Cold Context (Git history)                        |
|  +-- Full project history                                   |
|  +-- Previous run outcomes                                  |
|  +-- Archived decisions                                     |
|                                                              |
|  RETRIEVAL: Hot -> Warm -> Cold (only query cold if needed) |
|                                                              |
+-------------------------------------------------------------+

Why Git as Database? Git provides durable, diffable, auditable storage with zero infrastructure. Survives crashes, full audit trail, recovery to any point, sharing across agents, no heavyweight message bus.

Core Principles

GUPP: The Universal Propulsion Principle

"If there is work on your hook, YOU MUST RUN IT."

Work is "slung" to agent hooks (persistent git worktrees). When an agent starts, it checks its hook. If work exists, execution is immediate - no confirmation, no waiting.

Zero Framework Cognition

"Claude makes all decisions. The orchestrator executes."

All decisions delegated to AI. No heuristics, regex, or parsing in the orchestrator. The framework provides infrastructure, not intelligence.

Agents as Cattle, Not Pets

Kubernetes thinking applied to agents:

  • Individual sessions are ephemeral
  • The work graph is persistent
  • The agent is not the session
  • The agent is a Bead (identity with a global address)

Nondeterministic Idempotence

Workflows can be interrupted and resumed at any time:

  • Path is fully nondeterministic - different runs take different routes
  • Workflow eventually finishes as long as you keep throwing agents at it
  • Agents may make mistakes but self-correct because acceptance criteria are well-specified

Token Spend as Health Metric

"Token spend per developer per unit time is the new health metric that best represents how well your company is doing with AI."
  • Agent cost: ~$10-12 per operating hour
  • Recommended daily developer LLM budget: $100-300
  • Yegge's personal spend: ~$300/day, ~$80,000/year
  • Gas Town workshops generate 12,000 LOC/day per developer

When to Use Gas Town

Use Gas Town When:

  • You're at Stage 7+ (10+ agents, comfortable with chaos)
  • Budget is not a concern ($50-200/day)
  • You want team-scale output as a solo developer
  • You have clear product vision but need implementation velocity
  • You're comfortable with imperfection (some bugs fixed 2-3 times)
  • You have tmux proficiency

Don't Use Gas Town When:

  • You're below Stage 6 on the dev evolution scale
  • Budget is constrained
  • You need polish and stability
  • You need predictable, controlled outcomes
  • The project is simple enough for Ralph loops
  • You have any doubt whatsoever

Implement Gas Town

Prerequisite Check

Checkpoint: Are You Ready?
  1. Have you run 10+ agents in parallel before? (Stage 7 minimum)
  2. Are you comfortable with $50-200/day in API costs?
  3. Do you have tmux proficiency?
  4. Can you handle imperfection (some bugs fixed 2-3 times)?
  5. Do you have 100+ hours with multi-agent setups?

You should see: All 5 answers are "yes". If ANY is "no", use Ralph loops or CC Mirror instead. Gas Town will eat you alive.

Quick Wins (< 1 hour)

  1. Install Go if not present: brew install go
  2. Install Gas Town: go install github.com/steveyegge/gastown/cmd/gt@latest
  3. Initialize in your project: gt init
  4. Check status: gt status
Quick Win Verification

You should see: gt status shows "Gas Town initialized. No roles running." If you see errors, check Go installation and PATH.

Medium Lift (2-4 hours)

  1. Start the Mayor: gt mayor up
  2. Create your first rig: gt rig create my-feature
  3. Spawn a Polecat: gt polecat up my-feature alice
  4. File your first task: gt task "Implement user login form"
  5. Monitor via inbox: gt inbox

Deep Investment (days)

  1. Set up Beads for persistent memory: Install Beads
  2. Configure Dogs for your quality gates (tests, lints)
  3. Build your first multi-rig workflow
  4. Learn to read the work graph and unstick drifting agents
  5. Develop intuition for when to spawn Crew vs use Polecats

Troubleshooting

Problem: "Agents spinning but no progress"

Symptoms: Work graph shows tasks in progress, but commits aren't happening. Agents consuming tokens without output.

Causes:

  • Task specification too vague - agents don't know what "done" means
  • Beads state corrupted or out of sync
  • Agents stuck in a decision loop (needs human judgment)
# Check what agents are actually doing gt polecat logs my-feature alice --tail 50 # Check for stuck tasks gt task list --stuck # Force task reassignment gt task unstick <task-id> # Nuclear option: Reset the rig gt rig reset my-feature --soft # Keeps history gt rig reset my-feature --hard # Full reset

Prevention: Always include explicit acceptance criteria in task specs. The more specific, the less agents spin.

Problem: "Cost exploding unexpectedly"

Symptoms: Daily cost far exceeds $50-200 target. Tokens consumed faster than expected.

Causes:

  • Too many Crew spawned for simple tasks
  • Refinery decomposing tasks too granularly
  • Opus being used where Haiku would suffice
  • Agents retrying failed tasks indefinitely
# Check current token usage gt cost --today # See which roles are consuming most gt cost breakdown --by-role # Limit Crew spawning gt config set crew.max-per-rig 5 # Set task retry limits gt config set task.max-retries 3 # Downgrade models where possible # In .gastown/config.yaml: # witness: haiku (not sonnet) # crew: haiku (not sonnet)

Prevention: Steve says "If you care about costs, don't use this." But if you must, start with strict limits and relax them over time.

Problem: "Merge conflicts between agents"

Symptoms: Multiple agents modify same files. Git conflicts pile up. Work stalls in merge queue.

Causes:

  • Task decomposition didn't isolate file ownership
  • Agents not using isolated worktrees
  • Dogs not catching conflicts early enough
# Check conflict status gt conflicts list # Force sequential merge (slower but safer) gt config set merge.strategy sequential # Assign explicit file ownership in task specs: # Task: Implement login form # Ownership: src/pages/Login.tsx, src/hooks/useAuth.ts # Do NOT touch: src/components/* (owned by other task) # Have Dogs reject any PR touching non-owned files # In .gastown/dogs.yaml: # rules: # - reject-if: "modifies files outside ownership"

Prevention: When filing tasks, always specify file ownership. The Refinery should enforce this but double-check.

Problem: "Beads memory corrupted or lost"

Symptoms: Agents wake up with no context. Same mistakes repeated. "50 First Dates" syndrome.

Causes:

  • Beads git repo not synced
  • Hash conflicts in bead IDs
  • Memory decay over-aggressive (compacted too much)
# Check Beads health beads status # Sync from git beads sync # Restore from specific point beads restore --to <commit-hash> # Rebuild index if corrupted beads reindex # Adjust memory decay (in .beads/config.yaml) # decay: # days-until-cold: 14 # Default: 7 # days-until-archive: 90 # Default: 30

Prevention: Beads is git-backed - always push after significant work. Enable auto-sync: beads config set auto-sync true

Problem: "Town completely unresponsive"

Symptoms: gt status hangs. No agents responding. Inbox not updating.

Causes:

  • Mayor crashed or stuck
  • Git lock file preventing operations
  • Too many concurrent API requests hitting rate limits
# Check what's actually running ps aux | grep gt ps aux | grep claude # Kill everything and restart gt town down --force pkill -f "gt mayor" pkill -f "gt polecat" # Clear any git locks rm -f .git/index.lock rm -f .gastown/.lock # Restart fresh gt town reset --soft gt mayor up gt status # Should respond now

Prevention: Don't run Gas Town on machines that go to sleep. Use a dedicated server or cloud instance for 24/7 operation.

Emergency Recovery

# Town completely broken - nuclear reset gt town down --force gt town reset --hard # WARNING: Loses work in progress # Preserve work, reset infrastructure gt town down --force gt rig export-all ./backup-$(date +%Y%m%d) gt town reset --soft gt rig import-all ./backup-$(date +%Y%m%d) # Single rig chaos - isolate and restart gt rig down my-broken-rig gt rig reset my-broken-rig --hard gt polecat up my-broken-rig alice # Fresh start # API costs spiraling - emergency stop gt town down --force # Stop all agents immediately gt config set spending.daily-limit 50 # Set hard limit gt mayor up # Resume with limit # Beads completely corrupted cd ~/.beads git log --oneline -20 # Find last good state git reset --hard <good-commit> beads reindex # Started Gas Town prematurely (not ready for it) gt town down --force gt uninstall # Remove Gas Town # Go back to Ralph loops until you're truly ready
