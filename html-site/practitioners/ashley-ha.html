Why Context Matters

Most Claude Code users hit a wall when context windows overflow. Quality degrades, Claude loses track, responses become generic. Ashley's insight: treat context like a budget, not a bucket. You don't fill a bucket to the brim - you leave room for movement.

Core Philosophy

1"60% Is The Magic Number"

Never fill more than 60% of the context window. Reserve 40% for Claude's reasoning and response. This isn't arbitrary - it's a practical rule that prevents quality degradation as conversations grow.

"My personal rule: never let context exceed 60%. I'm always checking with /context during my sessions." - Ashley Ha

2"Phases Prevent Chaos"

Research, Plan, Implement, Validate. In that order. Every time. Don't skip phases, don't combine phases. The structure IS the methodology. Ad-hoc prompting leads to ad-hoc results.

3"Externalize Thinking"

The thoughts/ folder is Claude's brain outside the conversation. Persistent reasoning, queryable decisions, debug trails. Use it or lose institutional knowledge between sessions.

4"Systematic > Clever"

A systematic process beats brilliant chaos. Systematic wins long-term. Reproducibility compounds, cleverness doesn't.

Key Techniques

1The 60% Context Rule

Calculate your context budget before every major prompt. Here's how to estimate token usage:

Markdown Token Estimation

# Quick estimate: ~4 characters = 1 token # 200k context window = ~800k characters max # 60% budget = ~480k characters # Check file size in characters wc -c CLAUDE.md # Output: 12500 CLAUDE.md # ~3,125 tokens for this file # Check total project context find . -name "*.md" -exec wc -c {} + | tail -1 # Sum all markdown = total context estimate

Bash Token Budget Script

#!/bin/bash # context-budget.sh - Calculate context usage MAX_TOKENS=200000 BUDGET_PERCENT=60 BUDGET=$((MAX_TOKENS * BUDGET_PERCENT / 100)) # Estimate tokens from character count total_chars=$(find . -name "*.md" -exec cat {} + 2>/dev/null | wc -c) estimated_tokens=$((total_chars / 4)) used_percent=$((estimated_tokens * 100 / MAX_TOKENS)) echo "=== Context Budget Report ===" echo "Max tokens: $MAX_TOKENS" echo "Budget (60%): $BUDGET tokens" echo "Estimated usage: $estimated_tokens tokens ($used_percent%)" if [ $estimated_tokens -gt $BUDGET ]; then echo "WARNING: Over budget! Reduce context by $((estimated_tokens - BUDGET)) tokens" else echo "OK: $((BUDGET - estimated_tokens)) tokens remaining in budget" fi
Quick Win

Run wc -c on your CLAUDE.md right now. Divide by 4. That's your token estimate. Is it under 60% of your model's context window?

You should see: For a 200k context model, your CLAUDE.md should be under 480,000 characters (~120k tokens). Most effective CLAUDE.md files are 4,000-8,000 characters (1k-2k tokens).

2The 4-Phase Workflow

Every task follows the same progression. No exceptions:

1
Research Gather context, understand scope
2
Plan Define approach, identify risks
3
Implement Execute the plan, write code
4
Validate Test, verify, document

Phase 1: Research Prompt Template

# Phase 1: Research ## Objective Understand the scope and requirements for [TASK]. ## Questions to Answer 1. What existing code/patterns relate to this task? 2. What dependencies or constraints exist? 3. What are the edge cases? 4. What similar implementations can we reference? ## Output Create `thoughts/research-[TASK].md` with findings. Do NOT proceed to implementation yet.

Phase 2: Plan Prompt Template

# Phase 2: Plan ## Context Review `thoughts/research-[TASK].md` for background. ## Deliverable Create an implementation plan in `thoughts/plan-[TASK].md`: 1. Step-by-step approach 2. Files to create/modify 3. Test strategy 4. Rollback plan if needed ## Constraints - Plan must fit within current session context - Each step should be independently verifiable - Do NOT start implementation yet

Phase 3: Implement Prompt Template

# Phase 3: Implement ## Context - Research: `thoughts/research-[TASK].md` - Plan: `thoughts/plan-[TASK].md` ## Instructions Execute the plan step by step. After each step: 1. Verify the step completed successfully 2. Update `thoughts/progress-[TASK].md` with status 3. If blocked, stop and document the blocker ## Do NOT skip steps or combine phases.

Phase 4: Validate Prompt Template

# Phase 4: Validate ## Context Implementation complete per `thoughts/progress-[TASK].md`. ## Validation Checklist 1. Run all tests: `npm test` 2. Run type check: `npm run typecheck` 3. Run linter: `npm run lint` 4. Manual verification of key functionality 5. Update documentation if needed ## Output Create `thoughts/complete-[TASK].md` with: - Summary of changes - Test results - Any follow-up items identified

3The thoughts/ Directory Pattern

Externalize Claude's reasoning to persist across sessions:

project/ ├── thoughts/ │ ├── README.md # Directory purpose and conventions │ ├── research-auth.md # Phase 1 output for auth feature │ ├── plan-auth.md # Phase 2 output for auth feature │ ├── progress-auth.md # Phase 3 running log │ ├── complete-auth.md # Phase 4 summary │ ├── decisions/ # Architectural decisions │ │ ├── 001-database.md │ │ └── 002-auth-flow.md │ └── debug/ # Debug sessions and findings │ └── 2024-01-15-memory-leak.md └── src/

thoughts/README.md Template

# thoughts/ This directory contains Claude's externalized reasoning. ## Purpose - Persist context across sessions - Enable queryable decision history - Track progress through multi-phase work - Debug trails for complex problems ## Conventions - `research-*.md` - Phase 1 investigation findings - `plan-*.md` - Phase 2 implementation plans - `progress-*.md` - Phase 3 running status - `complete-*.md` - Phase 4 completion summaries - `decisions/*.md` - Numbered architectural decisions - `debug/*.md` - Dated debug session logs ## Usage When starting a new session, Claude should: 1. Read relevant thoughts/* files for context 2. Update progress files during work 3. Create new files for new investigations Never delete thoughts files - they are institutional memory.
Why This Works

Each new Claude session starts fresh. Without external state, all previous reasoning is lost. The thoughts/ directory acts as Claude's long-term memory - queryable, persistent, and version-controlled.

4Phase Gating Enforcement

Automate phase discipline - don't rely on willpower:

#!/bin/bash # phase-gate.sh - Enforce phase completion before advancing TASK=$1 PHASE=$2 check_phase_complete() { local task=$1 local phase=$2 local file="" case $phase in research) file="thoughts/research-$task.md" ;; plan) file="thoughts/plan-$task.md" ;; implement) file="thoughts/progress-$task.md" ;; validate) file="thoughts/complete-$task.md" ;; esac if [ -f "$file" ] && [ -s "$file" ]; then return 0 # Phase complete else return 1 # Phase incomplete fi } # Check prerequisites case $PHASE in plan) if ! check_phase_complete "$TASK" "research"; then echo "ERROR: Cannot start Plan phase - Research not complete" echo "Create thoughts/research-$TASK.md first" exit 1 fi ;; implement) if ! check_phase_complete "$TASK" "plan"; then echo "ERROR: Cannot start Implement phase - Plan not complete" echo "Create thoughts/plan-$TASK.md first" exit 1 fi ;; validate) if ! check_phase_complete "$TASK" "implement"; then echo "ERROR: Cannot start Validate phase - Implement not complete" echo "Update thoughts/progress-$TASK.md first" exit 1 fi ;; esac echo "Phase gate check passed. Proceeding with $PHASE for $TASK"

Usage

# Before starting plan phase ./phase-gate.sh auth plan # Before starting implement phase ./phase-gate.sh auth implement # Before starting validate phase ./phase-gate.sh auth validate
Implementation Checkpoint

Add this script to your project and run it before each Claude session. If it fails, you know exactly what prerequisite work is missing.

You should see: When you run ./phase-gate.sh auth plan:

  • If research is incomplete: ERROR: Cannot start Plan phase - Research not complete
  • If research is complete: Phase gate check passed. Proceeding with plan for auth

5Context Compression Techniques

When approaching the 60% budget, compress rather than cut:

Summarize Pattern

# Before: Full decision history (2000 tokens) [Complete discussion of 15 architectural decisions...] # After: Summary reference (200 tokens) See `thoughts/decisions/` for full context. Key decisions: - 001: PostgreSQL over MongoDB (query flexibility) - 002: JWT over sessions (stateless scaling) - 003: React Query over Redux (server state focus)

Archive Pattern

# Move completed work to archive, keep summary in active context # Active: thoughts/progress-auth.md ## Current Status Auth implementation complete. See archive for details. ## Active Items - [ ] Add rate limiting - [ ] Add password reset flow # Archived: thoughts/archive/auth-v1-complete.md [Full implementation details, decisions, debug logs...]

Reference Pattern

# Instead of including full file contents, reference by path ## Context Files For implementation details, read: - `src/auth/README.md` - Auth module documentation - `thoughts/decisions/002-auth-flow.md` - Why we chose this approach - `tests/auth.test.ts` - Expected behavior examples Claude will read these files as needed rather than having them pre-loaded in context.
Compression vs. Omission

Compression means making the same information available in less space. Omission means losing information. Always compress - never just delete context you might need.

Troubleshooting

Problem: "Context overflow despite following 60% rule"

Symptoms: Claude's responses become generic, loses track of project details, starts repeating itself or contradicting earlier statements.

Causes:

  • File reads during session weren't accounted for in budget
  • Multi-turn conversations accumulate faster than expected
  • Hidden context (system prompts, tools) consuming space
# Check actual context usage during session /context # If over 70%, take action immediately: # Option 1: Compact and continue /compact # This summarizes conversation history, freeing space # Option 2: Fresh session with context handoff # Before ending: > Create a summary of our current task state in thoughts/handoff-[DATE].md > Include: what's done, what's next, key decisions made # Start new session: claude > Read thoughts/handoff-[DATE].md for context on where we left off

Prevention: Check /context every 15-20 minutes during extended sessions. Start fresh sessions for major phase transitions.

Problem: "Claude forgets thoughts/ directory exists"

Symptoms: Claude doesn't reference existing thoughts/ files, asks questions already answered in research phase, repeats work.

Causes:

  • thoughts/ not mentioned in CLAUDE.md
  • Session started without explicit context loading
  • File paths incorrect or files not committed
# Add to CLAUDE.md (project root): ## Project Memory This project uses the thoughts/ directory for persistent context. Before starting any task: 1. Check thoughts/ for existing research, plans, and progress 2. Read relevant thoughts/*.md files for context 3. Update thoughts/ files as you work Key files: - thoughts/README.md - Directory conventions - thoughts/decisions/ - Architectural decisions (read before proposing changes) - thoughts/progress-*.md - Current task status

Session Start Script:

# session-start.sh - Always run this to start Claude with context #!/bin/bash echo "Loading thoughts/ context for Claude session..." # List recent thoughts files echo "Recent thoughts files:" ls -lt thoughts/*.md 2>/dev/null | head -5 # Show any in-progress tasks echo -e "\nIn-progress tasks:" grep -l "## Current Status" thoughts/progress-*.md 2>/dev/null echo -e "\nStarting Claude with context..." claude << EOF Read thoughts/README.md to understand project memory conventions. Then list any in-progress work from thoughts/progress-*.md files. EOF

Prevention: Always start sessions by explicitly asking Claude to read thoughts/ directory.

Problem: "Phase discipline breaking down - skipping phases"

Symptoms: Jumping from Research to Implement, skipping Plan. Starting tasks without research. Validation phase forgotten.

Causes:

  • Time pressure - feels faster to skip phases
  • Small tasks feel like they don't need phases
  • No enforcement mechanism

The Truth: Skipping phases for "small" tasks is where bugs compound. The phases take 2-3 minutes each. Debugging bad implementations takes hours.

# Add to CLAUDE.md to enforce discipline: ## Phase Discipline (REQUIRED) Every task MUST follow 4 phases. No exceptions. 1. **Research** → Create thoughts/research-[TASK].md Ask: "What do I need to understand first?" 2. **Plan** → Create thoughts/plan-[TASK].md Ask: "What's my step-by-step approach?" 3. **Implement** → Update thoughts/progress-[TASK].md Ask: "Am I following the plan?" 4. **Validate** → Create thoughts/complete-[TASK].md Ask: "Did I verify everything works?" If a phase file doesn't exist, I haven't completed that phase. Don't let me skip phases - prompt me to complete them.

Recovery: If you've already skipped phases, stop and backfill. Create the missing research and plan files retroactively. Document what you learned so it's not lost.

Problem: "thoughts/ directory becoming bloated"

Symptoms: Hundreds of files in thoughts/, can't find relevant context, reading everything takes too long.

Causes:

  • Never archiving completed work
  • Creating files for tiny tasks
  • No naming conventions making files hard to group
# Archive completed work monthly: mkdir -p thoughts/archive/2024-01 mv thoughts/complete-*.md thoughts/archive/2024-01/ mv thoughts/research-*.md thoughts/archive/2024-01/ mv thoughts/plan-*.md thoughts/archive/2024-01/ # Keep active: # - thoughts/progress-*.md (current work) # - thoughts/decisions/*.md (architectural decisions - always relevant) # - thoughts/debug/*.md (recent only, archive monthly) # Create an index for archived work: cat > thoughts/archive/2024-01/INDEX.md << 'EOF' # January 2024 Archive ## Completed Features - auth - User authentication system - profile - User profile management - settings - Settings page ## Key Decisions Made - Chose PostgreSQL (see decisions/001-database.md) - Chose JWT auth (see decisions/002-auth-flow.md) EOF

Prevention: Archive monthly. Keep active thoughts/ to 20-30 files max. Use consistent naming so related files group together.

Problem: "New sessions not picking up context"

Symptoms: Starting a new Claude session feels like starting from scratch. Previous session's understanding is lost.

Causes:

  • Previous session didn't externalize state properly
  • No handoff document created
  • thoughts/ files not updated before session end

Session End Checklist:

# Before ending ANY Claude session: > Before we end this session, please: > > 1. Update thoughts/progress-[CURRENT-TASK].md with: > - What we accomplished > - What's still pending > - Any blockers or decisions needed > > 2. If any decisions were made, add them to thoughts/decisions/ > > 3. Create thoughts/handoff-$(date +%Y%m%d).md with: > - Current task state > - Next actions > - Important context the next session needs

Session Start Checklist:

# Starting a new session: > Read these files for context: > 1. thoughts/handoff-*.md (most recent) > 2. thoughts/progress-*.md (any with active work) > 3. thoughts/decisions/*.md (relevant decisions) > > Then summarize: What were we working on? > What's the next action?

Prevention: Never end a session without creating a handoff. Make it a ritual.

Emergency Recovery

# Context completely overflowed - Claude is confused /compact # Summarize and free space # If still bad: exit # End session claude # Fresh start > Read thoughts/ for context # Reload from persistent memory # Lost track of what phase we're in ls thoughts/ # See what files exist # If progress-[TASK].md exists → we're in Implement # If plan-[TASK].md exists but no progress → we're ready for Implement # If research-[TASK].md exists but no plan → we're ready for Plan # Made changes without following phases (panic mode) # Retroactively document: > We made changes to [FILES]. Please create: > 1. thoughts/research-[TASK].md - What we learned > 2. thoughts/plan-[TASK].md - The approach we took > 3. thoughts/progress-[TASK].md - Current state > This will restore our phase discipline. # Accidentally deleted thoughts/ files git status # Check if tracked git checkout -- thoughts/ # Restore from git # If not tracked: check your editor's local history
