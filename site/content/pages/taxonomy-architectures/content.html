Essence (15 words)
Every agent architecture exists on one axis: orchestration separation. Master this, master all patterns.

2 The Core Abstraction

"An agent that orchestrates must NEVER directly execute. An agent that executes must NEVER directly orchestrate."
Orchestrator: Task*, Read (minimal) | Worker: Read, Write, Edit, Bash
This is the Iron Law. Every architecture in this taxonomy obeys it or fails.

The taxonomy organizes 13 architecture patterns across three complexity tiers. But every pattern is a variation on ONE theme: how strictly do you separate orchestration from execution? The strictest separation (CC Mirror) scales best. The loosest (single interactive session) is simplest. Everything else falls on this spectrum.

This isn't philosophy - it's physics. When an orchestrator writes code, it fills context with implementation details, losing strategic perspective. When a worker spawns sub-workers, coordination explodes exponentially. Mixing roles leads to context pollution, scope creep, verification bypass, and uncontrolled recursion.

3 Design Decisions

WHY ORCHESTRATOR/WORKER SEPARATION?
When an orchestrator writes code directly, its context fills with implementation details. It loses the strategic overview needed to coordinate multiple workers. This is called "context pollution" - the orchestrator becomes a worker with extra responsibilities it can't fulfill.
What this means for you
Define roles BEFORE starting. If you're the orchestrator: use only Task*, Read (1-2 files), AskUserQuestion. If you're a worker: use Read, Write, Edit, Bash, but NEVER Task*, NEVER spawn sub-agents. Tool ownership is non-negotiable.
WHY EXTERNAL STATE ACROSS ALL PATTERNS?
Every pattern in this taxonomy uses external state (files, git, task boards) rather than relying on agent memory. This isn't a Ralph-specific decision - it's universal. INV-003 (External state > internal memory) appears in Ralph, Gas Town, CC Mirror, and Panopticon because it's physics, not preference.
What this means for you
Whatever architecture you choose, design your persistence layer FIRST. For Ralph: prd.json + progress.txt. For CC Mirror: TaskCreate/TaskUpdate. For Gas Town: ~/.gt/. For Panopticon: domain directories. The agent can forget - the filesystem cannot.
WHY FLAT HIERARCHY (2 LEVELS MAX)?
Deep nesting (>2 levels) is an anti-pattern across ALL architectures. Each additional level compounds: context passing becomes lossy, coordination overhead multiplies, debugging becomes exponentially harder, and latency compounds. Even Gas Town's 3-level appearance is really 2 levels with role specialization.
What this means for you
If you're reaching for a third level, you've mis-decomposed. Workers that need to delegate should report back and let the orchestrator spawn. Never Orchestrator → Worker → SubWorker. Always Orchestrator → Worker A, Orchestrator → Worker B (flat).

6 Gotchas

Architecture mistakes that compound. Learn from others' failures.

Orchestrator writing code directly
Symptom: Main agent context fills up, loses strategic view, quality degrades
Cause: Violating Iron Law - orchestrator took on worker role
Fix: Add explicit worker preamble: "You are a WORKER. Do NOT spawn sub-agents."
Worker spawning sub-workers
Symptom: Exponential context explosion, coordination nightmare, lost tasks
Cause: Worker trying to orchestrate instead of execute
Fix: Workers NEVER spawn. They report back, orchestrator spawns next worker. Flat hierarchy only.
Wrong model for task type
Symptom: High costs on simple tasks, or poor quality on complex ones
Cause: Not matching model to task complexity
Fix: Haiku for lookups/errands. Sonnet for implementation. Opus for architecture/ambiguity.
Starting too complex
Symptom: Spending more time debugging orchestration than building features
Cause: Jumped to Gas Town when Ralph would suffice
Fix: Start simple, add complexity only when you hit limits. Ralph -> CC Mirror -> Gas Town.

7 What's Hard

Fundamental tensions that don't go away. Manage them, don't solve them.

Coordination Overhead vs. Parallelism
The tension: More agents = more parallelism, but also more coordination. At some point, coordination cost exceeds parallelism benefit.
Symptoms: Agents waiting for each other. Merge conflicts. Duplicated work.
Mitigation: Strict domain isolation. Each agent owns specific directories/files. Use worktrees or containers for physical isolation. Accept sequential when domains overlap.
Observability vs. Simplicity
The tension: Multi-agent systems need monitoring, but monitoring infrastructure adds complexity. You can't debug what you can't see.
Symptoms: "What is agent 5 doing?" unanswerable. Silent failures. Morning reveals chaos.
Mitigation: Start with rpai or Claude HUD. Add Gas Town's Witness pattern for audit trails. Accept that full observability has cost.
Architecture Selection Regret
The tension: You can't know the right architecture until you've tried it. But switching architectures mid-project is expensive.
Symptoms: "We should have used X" realizations at 60% completion.
Mitigation: Start one level simpler than you think you need. Migrate when you hit concrete limits, not anticipated ones. Preserve external state formats across migrations.
