Gas Town orchestrates 8 specialized agent roles into a factory-scale development system.

3 Design Decisions

WHY 8 SPECIALIZED ROLES?
Each role optimizes for a specific function. Mayor coordinates (needs Opus reasoning). Polecats implement (Sonnet is cost-effective). Witness observes (Haiku is cheap for monitoring). Specialization allows model selection that balances capability and cost.
What this means for you
Don't try to make one agent do everything. Spawn specialized agents for specialized tasks. Use the cheapest model that can do the job. Mayor doesn't write code; Polecats don't coordinate.
WHY BEADS DATA PLANE?
Agents suffer from "50 First Dates" syndrome - they wake up with no memory. Beads provides git-backed persistent memory. SQLite for fast queries, JSONL for mergeable git storage. ~225K lines of Go solving the agent memory problem.
What this means for you
Every agent reads from and writes to Beads. bd ready shows actionable work. bd sync persists to git. Without Beads, agents can't coordinate across sessions. Beads IS the shared memory.
WHY THE REFINERY (MERGE QUEUE)?
Multiple agents editing the same codebase create merge conflicts. At ~200K LOC, Yegge's agents started "savagely overwriting each other's work." The Refinery serializes merges: Alice completes, rebases, merges, THEN Bob rebases on top of Alice's changes.
What this means for you
All polecat work flows through the Refinery. No direct pushes to main. The Refinery rebases automatically and escalates conflicts to your inbox. This is how 10+ agents work on the same repo without destroying each other's work.

4 The 8-Role Hierarchy

Gas Town defines 8 distinct roles organized into three levels: Town (global), Rig (per-project), and Personal.

Mayor
Opus 4.5 | $15-20/hour
Your concierge and chief-of-staff. Receives high-level directives, decomposes work, spawns agents, distributes via hooks. 
gt mayor attach                 # Interactive session (primary interface)
gt mayor up                     # Start mayor
gt mayor status                 # Check state
gt mayor down                   # Stop mayor
Polecats (Named Workers)
Sonnet | $5-10/hour each | 3-30 per rig
Your implementation army. Ephemeral workers generating pull requests. Use human names (alice, bob, carol) for mental model clarity. They complete tasks, run tests, submit MRs to merge queue. 
gt polecat up myproject alice   # Start named polecat
gt polecat list myproject       # List all polecats
gt sling bd-001 myproject       # Assign task via sling
gt polecat status myproject alice # Check status
Refinery
Opus 4.5 | $15-20/hour | 1 per rig
The gatekeeper of code integration. Merges polecat changes sequentially to main, prevents conflicts via rebasing, resolves conflicts automatically where possible. 
gt refinery up myproject        # Start for rig
gt refinery status myproject    # Check queue status
gt refinery process myproject   # Force process queue
gt refinery resolve myproject   # Manual conflict resolution

5 The Path of Work

Follow a feature from your idea to merged code:

1
You define a feature: "Build user authentication"
2
Mayor receives directive via gt mayor attach
3
Mayor creates Beads issues: bd-001 (DB schema), bd-002 (API routes), bd-003 (Frontend)
4
Mayor slings work to Refinery: gt sling bd-001 myproject
5
Refinery assigns to available polecat: alice gets bd-001
6
Alice works in isolated git worktree, implements, runs tests
7
Alice submits PR to Refinery merge queue
8
Witness monitors alice's progress, unsticks if blocked
9
Refinery rebases and merges alice's work to main
10
Dogs run quality checks on merged code
11
Bob and Carol work on bd-002, bd-003 in parallel
12
Morning: You review completed PRs in gt inbox

6 Gotchas

Real problems you'll hit with concrete numbers and fixes:

Deacon kills workers (v0.3.x bug)
Symptom: Workers dying randomly during Deacon patrol
Quote: "The murderous rampaging Deacon who is spree-killing all the other workers while on his patrol" -- Yegge
Fix: Run without Deacon until v0.4.0. gt deacon down
Merge wall at 200K LOC
Symptom: Agents "savagely overwriting each other's work rather than try to deal with merge conflicts"
Threshold: ~200K lines of code
Fix: Use worktrees for isolation, Refinery for sequential merges, or split into multiple rigs
Cost spike from runaway agents
Symptom: API bill much higher than expected, multiple accounts hitting limits
Cause: Refinery misconfiguration spawning too many polecats
Fix: gt pause --all, then gt config set max_polecats 5
Beads sync conflicts
Symptom: Error: Merge conflict in .beads/issues.jsonl
Cause: Multiple agents modifying Beads simultaneously
Fix: rm .beads/beads.db && bd sync --force (JSONL is source of truth)

7 What's Hard

These are fundamental tensions, not bugs to fix:

Chaos Tolerance Required
The tension: Gas Town is early-stage software. Bugs happen. Work gets lost sometimes. You must accept imperfection in exchange for throughput.
Quote: "If you have any doubt whatsoever, then you can't use it." -- Steve Yegge
Mitigation: Start with Mayor-only tutorial mode. Build trust gradually. Don't deploy to production until you've run 50+ agent-hours.
Cost Visibility
The tension: No built-in cost tracking. You don't know what you're spending until the API bill arrives.
Quote: "If you care about costs, don't use this." -- Steve Yegge
Mitigation: Use API dashboard for monitoring. Set max_polecats and max_crew limits. Expect $50-200/day for moderate usage.
tmux-Native Interface
The tension: Gas Town is tmux-native. No GUI alternative exists. If you're not comfortable in terminal multiplexers, you'll struggle.
Community efforts: GTGUI (Age of Empires style) is in development, but not production-ready.
Mitigation: Learn tmux basics before starting. Use the gastown-start.sh script for session management.
Stage 7+ Prerequisite
The tension: Gas Town requires 100+ hours managing 10+ Claude instances. There's no shortcut to this experience.
Quote: "If you're not at Stage 7, you'll get your face ripped off by superintelligent chimpanzees." -- Steve Yegge
Path: Level 4 (Ralph) -> Level 5 (CC Mirror) -> Level 6 (Worktrees) -> Level 7 (Gas Town)

8 When to Use / When Not

USE GAS TOWN WHEN
You have 100+ hours managing multiple Claude instances
You want team-scale output from one person
You have $50-200/day budget tolerance
You're comfortable being Product Manager, not programmer
You want 24/7 autonomous development
DON'T USE GAS TOWN WHEN
You're new to Claude Code -> Start with Ralph (Level 4)
Cost is a primary concern -> Use CC Mirror
Need quick task, not ongoing factory -> Single session
Prefer GUI over tmux -> Wait for GTGUI
Windows without WSL2 -> Not supported

Quick Reference Commands

# Installation
go install github.com/steveyegge/gastown/cmd/gt@latest
go install github.com/steveyegge/beads/cmd/bd@latest
gt install ~/gt --git
gt rig add myproject <repo-url>

# Daily Workflow
gt mayor attach           # Primary interface
gt inbox                  # View pending decisions
gt sling <bead-id> <rig>  # Assign work
bd ready                  # List actionable tasks

# Emergency Commands
gt pause --all            # Stop all work
gt town down              # Kill all agents
gt town reset --soft      # Reset state
