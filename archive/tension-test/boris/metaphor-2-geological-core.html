Part I -- Identity Layer

Who Is Boris Cherny?

Boris Cherny is a Staff Engineer at Anthropic and the creator of Claude Code. His role extends beyond engineering leadership into product vision and design philosophy. He is the canonical source for understanding how Claude Code should be used because its design directly reflects his mental model.

His pre-Anthropic work included contributions to the TypeScript ecosystem and developer experience tools -- explaining Claude Code's emphasis on type safety as verification, developer experience over AI complexity, and simple composable primitives over opinionated frameworks.

Mineral Vein -- Key Signal

Boris explicitly describes his own setup as "surprisingly vanilla." Anti-complexity is core to his worldview: if the creator doesn't need heavy customization, the tool works well out of the box.

Part II -- Practice Layer

The Canonical Workflow

Boris's workflow is a reference implementation. Every element reflects intentional design. It is a system where each element amplifies the others.

Parallel Sessions (5-10 Concurrent)

Five terminal instances. Five to ten web sessions. Mobile sessions started in morning, checked later. Boris doesn't use one extremely capable agent -- he uses many independent workers.

"He's distributing cognition like compute: allocate it, queue it, keep it hot, switch contexts only when value is ready."

Always Opus + Extended Thinking

Total Time = (Generation Time) + (Correction Time) + (Re-prompting Time) Opus has higher Generation Time but dramatically lower Correction and Re-prompting time. Net result: faster despite slower generation.
"I use Opus 4.5 with thinking for everything. Even though it's bigger and slower than Sonnet, since you have to steer it less and it's better at tool use, it is almost always faster than using a smaller model in the end." -- Boris Cherny

Shared CLAUDE.md

Single file at git root. Team contributes multiple times weekly. Every mistake becomes a rule. GitHub Action enforces updates on relevant PRs.

"Anytime we see Claude do something incorrectly we add it to the CLAUDE.md, so Claude knows not to do it next time." -- Boris Cherny

Verification: The Most Important Thing

Chrome extensions for UI tests, bash scripts, mobile simulators, background agents. Verification is not optional -- it is the difference between 1x quality and 2-3x quality.

"Probably the most important thing to get great results out of Claude Code: give Claude a way to verify its work." -- Boris Cherny
Part III -- Philosophy Layer

Design Philosophy

Core Principle 1

"Do the Simple Thing First" -- the team considered embeddings, vector databases, semantic search. They shipped a markdown file.

SearchripgrepSemantic search

Core Principle 2

"Build for the Model Six Months From Now" -- don't add workarounds for current limitations. They may become unnecessary when models improve.

"Honestly, for a long time, Claude Code was not a great product. I used it for maybe 10% of my code because the model wasn't capable enough." -- Boris Cherny
Core Principle 3

"Raw Model Access as Design Goal" -- Boris explicitly rejected building "a Cursor or Windsurf." Minimal abstraction between user intent and model capability.

Core Principle 4

"Unix Philosophy as Foundation" -- do one thing well. Compose freely. Text as interface. Minimal surprise.

Part V -- Mechanism Layer

The Compounding System

Linear usage: Output = Capability x Time Compounding usage: Output = Capability x Time x (1 + Learning Rate)^Sessions

Each session that updates CLAUDE.md increases effective capability for all future sessions. Every correction becomes permanent context. The cost of a mistake pays dividends forever.

Enables Compounding
Shared knowledge, mistake-to-rule conversion, team contribution, verification loops
Breaks Compounding
Individual files, ignoring failures, CLAUDE.md bloat, skipping verification
Part IX -- Bedrock: Things Only The Creator Would Know

Fundamental Truths

Bedrock Insight 1

CLAUDE.md was "The Simple Thing." The team considered every memory architecture in the literature. They shipped a markdown file.

Bedrock Insight 2

Claude Code deletes code on model upgrades. Most teams add complexity. Anthropic removes it -- deleting half the system prompt per upgrade.

Bedrock Insight 3

"Claude Code was not a great product." The bet was on future model capability, not current workarounds. When capability arrived, the interface was ready.

Bedrock Insight 4

50-100 PRs per week. 100% AI-contributed code. Not 80%, not 90%. The entire system working together.

"Think of Claude Code as a compounding system -- not a one-shot tool, but an evolving teammate." -- Boris Cherny, Bedrock Layer
Part VII -- Fault Lines

Anti-Patterns Boris Warns Against

Fault Line 1

Skipping Plan Mode. "Plan Mode isn't training wheels -- it's measuring before you cut."

Fault Line 2

Sequential Instead of Parallel. Run five Claudes, context-switch between them as they work.

Fault Line 3

Optimizing for Token Speed Over Quality. Correction time dominates generation time. Smaller models are false economy.

Fault Line 4

Over-Customization. "My setup might be surprisingly vanilla!"
