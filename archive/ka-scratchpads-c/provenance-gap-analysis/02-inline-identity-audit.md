## 4. Identity vs Provenance: The Distinction

### What Inline Headers Capture

The inline headers capture **operational identity** -- the information an agent or auditor needs to understand, validate, and work with a specific file. This includes:

1. **Purpose and hypothesis** (WHY THIS EXISTS, THE QUESTION THIS ANSWERS) -- what this file is trying to prove.

2. **Dependency graph** (BUILT ON) -- what files this depends on, with paths and the specific values taken from each. This is directional: it only documents upstream dependencies, not downstream.

3. **Constraint enforcement** (MUST HONOR) -- what rules cannot be violated, with consequence statements. These are actionable: an agent reading them knows exactly what to check.

4. **Blast radius** (WHAT BREAKS IF THIS CHANGES) -- what files would be affected if this file is modified. This is forward-looking provenance: it anticipates downstream effects.

5. **Research citation** (Applied Research sections in original blocks) -- specific R-finding IDs, DD-F IDs, EXT IDs that were applied, with one-line descriptions of HOW they were applied.

6. **Soul compliance evidence** (SOUL ALIGNMENT) -- per-piece verification with line number references. This is auditable: an agent can go to the cited line and verify.

7. **Research debt** (RESEARCH DEBT) -- what was NOT done. This is negative provenance: documenting known gaps and deferrals.

8. **Diagnostic self-check** (DIAGNOSTIC QUESTIONS) -- verification queries that can be mechanically answered by an auditor.

9. **Anti-pattern avoidance** -- explicit listing of what was deliberately avoided.

10. **Fractal compliance at 4 scales** -- PAGE, SECTION, COMPONENT, CHARACTER descriptions of how DD-F-006 manifests.

### What Inline Headers Do NOT Capture

1. **Temporal history.** Headers do not record when the file was created, when it was modified, or the sequence of changes. OD-001 mentions "v2 enrichment" but does not timestamp when v1 ended and v2 began. No commit hashes. No session IDs.

2. **Decision rationale at the phase level.** Why were these 6 organizational patterns chosen? Why not 8? Why these density pairings? The headers document the WHAT (PULSE pairing for conversational) but not the WHY (why PULSE and not WAVE for conversational). That decision rationale lives in handoff documents and planning files.

3. **Cross-OD comparison.** Each header is self-contained. No header says "OD-001 scored higher than OD-002 because..." or "The ISLANDS pattern in OD-003 was more successful than CRESCENDO in OD-002 because...". The only cross-reference is in BUILT ON tables, which document dependency, not comparative analysis.

4. **Aggregate statistics.** No header contains counts like "this file applies 14 out of 337 total findings" or "research coverage is 4.2%". Each header is locally complete but globally unaware.

5. **Score justification.** OD-001 mentions "~35/40" and OD-002 mentions "~33/40", but no header explains what scoring rubric was used, who scored it, or why it got that score. The scoring methodology is external.

6. **Agent execution metadata.** No header records which Claude agent built it, how long it took, what session it was built in, or what errors occurred during construction.

7. **Audit results.** The directory CLAUDE.md notes "Visual Audit PASS (0 violations)" for all 6 ODs, but the individual file headers do not contain audit results. The granular audit report (688 lines, 89 findings) lives in a separate file entirely.

8. **Finding synthesis.** OD-F-001 through OD-F-004 are described in OD-006's header but not traced back to their origin OD files. There is no "finding provenance" -- no record of when OD-F-001 was first identified, whether it was validated, or what evidence supports it.

9. **Pattern evolution.** If OD-003's ISLANDS implementation informed OD-005's spoke isolation, that evolution is not documented. The BUILT ON tables show dependency but not how the understanding of a pattern evolved across files.

10. **Retirement and supersession.** No header documents whether it has been superseded. If OD-001 v2 replaced OD-001 v1, the v1 version is not referenced or archived.

### What Formal Provenance Documents Would Capture (That Headers Do Not)

Based on the provenance structure documented in the explorations CLAUDE.md (which references `DESIGN-SYSTEM/provenance/stage-3-organization-od/`):

- **Stage-level narrative**: Why the OD stage exists, what it inherits from DD, what it must produce for AD.
- **Handoff documents**: HANDOFF-DD-TO-OD.md (what constraints cross the boundary).
- **Cross-exploration synthesis**: Which patterns won, which lost, comparative scoring.
- **Finding registry**: All OD-F findings in one place, with origin, evidence, and validation status.
- **Checkpoint tracking**: OD-CHECKPOINT.md with per-file state rows.
- **Accumulated identity**: ACCUMULATED-IDENTITY-v1.md (539 lines) that OD-004 and OD-006 reference -- a synthesized constraint document.

## 5. Reproducibility Test

### If you had ONLY the inline headers (no formal provenance docs), could you:

**Reconstruct what each OD exploration demonstrates?**
YES, partially. Each header contains the hypothesis, the density pairing, the organizational pattern, and the question it answers. You would know that OD-001 demonstrates Q&A conversational structure creating PULSE density, OD-002 demonstrates narrative arc creating CRESCENDO, etc. However, you would NOT know:
- Why these 6 patterns were chosen
- What alternatives were considered and rejected
- What the scoring criteria are
- How to compare them to each other

**Understand the constraints it operated under?**
YES, almost fully. The MUST HONOR table is explicit and actionable. The SOUL ALIGNMENT table documents compliance. Anti-patterns are listed. The gap: you would not know the SOURCE of these constraints (MASTER-SOUL-SYNTHESIS.md and DESIGN-TOKEN-SUMMARY.md are referenced by name and path, but their content is not inlined). You would need to follow the paths to reconstruct the full constraint set.

**Know what findings it produced for the next stage?**
PARTIALLY. OD-006's header declares OD-F-005 as a new finding. OD-F-006 through OD-F-008 appear in OD-006's HTML body but not in any header. OD-F-001 through OD-F-004 are listed in OD-006's header as applied research, but their origin and validation status are not documented. ## 7. Key Observations

### Observation 1: The Dual-Block Architecture Creates Information Duplication

Every OD file has two comment blocks that partially overlap. The original block lists applied research, anti-patterns, and fractal compliance. The threading header repeats some of this information in structured tables (SOUL ALIGNMENT, MUST HONOR) while adding new dimensions (BUILT ON, CONSUMED BY, WHAT BREAKS, RESEARCH DEBT, DIAGNOSTIC QUESTIONS). The overlap is not exact -- the original block contains research details (specific EXT finding descriptions) that the threading header does not repeat, while the threading header contains dependency and blast radius information that the original block does not.

### Observation 2: OD-F Finding Propagation Is Asymmetric

OD-F-001 through OD-F-004 describe findings from OD-001 but appear ONLY in OD-006's header. OD-001's own header contains zero OD-F references. This means findings were retroactively cataloged and assigned IDs during the OD-006 synthesis, not discovered and documented incrementally. If OD-001 were read in isolation, the reader would not know it produced any formal findings.

### Observation 3: Research Depth Varies Dramatically

OD-002 has zero EXT findings (explicitly noted as research debt), while OD-001 has 13 EXT findings. OD-004 has 12 EXT-CONF findings but uses non-standard naming (descriptive labels without numeric suffixes in the original block, with numeric IDs appearing only in downstream CSS comments and diagnostic questions). This naming inconsistency means automated extraction of EXT finding IDs from headers would produce unreliable results for OD-004.

### Observation 4: Headers Are Build-Time Artifacts, Not Audit-Time Artifacts

The headers were written when the files were built and threaded. They do NOT contain post-construction audit results. The 89-finding granular audit, the 16 fixes, the verification results -- none of this appears in the headers. The headers describe intent and constraints; the audit describes reality. These are fundamentally different temporal perspectives.

### Observation 5: Forward References Are Speculative

Multiple CONSUMED BY entries reference "(Future)" documents that do not yet exist: organizational-patterns.md, OD-SYNTHESIS.md, HANDOFF-OD-TO-AD.md, ACCUMULATED-IDENTITY-v2.md. These are predictions, not facts. If the project direction changes, these forward references become misleading dead links.

### Observation 6: The Original Block Is the Richer Research Record

For every OD file, the original comment block (lines 1-N) contains more granular research citation than the threading header. The threading header's SOUL ALIGNMENT and MUST HONOR tables are more structured, but the original block's "Applied Research" section lists specific finding IDs with one-line descriptions of how each was applied. If forced to choose one block, the original block is better for understanding WHAT research was applied; the threading header is better for understanding HOW the file fits into the system.

### Observation 7: The Threading Header Is a Dependency Document, Not an Identity Document

Despite being called an "identity" header, the threading header's most unique contribution is dependency tracking: BUILT ON (upstream), CONSUMED BY (downstream), WHAT BREAKS IF THIS CHANGES (blast radius), and MUST HONOR (constraint inheritance). These are relational properties -- they define the file's position in a graph, not its intrinsic identity. The intrinsic identity (hypothesis, research applied, density pairing, fractal compliance) lives primarily in the original block.

## 8. The Core Question: Can Inline Headers Substitute for Formal Provenance?

**No. They capture fundamentally different information.**

The inline headers capture **file-level operational identity**: what this file is, what it depends on, what constraints it honors, and what research it applies. This is a **node description** in a graph.

Formal provenance documents capture **stage-level strategic narrative**: why this stage exists, what decisions were made across all files, what findings emerged from the stage as a whole, how files compare to each other, and what gets carried forward to the next stage. This is an **edge and subgraph description** in the same graph.

Neither can substitute for the other because:

1. **Aggregation**: Headers are per-file. Provenance is per-stage. The insight "OD-003's ISLANDS pattern scored higher than OD-002's CRESCENDO" requires both files' data plus a comparison framework that lives in neither header.

2. **Temporal ordering**: Headers are static snapshots. Provenance documents record evolution: what was tried first, what failed, what changed. The "OD-001 Key Lesson Applied" section in OD-002's original block is a partial exception, but it is one-directional (OD-002 knows about OD-001; OD-001 does not know it taught OD-002).

3. **Finding lifecycle**: OD-F findings are born, validated, and propagated. Headers record findings at their declaration point (OD-006 for OD-F-001 through OD-F-005) but not their origin point (OD-001 for OD-F-001 through OD-F-004). The lifecycle requires a registry external to any single header.

4. **Audit integration**: The 89-finding audit produced 16 fixes. None of this is reflected in the headers. The audit is a separate temporal layer that formal provenance would integrate but headers cannot.

5. **Strategic decisions**: Why PULSE for conversational? Why not WAVE? Why 6 explorations and not 4 or 8? These decisions precede the files and cannot live inside them.

The headers and formal provenance are complementary layers: headers provide the micro-view (per-file), provenance provides the macro-view (per-stage). Both are needed. Neither is redundant.
